Why Python — and not something else

Every major AI and agent framework — LangChain, LangGraph, Pydantic, FastAPI, the Anthropic SDK, the OpenAI SDK — is written in Python. This isn't an accident. Python reads like pseudocode, which means AI researchers could write their ideas as code without becoming software engineers first. That culture compounded over twenty years, and now the entire AI ecosystem lives here.

You don't need to be a Python expert before you start building agents. But you do need a solid grip on the five or six features that agent frameworks use constantly. Skip any of them and you'll spend your time debugging framework internals instead of building things.

Note
We're not trying to teach you all of Python here — that would take months. We're teaching the 20% that appears 80% of the time in AI engineering code.

Variables, types, and why types matter now

Python is dynamically typed — you never write int x = 5 like you would in Java. You write x = 5 and Python figures out the type at runtime. This is great for experimentation and terrible for large systems, which is why modern Python code uses type hints.

python 
# Without type hints — works but gives you no help when something breaks
def greet(name):
    return f"Hello, {name}"

# With type hints — same code, but now your editor and Pydantic understand it
def greet(name: str) -> str:
    return f"Hello, {name}"

# Why this matters: LangChain reads your type hints to build tool schemas automatically
# If you skip them, your agent tools silently break in weird ways

The wrong way vs the right way

✗ Common mistake
# Mutable default argument
def add_item(item, lst=[]):
    lst.append(item)
    return lst

add_item("a")  # ['a']
add_item("b")  # ['a', 'b'] ← bug!
✓ Correct
# Use None, create fresh list each call
def add_item(item, lst=None):
    if lst is None:
        lst = []
    lst.append(item)
    return lst

add_item("a")  # ['a']
add_item("b")  # ['b'] ✓

Decorators — because agent frameworks are full of them

A decorator is a function that wraps another function to add behaviour without changing the original. This sounds abstract until you see it in context. LangChain's @tool decorator turns a regular Python function into something an LLM can call. FastAPI's @app.get("/chat") turns a function into an HTTP endpoint.

Mental model
Think of a decorator like a pizza box. The pizza (your function) doesn't change. The box (the decorator) adds structure around it — a label, a price, rules about how it gets delivered.
python — decorator from scratch, then the real thing 
# Step 1 — write a decorator from scratch to understand what's happening
def log_calls(func):
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        result = func(*args, **kwargs)
        print(f"Done: {result}")
        return result
    return wrapper

@log_calls
def add(a: int, b: int) -> int:
    return a + b

add(2, 3)
# Calling add
# Done: 5

# Step 2 — same pattern in LangChain (this is real code you'll write in Phase 5)
from langchain_core.tools import tool

@tool
def search_docs(query: str) -> str:
    """Search internal documentation for a given query."""
    return run_vector_search(query)
Common mistake
Forgetting the docstring on a @tool function. LangChain uses the docstring as the tool description it sends to the LLM. No docstring = the LLM doesn't know when to call your tool.
