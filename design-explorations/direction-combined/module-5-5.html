LangChain Agents

In modules 5.1–5.4 you built tool-calling loops and ReAct agents from scratch. LangChain (specifically LangGraph, its stateful agent framework) provides production-ready scaffolding for these patterns — handling the message loop, tool dispatch, checkpointing, and streaming. This module covers the core LangGraph agent primitives: create_react_agent, the @tool decorator with automatic schema generation, parallel execution, and structured outputs.

Key Takeaways
  • create_react_agent(model, tools) assembles a full ReAct loop — you write tools, it handles the message loop, parallel tool dispatch, and finish detection
  • The @tool decorator reads the function's docstring and type hints to auto-generate the JSON Schema tool definition — no manual schema writing
  • LangGraph's MemorySaver checkpoints agent state between calls — enables stateful multi-turn conversations with the same agent
  • Parallel tool execution is built-in — when the model calls multiple tools in one step, LangGraph executes them concurrently via asyncio
  • Use with_structured_output(PydanticModel) when you need guaranteed JSON output — the model is forced to produce a schema-valid response

create_react_agent — the building block #

LangGraph's create_react_agent assembles a complete ReAct loop from a model and a list of tools. It handles: the message history, detecting when the model wants to call tools, dispatching tool calls (in parallel if multiple), injecting results, and detecting when to stop. You write tools; LangGraph handles the loop.

pythonlanggraph_agent.py — create_react_agent with tools and memory
from langchain_openai import ChatOpenAI
from langchain.tools import tool
from langgraph.prebuilt import create_react_agent
from langgraph.checkpoint.memory import MemorySaver

# ── Step 1: Define tools with @tool decorator ─────────────────────────
@tool
def get_weather(city: str, unit: str = "celsius") -> dict:
    """Get current weather for a city.

    Call this when the user asks about weather, temperature, or conditions
    for a specific location. Do NOT call for forecasts (use get_forecast).

    Args:
        city: City name in English, e.g. 'Berlin' or 'Tokyo'
        unit: Temperature unit: 'celsius' or 'fahrenheit'
    """
    return {"city": city, "temperature": 18, "condition": "Partly cloudy", "unit": unit}

@tool
def convert_currency(amount: float, from_currency: str, to_currency: str) -> dict:
    """Convert an amount between currencies using current exchange rates.

    Args:
        amount: Amount to convert (positive number)
        from_currency: Source currency code, e.g. 'USD', 'EUR', 'GBP'
        to_currency: Target currency code
    """
    rates = {("USD", "EUR"): 0.92, ("EUR", "USD"): 1.09}
    rate = rates.get((from_currency.upper(), to_currency.upper()), 1.0)
    return {"from": from_currency, "to": to_currency,
             "amount": amount, "result": round(amount * rate, 2), "rate": rate}

# ── Step 2: Build the agent ───────────────────────────────────────────
model = ChatOpenAI(model="gpt-4o-mini")
memory = MemorySaver()  # enables multi-turn stateful conversations

agent = create_react_agent(
    model=model,
    tools=[get_weather, convert_currency],
    checkpointer=memory,
)

# ── Step 3: Invoke with thread_id for stateful sessions ───────────────
config = {"configurable": {"thread_id": "user-123"}}

result = agent.invoke(
    {"messages": [{"role": "user", "content": "What's the weather in Berlin, and how much is 100 EUR in USD?"}]},
    config=config,
)
print(result["messages"][-1].content)

# Follow-up in same thread — agent remembers the conversation
result2 = agent.invoke(
    {"messages": [{"role": "user", "content": "And what about Tokyo weather?"}]},
    config=config,  # same thread_id = same conversation
)
print(result2["messages"][-1].content)
ℹ
@tool reads your docstring as the tool description The @tool decorator parses the function's docstring — the first paragraph becomes the tool description, and the Args section becomes parameter descriptions. This is the same principle as module 5.2: write docstrings as if you're writing a user-facing API spec. LangChain also reads type hints to infer JSON Schema types, so annotate all parameters.

Streaming agent responses #

For production UIs you want to stream tokens as they're generated rather than waiting for the complete response. LangGraph's .stream() method yields events including token chunks, tool call starts, tool results, and final messages.

pythonstreaming_agent.py — streaming tokens and events from a LangGraph agent
import sys

# Stream events from the agent
for event in agent.stream(
    {"messages": [{"role": "user", "content": "Weather in Berlin and convert 100 EUR to USD"}]},
    config=config,
    stream_mode="values",
):
    last_msg = event["messages"][-1]

    if hasattr(last_msg, "tool_calls") and last_msg.tool_calls:
        for tc in last_msg.tool_calls:
            print(f"\n[Tool call] {tc['name']}({tc['args']})")

    elif hasattr(last_msg, "content") and isinstance(last_msg.content, str):
        # Print final text content as it arrives
        sys.stdout.write(last_msg.content)
        sys.stdout.flush()

# For token-level streaming, use stream_mode="messages":
for chunk, metadata in agent.stream(
    {"messages": [{"role": "user", "content": "What's 50 USD in EUR?"}]},
    config={"configurable": {"thread_id": "stream-demo"}},
    stream_mode="messages",
):
    if chunk.content and not isinstance(chunk.content, list):
        sys.stdout.write(chunk.content)
        sys.stdout.flush()

Structured output #

Sometimes you need the agent to produce structured data rather than prose — for example, extracting a JSON report or filling in a form. model.with_structured_output(PydanticModel) forces the model to produce a schema-valid response. When combined with tools, the agent can gather information via tools and then output it in the exact schema you define.

pythonstructured_agent.py — forcing Pydantic-schema output
from pydantic import BaseModel, Field
from langchain_openai import ChatOpenAI
from langchain.tools import tool

class WeatherReport(BaseModel):
    city: str
    temperature_celsius: float
    condition: str
    recommendation: str = Field(description="Clothing recommendation based on conditions")

@tool
def get_weather_data(city: str) -> dict:
    """Get raw weather data for a city."""
    return {"city": city, "temp": 8, "condition": "Rainy"}

# Bind structured output to the model
structured_model = ChatOpenAI(model="gpt-4o-mini").with_structured_output(WeatherReport)

# Assemble agent that outputs a WeatherReport — NOT prose
structured_agent = create_react_agent(
    model=structured_model,
    tools=[get_weather_data],
)

result = structured_agent.invoke({
    "messages": [{"role": "user", "content": "Give me a weather report for Hamburg"}]
})

report = result["messages"][-1].content
if isinstance(report, WeatherReport):
    print(f"City: {report.city}")
    print(f"Temp: {report.temperature_celsius}°C, {report.condition}")
    print(f"Wear: {report.recommendation}")
⚠
LangChain abstracts the loop — but you still need to understand 5.1–5.4 LangChain and LangGraph change rapidly — their APIs have broken between minor versions multiple times. If you only know LangChain and not the underlying tool-calling protocol, you will be unable to debug when something breaks at the API level, unable to switch frameworks, and unable to implement custom behaviour not supported by LangChain's abstractions. Build from scratch first (modules 5.1–5.4), then use LangGraph for production velocity. Know the foundations.

Self-check #

What does MemorySaver do in a LangGraph agent, and what problem does it solve compared to a stateless agent invocation?
MemorySaver is a checkpointer that persists the agent's message state between invocations. Without it, every call to agent.invoke() is stateless — the agent starts fresh with no memory of prior turns. With MemorySaver and a thread_id in the config, the agent stores the full message history for each thread in memory (or a database if using a persistent checkpointer like SqliteSaver). When the user asks a follow-up question ("And what about Tokyo?"), the agent receives both the original conversation history and the new message, so it knows "Berlin was already covered" and doesn't repeat it. The thread_id is the session identifier — different thread_ids are separate conversations. In production, you'd use a persistent checkpointer (PostgreSQL-backed) rather than MemorySaver (in-process RAM only, lost on restart) so conversations survive server restarts.
When does parallel tool execution happen in a LangGraph agent, and what is the performance benefit?
Parallel tool execution happens when the model returns multiple tool_calls in a single step — for example, when the user asks "Weather in Berlin and Tokyo" and the model decides both can be resolved independently. LangGraph detects that there are multiple tool calls and executes them concurrently using asyncio.gather (or a thread pool for sync tools), rather than executing them sequentially. The performance benefit: if each weather API call takes 500ms, sequential execution takes 1000ms, parallel takes ~500ms. For agents that make 3–5 API calls per reasoning step, this can reduce latency by 60–80%. The model controls whether parallel execution happens — it groups independent calls in a single response. You cannot force it (the model decides) but you can encourage it by having a system prompt that says "when you need information from multiple independent sources, call all relevant tools at once."
You're building an agent that extracts structured data from customer emails (sender, issue type, urgency, summary). Why use with_structured_output instead of asking the model to "reply in JSON"?
with_structured_output(PydanticModel) uses the OpenAI function-calling protocol to force structured output — the API enforces that the response matches the JSON Schema derived from the Pydantic model. If it doesn't, the API rejects it. "Reply in JSON" is a prompt instruction that the model tries to follow but occasionally fails: it may wrap the JSON in markdown code fences, add explanatory text before or after the JSON, use inconsistent key names, omit optional fields incorrectly, or produce JSON that's syntactically invalid. All of these require defensive parsing logic on your end, and failures surface silently at runtime. With structured_output, you get a typed Python object you can access with dot notation, a 400 error if the model fails (never a silently wrong object), and automatic documentation of the expected schema. For any production extraction pipeline, always use structured_output over prompt-based JSON instructions.
