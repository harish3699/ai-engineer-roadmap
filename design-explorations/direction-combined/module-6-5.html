Episodic Memory

Short-term memory (Module 6.3) keeps the recent conversation alive. Episodic memory goes further — it stores facts extracted from past conversations in a persistent vector database, so an agent can say "you mentioned last month that you prefer TypeScript" even though that conversation is nowhere in the current context window. This module covers the write path (memory extraction), the read path (query-time retrieval), memory deduplication, and the privacy/consent considerations you cannot skip in production.

~24 min read 4 code examples Last updated 2026-05-07
Key Takeaways
  • Episodic memory = facts extracted from past conversations, stored in a vector store, retrieved at query time by semantic similarity
  • The write path is separate from the read path — memory extraction happens at end-of-session (async), not inline during the conversation
  • Memory deduplication is essential — the same fact stated in 10 sessions will create 10 near-duplicate entries; periodically merge or deduplicate by clustering
  • Always namespace memories per user (user_id metadata filter) — never let one user's episodic memory influence another user's responses
  • Memory extraction must comply with privacy regulations — in GDPR contexts, users must be able to view, correct, and delete their stored memories on request

The write path and the read path #

Episodic memory involves two completely separate operations that run at different times. Understanding this separation prevents the most common implementation mistake — trying to write memories during the live conversation turn (which adds latency the user feels directly).

Episodic Memory Architecture
WRITE PATH — end of session
1. Session ends (user closes chat)
2. Async job: pass full session to LLM extractor
3. LLM returns list of extractable facts
4. Deduplicate against existing memories
5. Embed each fact and upsert to vector store
READ PATH — each query
1. New user query arrives
2. Embed query → similarity search in vector store
3. Filter by user_id metadata
4. Top-k memories → inject into context as <memories>
5. Model answers with cross-session knowledge

Building the memory extractor #

The extractor is an LLM call at the end of each session that identifies which facts are worth storing long-term. The prompt matters — it determines what quality of memories you accumulate:

pythonmemory_extractor.py — LLM-driven fact extraction from session transcript
import json
from openai import OpenAI

client = OpenAI()

EXTRACT_PROMPT = """You extract memorable facts from a conversation transcript.
Return a JSON array of strings. Each string is ONE discrete fact about the user.

Rules:
- Only extract facts the user stated about themselves (preferences, expertise, constraints, goals)
- Do NOT extract general knowledge, questions asked, or assistant responses
- Do NOT extract transient details ("what I had for breakfast")
- Use second person: "The user prefers Python over JavaScript"
- Limit to 5 facts maximum per session

Example output: ["The user prefers Python over JavaScript", "The user is building a SaaS for HR teams", "The user has experience with LangChain"]

If no memorable facts exist, return: []"""

def extract_memories(session_messages: list[dict]) -> list[str]:
    transcript = "\n".join(
        f"{m['role'].upper()}: {m['content']}"
        for m in session_messages
        if m["role"] in ("user", "assistant")
    )
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": EXTRACT_PROMPT},
            {"role": "user",   "content": transcript},
        ],
        response_format={"type": "json_object"},
        max_tokens=300,
    )
    data = json.loads(resp.choices[0].message.content)
    return data if isinstance(data, list) else data.get("memories", [])

Storing and retrieving memories #

pythonepisodic_store.py — ChromaDB-backed per-user memory store
import chromadb, uuid
from chromadb.utils.embedding_functions import OpenAIEmbeddingFunction

embed_fn = OpenAIEmbeddingFunction(
    api_key="YOUR_KEY", model_name="text-embedding-3-small"
)
db = chromadb.PersistentClient(path="./episodic_memory_db")
collection = db.get_or_create_collection(
    name="user_memories", embedding_function=embed_fn
)


def save_memories(user_id: str, memories: list[str]) -> None:
    if not memories:
        return
    collection.upsert(
        documents=memories,
        ids=[str(uuid.uuid4()) for _ in memories],
        metadatas=[{"user_id": user_id} for _ in memories],
    )


def retrieve_memories(user_id: str, query: str, k: int = 5) -> list[str]:
    results = collection.query(
        query_texts=[query],
        n_results=k,
        where={"user_id": user_id},   # critical: scope to this user only
    )
    return results["documents"][0] if results["documents"] else []


def list_memories(user_id: str) -> list[dict]:
    """Return all memories for a user — for the user-facing memory management UI."""
    results = collection.get(where={"user_id": user_id})
    return [
        {"id": id_, "memory": doc}
        for id_, doc in zip(results["ids"], results["documents"])
    ]


def delete_memory(memory_id: str) -> None:
    """User-initiated deletion — GDPR right to be forgotten."""
    collection.delete(ids=[memory_id])

Injecting memories into context #

pythonagent_with_episodic.py — full query-time memory injection
async def chat(user_id: str, user_query: str, history: list[dict]) -> str:
    # 1. Retrieve relevant memories for this user
    memories = retrieve_memories(user_id=user_id, query=user_query, k=5)

    # 2. Build context block with memories injected
    memories_block = ""
    if memories:
        formatted = "\n".join(f"- {m}" for m in memories)
        memories_block = f"\n\n<memories>\nKnown facts about this user:\n{formatted}\n</memories>"

    system = (
        "You are a helpful assistant."
        "Use the facts in <memories> to personalise your response where appropriate."
        "Do not mention that you have memories unless the user asks."
        + memories_block
    )

    # 3. Assemble messages with history + current query
    messages = [
        {"role": "system", "content": system},
        *history,
        {"role": "user",   "content": user_query},
    ]
    resp = await client.chat.completions.acreate(model="gpt-4o", messages=messages)
    return resp.choices[0].message.content


# After the session ends, extract and save new memories asynchronously
async def on_session_end(user_id: str, session_messages: list[dict]) -> None:
    new_memories = extract_memories(session_messages)
    save_memories(user_id=user_id, memories=new_memories)

Memory deduplication #

Without deduplication, a user who says "I prefer Python" in every session will accumulate hundreds of near-identical entries. This wastes storage, clutters the retrieved context, and can confuse the model with redundant facts. Run a deduplication pass periodically or at session end:

ℹ
Deduplication strategy Before inserting a new memory, retrieve the top-1 existing memory for that user with the same query as the candidate fact. If similarity ≥ 0.95, skip the insert — the fact is already represented. This is a single vector query per extracted fact, so the overhead is small. Alternatively, run a nightly batch job that clusters all memories per user and keeps only the cluster centroid.

Privacy and consent #

⚠
Episodic memory has real regulatory implications
  • GDPR (EU) / CCPA (CA): users have the right to see, correct, and delete their stored data. Ship a /memories endpoint before going to production.
  • Data minimisation: only extract facts the user consciously shared. Never infer or store inferences (e.g., "user seems anxious" from tone analysis).
  • Consent: disclose to users that you are storing facts about them. "I remembered you prefer Python from last time" surprises users who didn't know this was happening.
  • Retention policy: define how long memories persist. Auto-expire memories older than 12 months unless refreshed.

Using Mem0 for managed episodic memory #

Mem0 is an open-source managed memory layer designed specifically for LLM agents. It handles extraction, storage, deduplication, and retrieval out of the box, with both a self-hosted option and a cloud API:

pythonmem0_example.py — drop-in episodic memory with Mem0
# pip install mem0ai
from mem0 import Memory

mem = Memory()  # self-hosted by default (SQLite + FAISS locally)

user_id = "user-harish"

# Add a memory after a session
mem.add(
    messages=[
        {"role": "user",      "content": "I always write TypeScript, not JavaScript"},
        {"role": "assistant", "content": "Got it, I'll use TypeScript in future examples."},
    ],
    user_id=user_id,
)

# Query relevant memories for the next session
results = mem.search(query="what language should I use?", user_id=user_id, limit=5)
# → [{"memory": "User prefers TypeScript over JavaScript", "score": 0.94}]

# List all memories for a user (for the /memories management endpoint)
all_memories = mem.get_all(user_id=user_id)

# Delete a specific memory (GDPR right to erasure)
mem.delete(memory_id="mem-id-here")
