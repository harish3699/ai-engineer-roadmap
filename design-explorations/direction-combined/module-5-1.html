Function Calling / Tool Use

Phase 4 gave your LLM the ability to retrieve information. Phase 5 gives it the ability to act. Function calling — also called tool use — is the mechanism by which an LLM can invoke external code: search the web, query a database, send a message, execute a calculation. This module covers the full protocol from tool schema definition through the model's decision loop to parsing responses and handling errors. Understanding this mechanism precisely is the foundation for everything in Phase 5.

~22 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • The LLM does not execute tools — it outputs a structured JSON request describing which tool to call and with what arguments; your code executes it and returns the result
  • Tool definitions are JSON Schema objects; the model reads the name, description, and parameter schema to decide when and how to call each tool — description quality directly determines call accuracy
  • The conversation is stateful: tool call → your code runs → tool result injected back as a message → model continues generation using that result
  • Always validate model-supplied arguments before executing tools — the model can hallucinate parameter values, especially for IDs and enums
  • Parallel tool calling (multiple tool_calls in one response) lets the model batch independent lookups — handle all calls before continuing the loop

    What is tool use? #

    An LLM with no tools is a text-in, text-out system. It can reason, summarise, and generate — but it cannot check today's weather, read your calendar, or write to a database. Tool use is the mechanism that connects LLM reasoning to real-world actions and live data.

    The key insight is that the LLM never executes tools directly. Instead, the protocol works like this:

    Tool calling protocol — 4 steps
    1
    You define tools — a JSON Schema object with name, description, and parameters. The model reads these descriptions to understand when to call each tool.
    2
    Model decides to call a tool — instead of generating a text reply, the model outputs a tool_calls object: tool name + JSON arguments. Your chat loop detects this.
    3
    Your code executes the tool — you run the actual function with the model-supplied arguments and get a result.
    4
    Result injected as a message — you append the tool result to the message list as a tool role message, then call the API again. The model continues its response using the result.

    Defining tools with JSON Schema #

    A tool definition has three fields that matter: name (used to route calls to your functions), description (read by the model — this is your API contract), and parameters (a JSON Schema object defining the expected arguments). The description is the most important field — think of it as the docstring the model reads to decide when and how to call the tool.

    pythontool_definition.py — defining a tool and calling the API
    import os, json
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# ── Step 1: Define the tool ────────────────────────────────────────────
TOOLS = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": (
                "Get the current weather for a city. Call this whenever the user asks "
                "about weather conditions, temperature, or forecast for a specific location."
            ),
            "parameters": {
                "type": "object",
                "properties": {
                    "city": {
                        "type": "string",
                        "description": "City name, e.g. 'Berlin' or 'New York'",
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "Temperature unit. Default: celsius.",
                    },
                },
                "required": ["city"],
            },
        },
    }
]

# ── Step 2: Your actual function ──────────────────────────────────────
def get_weather(city: str, unit: str = "celsius") -> dict:
    # In production: call a real weather API (OpenWeatherMap, WeatherAPI, etc.)
    return {"city": city, "temp": 18, "condition": "Partly cloudy", "unit": unit}

# ── Step 3: First API call ─────────────────────────────────────────────
messages = [{"role": "user", "content": "What's the weather like in Berlin right now?"}]

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=messages,
    tools=TOOLS,
    tool_choice="auto",  # model decides when to call tools
)

msg = response.choices[0].message
print(f"finish_reason: {response.choices[0].finish_reason}")  # "tool_calls"
print(f"tool called: {msg.tool_calls[0].function.name}")       # "get_weather"
print(f"arguments: {msg.tool_calls[0].function.arguments}")    # '{"city": "Berlin"}'
    ℹ
    finish_reason tells you the state When the model wants to call a tool, finish_reason is "tool_calls". When it produces a final text answer, it's "stop". Always check finish_reason in your loop rather than assuming a text response.

    The complete tool-calling loop #

    After the first API call returns a tool request, you need to: execute the function, append both the assistant's tool-call message and your tool result to the message list, then call the API again. The loop continues until finish_reason == "stop".

    pythontool_loop.py — complete multi-turn tool calling loop
    def run_with_tools(user_message: str) -> str:
    messages = [{"role": "user", "content": user_message}]
    tool_map = {"get_weather": get_weather}  # map name → callable

    while True:
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            tools=TOOLS,
            tool_choice="auto",
        )
        msg = response.choices[0].message

        if response.choices[0].finish_reason == "stop":
            return msg.content  # final answer

        # Model requested tool calls — execute all of them
        messages.append(msg)  # IMPORTANT: append the assistant message first

        for call in msg.tool_calls:
            fn_name = call.function.name
            fn_args = json.loads(call.function.arguments)

            try:
                result = tool_map[fn_name](**fn_args)
                result_str = json.dumps(result)
            except Exception as e:
                result_str = f"Error: {e}"

            messages.append({
                "role": "tool",
                "tool_call_id": call.id,  # must match the call ID
                "content": result_str,
            })

# Run it
answer = run_with_tools("What's the weather in Berlin and Tokyo?")
print(answer)
# → Model may issue two parallel get_weather calls, then compose a final answer
    ⚠
    Always append the assistant message before tool results The OpenAI API requires the message list to include the assistant's tool_calls message (step 3 in the loop) before any tool role messages. If you only append tool results without the assistant message, the API returns a 400 error. The order must be: user → assistant (with tool_calls) → tool result(s) → next assistant turn.

    Parallel tool calls #

    When the model can satisfy multiple tool calls independently — for example, getting the weather in both Berlin and Tokyo — it may return multiple tool_calls in a single response. You must handle all of them before calling the API again. The loop above already does this with the for call in msg.tool_calls iteration. In production, you can execute independent calls concurrently with asyncio.gather or a thread pool.

    pythonparallel_tools.py — parallel execution with asyncio
    import asyncio, json

async def execute_tool_calls_parallel(tool_calls, tool_map):
    """Execute all tool calls concurrently, return list of (call_id, result_str)."""

    async def run_one(call):
        fn_name = call.function.name
        fn_args = json.loads(call.function.arguments)
        try:
            fn = tool_map[fn_name]
            # Support both sync and async tools
            if asyncio.iscoroutinefunction(fn):
                result = await fn(**fn_args)
            else:
                result = await asyncio.get_event_loop().run_in_executor(None, lambda: fn(**fn_args))
            return call.id, json.dumps(result)
        except Exception as e:
            return call.id, f"Error calling {fn_name}: {e}"

    results = await asyncio.gather(*[run_one(c) for c in tool_calls])
    return [
        {"role": "tool", "tool_call_id": cid, "content": res}
        for cid, res in results
    ]

    Controlling when tools are called #

    The tool_choice parameter controls the model's behaviour:

    tool_choice options
    "auto"
    Model decides — calls tools when useful, generates text otherwise. Use this for general agents.
    "required"
    Model MUST call at least one tool. Use when you want to force tool use regardless of query type.
    "none"
    Model cannot call tools. Use for final summarisation steps where you want text output only.
    {"type":"function","function":{"name":"X"}}
    Forces the model to call a specific tool. Use for structured data extraction where you always want a Pydantic-validated JSON response.

    Handling tool errors gracefully #

    Tools fail: APIs time out, arguments are invalid, rate limits hit. The model can handle tool errors intelligently — but only if you pass the error string back as the tool result rather than raising an exception in the loop. Return a descriptive error string and the model will typically either retry with corrected arguments or explain to the user that the tool is unavailable.

    💡
    Return errors as strings, not exceptions If your tool raises an exception in the calling loop, the loop crashes. Instead, catch all exceptions inside run_one and return f"Error: {e}" as the tool result. The model reads this error string and decides how to proceed — it might try a different query, ask the user for clarification, or explain the failure. This graceful degradation is what makes agents robust in production.

    Self-check #

    Explain precisely what happens when you set tool_choice="auto" and the model decides to call a tool. What does the API return and what must your code do next?
    When tool_choice="auto" and the model decides a tool is needed, the API response has finish_reason="tool_calls" and the message object contains a tool_calls list rather than a text content. Each element in tool_calls has an id (a unique identifier for this call), a function.name (matching one of your tool definitions), and function.arguments (a JSON string of the arguments the model chose). Your code must: (1) append the entire assistant message (including its tool_calls) to the message list, (2) parse each tool call's arguments, execute your actual function, and collect results, (3) append each result as a {"role": "tool", "tool_call_id": call.id, "content": result_str} message, (4) call the API again with the updated message list. Only then will the model generate its final text response. Skipping any of these steps — especially forgetting to append the assistant message before tool results — causes an API error or a confused model response.
    Why must you validate model-supplied arguments before executing tools, and what is a safe validation approach?
    The model generates tool arguments by predicting the most likely JSON given the conversation context — it does not have access to your actual data or system state. This means it can hallucinate values: inventing record IDs that don't exist, providing strings where numbers are expected, supplying enum values outside the allowed set, or generating structurally invalid JSON. If you pass these arguments directly to your functions (especially database queries or API calls), you risk: incorrect results, database errors, accidental data modification with wrong IDs, or exceptions that crash the loop. A safe validation approach: use Pydantic to parse and validate arguments against a typed model that mirrors the JSON Schema — if validation fails, return an error string to the model rather than executing. Additionally: never expose destructive operations (DELETE, overwrite) without adding an explicit confirmation step, and sanitise any arguments that reach SQL or shell commands to prevent injection attacks.
    The model returns two tool_calls in a single response. What is the correct order for appending messages before the next API call?
    The correct order is: (1) append the complete assistant message containing both tool_calls, (2) for each tool call, execute the function and append its tool result as a separate {"role":"tool","tool_call_id":call.id,"content":result} message. Both tool result messages must be appended before making the next API call. The tool_call_id in each tool result must exactly match the id of the corresponding tool call — the API uses this to correlate results with requests. You may execute the two tools in parallel (e.g., asyncio.gather) but you must still append all results before calling the API again. Appending results in the wrong order or with mismatched IDs causes a 400 validation error.
