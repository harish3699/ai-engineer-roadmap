LLMOps — Observability

You can't debug what you can't see. LLM applications fail in ways that don't surface in conventional application monitoring: a model that starts returning shorter answers, a retrieval pipeline that silently degrades, a tool that costs three times more than last week. This module builds the observability stack that makes these problems visible — distributed traces with LangSmith and LangFuse, token cost dashboards, latency percentile tracking, and failure rate breakdowns by tool, route, and model.

~22 min read 4 code examples Last updated 2026-05-07
Key Takeaways
  • LangSmith vs LangFuse: LangSmith is tightly integrated with LangChain/LangGraph and the default for those stacks; LangFuse is open-source, self-hostable, provider-agnostic — better for non-LangChain stacks or when you need data sovereignty
  • The four metrics that matter: total cost/request, p95 latency per node, tool error rate, and retrieval relevance score — everything else is derived from these
  • Latency percentiles: p50 tells you what normal looks like; p95 and p99 tell you what your worst users experience — alert on p95, capacity-plan on p99
  • Failure rate by dimension: a global 2% error rate hides a 40% error rate on one tool — always break down by tool name, route, and model before investigating
  • Cost attribution: tag every LLM call with the workflow name and node name so you know which part of a multi-agent pipeline is the cost driver

LangSmith vs LangFuse #

Both platforms capture the same core data: a hierarchy of spans representing each LLM call, tool invocation, and chain step, with input/output text, latency, and token counts at each level. The practical difference is ecosystem and hosting.

Feature LangSmith LangFuse
Hosting Cloud SaaS (Langchain) Cloud SaaS or self-hosted (Docker/K8s)
LangChain/Graph integration Native — 3 env vars, zero code Via LangFuse callback handler
Provider-agnostic Partial — works with any, best with Langchain Yes — OpenAI, Anthropic, raw HTTP
Data sovereignty Data sent to Langchain servers Self-host keeps data on-prem
Pricing Free tier, then per-seat + usage Free tier + open-source self-host
Best for LangGraph-native teams Non-LangChain stacks, compliance-heavy orgs

LangFuse SDK instrumentation #

LangFuse's Python SDK lets you instrument any LLM code — not just LangChain — with a clean context-manager API. Each trace represents one user request; spans are nested operations inside it.

pythonlangfuse_instrumentation.py — provider-agnostic tracing
# pip install langfuse
from langfuse import Langfuse
from datetime import datetime
import anthropic

langfuse = Langfuse(
    public_key="pk-lf-...",
    secret_key="sk-lf-...",
    host="https://cloud.langfuse.com",   # or your self-hosted URL
)

client = anthropic.Anthropic()

async def handle_request(user_id: str, user_message: str, request_id: str):
    trace = langfuse.trace(
        name="customer-support-agent",
        user_id=user_id,
        session_id=request_id,
        metadata={"channel": "web"},
    )

    # ── Retrieval span ──────────────────────────────────────────────────
    retrieval_span = trace.span(name="retrieval", input={"query": user_message})
    passages = await retrieve_context(user_message)
    retrieval_span.end(output={"passages_count": len(passages), "passages": passages[:2]})

    # ── LLM generation span ────────────────────────────────────────────
    generation = trace.generation(
        name="claude-response",
        model="claude-sonnet-4-6",
        input=[{"role": "user", "content": user_message}],
    )
    t0 = datetime.utcnow()
    response = client.messages.create(
        model="claude-sonnet-4-6-20250219",
        max_tokens=1024,
        messages=[{"role": "user", "content": user_message}],
    )
    generation.end(
        output=response.content[0].text,
        usage={
            "input":   response.usage.input_tokens,
            "output":  response.usage.output_tokens,
        },
        metadata={"latency_ms": (datetime.utcnow() - t0).microseconds // 1000},
    )

    # Scores can be added later (e.g., from user feedback or evaluation runs)
    trace.score(name="user_rating", value=1)    # 1 = thumbs-up, 0 = thumbs-down
    return response.content[0].text

Token cost dashboards #

LLM costs compound in multi-agent systems. The dashboard metric that matters most is cost per successful request — not total cost, which rises naturally with volume, but the unit cost. A rising unit cost signals prompt bloat, missing caching, or a workflow regression.

pythoncost_tracking.py — per-request cost attribution
# Model pricing (update when providers change rates)
PRICE_PER_MTok = {
    "claude-sonnet-4-6":     {"input": 3.00,  "output": 15.00},
    "claude-haiku-4-5":      {"input": 0.80,  "output": 4.00},
    "gpt-4o":                {"input": 2.50,  "output": 10.00},
    "gpt-4o-mini":           {"input": 0.15,  "output": 0.60},
}

def calculate_cost(model: str, input_tokens: int, output_tokens: int) -> float:
    prices = PRICE_PER_MTok.get(model, {"input": 0, "output": 0})
    return (input_tokens * prices["input"] + output_tokens * prices["output"]) / 1_000_000

# ── Prometheus metrics for Grafana dashboards ─────────────────────────
from prometheus_client import Counter, Histogram, Gauge

llm_tokens_total = Counter(
    "llm_tokens_total", "Total tokens used",
    ["model", "workflow", "node", "token_type"]   # label dimensions for slicing
)
llm_cost_dollars = Counter(
    "llm_cost_dollars_total", "Estimated LLM cost in USD",
    ["model", "workflow"]
)
llm_latency_ms = Histogram(
    "llm_latency_milliseconds", "LLM call latency",
    ["model", "workflow", "node"],
    buckets=[100, 250, 500, 1000, 2000, 5000, 10000, 30000]
)
tool_errors_total = Counter(
    "tool_errors_total", "Tool invocation errors",
    ["tool_name", "error_type", "workflow"]
)

def record_llm_call(model: str, workflow: str, node: str,
                    input_tok: int, output_tok: int, latency_ms: float):
    llm_tokens_total.labels(model, workflow, node, "input").inc(input_tok)
    llm_tokens_total.labels(model, workflow, node, "output").inc(output_tok)
    llm_cost_dollars.labels(model, workflow).inc(calculate_cost(model, input_tok, output_tok))
    llm_latency_ms.labels(model, workflow, node).observe(latency_ms)

Latency percentiles #

p50 is the median — half of requests are faster, half are slower. p95 is what 1 in 20 users experiences. p99 is what 1 in 100 experiences. In LLM applications p99 is often 10–50× higher than p50 because of occasional model timeouts, cold-start penalties, and retrieval outliers.

Alert when p95 crosses 2× the rolling 7-day baseline. Alert when p99 crosses 5 s for interactive endpoints. Use p99 for capacity planning — if your SLA is "99% of requests under 10 s" you need to know p99, not p50.

Failure rate by dimension #

A global error rate of 2% often hides a 40% error rate on one specific tool, model, or workflow route. Always slice failure rates by at least three dimensions before investigating.

pythonfailure_analysis.py — SQL to surface failure hot-spots
-- LangFuse SQL (works in Metabase, Superset, or direct Postgres)
-- Failure rate by tool name over the last 7 days
SELECT
    observations.name                                               AS tool_name,
    COUNT(*)                                                       AS total_calls,
    COUNT(*) FILTER (WHERE observations.level = 'ERROR')           AS errors,
    ROUND(
        100.0 * COUNT(*) FILTER (WHERE observations.level = 'ERROR') / COUNT(*), 1
    )                                                              AS error_pct,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY observations.latency) AS p95_ms
FROM observations
WHERE
    observations.type = 'TOOL'
    AND observations.start_time >= NOW() - INTERVAL '7 days'
GROUP BY tool_name
ORDER BY error_pct DESC
LIMIT 20;
✦
The four alerts that cover 80% of production incidents Set up exactly these four alerts and you'll catch most problems before users report them: (1) p95 latency > 2× 7-day rolling average, (2) tool error rate > 5% on any single tool over a 15-min window, (3) cost/request > 2× 7-day rolling average, (4) guardrail intervention rate > 10% over a 1-hour window (signals a new attack pattern or prompt regression).
