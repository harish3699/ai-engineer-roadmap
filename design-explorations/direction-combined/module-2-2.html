How an LLM Thinks

You know what an LLM is — a probabilistic text generator trained on a fixed snapshot. Now zoom in one level. This module covers how text gets converted to tokens (and why that matters for cost), what a context window actually is and what happens at its edge, the sampling parameters that control output quality, and the "lost in the middle" failure mode that will catch you off guard in production.

~25 min read 1 video · 1 recording script 2 code examples Last updated 2026-05-07
Key Takeaways
  • Tokens are not words — common English words are 1 token, rare/long words split into several; non-English text and code tokenise less efficiently (more tokens = more cost)
  • The context window is the model's entire working memory for one request — everything outside it is invisible; silently truncated, not summarised
  • top_p controls vocabulary breadth per step; top_k hard-caps the candidate list — combining them with temperature gives fine-grained output control
  • "Lost in the middle" — retrieval accuracy drops for content placed in the middle of a long context; keep the most important context at the beginning or end
  • Counting tokens before sending a request is not optional in production — unexpected truncation is a silent bug that changes model behaviour without any error

BPE Tokenisation #

Models don't process characters or words — they process tokens. A token is a chunk of text, typically 3–4 characters for common English. The tokeniser uses Byte-Pair Encoding (BPE): it starts with individual characters and repeatedly merges the most frequent pairs until it has a vocabulary of ~50,000–100,000 tokens. The result is that common words get a single token, while rare, long, or non-English words split into many.

Token count examples — GPT-4 tokeniser
"hello"1 token
"Hello world"2 tokens
"antidisestablishmentarianism"6 tokens
"Künstliche Intelligenz"7 tokens(German tokenises ~2× less efficiently)
"人工智能"6 tokens(Chinese: each character ~2 tokens)
"def calculate_embeddings():"7 tokens(code tokenises well)

Why this matters for you as an engineer: you pay per token. A 1000-word English document is roughly 1300 tokens. The same document in Chinese or Arabic costs 2–3× more. Code is efficient. Whitespace counts. Numbers tokenise unpredictably (each digit is often its own token). Always count tokens before you budget an API call.

pythoncount_tokens.py — count tokens before sending to the API
import tiktoken

# cl100k_base = GPT-4 / GPT-3.5 tokeniser. Use as a rough estimate for Claude too.
enc = tiktoken.get_encoding("cl100k_base")

def count_tokens(text: str) -> int:
    return len(enc.encode(text))

def estimate_cost(text: str, price_per_million: float = 3.0) -> float:
    return count_tokens(text) / 1_000_000 * price_per_million

system_prompt = "You are an expert AI engineer helping students learn about LLMs."
user_message  = "Explain context windows in plain English, under 100 words."

total = count_tokens(system_prompt) + count_tokens(user_message)
print(f"Input tokens: {total}")                           # ~30
print(f"Estimated cost: ${estimate_cost(system_prompt + user_message):.6f}")

# ── Check if a document fits in the context window ───────────────────
CLAUDE_SONNET_CONTEXT = 200_000   # tokens
GPT4O_CONTEXT         = 128_000

with open("big_document.txt") as f:
    doc = f.read()

tokens = count_tokens(doc)
if tokens > CLAUDE_SONNET_CONTEXT - 4000:   # leave 4k for output
    print(f"Document too large ({tokens} tokens) — needs chunking")
else:
    print(f"Fits in context ({tokens}/{CLAUDE_SONNET_CONTEXT} tokens)")

Context windows #

The context window is everything the model can "see" at once — your system prompt, the conversation history, any retrieved documents, and the space reserved for the model's reply. Everything outside the window simply doesn't exist to the model. It isn't summarised or compressed — it's gone.

Context window — how tokens are spent (128k window example)
~2k
System prompt (instructions, persona, rules)
~50k
Retrieved documents / RAG context
~30k
Conversation history (sliding window)
~1k
User message
~4k reserved
Output (max_tokens budget)
Total: ~87k / 128k used. Anything pushed beyond 128k is silently dropped — the model never sees it.
⚠
Silent truncation is a production bug When input exceeds the context limit, most APIs truncate silently — they drop tokens from the middle or end of the input without raising an error. Your agent responds as if everything was received, but it's missing half the context. Always count tokens before sending and enforce a hard limit in your code.

Sampling parameters: top-p and top-k #

You set temperature in Module 2.1. Two more parameters shape every token sample: top_k and top_p. They're applied before temperature scaling, filtering the candidate vocabulary down before sampling happens.

pythonsampling_params.py — when to use which combination
import anthropic

client = anthropic.Anthropic()

# ── Deterministic structured output (extraction, classification) ───────
extraction = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    temperature=0,       # no randomness
    messages=[{"role": "user", "content": "Extract entities from: ..."}],
)

# ── Balanced chat response (general Q&A, summarisation) ────────────────
chat = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    temperature=0.7,
    top_p=0.9,   # sample from tokens covering 90% of probability mass
    messages=[{"role": "user", "content": "Explain RAG in plain English."}],
)

# ── Creative generation (marketing copy, story, brainstorm) ────────────
creative = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=2048,
    temperature=1.0,
    top_p=0.95,
    messages=[{"role": "user", "content": "Write 5 taglines for an AI course."}],
)

# ── Parameter cheat sheet ─────────────────────────────────────────────
# temperature=0, top_p=1   → deterministic, always pick top token
# temperature=0.7, top_p=0.9 → balanced — good default for most tasks
# temperature=1.0, top_p=0.95 → creative, more variety, higher hallucination risk
# top_k=40 (OpenAI default) → hard limit vocabulary to top 40 tokens per step
# top_p=0.1 → extreme focus — only tokens in top 10% of probability mass

Lost in the middle #

A 2023 Stanford paper ("Lost in the Middle") showed that LLMs reliably attend to content at the beginning and end of a long context window. Content buried in the middle gets significantly less attention — the model is demonstrably worse at recalling and reasoning about it, even though it was technically "in context."

Attention strength vs position in context
↑ Beginning (high attention) ↓ Middle (low attention) ↑ End (high attention)
💡
How to work around lost-in-the-middle For RAG: Put your highest-relevance retrieved chunks at the top of the context, not buried after 20 other chunks.
For long documents: Put your instructions and the specific question both at the start AND end of the context window (repeat them).
For multi-document: Fewer, more relevant chunks outperform many loosely relevant ones — retrieval quality matters more than retrieval quantity.

Self-check #

A 10,000-word English document is roughly how many tokens? What about the same content in Chinese?
A rough rule: 1 English word ≈ 1.3 tokens, so 10,000 words ≈ 13,000 tokens. The same content in Chinese tokenises at roughly 2–3× the cost, so approximately 26,000–39,000 tokens. This has direct cost implications — multilingual applications need to budget per language, not per word count.
What is the difference between top_p and top_k?
top_k is a hard count limit — it restricts sampling to the top K most probable tokens regardless of how much probability mass they represent. top_p (nucleus sampling) is probability-mass-based — it keeps the smallest set of tokens whose cumulative probability exceeds p. top_p=0.9 adapts to the distribution: when the model is confident, it may sample from 5 tokens; when uncertain, it may sample from 50. top_p is generally preferred for production use.
Your RAG system retrieves 20 chunks and puts the most relevant one at position 10 out of 20. What is the likely problem?
The "lost in the middle" failure — LLMs attend more strongly to content at the start and end of the context window than to content in the middle. Chunk at position 10 of 20 is in the middle of the retrieved context and will receive less attention than chunks at positions 1 or 20. Solution: always put the most relevant chunks at the top of the retrieved context, not in the middle. Fewer, higher-quality chunks also outperform many mediocre ones.
You send a request that is 130,000 tokens to GPT-4o (128k context limit). What happens?
The API will either return an error (if you're over the hard limit) or silently truncate the input to fit within the context window — typically dropping tokens from the end or middle of the input without raising an exception. The model responds as if nothing happened, but is missing 2,000+ tokens of context. This is why you must count tokens before sending and enforce limits in code, not rely on error handling.
