Cost & Capacity Control

The system runs. Now make it run cheaply and not fall over under load. LLM applications fail under concurrency in ways that traditional web applications don't — the bottleneck is usually your application server, not the LLM API. This final module covers semantic cache hit rate as the key cost KPI, model routing to avoid over-spending on simple queries, prompt compression, max-token caps, and load testing with Locust to find your breaking point before users do.

~20 min read 4 code examples Last updated 2026-05-07
Key Takeaways
  • Cache hit rate is your cost KPI: a 40% hit rate means 40% of requests are served for ~$0 in model cost — track it on your dashboard and treat a drop as an incident
  • Model routing: classify query complexity (simple/medium/complex) and route to Haiku/Sonnet/Opus accordingly — the 80% of simple queries run at 10× lower cost without quality loss
  • Prompt compression reduces both cost and latency: drop low-information tokens before the LLM call, not after — every token you remove saves money and speeds up TTFT
  • Agents fall over at concurrency, not at single-request load: 50 concurrent users with 30-second agent runs means 1500 seconds of active agent work — Locust reveals this before you deploy
  • Rate-limit at the gateway, not the model: your API Gateway rate limit protects your infra; the model provider's rate limit protects theirs — you need both, and they're independent

Semantic cache hit rate as a KPI #

The semantic cache from module 6.4 is your most powerful cost lever. A query that hits the cache costs zero model tokens — only the embedding call (~$0.00002). Monitor cache hit rate as a first-class KPI alongside latency and error rate.

Target ranges: <20% hit rate means your cache threshold is too strict or your TTL too short. >70% hit rate is excellent for FAQ-heavy applications. 30–50% is typical for general-purpose assistants. A sudden drop in hit rate often means a new type of user query — investigate whether the cache should be expanded or the threshold adjusted.

Model routing — cheap for simple, expensive for complex #

Not every query needs your most capable model. A "what is your return policy?" question doesn't need Claude Sonnet — it needs Claude Haiku at 10× lower cost. Route by complexity classification before invoking the main agent.

pythonmodel_router.py — complexity-based model selection
from pydantic import BaseModel
from enum import Enum

class Complexity(Enum):
    SIMPLE  = "simple"   # factual lookup, FAQ — Haiku
    MEDIUM  = "medium"   # multi-step reasoning, synthesis — Sonnet
    COMPLEX = "complex"  # deep research, code generation, long output — Opus/Sonnet Max

class RoutingDecision(BaseModel):
    complexity: Complexity
    reason: str

MODEL_MAP = {
    Complexity.SIMPLE:  "claude-haiku-4-5-20251001",      # $0.80/$4.00 per MTok
    Complexity.MEDIUM:  "claude-sonnet-4-6-20250219",     # $3.00/$15.00 per MTok
    Complexity.COMPLEX: "claude-opus-4-7-20250514",       # $15.00/$75.00 per MTok
}

router_llm = haiku_llm.with_structured_output(RoutingDecision)

ROUTER_SYSTEM = """Classify the complexity of this user query.
simple: factual single-answer, FAQ, definition lookup
medium: requires reasoning, multi-step, summarisation
complex: open-ended research, code generation, long analysis"""

def route_query(user_message: str) -> str:
    # Use Haiku to classify — cheaper than the task it's routing
    decision: RoutingDecision = router_llm.invoke([
        {"role": "system", "content": ROUTER_SYSTEM},
        {"role": "user",   "content": user_message},
    ])
    model = MODEL_MAP[decision.complexity]
    log_info("ModelRouter", "route_query",
             f"routed to {decision.complexity.value}: {model}")
    return model

Prompt compression #

Every token in the prompt costs money and adds to TTFT (time to first token). Compression techniques from module 6.6 apply directly here: drop boilerplate system prompt sections that are never triggered for a given query, summarise tool results before appending them, and trim message history to a token budget. A typical RAG system can reduce input tokens by 30–50% with no quality loss.

Load testing with Locust #

Agents fail differently under load than regular web APIs. A single request takes 5–30 s. 50 concurrent users means 50 simultaneous long-lived connections. Your FastAPI server runs out of worker threads long before the LLM rate limit is hit. Find this out with Locust before users do.

pythonlocustfile.py — load testing an agent streaming endpoint
from locust import HttpUser, task, between
import json, random

TEST_MESSAGES = [
    "What is your return policy?",
    "How do I integrate your API with Python?",
    "Summarise the key differences between your Enterprise and Growth plans.",
    "I need help debugging a webhook that's not firing.",
]

class AgentUser(HttpUser):
    wait_time = between(2, 8)   # realistic inter-request pause (seconds)

    @task
    def chat_stream(self):
        message = random.choice(TEST_MESSAGES)

        # Use stream=True to consume SSE without buffering the full response
        with self.client.post(
            "/chat/stream",
            json={"message": message},
            stream=True,
            catch_response=True,
            timeout=60,
        ) as response:
            if response.status_code != 200:
                response.failure(f"Got {response.status_code}")
                return

            chunks_received = 0
            for line in response.iter_lines():
                if line.startswith(b"data: "):
                    data = json.loads(line[6:])
                    if data["type"] == "done":
                        break
                    chunks_received += 1

            if chunks_received == 0:
                response.failure("No tokens received")
            else:
                response.success()

# Run: locust -f locustfile.py --host=http://your-alb-url
# Start with 10 users, ramp to 100 over 2 minutes
# Watch: response time percentiles, failure rate, Fargate CPU/memory
⚠
The concurrency bottleneck is your app server, not the LLM Uvicorn with 2 workers handles ~100 concurrent requests comfortably. At 200 concurrent users with 20-second average response times that's 4000 request-seconds of work — you'll need 4 Fargate tasks at minimum, plus API Gateway rate limiting to smooth out spikes. Run Locust first, then size your service accordingly.
