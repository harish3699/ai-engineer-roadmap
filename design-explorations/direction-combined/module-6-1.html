The Context Window as Working Memory

Your agent doesn't have memory in the biological sense. It has a context window — a finite sequence of tokens that is the entirety of what it "knows" at the moment of generation. Understanding this precisely is the foundation of Phase 6. This module covers why agents forget, how token budgeting works, the lost-in-the-middle problem (information buried in the middle of a long context is less attended to), and recency bias. These aren't bugs — they're the mechanical properties of the transformer architecture you need to design around.

~20 min read 1 video · 1 recording script 2 code examples Last updated 2026-05-07
Key Takeaways
  • The context window is the model's entire "working memory" — it has no access to anything outside the tokens currently in context
  • The lost-in-the-middle problem: information in the first ~20% and last ~20% of the context receives significantly more attention than information in the middle — critical content belongs at the edges
  • Recency bias: the last few messages in context are weighted most heavily — the model's answer tends to reflect whatever was said most recently, regardless of importance
  • Token budgeting is not optional — without explicit section budgets, retrieved content can crowd out the conversation history, and vice versa
  • Throwing more tokens at a problem (increasing context size) often makes agent quality worse, not better — longer contexts dilute attention over more tokens

The context window is all the model has #

Between API calls, the model retains absolutely nothing. There is no persistent state, no hidden memory, no background process accumulating knowledge. When you call the API, the model sees exactly the tokens in the messages array and nothing else. When users say "the agent forgot what I told it three turns ago," the cause is almost always one of three things: the message was pushed out of the context window by truncation, it's buried in the middle of a long context and receiving less attention, or it was in a tool result that was stripped from the history.

Attention distribution in a long context (lost-in-the-middle)
← Start of context (high attention) Middle (low attention) End of context (high attention) →
Place your system prompt and the most critical retrieved passages at the start. Place the most recent conversation and the user's current question at the end. Avoid putting must-read content in the middle.

Token counting and budgeting #

A token is roughly 3–4 characters in English. The model's context limit is measured in tokens, not words or characters. Accurate token counting is essential for budget enforcement — the naive approach of using len(text) or word count will be wrong by 20–40%.

pythontoken_budget.py — counting tokens and enforcing section budgets
import tiktoken

# tiktoken is OpenAI's tokeniser — use cl100k_base for gpt-4/gpt-4o models
enc = tiktoken.get_encoding("cl100k_base")

def count_tokens(text: str) -> int:
    return len(enc.encode(text))

def count_messages_tokens(messages: list[dict]) -> int:
    """Approximate token count for a messages list (OpenAI format)."""
    total = 3  # every reply is primed with <|start|>assistant<|message|>
    for msg in messages:
        total += 4  # role + framing tokens
        total += count_tokens(msg.get("content", ""))
    return total

# ── Per-section token budgets ─────────────────────────────────────────
CONTEXT_LIMIT = 8000
BUDGETS = {
    "system":   800,   # system prompt — precise instructions
    "context":  3000,  # retrieved RAG passages
    "history":  2500,  # conversation history
    "user":     500,   # current user message
    "response": 1200,  # headroom for assistant response
}
# Total = 8000 — leaves nothing on the table

def trim_to_budget(text: str, budget: int) -> str:
    """Truncate text to fit within token budget, preserving whole sentences."""
    tokens = enc.encode(text)
    if len(tokens) <= budget:
        return text
    truncated = enc.decode(tokens[:budget])
    # Find last sentence boundary to avoid mid-sentence cuts
    last_period = truncated.rfind(".")
    return truncated[:last_period + 1] + " [truncated]" if last_period > 0 else truncated

# Usage: before assembling the messages array
system_prompt = trim_to_budget(raw_system_prompt, BUDGETS["system"])
context_block  = trim_to_budget(retrieved_passages, BUDGETS["context"])

Recency bias and its implications #

The last message in the context window receives disproportionately high attention. This is useful — the model naturally focuses on the most recent user question — but harmful when earlier, equally important instructions are forgotten. Common failure mode: the user says "always respond in German" in turn 1, and by turn 15 (after many tool calls and retrieved passages filling the middle), the model starts responding in English because the German instruction is buried and the last user message was in English.

⚠
Critical instructions belong in the system prompt, not the conversation Any instruction you need the model to follow throughout the entire conversation — language, persona, constraints, output format — must be in the system prompt (position: first). Do not put standing instructions in a user turn and hope the model remembers them after 10 more exchanges. The system prompt is the only position in context that is guaranteed to stay prominent regardless of conversation length.

Designing context layout for attention #

Given the U-shaped attention curve (high at start, low in middle, high at end), the optimal context layout is:

pythoncontext_layout.py — optimal message assembly order
def build_messages(
    system_prompt: str,
    retrieved_context: str,
    history: list[dict],
    user_query: str,
) -> list[dict]:
    """
    Optimal layout for long-context agents:
    1. System prompt  ← START (high attention) — instructions, persona, constraints
    2. Retrieved context ← just after system — important reference material
    3. Conversation history ← middle (low attention) — older turns compress to summaries
    4. User message ← END (highest attention) — current question always last
    """
    return [
        {"role": "system", "content": trim_to_budget(system_prompt, BUDGETS["system"])},
        {"role": "user",   "content": f"[CONTEXT]\n{trim_to_budget(retrieved_context, BUDGETS['context'])}"},
        {"role": "assistant", "content": "Understood. I'll answer using only the provided context."},
        *history,  # middle — compressed by module 6.6 once it grows
        {"role": "user",   "content": user_query},  # always last
    ]
ℹ
Larger context ≠ better answers GPT-4o supports 128k tokens and Claude supports up to 200k. It is tempting to just dump everything in and let the model figure it out. Research consistently shows that answer quality peaks at a fraction of the maximum context and degrades as the context grows — the model attends to more tokens but less reliably to the right ones. The discipline of Phase 6 is not filling the context window — it's engineering what goes in, in what order, at what position, and compressing or externalising what doesn't need to be there.

Self-check #

Explain the "lost in the middle" problem. How does it affect a RAG agent that injects 10 retrieved passages into the middle of a long conversation history?
The lost-in-the-middle problem, documented in research by Liu et al. (2023), shows that language models exhibit a U-shaped attention pattern over their context window: tokens near the beginning and end receive significantly more attention than tokens in the middle. In a RAG agent that places 10 retrieved passages between a long conversation history and the current user question, those passages land in the middle-to-lower-attention zone. The model is technically "reading" them, but its attention is lower — it is more likely to miss a key fact buried in passage 5 of 10 than it would be if that passage were near the start. The practical consequence: the agent may answer incorrectly or incompletely even when the relevant information is present in context. The fix: place retrieved context immediately after the system prompt (position 2), never in the middle of the conversation history. Apply reranking (module 4.6) to ensure the top-1 passage is the most relevant one, and keep the retrieved set small (3–5 passages) rather than injecting everything.
A user complains that your agent "forgot" their preference (stated in turn 2) by turn 20. What are the three most likely causes and how do you diagnose each?
Cause 1 — Recency bias: the preference message is still in context but it's buried 18 turns back in the middle of the context window. The most recent messages (tool calls, assistant responses) are dominating attention. Diagnose by checking where the preference message sits in the message list and estimating its position as a percentage of total tokens. Fix: move standing preferences to the system prompt, or re-inject them at the end of context before each LLM call. Cause 2 — Context truncation: the sliding window (module 6.3) or compression (module 6.6) removed the message entirely to fit within the token budget. Diagnose by logging the exact messages array sent to the API and checking whether the preference message is still present. Fix: mark important messages as "pinned" so the window/compression logic never removes them. Cause 3 — Tool call stripping: the preference was stated in the same turn as a tool result, and the tool message was stripped from history as part of history trimming. Diagnose by checking your history management code — does it remove tool role messages? Fix: ensure at minimum the user messages from important turns are preserved even if the tool results are stripped.
