Working with HTTP APIs

Every LLM, vector database, and tool you'll integrate in this roadmap is accessed via HTTP. This module gives you the requests library, authentication patterns, and the production retry logic that keeps your agent alive when APIs hiccup.

~25 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-06
Key Takeaways
  • Always check response.raise_for_status() — a 200 response doesn't guarantee correct data; a 4xx/5xx without this call silently returns garbage
  • Use a requests.Session() for multiple calls to the same host — it reuses TCP connections and applies headers globally
  • API keys belong in environment variables, never in code — use os.environ or python-dotenv
  • tenacity retry decorators are the production standard — they handle exponential backoff, jitter, and per-exception retry rules in four lines
  • Always set a timeout= on every request — an LLM API can hang for 60+ seconds without one

GET and POST with requests #

The requests library is the standard for synchronous HTTP in Python. Phase 5 will introduce httpx for async — for now, requests covers everything you need.

pythonhttp_basics.py
import requests

# ── GET — query parameters ────────────────────────────────────────────
resp = requests.get(
    "https://api.openai.com/v1/models",
    headers={"Authorization": "Bearer sk-..."},
    timeout=10,              # ALWAYS set — prevents indefinite hangs
)
resp.raise_for_status()    # raises HTTPError on 4xx/5xx — never skip this
models = resp.json()       # parses response body as JSON

# ── POST — JSON body ──────────────────────────────────────────────────
payload = {
    "model": "gpt-4o",
    "messages": [{"role": "user", "content": "Hello"}],
    "max_tokens": 100,
}
resp = requests.post(
    "https://api.openai.com/v1/chat/completions",
    headers={"Authorization": "Bearer sk-...", "Content-Type": "application/json"},
    json=payload,             # use json= not data= — sets Content-Type automatically
    timeout=30,
)
resp.raise_for_status()
reply = resp.json()["choices"][0]["message"]["content"]

# ── Session — reuse connection + set base headers once ───────────────
session = requests.Session()
session.headers.update({"Authorization": "Bearer sk-..."})

r1 = session.get("https://api.openai.com/v1/models", timeout=10)
r2 = session.post("https://api.openai.com/v1/chat/completions", json=payload, timeout=30)
# Both calls share the auth header — no repetition, and TCP is reused

Authentication patterns #

Every LLM API uses one of two auth patterns. Know both, and know how to store secrets safely.

pythonauth.py — Bearer token + API key + env var loading
import os
from dotenv import load_dotenv
import requests

load_dotenv()   # reads .env file into os.environ — never commit .env to git

# ── Pattern 1: Bearer token (OpenAI, Anthropic, most LLM APIs) ───────
openai_key = os.environ["OPENAI_API_KEY"]   # KeyError if missing — fail fast

headers_bearer = {"Authorization": f"Bearer {openai_key}"}

# ── Pattern 2: API key in header (Pinecone, many vector DBs) ─────────
pinecone_key = os.environ["PINECONE_API_KEY"]

headers_apikey = {"Api-Key": pinecone_key}   # header name varies by service

# ── Anthropic — x-api-key header (non-standard) ──────────────────────
anthropic_key = os.environ["ANTHROPIC_API_KEY"]

headers_anthropic = {
    "x-api-key": anthropic_key,
    "anthropic-version": "2023-06-01",
    "content-type": "application/json",
}

# Note: in production you will use the SDK (anthropic.Anthropic()),
# not raw requests. This shows you what the SDK is doing underneath.
🚫
API keys in code = instant security incident Never write api_key = "sk-abc123..." in your source code. It will end up in git history, a pastebin, or a public repo. Always load from environment variables. Add .env to .gitignore on day one.

Rate limits and retry with tenacity #

LLM APIs return 429 Too Many Requests when you exceed rate limits. The correct response is exponential backoff with jitter — wait longer after each failure, add randomness to avoid thundering-herd when multiple processes retry simultaneously. tenacity handles all of this in one decorator.

pythonretry.py — production retry pattern with tenacity
import requests
from tenacity import (
    retry, stop_after_attempt, wait_exponential_jitter,
    retry_if_exception_type, before_sleep_log
)
import logging

logger = logging.getLogger(__name__)

@retry(
    retry=retry_if_exception_type(requests.exceptions.HTTPError),
    stop=stop_after_attempt(5),
    wait=wait_exponential_jitter(initial=1, max=60),   # 1s → 2s → 4s… + jitter
    before_sleep=before_sleep_log(logger, logging.WARNING),
    reraise=True,
)
def call_llm(prompt: str, model: str = "gpt-4o") -> str:
    """Call the OpenAI chat API with automatic retry on rate limit."""
    resp = requests.post(
        "https://api.openai.com/v1/chat/completions",
        headers={"Authorization": "Bearer ..."},
        json={"model": model, "messages": [{"role": "user", "content": prompt}]},
        timeout=30,
    )
    resp.raise_for_status()   # 429 raises HTTPError → triggers retry
    return resp.json()["choices"][0]["message"]["content"]


# Only retry on transient errors — not on 401 (bad key) or 400 (bad request)

Self-check #

What does response.raise_for_status() do and why should you always call it?
It raises an HTTPError exception if the response status code indicates a failure (4xx or 5xx). Without it, your code sees a Response object regardless of whether the API call succeeded — response.json() on a 429 rate-limit response will return the error body (or raise a decode error), not your expected data. Always call raise_for_status() immediately after every request so failures surface explicitly rather than causing silent downstream corruption.
Why use requests.Session() instead of top-level requests.get()?
A Session persists headers, auth, and cookies across multiple requests, so you don't repeat them on every call. More importantly, it maintains a connection pool — TCP connections to the same host are reused rather than re-established on each request. For a pipeline that calls an embedding API 100 times, connection reuse can halve latency.
What is exponential backoff with jitter and why does "jitter" matter?
Exponential backoff waits progressively longer between retries (1s → 2s → 4s → 8s). Jitter adds a random amount to each wait. Without jitter, if 100 clients hit a rate limit simultaneously and all retry at exactly 2 seconds, they cause another burst at 2 seconds — the thundering herd problem. Jitter spreads retries across the wait window, smoothing out the load on the API and increasing the chance of success.
What is the difference between passing json=payload vs data=payload to requests.post()?
json=payload serializes the dict to a JSON string and automatically sets Content-Type: application/json. data=payload form-encodes the dict (like an HTML form submission) and sets Content-Type: application/x-www-form-urlencoded. LLM APIs expect JSON — always use json= for them. Using data= by mistake is a common source of cryptic 400 errors.
