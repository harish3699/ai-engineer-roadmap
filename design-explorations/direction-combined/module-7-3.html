Common Multi-Agent Patterns

Five patterns cover the vast majority of multi-agent architectures you'll encounter in practice. Each has a distinct graph topology, a clear use case, and specific failure modes. This module covers all five with working LangGraph code: supervisor + workers (hierarchical delegation), sequential pipeline (assembly-line processing), parallel fan-out/fan-in (independent parallel tasks), plan-and-execute (upfront planning then step execution), and reflection loops (generate → critique → refine).

~30 min read 5 code examples Last updated 2026-05-07
Key Takeaways
  • Supervisor + workers: a supervisor decides which worker to call next; workers report back to supervisor, not to each other — avoids uncontrolled agent chaining
  • Sequential pipeline: each agent's output is the next agent's input; the cleanest pattern for assembly-line transformations with clear data contracts between stages
  • Parallel fan-out/fan-in: a splitter launches N independent workers concurrently; a merger aggregates their outputs — for tasks with truly independent sub-problems
  • Plan-and-execute: a planner agent produces a structured plan upfront; an executor agent follows it step by step — reduces mid-task drift on complex multi-step tasks
  • Reflection loop: a generator produces output; a critic evaluates it; the generator revises — keeps cycling until quality threshold or max iterations; the most expensive pattern but best for quality-critical output

Pattern 1 — Supervisor + Workers #

The supervisor is a central routing agent that decides which worker to call based on the current state. Workers complete specific sub-tasks and always return control to the supervisor. This pattern ensures one agent remains in control of the overall workflow — no agent chains without the supervisor's knowledge.

pythonsupervisor.py — hierarchical supervisor routing workers
from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated, Literal
from langgraph.graph.message import add_messages

WORKERS = ["researcher", "analyst", "writer"]

class SupervisorState(TypedDict):
    messages: Annotated[list, add_messages]
    next: str   # which worker to call next, or "FINISH"

supervisor_llm = llm.with_structured_output(class Route(TypedDict): next: Literal["researcher","analyst","writer","FINISH"])

def supervisor_node(state: SupervisorState) -> dict:
    result = supervisor_llm.invoke([
        {"role": "system", "content": f"You are a supervisor. Workers: {WORKERS}. Choose next or FINISH."},
        *state["messages"],
    ])
    return {"next": result["next"]}

def make_worker(name: str, system_prompt: str):
    def worker_node(state: SupervisorState) -> dict:
        response = llm.invoke([{"role":"system","content":system_prompt}, *state["messages"]])
        return {"messages": [{"role":"assistant","content":f"[{name}] {response.content}"}]}
    return worker_node

builder = StateGraph(SupervisorState)
builder.add_node("supervisor", supervisor_node)
builder.add_node("researcher", make_worker("researcher", "You are a research expert."))
builder.add_node("analyst",    make_worker("analyst",    "You are a data analyst."))
builder.add_node("writer",     make_worker("writer",     "You are a technical writer."))

builder.add_edge(START, "supervisor")
builder.add_conditional_edges("supervisor", lambda s: s["next"],
    {"researcher":"researcher", "analyst":"analyst", "writer":"writer", "FINISH":END})
for w in WORKERS:
    builder.add_edge(w, "supervisor")   # all workers return to supervisor
graph = builder.compile()

Pattern 2 — Sequential Pipeline #

Each stage transforms the output of the previous stage. The data contract between stages is explicit — each node produces exactly what the next node expects. Best for assembly-line tasks where each transformation is independent.

pythonpipeline.py — sequential 4-stage document processing pipeline
class PipelineState(TypedDict):
    raw_document: str
    extracted_facts: list[str]
    structured_report: str
    final_summary: str

def extract_node(state): ...   # raw_document → extracted_facts
def structure_node(state): ... # extracted_facts → structured_report
def summarise_node(state): ... # structured_report → final_summary

builder = StateGraph(PipelineState)
builder.add_node("extract",   extract_node)
builder.add_node("structure", structure_node)
builder.add_node("summarise", summarise_node)
builder.add_edge(START,       "extract")
builder.add_edge("extract",   "structure")
builder.add_edge("structure", "summarise")
builder.add_edge("summarise", END)

Pattern 3 — Parallel Fan-out / Fan-in #

A splitter creates N independent sub-tasks and dispatches them to worker nodes simultaneously. The Send primitive in LangGraph allows dynamic fan-out — you can dispatch a variable number of workers based on the state at runtime, not at graph-definition time.

pythonfan_out.py — dynamic parallel fan-out with Send
from langgraph.types import Send
import operator

class FanOutState(TypedDict):
    topics: list[str]                               # input: N topics to research
    reports: Annotated[list[str], operator.add]    # aggregated results
    final: str

class WorkerState(TypedDict):
    topic: str
    reports: Annotated[list[str], operator.add]

def dispatch_workers(state: FanOutState) -> list[Send]:
    # Returns a Send for each topic → LangGraph launches them concurrently
    return [Send("research_worker", {"topic": topic, "reports": []}) for topic in state["topics"]]

def research_worker(state: WorkerState) -> dict:
    result = llm.invoke(f"Research: {state['topic']}").content
    return {"reports": [result]}

def merge_node(state: FanOutState) -> dict:
    combined = "\n\n".join(state["reports"])
    final = llm.invoke(f"Synthesise: {combined}").content
    return {"final": final}

builder = StateGraph(FanOutState)
builder.add_node("research_worker", research_worker)
builder.add_node("merge", merge_node)
builder.add_conditional_edges(START, dispatch_workers, ["research_worker"])
builder.add_edge("research_worker", "merge")
builder.add_edge("merge", END)

Pattern 4 — Plan and Execute #

The planner produces a structured plan with discrete numbered steps. The executor works through the steps one by one, updating the plan's completion status as it goes. This pattern reduces mid-task drift on long, complex workflows where a reactive ReAct loop might get sidetracked.

pythonplan_execute.py — planner + step executor
from pydantic import BaseModel

class Plan(BaseModel):
    steps: list[str]

class PlanExecState(TypedDict):
    task: str
    plan: list[str]
    completed_steps: Annotated[list[str], operator.add]
    current_step: int
    response: str

planner_llm = llm.with_structured_output(Plan)

def planner_node(state: PlanExecState) -> dict:
    plan = planner_llm.invoke(f"Create a step-by-step plan for: {state['task']}")
    return {"plan": plan.steps, "current_step": 0}

def executor_node(state: PlanExecState) -> dict:
    step = state["plan"][state["current_step"]]
    context = "\n".join(state["completed_steps"])
    result = llm.invoke(f"Context:\n{context}\n\nExecute step: {step}").content
    return {"completed_steps": [f"Step {state['current_step']+1}: {result}"],
            "current_step": state["current_step"] + 1,
            "response": result}

def has_more_steps(state: PlanExecState) -> str:
    return "execute" if state["current_step"] < len(state["plan"]) else "end"

builder = StateGraph(PlanExecState)
builder.add_node("plan", planner_node)
builder.add_node("execute", executor_node)
builder.add_edge(START, "plan")
builder.add_edge("plan", "execute")
builder.add_conditional_edges("execute", has_more_steps, {"execute":"execute", "end":END})

Pattern 5 — Reflection Loop #

A generator produces output; a critic evaluates it against quality criteria and returns structured feedback; the generator revises based on the feedback. The loop continues until the critic approves or max iterations is reached. Most expensive pattern — uses 2N LLM calls for N revisions — but produces the highest-quality output for writing, code review, and plan validation.

pythonreflection.py — generate-critique-revise loop
class CritiqueResult(BaseModel):
    approved: bool
    feedback: str

class ReflectState(TypedDict):
    task: str
    draft: str
    feedback: str
    revisions: int
    max_revisions: int
    approved: bool

critic_llm = llm.with_structured_output(CritiqueResult)

def generator_node(state: ReflectState) -> dict:
    prompt = state["task"]
    if state["feedback"]:
        prompt += f"\n\nPrevious draft:\n{state['draft']}\n\nRevise based on: {state['feedback']}"
    draft = llm.invoke(prompt).content
    return {"draft": draft, "revisions": state["revisions"] + 1}

def critic_node(state: ReflectState) -> dict:
    result = critic_llm.invoke(f"Evaluate this draft for the task '{state['task']}':\n\n{state['draft']}")
    return {"feedback": result.feedback, "approved": result.approved}

def should_revise(state: ReflectState) -> str:
    if state["approved"] or state["revisions"] >= state["max_revisions"]:
        return "end"
    return "generate"

builder = StateGraph(ReflectState)
builder.add_node("generate", generator_node)
builder.add_node("critique", critic_node)
builder.add_edge(START, "generate")
builder.add_edge("generate", "critique")
builder.add_conditional_edges("critique", should_revise, {"generate":"generate", "end":END})
