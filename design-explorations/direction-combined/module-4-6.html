Reranking

Retrieval gives you a coarse shortlist — typically 20–100 candidates ranked by cosine similarity or BM25. Reranking gives you a precise final list — typically 3–10 chunks — using a model that reads query and passage together, not as separate vectors. This cross-encoder architecture is the single most impactful RAG upgrade per unit of engineering effort: it consistently moves answer quality from "usually good" to "reliable" by fixing the ranking mistakes that bi-encoders make when the relevant chunk is on the edge of the semantic neighbourhood.

~20 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • A cross-encoder reads the query and passage together in one forward pass — full attention between all tokens — producing a relevance score that is far more accurate than bi-encoder cosine similarity
  • Reranking is applied in two stages: bi-encoder retrieval gets a broad candidate set quickly; the cross-encoder re-scores only those candidates precisely — this keeps latency manageable
  • Cohere Rerank is the easiest managed option — one API call, handles English and multilingual corpora, no model hosting required
  • sentence-transformers cross-encoders (e.g. cross-encoder/ms-marco-MiniLM-L-6-v2) are open-source, free to run locally, and competitive with managed options on English text
  • The LLM context window is the most expensive resource in RAG — reranking ensures only the highest-relevance chunks consume those tokens, reducing hallucination from distracting context

▶ Cross-encoder vs bi-encoder architecture and when to use reranking in your RAG pipeline

Bi-encoder vs cross-encoder #

Everything you built in modules 4.3–4.5 is bi-encoder retrieval: encode the query independently, encode each document independently, compare the resulting vectors. The query and document never interact during encoding — cosine similarity is the only comparison mechanism. This is fast to scale because you pre-compute document embeddings once and query any number of times at low latency.

A cross-encoder is different: it takes the concatenation of query and passage as a single input — [CLS] query [SEP] passage [SEP] — and runs one transformer forward pass where every query token attends to every passage token. The output is a single relevance score, not a vector. This full cross-attention produces much more accurate relevance judgements than cosine similarity, but it is O(n) per query (you must run a new forward pass for each candidate), which is why you can't use it as your primary retrieval mechanism across millions of documents.

The bi-encoder gets a broad candidate set fast. The cross-encoder re-ranks those candidates with high accuracy. The LLM receives only the most relevant 3–5 chunks. Total latency: retrieval (~20ms) + reranking (~200ms for 50 candidates) + LLM (~800ms) = ~1s end-to-end.

Local reranking with sentence-transformers #

The sentence-transformers library includes cross-encoder models you can run locally at zero cost. The MS MARCO family of cross-encoders (ms-marco-MiniLM-L-6-v2, ms-marco-MiniLM-L-12-v2) is trained on Microsoft's BING web search passages — they generalise well to most English retrieval tasks.

pythonlocal_reranker.py — cross-encoder reranking with sentence-transformers
from sentence_transformers import CrossEncoder
import numpy as np

# Downloaded on first run — ~80MB for L-6-v2
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

query = "What is the allowance for ergonomic furniture?"

# Bi-encoder retrieved these candidates (in their original order)
candidates = [
    "Employees may claim up to $150 per month for home internet.",       # rank 1 from retrieval
    "A one-time $600 allowance is available for ergonomic furniture.",    # rank 2 — correct answer
    "All expense claims must be submitted via Workday by the 5th.",
    "AcmeCorp was founded in Austin, Texas in 2018.",
    "Manager approval is required for all claims over $200.",
    "Software subscriptions may be expensed up to $50 per month.",
]

# Score (query, passage) pairs — one forward pass per pair
pairs = [(query, c) for c in candidates]
scores = reranker.predict(pairs)  # logits, not probabilities — higher = more relevant

# Sort by reranker score
ranked = sorted(zip(scores, candidates), reverse=True)
print("── After reranking ──")
for score, text in ranked[:3]:
    print(f"  score={score:.2f}  {text}")
# → score=8.71  "one-time $600 allowance for ergonomic furniture"  ← moved to #1
# → score=3.12  "claims over $200 require manager approval"
# → score=1.48  "claims must be submitted via Workday"

# ── Batch prediction (GPU-accelerated) ───────────────────────────────
# For 50+ candidates, pass show_progress_bar=True and batch_size=32
scores_batch = reranker.predict(pairs, batch_size=32, show_progress_bar=False)
ℹ
Cross-encoder scores are not probabilities — they are logits The raw output of reranker.predict() is a logit from the final linear layer. It can be any real number — 8.7 is not "87% relevant". You use these scores only for ranking (sort descending), not for thresholding. If you want a calibrated 0–1 relevance estimate, apply a sigmoid: import torch; prob = torch.sigmoid(torch.tensor(score)).item(). But for RAG context selection, ranking by raw logit is sufficient and is the standard practice.

Cohere Rerank API #

Cohere Rerank is a managed API that accepts a query and a list of documents and returns them sorted by relevance with a relevance score. It supports English and multilingual corpora, handles documents up to ~4000 tokens each, and removes the need to host a model. The API cost as of 2025 is $2 per 1000 searches.

pythoncohere_rerank.py — managed reranking with Cohere API
import os
import cohere

co = cohere.Client(os.environ["CO_API_KEY"])

query = "What is the allowance for ergonomic furniture?"

# Bi-encoder candidates (just the text — Cohere accepts strings)
candidates = [
    "Employees may claim up to $150 per month for home internet.",
    "A one-time $600 allowance is available for ergonomic furniture.",
    "All expense claims must be submitted via Workday by the 5th.",
    "AcmeCorp was founded in Austin, Texas in 2018.",
    "Manager approval is required for all claims over $200.",
]

response = co.rerank(
    query=query,
    documents=candidates,
    model="rerank-english-v3.0",  # or "rerank-multilingual-v3.0"
    top_n=3,                     # return only top 3
    return_documents=True,
)

for result in response.results:
    print(f"[rank {result.index}] score={result.relevance_score:.4f}  {result.document.text}")

# ── Integrate into a RAG pipeline ─────────────────────────────────────
def retrieve_and_rerank(
    query: str,
    vector_results: list[str],  # raw chunk texts from bi-encoder retrieval
    top_k: int = 5,
) -> list[str]:
    """Rerank vector results, return top_k for LLM context."""
    if len(vector_results) <= top_k:
        return vector_results  # skip reranking if already small

    resp = co.rerank(
        query=query,
        documents=vector_results,
        model="rerank-english-v3.0",
        top_n=top_k,
        return_documents=True,
    )
    return [r.document.text for r in resp.results]

Where to place reranking in your pipeline #

Reranking sits between retrieval and generation. The pattern is always the same: retrieve many, rerank few, generate with the top results.

Wrong: rerank all chunks in your index
  • Reranking 100k chunks takes minutes — cross-encoder inference is O(n)
  • Defeats the purpose of ANN retrieval — you lose the speed advantage
  • No practical scenario requires scoring every document against every query
Right: retrieve broad, rerank narrow
  • Retrieve top-20 to top-100 via bi-encoder (milliseconds)
  • Rerank those candidates with cross-encoder (100–400ms for 20–100 pairs)
  • Take top-3 to top-10 for LLM context
  • Total added latency: 100–400ms — acceptable for most use cases
pythonfull_retrieve_rerank.py — complete two-stage pipeline
from sentence_transformers import SentenceTransformer, CrossEncoder
import numpy as np

bi_encoder  = SentenceTransformer("BAAI/bge-small-en-v1.5")
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

chunks = [
    "Employees may claim $150/mo for home internet.",
    "One-time $600 allowance for ergonomic furniture.",
    "Claims submitted via Workday by the 5th of each month.",
    "Manager approval required for claims over $200.",
    "Software subscriptions: up to $50/mo with pre-approval.",
    "AcmeCorp founded 2018, headquartered in Austin, Texas.",
    "Two-week sprint cycles with Friday retrospectives.",
    "Receipts required for all claims above $50.",
]

# Pre-compute document embeddings at index time
doc_vecs = bi_encoder.encode(chunks, normalize_embeddings=True)

def rag_retrieve(query: str, retrieve_k: int = 20, final_k: int = 3) -> list[str]:
    # Stage 1: bi-encoder — fast candidate retrieval
    q_prefix = f"Represent this sentence for searching relevant passages: {query}"
    q_vec = bi_encoder.encode([q_prefix], normalize_embeddings=True)[0]
    scores = doc_vecs @ q_vec
    top_indices = np.argsort(scores)[-retrieve_k:][::-1]
    candidates = [(idx, chunks[idx]) for idx in top_indices]

    # Stage 2: cross-encoder — precise reranking of candidates
    pairs = [(query, text) for _, text in candidates]
    rerank_scores = cross_encoder.predict(pairs)

    # Sort candidates by cross-encoder score
    reranked = sorted(
        zip(rerank_scores, [text for _, text in candidates]),
        reverse=True,
    )
    return [text for _, text in reranked[:final_k]]

context = rag_retrieve("What is the ergonomic furniture allowance?")
for i, chunk in enumerate(context):
    print(f"[{i+1}] {chunk}")

Choosing a reranker #

The choice between managed and local reranking comes down to privacy, infrastructure, and cost:

Reranker comparison
Model Latency (50 pairs) Cost Notes
Cohere rerank-english-v3.0 ~150ms $2/1k searches Best English quality, zero hosting, easy integration
Cohere rerank-multilingual-v3.0 ~200ms $2/1k searches 100+ languages; use for non-English or mixed-language corpora
ms-marco-MiniLM-L-6-v2 ~80ms CPU Free Best open-source for speed; good quality on English web/FAQ text
ms-marco-MiniLM-L-12-v2 ~200ms CPU Free Higher quality than L-6-v2; 12 layers vs 6; worth it on GPU
BAAI/bge-reranker-large ~400ms CPU Free MTEB-competitive open-source; multilingual BGE reranker
💡
Increase retrieve_k before adding reranking Reranking can only promote candidates that made it into the initial retrieval set. If the correct answer is ranked #25 by your bi-encoder and you retrieve only top-10, the reranker never sees it. Before adding reranking, increase retrieve_k to 40–100. The overhead is negligible for bi-encoder retrieval (it's fast). The reranker then has a better candidate pool to work with, which is often more impactful than the reranker model itself.

Self-check #

Explain the architectural difference between a bi-encoder and a cross-encoder. Why can't you use a cross-encoder as your primary retrieval mechanism across a million-document corpus?
A bi-encoder processes query and document independently through a transformer, producing a vector for each. Similarity is computed as cosine distance between pre-computed vectors — O(1) per query because document vectors are pre-indexed. A cross-encoder concatenates query and document as a single input: [CLS] query [SEP] document [SEP]. Every query token attends to every document token in a full forward pass, producing one relevance logit. This cross-attention allows far more nuanced relevance judgement but requires a new forward pass for every (query, document) pair — O(n) per query. At 1M documents and 50ms per forward pass, querying all documents takes 50 million milliseconds (14 hours). This is why cross-encoders are used only for reranking a small candidate set (20–100 documents) produced by fast bi-encoder retrieval.
Why is increasing retrieve_k (the number of bi-encoder candidates) important before adding reranking, and what is the tradeoff?
The reranker can only re-order candidates it has seen. If the correct answer is ranked #15 by the bi-encoder and you retrieve only top-10, the reranker never evaluates it — the answer is excluded before reranking even starts. This is called the "first-stage recall ceiling." Increasing retrieve_k from 10 to 50 or 100 raises the ceiling: more of the correct answers are guaranteed to be in the candidate pool. The tradeoff is reranker latency: a cross-encoder scoring 100 pairs takes roughly 10× as long as scoring 10 pairs. On CPU, ms-marco-MiniLM-L-6-v2 scores 100 pairs in about 200–400ms — acceptable for most RAG response time budgets. Always measure first-stage recall@K at your bi-encoder retrieve_k before committing to a reranker; if recall@10 is already 98%, increasing to 50 adds latency for negligible recall gain.
Cross-encoder scores are logits. What does this mean, and how does it affect how you use them?
A logit is the raw output of the final linear layer before any activation function — it can be any real number (e.g., -3.1, 0, 8.7). It is not bounded to [0,1] and is not a probability. The absolute value is not interpretable — a score of 8.7 does not mean "87% relevant." What matters is the relative ordering: if chunk A scores 8.7 and chunk B scores 3.1, chunk A is more relevant for this query. This means you use cross-encoder scores exclusively for ranking (sort descending), not for thresholding (e.g., "drop chunks with score < 0.5"). If you want a calibrated probability for downstream use, apply a sigmoid function to convert logits to [0,1] probabilities. But for selecting top-K context chunks, raw logit ranking is correct and sufficient.
You have a RAG system serving both English and German users from the same corpus. What reranker configuration would you use?
Use a multilingual reranker: either Cohere's rerank-multilingual-v3.0 (managed, $2/1k) or BAAI/bge-reranker-large (open-source, local). The key reason: query-document language matching matters. If a German-language query is encoded by an English-only reranker, the model has not seen German training pairs and will score incorrectly. Multilingual models are trained on (query, passage) pairs across languages including cross-language pairs (German query, English answer), so they handle mixed-language corpora correctly. Also check your bi-encoder — use a multilingual bi-encoder like BAAI/bge-m3 at the retrieval stage so German queries can match German documents. Both retrieval and reranking need multilingual support end-to-end.
