Retrieval Techniques

Basic cosine similarity retrieval works, but it fails in predictable ways: it misses exact-match keywords that happen to have low vector similarity, it returns redundant chunks that all say the same thing, and it struggles with complex multi-part questions. This module covers the techniques that fix these failures — hybrid BM25+vector search with Reciprocal Rank Fusion, MMR for diverse result sets, multi-query decomposition, HyDE for vocabulary mismatch, and parent-child retrieval for better context windows.

~26 min read 1 video · 1 recording script 5 code examples Last updated 2026-05-07
Key Takeaways
  • Vector similarity alone misses exact-match keywords; BM25 alone misses semantic paraphrases — hybrid search with Reciprocal Rank Fusion combines both signals reliably
  • MMR (Maximal Marginal Relevance) trades a little relevance for diversity — it prevents the top-K context window from being filled with near-duplicate chunks all saying the same thing
  • HyDE (Hypothetical Document Embeddings) generates a plausible answer to the question and searches with that answer's embedding rather than the raw question — it dramatically improves retrieval when query vocabulary differs from corpus vocabulary
  • Multi-query retrieval decomposes a complex question into simpler sub-queries, retrieves independently for each, then deduplicates — it improves recall on multi-hop questions
  • Parent-child retrieval indexes small child chunks for precise matching but returns their larger parent chunk as context — the LLM sees more surrounding information without sacrificing retrieval precision

Why vector-only retrieval fails #

Dense vector retrieval works by capturing semantic meaning — but it has two systematic blind spots. First, keyword mismatch: if a user asks "What is the EBITDA adjustment for Q3?" and the corpus contains that exact phrase, a dense retrieval model may score a general "financial reporting overview" chunk higher than the specific EBITDA chunk, because the query's semantic neighbourhood includes finance broadly. Second, redundancy: the top-5 results can all be near-identical chunks from the same section of the same document, wasting your context window on repetition.

Hybrid search: BM25 + vector with RRF #

BM25 is a classical keyword retrieval algorithm that ranks documents by how often query terms appear in them, adjusted for document length and term frequency across the corpus. It excels at exact-match retrieval that dense search misses. Combining BM25 and vector scores via Reciprocal Rank Fusion (RRF) is the simplest and most robust fusion method — it ranks each candidate by the sum of the reciprocals of its BM25 rank and vector rank.

ℹ
RRF formula: 1/(k + rank_bm25) + 1/(k + rank_vector) The constant k=60 (common default) dampens the influence of very high ranks. A result that ranks #1 in BM25 and #3 in vector gets score 1/61 + 1/63 ≈ 0.0322. A result that ranks #2 in both gets 1/62 + 1/62 ≈ 0.0323 — effectively the same. RRF does not require score normalisation between BM25 and cosine similarity, which have incompatible ranges, making it far more robust than weighted score addition.
pythonhybrid_search.py — BM25 + vector search with Reciprocal Rank Fusion
from rank_bm25 import BM25Okapi
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("BAAI/bge-small-en-v1.5")

chunks = [
    "Employees may claim up to $150 per month for home internet.",
    "A one-time $600 allowance is available for ergonomic furniture.",
    "All claims must be submitted via Workday by the 5th of the month.",
    "AcmeCorp was founded in Austin, Texas in 2018 and employs 400 people.",
    "Software subscriptions may be expensed up to $50 per month with approval.",
    "The EBITDA adjustment for Q3 is detailed in the supplemental finance report.",
    "Code reviews are mandatory for all pull requests before merging to main.",
]

# ── BM25 index ────────────────────────────────────────────────────────
tokenised = [doc.lower().split() for doc in chunks]
bm25 = BM25Okapi(tokenised)

# ── Vector index ──────────────────────────────────────────────────────
doc_vecs = model.encode(chunks, normalize_embeddings=True)

def rrf_search(query: str, k: int = 3, rrf_k: int = 60) -> list[tuple[int, float]]:
    n = len(chunks)

    # BM25 ranks
    bm25_scores = bm25.get_scores(query.lower().split())
    bm25_ranks = np.argsort(bm25_scores)[::-1]   # descending

    # Vector ranks
    q_prefix = f"Represent this sentence for searching relevant passages: {query}"
    q_vec = model.encode([q_prefix], normalize_embeddings=True)[0]
    vec_scores = doc_vecs @ q_vec
    vec_ranks = np.argsort(vec_scores)[::-1]

    # Build rank lookup dicts
    bm25_rank = {idx: r for r, idx in enumerate(bm25_ranks)}
    vec_rank  = {idx: r for r, idx in enumerate(vec_ranks)}

    # RRF score for every candidate
    rrf_scores = {
        i: 1 / (rrf_k + bm25_rank[i]) + 1 / (rrf_k + vec_rank[i])
        for i in range(n)
    }
    ranked = sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
    return ranked[:k]

for idx, score in rrf_search("EBITDA adjustment Q3"):
    print(f"[{idx}] rrf={score:.4f}  {chunks[idx]}")
# → [5] rrf=0.0323  "The EBITDA adjustment for Q3..."  ← exact match wins

MMR: maximum marginal relevance #

MMR selects results iteratively. It picks the most relevant chunk first, then at each step selects the chunk that maximises λ · relevance_to_query − (1−λ) · max_similarity_to_already_selected. The result set is diverse — high-relevance but non-redundant. The parameter λ ∈ [0, 1] controls the diversity/relevance tradeoff: λ=1 is pure relevance (standard top-K); λ=0 is pure diversity; λ=0.6–0.7 is a good default for RAG.

pythonmmr.py — iterative diverse chunk selection
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-small-en-v1.5")

chunks = [
    "Employees may claim $150/mo for home internet.",
    "Home internet allowance is $150 per month for all employees.",   # near-duplicate
    "The internet expense cap of $150 applies to primary residence only.",  # near-duplicate
    "Ergonomic furniture: one-time $600 allowance.",
    "Claims must be submitted via Workday by the 5th.",
]

doc_vecs = model.encode(chunks, normalize_embeddings=True)

def mmr(
    query: str,
    k: int = 3,
    lambda_: float = 0.65,
    fetch_k: int = 10,
) -> list[int]:
    """Return k diverse chunk indices via MMR selection."""
    q_vec = model.encode([query], normalize_embeddings=True)[0]

    # Start with top-fetch_k by cosine similarity
    relevance = doc_vecs @ q_vec
    candidates = list(np.argsort(relevance)[::-1][:fetch_k])

    selected = []
    while candidates and len(selected) < k:
        if not selected:
            # First pick: highest relevance
            best = candidates[0]
        else:
            selected_vecs = doc_vecs[selected]  # shape: (|selected|, dim)
            scores = []
            for idx in candidates:
                rel = relevance[idx]
                redundancy = float(np.max(doc_vecs[idx] @ selected_vecs.T))
                mmr_score = lambda_ * rel - (1 - lambda_) * redundancy
                scores.append((idx, mmr_score))
            best = max(scores, key=lambda x: x[1])[0]

        selected.append(best)
        candidates.remove(best)

    return selected

for idx in mmr("internet expense allowance", k=3, lambda_=0.65):
    print(f"[{idx}] {chunks[idx]}")
# → [0] "Employees may claim $150/mo..." ← top result
# → [3] "Ergonomic furniture: $600..." ← different topic, not the duplicate
# → [4] "Claims must be submitted..." ← another different topic

HyDE: hypothetical document embeddings #

Queries and documents live in different parts of the embedding space — a question like "What is the expense reimbursement deadline?" embeds differently from the answer "Claims must be submitted by the 5th of the month." HyDE addresses this by using an LLM to generate a hypothetical answer to the query, then embedding that answer and searching with it. The hypothetical answer's embedding lands closer to actual answer documents in the vector space.

pythonhyde.py — generate a hypothetical answer, search with its embedding
import os
from openai import OpenAI
from sentence_transformers import SentenceTransformer
import numpy as np

oai = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
model = SentenceTransformer("BAAI/bge-small-en-v1.5")

chunks = [
    "Employees may claim $150/mo for home internet.",
    "One-time $600 allowance for ergonomic furniture.",
    "Claims must be submitted via Workday by the 5th of the month.",
    "AcmeCorp was founded in Austin, Texas in 2018.",
]
doc_vecs = model.encode(chunks, normalize_embeddings=True)

def hyde_search(query: str, k: int = 3) -> list[tuple[int, float]]:
    # Step 1: generate a hypothetical answer
    resp = oai.chat.completions.create(
        model="claude-haiku-4-5-20251001",  # fast, cheap for hypothesis generation
        messages=[{
            "role": "user",
            "content": (
                f"Write a one-paragraph passage from a company policy document "
                f"that directly answers this question:\n\n{query}\n\n"
                f"Answer only with the passage, no preamble."
            ),
        }],
        max_tokens=120,
    )
    hypothesis = resp.choices[0].message.content.strip()

    # Step 2: embed the hypothesis and search
    h_vec = model.encode([hypothesis], normalize_embeddings=True)[0]
    scores = doc_vecs @ h_vec
    ranked = sorted(enumerate(scores), key=lambda x: x[1], reverse=True)

    print(f"\nHypothesis: {hypothesis[:100]}…")
    return ranked[:k]

query = "When do I need to submit my expense claim?"
for idx, score in hyde_search(query):
    print(f"[{idx}] score={score:.3f}  {chunks[idx]}")
⚠
HyDE adds latency and LLM cost to every query Generating a hypothesis requires one extra LLM call before retrieval. This adds 200–800ms depending on the model. Use HyDE selectively: (1) when your eval shows low context recall on queries where vocabulary mismatch is likely (user query uses different terms than the corpus), (2) not by default on every query. You can run HyDE in parallel with standard vector search and merge results with RRF — this hedges against the hypothesis being wrong.

Multi-query retrieval #

Complex questions often have multiple sub-questions embedded in them. "What are the expense limits for internet and equipment, and how do I submit a claim?" contains three retrievable intents. Multi-query retrieval uses an LLM to decompose the query into sub-questions, retrieves independently for each, then deduplicates the result set before passing to the LLM.

pythonmulti_query.py — LLM-decomposed query retrieval with deduplication
import os, json
from openai import OpenAI
from sentence_transformers import SentenceTransformer
import numpy as np

oai = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
model = SentenceTransformer("BAAI/bge-small-en-v1.5")

chunks = [
    "Employees may claim $150/mo for home internet.",
    "One-time $600 allowance for ergonomic furniture.",
    "Claims must be submitted via Workday by the 5th of the month.",
    "Manager approval required for claims over $200.",
    "Software subscriptions: up to $50/mo with pre-approval.",
]
doc_vecs = model.encode(chunks, normalize_embeddings=True)

def decompose_query(query: str) -> list[str]:
    resp = oai.chat.completions.create(
        model="claude-haiku-4-5-20251001",
        messages=[{
            "role": "user",
            "content": (
                f"Decompose this question into at most 3 simpler sub-questions, "
                f"each independently answerable. Return a JSON array of strings.\n\n"
                f"Question: {query}"
            ),
        }],
        max_tokens=200,
    )
    return json.loads(resp.choices[0].message.content)

def multi_query_search(query: str, k: int = 5) -> list[int]:
    sub_queries = decompose_query(query)
    print(f"Sub-queries: {sub_queries}")

    all_indices = set()
    for sq in sub_queries:
        sq_vec = model.encode([sq], normalize_embeddings=True)[0]
        scores = doc_vecs @ sq_vec
        top = np.argsort(scores)[-3:][::-1]  # top-3 per sub-query
        all_indices.update(top.tolist())

    return list(all_indices)[:k]  # deduplicate, return up to k

complex_query = "What are the expense limits for internet and equipment, and how do I submit a claim?"
result_indices = multi_query_search(complex_query)
for idx in result_indices:
    print(f"[{idx}] {chunks[idx]}")

Parent-child chunk retrieval #

Small chunks produce precise embeddings but leave the LLM without surrounding context. Large chunks produce noisy embeddings but give the LLM rich context. Parent-child retrieval gets the best of both: index small child chunks (high precision), but when a child chunk is retrieved, return its parent chunk (rich context) to the LLM.

pythonparent_child_retrieval.py — index children, return parents
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("BAAI/bge-small-en-v1.5")

# Parent chunks — large, context-rich sections
parents = {
    "p0": """EXPENSE POLICY — REMOTE WORK.
Employees working from home may claim up to $150 per month for home internet.
A one-time $600 allowance is available for ergonomic furniture.
Software subscriptions related to work may be expensed up to $50 per month.
All claims require manager pre-approval for amounts over $200.""",

    "p1": """EXPENSE CLAIM PROCESS.
All expense claims must be submitted via Workday.
The deadline is the 5th of the following month.
Claims submitted after the deadline are processed in the next cycle.
Receipts must be attached for all claims above $50.""",
}

# Child chunks — small, precision-focused sentences
children = [
    {"text": "Employees may claim $150/mo for home internet.",              "parent_id": "p0"},
    {"text": "One-time $600 allowance for ergonomic furniture.",            "parent_id": "p0"},
    {"text": "Software subscriptions: up to $50/mo with pre-approval.",     "parent_id": "p0"},
    {"text": "Expense claims submitted via Workday by the 5th.",            "parent_id": "p1"},
    {"text": "Late claims are processed in the following cycle.",           "parent_id": "p1"},
    {"text": "Receipts required for claims over $50.",                      "parent_id": "p1"},
]

# Embed children (not parents)
child_vecs = model.encode([c["text"] for c in children], normalize_embeddings=True)

def parent_child_search(query: str, k: int = 2) -> list[str]:
    q_vec = model.encode([query], normalize_embeddings=True)[0]
    scores = child_vecs @ q_vec
    top_child_indices = np.argsort(scores)[-k*3:][::-1]  # over-retrieve children

    # Deduplicate: collect unique parent_ids from top children
    seen_parents, context_chunks = set(), []
    for ci in top_child_indices:
        pid = children[ci]["parent_id"]
        if pid not in seen_parents:
            seen_parents.add(pid)
            context_chunks.append(parents[pid])
        if len(context_chunks) >= k:
            break

    return context_chunks  # full parent text goes to LLM context

for ctx in parent_child_search("standing desk reimbursement deadline"):
    print(f"--- PARENT CONTEXT ---\n{ctx}\n")

Which technique to use #

Stack these techniques based on the failure mode you observe in evaluation (module 4.8), not by default. Each technique adds latency or complexity — apply only what your evals show is needed.

Technique selection by failure mode
Exact-match terms not found
Hybrid (BM25 + RRF)
BM25 catches keyword matches that dense search misses. Low complexity, high impact.
Redundant top-K context
MMR (λ=0.65)
Diversifies results when many chunks say the same thing from the same section.
Query vocabulary ≠ corpus vocabulary
HyDE
User asks in casual language; documents use formal terminology. HyDE bridges the gap.
Multi-part complex questions
Multi-query
Decomposes multi-hop questions; each sub-question retrieves its own relevant chunks.
Context too narrow / missing context
Parent-child
Precise child chunk retrieval + rich parent context delivery to LLM.
💡
The recommended production stack: hybrid + parent-child + reranker If you only implement three techniques, these give the highest average ROI across diverse RAG workloads: (1) Hybrid BM25 + vector with RRF — fixes keyword misses at near-zero latency cost. (2) Parent-child chunks — fixes context narrowness without hurting precision. (3) A cross-encoder reranker on the merged result set (module 4.6) — fine-grained relevance scoring after coarse retrieval. Add MMR and HyDE only if eval shows they fix specific failures in your corpus.

Self-check #

Explain why Reciprocal Rank Fusion is preferred over weighted score addition when combining BM25 and vector search results.
BM25 scores are frequencies (typically 0–10 or more, corpus-dependent). Cosine similarity scores are bounded in [-1, 1] (practically 0–1 for similar texts). These ranges are incompatible — any fixed weighting like 0.5 × BM25_score + 0.5 × cosine is dominated by whichever score has larger magnitude, which varies per query. Normalising both to [0,1] works but requires knowing their ranges in advance. RRF avoids this entirely: it works only with ranks (ordinal positions), which are comparable across any two retrieval systems. A result ranked #1 contributes the same RRF score regardless of whether BM25 gave it a score of 0.3 or 8.7. This makes RRF robust, requiring no tuning for score normalisation.
In MMR, what happens as λ approaches 0, and when would you want that behaviour?
As λ → 0, the MMR objective becomes pure diversity: −max_similarity_to_already_selected. The algorithm selects the next chunk that is most different from all already-selected chunks, ignoring relevance to the query entirely. You would want this in an exploratory retrieval scenario — for example, a "show me diverse perspectives on this topic" use case, or when building a summary that needs to cover multiple different aspects of a document. For standard Q&A RAG where the user wants precise answers, λ=0 is counterproductive — it will select off-topic chunks. The practical RAG range is λ=0.5–0.7, balancing relevance with enough diversity to prevent context window waste on near-duplicates.
What is the vocabulary mismatch problem that HyDE addresses, and why does it arise in embedding-based retrieval?
Vocabulary mismatch is the phenomenon where a user query and the relevant document express the same concept using different words. Example: query "when is the form due?" → relevant chunk "claims must be submitted by the 5th of the month." The embedding models are trained to encode semantic meaning, but query–document asymmetry means the embedding of a question often lands in a different region of the vector space from the embedding of the answer, even when they are about the same concept. This is because question phrasing activates question-type semantic features while answer phrasing activates declarative statement features. HyDE bypasses this by having the LLM generate a plausible answer in the same declarative register as the corpus documents, then embedding that answer. The hypothesis embedding lands in the same region of the vector space as actual answer chunks, even though the hypothesis may be factually wrong — the point is the vector similarity, not factual correctness.
Describe the parent-child chunking architecture: what is indexed, what is retrieved, and why this is better than either small-chunk or large-chunk retrieval alone.
Architecture: Documents are split at two granularities. Child chunks are small (50–150 tokens) — single sentences or short passages optimised for embedding precision. Parent chunks are large (300–800 tokens) — full paragraphs or sections that contain the child plus surrounding context. Each child chunk stores a reference to its parent ID. Index: Only child chunk embeddings are stored in the vector index. Retrieval: The query is matched against child embeddings (high precision — the dense signal is undiluted by surrounding text). When child chunks are retrieved, the retriever fetches their parent chunk IDs and returns the full parent text to the LLM. Why better: Pure small-chunk retrieval: precise retrieval, but the LLM gets narrow context and may miss necessary surrounding information. Pure large-chunk retrieval: noisy embedding (multiple topics dilute the signal), lower retrieval precision. Parent-child gives retrieval precision of small chunks and context richness of large chunks.
