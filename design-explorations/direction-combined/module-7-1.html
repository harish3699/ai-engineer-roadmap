When to Go Multi-Agent

Multi-agent systems are the most over-applied pattern in AI engineering. Developers reach for them when their single-agent gets complex, then spend three weeks debugging inter-agent communication that didn't need to exist. This module is deliberately contrarian: you will spend most of it learning the reasons not to go multi-agent. When you do understand the genuine use cases — tasks with fundamentally different specialisation requirements, parallel workloads that need independent context isolation, or workflows where one agent's output is another's input and the boundary is real — multi-agent earns its complexity budget.

~18 min read 2 code examples Last updated 2026-05-07
Key Takeaways
  • A single agent with well-designed tools solves ~80% of tasks with less latency, lower cost, and simpler debugging than a multi-agent system
  • Multi-agent genuinely earns its complexity when: tasks need different context isolation, different system prompts, or parallel specialisation that exceeds one agent's capacity to hold in a single context window
  • The Tableau→QuickSight migration case: code analysis, migration planning, code generation, and testing each need different tools, context, and prompting strategies — this is a real multi-agent use case
  • Two telltale signs you need multi-agent: (1) your single agent's system prompt has grown to handle 4+ completely different sub-tasks, or (2) you need true parallel execution of independent steps
  • The cost is real: every agent hop adds an LLM call, latency, and a new failure point. Don't pay that cost without a clear benefit

Default to single agent #

Before adding a second agent, ask: can I solve this with one agent and better tools? In the vast majority of cases, the answer is yes. A single agent with 10 well-designed tools can handle complex, multi-step tasks, conditional routing, error recovery, and parallelism via asyncio.gather — without the inter-agent communication overhead, without multiple LLM calls per step, and with a single trace to debug.

⚠
The complexity tax Every agent you add multiplies the failure surface. With 2 agents, one can produce bad output that the second accepts without validation. With 4 agents, you get exponential combinations of failure modes. You also pay: N × LLM call latency (sequential), added token cost for state passing, and N × debugging surface when something goes wrong. That tax must be justified by a real architectural benefit, not by complexity avoidance.

The multi-agent decision framework #

Multi-agent architecture is justified when at least one of these three conditions is true:

When Multi-Agent is Justified
1
Context isolation
Sub-tasks need completely different context, retrieved documents, or system prompts that would conflict in a single context window. Example: a "code analysis" agent that reads 5,000 tokens of legacy code has context that would pollute a "test generation" agent if mixed together.
2
Specialised reasoning
Sub-tasks need fundamentally different prompting strategies (e.g., creative writing vs SQL generation vs code review). A single system prompt trying to do all three produces worse results than three agents each specialised with precision instructions.
3
True parallelism
Multiple independent sub-tasks can run in parallel and their results need to be merged. A single agent cannot achieve true parallelism across LLM calls — it must run sequentially. Multiple agents can run concurrently via asyncio.gather across different LLM calls.

Case study: Tableau → QuickSight migration #

The canonical example from the data.js curriculum. A customer wants to migrate 200 Tableau workbooks to AWS QuickSight. This is a real multi-agent use case because each phase of the migration requires genuinely different capabilities:

pythonmigration_agents.py — why this needs multiple agents
# Why a single agent fails here:
# ------------------------------------
# Agent turn 1: "Analyse this 200-workbook codebase for migration complexity"
#   → Loads 150,000 tokens of Tableau XML + VizQL into context
#   → Context is now filled with Tableau-specific analysis
#
# Agent turn 2: "Now write the QuickSight migration code"
#   → 150,000 tokens of Tableau XML still in context
#   → Model is now reasoning about QS with Tableau noise bleeding through
#   → Quality degrades 20-30% vs clean context
#
# Agent turn 3: "Now write tests for the migration"
#   → Context has BOTH Tableau analysis AND QS code
#   → At 128k context: lost-in-the-middle kills the test quality
#
# ────────────────────────────────────────────────────────────────────────
# Why multi-agent works:
# ────────────────────────────────────────────────────────────────────────

ANALYST_SYSTEM = """You are a Tableau expert. Analyse workbook XML for:
- Custom SQL complexity (1-5 scale)
- Calculated field patterns
- Cross-database joins
- Filter dependencies
Output a structured migration complexity report. Nothing else."""

PLANNER_SYSTEM = """You are a migration architect. Given a complexity report,
produce a phased migration plan with QuickSight equivalents.
You do not see the original Tableau XML — only the summary."""

GENERATOR_SYSTEM = """You are a QuickSight expert. Given a migration plan,
produce the equivalent QuickSight dataset definitions and analysis configs.
You do not see the Tableau XML or the migration plan rationale."""

TESTER_SYSTEM = """You are a QA engineer. Given QuickSight config, generate
pytest tests that validate data equivalence between old and new dashboards."""

# Each agent gets only what it needs — no context bleed, clean specialisation
# Analyst → Planner → Generator → Tester: ~4 LLM calls, each with clean context
# vs. single-agent: 4 turns × accumulating context = degrading quality

The cost model #

Before choosing multi-agent, estimate the per-hop penalty in your specific case:

Architecture Latency LLM calls Failure surfaces Justified when
Single agent + tools 1× per step 1 per turn 1 (the agent) Default choice
Supervisor + 2 workers 3× sequential 3 per request 3 Condition 1 or 2 met
Parallel fan-out (4 workers) 2× (fan-out + merge) 5 per request 5 Condition 3 met, 4× parallelism needed
Full pipeline (4 sequential) 4× sequential 4 per request 4 Conditions 1 AND 2 both met

Warning signs you're adding unnecessary complexity #

◈
Red flags for premature multi-agent
  • You're adding a second agent because the first agent "feels too complex" — the fix is better tool design, not more agents
  • The second agent mostly re-formats what the first agent produced — use a structured output schema instead
  • You need agents to share context extensively — they're passing most of their state to each other anyway, which defeats context isolation
  • The tasks are strictly sequential AND the output of step N is exactly what step N+1 needs verbatim — a pipeline of function calls is simpler
