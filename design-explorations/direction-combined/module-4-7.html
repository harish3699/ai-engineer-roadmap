Key Takeaways
  • A production RAG pipeline separates indexing (run once, or on document update) from querying (run per user request) — never re-index on every query
  • The system prompt for RAG has three jobs: instruct the model to answer only from context, tell it to say "I don't know" when context is insufficient, and format source citations
  • Streaming the LLM response dramatically improves perceived latency — users see the first tokens within one second instead of waiting for the full response
  • A relevance threshold on reranker scores lets you detect "no relevant context found" before calling the LLM — avoiding hallucinated answers from an empty context window
  • Modular design matters: keep indexing, retrieval, and generation as separate, testable functions — this makes it easy to swap components (e.g. upgrade the embedding model) without rewriting the whole pipeline

Pipeline architecture #

A production RAG pipeline has two distinct phases: indexing (run once when documents change) and querying (run per user request). Keeping them separate is the most important architectural decision — confusing them leads to systems that re-embed the entire corpus on every query, which is both slow and expensive.

The RAGPipeline class #

The full pipeline below integrates hybrid retrieval (BM25 + FAISS + RRF), cross-encoder reranking, and streaming generation into a single class with clean index/query interfaces.

pythonrag_pipeline.py — complete production-ready RAG pipeline
import os
from dataclasses import dataclass, field
from openai import OpenAI
from rank_bm25 import BM25Okapi
from sentence_transformers import SentenceTransformer, CrossEncoder
import faiss
import numpy as np

@dataclass
class Chunk:
    text:     str
    source:   str = ""    # filename or document title
    section:  str = ""    # section header if available

class RAGPipeline:
    def __init__(
        self,
        bi_encoder_model:    str = "BAAI/bge-small-en-v1.5",
        cross_encoder_model: str = "cross-encoder/ms-marco-MiniLM-L-6-v2",
        llm_model:           str = "claude-sonnet-4-6",
        retrieve_k:          int = 30,
        final_k:             int = 5,
        relevance_threshold: float = 0.0,  # cross-encoder logit floor
    ):
        self.bi_encoder   = SentenceTransformer(bi_encoder_model)
        self.cross_encoder = CrossEncoder(cross_encoder_model)
        self.llm          = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
        self.llm_model    = llm_model
        self.retrieve_k   = retrieve_k
        self.final_k      = final_k
        self.relevance_threshold = relevance_threshold

        self.chunks: list[Chunk] = []
        self.faiss_index  = None
        self.bm25         = None

    # ── INDEXING ─────────────────────────────────────────────────────────

    def index(self, chunks: list[Chunk]) -> None:
        """Embed chunks and build FAISS + BM25 indexes. Call once."""
        self.chunks = chunks
        texts = [c.text for c in chunks]

        # Bi-encoder vectors → FAISS
        vecs = self.bi_encoder.encode(texts, normalize_embeddings=True, show_progress_bar=True)
        dim = vecs.shape[1]
        self.faiss_index = faiss.IndexFlatIP(dim)
        self.faiss_index.add(vecs.astype("float32"))

        # BM25 index on tokenised text
        self.bm25 = BM25Okapi([t.lower().split() for t in texts])
        print(f"Indexed {len(chunks)} chunks.")

    # ── RETRIEVAL ────────────────────────────────────────────────────────

    def _hybrid_retrieve(self, query: str) -> list[int]:
        n = len(self.chunks)
        k = min(self.retrieve_k, n)

        # BM25 ranks
        bm25_scores = self.bm25.get_scores(query.lower().split())
        bm25_ranks = {idx: r for r, idx in enumerate(np.argsort(bm25_scores)[::-1])}

        # Vector ranks
        prefix = f"Represent this sentence for searching relevant passages: {query}"
        q_vec = self.bi_encoder.encode([prefix], normalize_embeddings=True).astype("float32")
        _, idx_arr = self.faiss_index.search(q_vec, k)
        vec_ranks = {idx: r for r, idx in enumerate(idx_arr[0].tolist())}

        # RRF fusion over all candidates seen in either rank list
        candidates = set(bm25_ranks) | set(vec_ranks)
        rrf = {
            i: 1 / (60 + bm25_ranks.get(i, n)) + 1 / (60 + vec_ranks.get(i, n))
            for i in candidates
        }
        return [i for i, _ in sorted(rrf.items(), key=lambda x: x[1], reverse=True)[:k]]

    def retrieve(self, query: str) -> list[Chunk]:
        """Hybrid retrieval → cross-encoder reranking → top-final_k chunks."""
        candidate_indices = self._hybrid_retrieve(query)
        candidates = [self.chunks[i] for i in candidate_indices]

        pairs = [(query, c.text) for c in candidates]
        scores = self.cross_encoder.predict(pairs)

        ranked = sorted(zip(scores, candidates), key=lambda x: x[0], reverse=True)
        return [
            c for score, c in ranked[:self.final_k]
            if score >= self.relevance_threshold
        ]

    # ── GENERATION ───────────────────────────────────────────────────────

    def query(self, question: str, stream: bool = True) -> str:
        """Retrieve context and generate an answer. Streams by default."""
        context_chunks = self.retrieve(question)

        if not context_chunks:
            return "I couldn't find relevant information in the knowledge base to answer this question."

        context_text = "\n\n".join(
            f"[Source: {c.source or 'unknown'}, Section: {c.section or 'general'}]\n{c.text}"
            for c in context_chunks
        )

        messages = [
            {"role": "system", "content": _system_prompt()},
            {
                "role": "user",
                "content": (
                    f"CONTEXT:\n{context_text}\n\n"
                    f"QUESTION: {question}"
                ),
            },
        ]

        if stream:
            response = self.llm.chat.completions.create(
                model=self.llm_model, messages=messages, stream=True, max_tokens=600
            )
            full = ""
            for chunk in response:
                delta = chunk.choices[0].delta.content or ""
                print(delta, end="", flush=True)
                full += delta
            print()
            return full
        else:
            resp = self.llm.chat.completions.create(
                model=self.llm_model, messages=messages, max_tokens=600
            )
            return resp.choices[0].message.content

The RAG system prompt #

The system prompt for a RAG pipeline has three specific jobs beyond general instruction: it must instruct the LLM to stay within the provided context, to refuse confidently when context is insufficient, and to cite sources so users can verify answers. These three constraints collectively prevent hallucination in the RAG context — the most common production failure mode.

pythonsystem_prompt.py — the three-part RAG system prompt
def _system_prompt() -> str:
    return """You are a precise knowledge-base assistant.

RULES — follow exactly:
1. Answer ONLY using the CONTEXT passages provided below the question.
   Do not use any information from your training data or prior knowledge.
2. If the CONTEXT does not contain sufficient information to answer the question,
   respond with exactly: "I don't have enough information in the knowledge base to answer this question."
   Do not guess, speculate, or fill in gaps from general knowledge.
3. When you use a fact from the context, cite its source inline using the format:
   (Source: ) — e.g. "(Source: HR Policy v3.2)"
4. Be concise. Answer in 2–5 sentences unless the question requires more detail.
5. If the question can be answered partially from the context, state what you know
   and explicitly note what is not covered: "The context doesn't include information about X."

TONE: direct, professional, no preamble ("Sure!", "Certainly!", "Great question!")."""
⚠
The most common RAG prompt mistake: not explicitly forbidding prior knowledge Without a clear prohibition, LLMs will blend retrieved context with training knowledge — seamlessly and invisibly. The answer will appear grounded in the context but contain facts the model added from training. This is especially dangerous in regulated domains (legal, medical, HR policy) where the context may contain the accurate, current version of a policy and the model's training data may contain an outdated version. The phrase "Do not use any information from your training data" must be explicit, not implied.

Streaming responses #

Streaming is not a performance optimisation — it is a UX requirement. Without streaming, users stare at a blank screen for 1–3 seconds before seeing any output. With streaming, they see the first tokens within 200–400ms and can start reading while the rest generates. For a RAG system where the retrieval phase already takes 200–500ms, streaming generation is the difference between feeling responsive and feeling slow.

pythonstreaming_rag.py — FastAPI endpoint with streaming SSE response
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio, json

app = FastAPI()
pipeline = RAGPipeline()  # already indexed

async def stream_answer(question: str):
    """Async generator yielding SSE-formatted tokens."""
    context_chunks = pipeline.retrieve(question)

    if not context_chunks:
        yield f"data: {json.dumps({'token': 'No relevant context found.'})}\n\n"
        return

    context_text = "\n\n".join(f"[{c.source}]\n{c.text}" for c in context_chunks)
    messages = [
        {"role": "system", "content": _system_prompt()},
        {"role": "user", "content": f"CONTEXT:\n{context_text}\n\nQUESTION: {question}"},
    ]

    client = pipeline.llm
    with client.chat.completions.create(
        model=pipeline.llm_model, messages=messages, stream=True, max_tokens=600
    ) as response:
        for chunk in response:
            token = chunk.choices[0].delta.content or ""
            if token:
                yield f"data: {json.dumps({'token': token})}\n\n"
                await asyncio.sleep(0)  # yield control to event loop
    yield "data: [DONE]\n\n"

@app.get("/ask")
async def ask(q: str):
    return StreamingResponse(
        stream_answer(q),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
    )

Running the complete pipeline #

Here is how you wire everything together for a real document corpus:

pythonmain.py — index documents and run queries
from rag_pipeline import RAGPipeline, Chunk

# ── 1. Prepare chunks from your document corpus ───────────────────────
# In production: load PDFs/markdown, run your chunking strategy (4.2),
# and attach source metadata to each chunk.
chunks = [
    Chunk("Employees may claim $150/mo for home internet.",    source="HR Policy v3.2", section="Remote Work Expenses"),
    Chunk("One-time $600 allowance for ergonomic furniture.",   source="HR Policy v3.2", section="Remote Work Expenses"),
    Chunk("Claims submitted via Workday by the 5th of each month.", source="HR Policy v3.2", section="Claims Process"),
    Chunk("Manager approval required for claims over $200.",     source="HR Policy v3.2", section="Claims Process"),
    Chunk("Software subscriptions: up to $50/mo with pre-approval.", source="HR Policy v3.2", section="Software"),
    Chunk("AcmeCorp founded 2018, headquartered in Austin, TX.",  source="Company Handbook", section="About"),
]

# ── 2. Initialise and index (run once) ────────────────────────────────
rag = RAGPipeline(
    bi_encoder_model="BAAI/bge-small-en-v1.5",
    cross_encoder_model="cross-encoder/ms-marco-MiniLM-L-6-v2",
    llm_model="claude-sonnet-4-6",
    retrieve_k=20,    # retrieve this many candidates before reranking
    final_k=4,        # pass this many to the LLM context
    relevance_threshold=1.0,  # drop chunks with cross-encoder logit < 1.0
)
rag.index(chunks)

# ── 3. Query (per user request) ───────────────────────────────────────
questions = [
    "What is the allowance for ergonomic furniture?",
    "How much can I claim for my standing desk?",     # vocabulary mismatch
    "When is the deadline to submit expense claims?",
    "What is the company's stock price?",              # out-of-corpus — should refuse
]

for q in questions:
    print(f"\n❓ {q}")
    print("─" * 60)
    rag.query(q, stream=True)
💡
Set the relevance threshold by observing cross-encoder score distributions Run 50–100 representative queries on your corpus and print the cross-encoder score of the top-ranked chunk. For queries where the answer exists in the corpus, scores typically land above 2–5. For out-of-corpus queries, scores often land below 0–2. Set your relevance_threshold at the 5th percentile of "good query" scores — this catches most "no relevant context" cases while rarely misclassifying answerable queries. If you have no eval data yet, start at 0.0 (off) and add it after collecting query logs.

Self-check #

Why must indexing and querying be separate phases in a production RAG pipeline, and what goes wrong when they are conflated?
Indexing is O(n) — every document in the corpus must be chunked and embedded. For a 10,000-document corpus with 500ms per chunk embedding, indexing takes hours. Querying is O(1) in corpus size — the query is embedded once and ANN retrieval is sub-linear. Conflating the two means re-embedding the entire corpus on every user query, making each query take hours. The correct design is: index runs once at startup, or on an incremental basis when documents change (add/update the new chunks, don't re-index everything). Query runs per user request, using the pre-built indexes. This distinction also matters for scaling: you can deploy multiple query replicas (fast, stateless) while running indexing as a separate background process.
Explain the role of the relevance_threshold parameter and how you would calibrate it for a new corpus.
The relevance threshold is a minimum cross-encoder logit score that retrieved chunks must exceed to be passed to the LLM. Chunks below the threshold are dropped. If all chunks are dropped, the pipeline returns a "no relevant context" refusal without calling the LLM — preventing hallucination from an empty context window. Calibration: (1) Run 50–100 representative queries split between "answerable from the corpus" and "out-of-corpus" questions. (2) For each, print the cross-encoder score of the top retrieved chunk. (3) Observe the score distributions — in-corpus queries typically score much higher than out-of-corpus queries. (4) Set the threshold at the value that best separates these distributions, erring toward false negatives (occasionally refusing answerable queries) over false positives (hallucinating answers for unanswerable ones). Start at 0.0 (disabled) until you have enough query data to calibrate.
What are the three specific jobs of a RAG system prompt, and why is each one necessary?
1. Answer only from context: LLMs blend training knowledge with retrieved context invisibly. An explicit "do not use information from your training data" forces the model to stay in context. Without this, answers may silently contain outdated or domain-mismatched training data. 2. Refuse when context is insufficient: If the answer is not in the corpus, the model must say so rather than guessing. Without this, the model fabricates plausible-sounding answers ("hallucination with confidence"). In regulated domains (HR policy, medical) a confident but wrong answer is worse than an explicit refusal. 3. Cite sources: Inline citations allow users to verify answers in the source documents. This is critical for trust in enterprise RAG systems. Without citations, users cannot distinguish between grounded answers and well-phrased hallucinations. Citations also make debugging easier: if the answer is wrong, you know which chunk to investigate.
A user reports that your RAG system is slow — responses take 4–5 seconds. Walk through the pipeline stages and identify where the latency is likely coming from.
Measure each stage individually: (1) Bi-encoder retrieval: should be 10–50ms for a FAISS IndexFlatIP over 100k vectors. If this is slow, check if you are re-encoding all document vectors on each query (indexing conflation bug), or if the FAISS index is on CPU and you have too many vectors for memory. (2) Cross-encoder reranking: depends on retrieve_k and model size. ms-marco-MiniLM-L-6-v2 on CPU: ~80ms for 30 pairs. If this is 1s+, your retrieve_k is too high (reduce to 20–30) or you're using a large model (L-12 or bge-large) on CPU without GPU. (3) LLM generation: typically 1–3s for the first token (TTFT) for Claude Sonnet. If you're not streaming, users wait the full generation time. Enable streaming to improve perceived latency to TTFT. If TTFT itself is slow, your final_k is too high — each additional context chunk adds tokens to the prompt, slowing the first-token response.
