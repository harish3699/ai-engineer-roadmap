Storage & Data

Every agent system needs three types of storage working together: durable object storage for raw documents and artefacts, a relational database for structured state and conversation history, and a key-value store for high-throughput pipeline metadata. On AWS those are S3, RDS PostgreSQL, and DynamoDB. This module shows exactly how each one is used in an agent deployment — not a general AWS tutorial, but the specific patterns you'll reach for when building production LLM systems.

~18 min read 3 code examples Last updated 2026-05-07
Key Takeaways
  • S3: document lake for raw files uploaded by users, crawled web pages, embeddings export/import, model artefacts, and LLM trace archives — virtually unlimited, pay-per-GB-stored
  • RDS PostgreSQL: agent conversation state, long-term memory records, evaluation results, user preferences — anything that needs ACID transactions, joins, or complex queries. Use pgvector for embeddings stored alongside structured data
  • DynamoDB: ingestion pipeline job state, document processing status, idempotency keys — anything that needs single-digit-ms reads at arbitrary throughput with a known access pattern
  • The decision rule: S3 for blobs, Postgres for queryable structured data and anything you'll JOIN, DynamoDB for high-throughput KV with known access patterns and no joins

S3 — document lakes and artefact storage #

In an agent system, S3 is where raw material lives before it's processed, and where outputs are persisted for durability. The typical bucket layout for a RAG pipeline:

texts3-layout.txt — bucket structure for an agent deployment
s3://my-agent-prod/
├── documents/raw/          ← user uploads (PDF, DOCX, TXT) — immutable once written
│   └── {user_id}/{doc_id}.{ext}
├── documents/processed/    ← chunked JSON after text extraction
│   └── {user_id}/{doc_id}.jsonl
├── embeddings/             ← exported vectors for backup / re-indexing
│   └── {collection_id}/{date}.parquet
├── traces/                 ← LangSmith/LangFuse trace exports for offline analysis
│   └── {year}/{month}/{day}/{trace_id}.json
└── models/                 ← fine-tuned model artefacts, eval checkpoints
    └── {experiment_id}/
pythons3_document_store.py — upload, presign, and retrieve
import boto3
from pathlib import Path

s3 = boto3.client("s3", region_name="us-east-1")
BUCKET = "my-agent-prod"

def store_document(user_id: str, doc_id: str, file_bytes: bytes, extension: str) -> str:
    key = f"documents/raw/{user_id}/{doc_id}.{extension}"
    s3.put_object(
        Bucket=BUCKET, Key=key, Body=file_bytes,
        ServerSideEncryption="AES256",          # encrypt at rest
        Metadata={"user_id": user_id, "doc_id": doc_id},
    )
    return key

def get_download_url(key: str, expires_in: int = 3600) -> str:
    """Pre-signed URL — never expose the bucket directly to users."""
    return s3.generate_presigned_url(
        "get_object",
        Params={"Bucket": BUCKET, "Key": key},
        ExpiresIn=expires_in,
    )

def stream_document(key: str) -> bytes:
    response = s3.get_object(Bucket=BUCKET, Key=key)
    return response["Body"].read()

RDS PostgreSQL — agent state and memory #

Postgres is the right home for anything that needs ACID guarantees, rich queries, or a relation to another table. For agents that means: conversation history (LangGraph checkpoint storage), long-term memory records, evaluation results, and user preferences. The pgvector extension lets you store embeddings in the same database as structured metadata — eliminating the need for a separate vector database for smaller deployments.

sqlschema.sql — agent state tables on RDS Postgres
-- Enable pgvector for embedding storage
CREATE EXTENSION IF NOT EXISTS vector;

-- Conversation sessions
CREATE TABLE sessions (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     TEXT NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT now(),
    ended_at    TIMESTAMPTZ,
    metadata    JSONB DEFAULT '{}'
);

-- LangGraph checkpoints (compatible with PostgresSaver)
CREATE TABLE checkpoints (
    thread_id       TEXT NOT NULL,
    checkpoint_ns   TEXT NOT NULL DEFAULT '',
    checkpoint_id   TEXT NOT NULL,
    parent_id       TEXT,
    type            TEXT,
    checkpoint      BYTEA NOT NULL,
    metadata        BYTEA NOT NULL,
    PRIMARY KEY (thread_id, checkpoint_ns, checkpoint_id)
);

-- Long-term memory with pgvector
CREATE TABLE memories (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         TEXT NOT NULL,
    content         TEXT NOT NULL,
    memory_type     TEXT NOT NULL,
    importance      FLOAT NOT NULL DEFAULT 0.5,
    embedding       vector(1536),            -- text-embedding-3-small dimension
    created_at      TIMESTAMPTZ DEFAULT now(),
    last_accessed   TIMESTAMPTZ DEFAULT now(),
    access_count    INT DEFAULT 0
);

CREATE INDEX ON memories USING hnsw (embedding vector_cosine_ops);
CREATE INDEX ON memories (user_id, importance DESC);

-- ANN search: top-5 memories by cosine similarity
SELECT content, memory_type, importance,
       1 - (embedding <<=>> $1::vector) AS similarity
FROM memories
WHERE user_id = $2 AND importance > 0.2
ORDER BY embedding <<=>> $1::vector
LIMIT 5;

DynamoDB — ingestion pipeline KV state #

DynamoDB excels when you have a high-throughput operation with a known access pattern — always fetched by a single key, no joins required. Document ingestion pipelines are a perfect fit: track processing status per document, store idempotency keys to prevent duplicate processing, record chunk counts and error flags.

The single-table design pattern: store all entity types in one table, use the PK/SK combination to encode the entity type and identifier. This gives you the flexibility to add access patterns later without schema changes.

✦
Postgres vs DynamoDB decision If you need to query "all documents for user X that failed in the last 7 days" with a WHERE clause — that's Postgres. If you need to check "is document Y already processed?" at 50,000 requests/second — that's DynamoDB. When in doubt: start with Postgres (easier to iterate on), migrate the hot paths to DynamoDB when you have real throughput data.
