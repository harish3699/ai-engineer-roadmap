Applied Prompt Patterns

Every production LLM task is one of five primitives: extract structure from text, classify it into categories, transform it into a different form, generate new content from a spec, or decompose a complex task into sub-steps. Each primitive has a repeatable template. Master all five and you can assemble any pipeline you'll encounter in the field — no trial-and-error from scratch each time.

~30 min read 1 video · 1 recording script 5 code examples Last updated 2026-05-07
Key Takeaways
  • Every LLM task reduces to one of five primitives — extract, classify, transform, generate, decompose — each with its own reliable template
  • Extraction is most reliable with a tool-call schema; the model can't hallucinate fields that aren't in your JSON Schema definition
  • Classification at scale needs a confidence score alongside the label — low-confidence results should route to a human fallback, not be silently accepted
  • Transformation tasks must specify the preservation constraint — what must stay the same — or the model will over-edit
  • Decomposition is the gateway to agents: breaking one complex task into a sequential pipeline of simple LLM calls is more reliable than one mega-prompt trying to do everything

The five primitives #

Before writing a single line of prompt, identify which primitive your task is. The primitive determines your template, your output validation strategy, and how you handle failures.

The five primitives — input → output shape
Extraction
Pull structured fields from unstructured text. Contracts, emails, docs, transcripts.
text → structured JSON
Classification
Assign one or more labels from a defined set. Routing, tagging, scoring, intent detection.
text → label(s) + score
Transformation
Convert text to a different form — same meaning, different format, style, or language.
text → text (reshaped)
Generation
Produce new content from a spec or brief. Emails, descriptions, code, plans.
spec → new content
Decomposition
Break a complex task into sequential sub-tasks. Each step is a simple LLM call.
task → pipeline of calls

Pattern 1: Extraction #

Extraction pulls specific fields from unstructured text. The output is always structured — JSON, a dataclass, or a named tuple. Use a tool-call schema (from module 3.2) to constrain the output. The model can only return fields defined in the schema — hallucination of extra fields is eliminated at the API level.

pythonextraction.py — contract key terms extraction
import anthropic, json
client = anthropic.Anthropic()

# ── Schema: only these fields can appear in the output ───────────────
CONTRACT_SCHEMA = {
    "name": "extract_contract_terms",
    "description": "Extract key commercial terms from a contract excerpt.",
    "input_schema": {
        "type": "object",
        "properties": {
            "parties":        {"type": "array",   "items": {"type": "string"}},
            "contract_value": {"type": "string",  "description": "Total value with currency"},
            "start_date":     {"type": "string",  "description": "ISO 8601 date"},
            "end_date":       {"type": "string",  "description": "ISO 8601 date"},
            "renewal_terms":  {"type": "string"},
            "governing_law":  {"type": "string"},
        },
        "required": ["parties", "contract_value", "start_date"],
    },
}

contract_text = """This Software License Agreement is entered into as of January 1, 2025,
between AcmeCorp Inc. ("Licensor") and BetaWorks Ltd. ("Licensee"). The total
license fee is USD 84,000 per annum, payable quarterly. The agreement commences
on 2025-01-01 and expires on 2026-12-31, with automatic one-year renewal unless
either party provides 90 days written notice. This agreement is governed by the
laws of the State of Delaware."""

resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=256,
    system="Extract contract terms precisely. Use null for any field not present in the text.",
    tools=[CONTRACT_SCHEMA],
    tool_choice={"type": "tool", "name": "extract_contract_terms"},
    messages=[{"role": "user", "content": f"<contract>\n{contract_text}\n</contract>"}],
)

result = next(b for b in resp.content if b.type == "tool_use").input
print(json.dumps(result, indent=2))
# → {"parties": ["AcmeCorp Inc.", "BetaWorks Ltd."],
#    "contract_value": "USD 84,000 per annum",
#    "start_date": "2025-01-01", "end_date": "2026-12-31", ...}

Pattern 2: Classification #

Classification assigns labels from a predefined set. The most important production detail: always return a confidence score alongside the label. Low-confidence results should not be silently accepted — route them to a human review queue. Without a confidence signal, you have no way to distinguish a certain result from a guess.

pythonclassification.py — multi-label support ticket routing with confidence
import anthropic
client = anthropic.Anthropic()

ROUTING_TOOL = {
    "name": "route_ticket",
    "description": "Classify a support ticket and route it to the correct team.",
    "input_schema": {
        "type": "object",
        "properties": {
            "primary_team": {
                "type": "string",
                "enum": ["billing", "technical", "onboarding", "security", "account"],
            },
            "secondary_teams": {
                "type": "array",
                "items": {"type": "string", "enum": ["billing", "technical", "onboarding", "security", "account"]},
            },
            "priority":      {"type": "string", "enum": ["low", "medium", "high", "critical"]},
            "confidence":    {"type": "number", "minimum": 0, "maximum": 1},
            "reason":        {"type": "string"},
        },
        "required": ["primary_team", "priority", "confidence"],
    },
}

def route_ticket(ticket: str) -> dict:
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=128,
        system="You are a support triage system. Route tickets accurately. Be strict with priority levels.",
        tools=[ROUTING_TOOL],
        tool_choice={"type": "tool", "name": "route_ticket"},
        messages=[{"role": "user", "content": ticket}],
    )
    return next(b for b in resp.content if b.type == "tool_use").input

CONFIDENCE_THRESHOLD = 0.80  # route below this to human review

tickets = [
    "I was charged twice for my subscription this month.",
    "My API keys stopped working after the password reset — also I think someone accessed my account.",
    "How do I export my data to CSV? I can't find the button.",
]

for ticket in tickets:
    result = route_ticket(ticket)
    destination = result["primary_team"] if result["confidence"] >= CONFIDENCE_THRESHOLD else "human_review"
    print(f"→ {destination} [{result['priority']}] conf={result['confidence']:.2f}")

Pattern 3: Transformation #

Transformation converts text to a different form while preserving meaning. Translate, simplify, reformat, or change tone. The critical instruction: always specify what must not change. Without a preservation constraint, the model treats everything as fair game and over-edits — changing meaning, dropping facts, or rewriting in ways you didn't intend.

pythontransformation.py — error message → user-friendly copy, preserving facts
import anthropic
client = anthropic.Anthropic()

TRANSFORM_SYSTEM = """You rewrite technical error messages as friendly, actionable user copy.

## Preservation constraints (MUST NOT CHANGE)
- All specific values: error codes, file names, limits, URLs, time durations
- The core meaning of what went wrong
- Any instructions about what the user should do next

## What you change
- Technical jargon → plain English
- Passive voice → active voice
- Cold/robotic tone → warm, helpful tone

## Format
One short paragraph. No markdown. Under 60 words."""

errors = [
    "ERR_UPLOAD_LIMIT_EXCEEDED: The uploaded file (24.3 MB) exceeds the maximum permitted size (20 MB). Request aborted. See https://docs.acme.com/limits for upgrade options.",
    "Authentication token expired. Token TTL: 3600s. Re-authenticate at /auth/refresh to obtain a new bearer token.",
]

for error in errors:
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=128,
        system=TRANSFORM_SYSTEM,
        messages=[{"role": "user", "content": f"<error>\n{error}\n</error>"}],
    )
    print(f"Original: {error[:60]}…")
    print(f"Friendly: {resp.content[0].text}\n")
# → "Your file (24.3 MB) is a bit too large — our limit is 20 MB.
#    Try compressing it or visit docs.acme.com/limits to upgrade your plan."

Pattern 4: Generation #

Generation creates new content from a structured specification. The spec is the entire user message — the more complete it is, the less creative latitude the model takes. Pass it as structured data (XML-tagged fields or a Python dict serialised to a string) rather than a conversational request. This makes generation reproducible: same spec in → same quality output every time.

pythongeneration.py — product descriptions from a spec dict
import anthropic
client = anthropic.Anthropic()

GENERATION_SYSTEM = """You write e-commerce product descriptions.
Style: benefit-first, conversational, no superlatives ("best", "amazing").
Format: one short paragraph (50–70 words), then three bullet points.
Audience: online shoppers comparing similar products."""

def generate_description(spec: dict) -> str:
    # Serialise spec as XML-tagged fields — unambiguous for the model
    spec_str = "\n".join(f"<{k}>{v}</{k}>" for k, v in spec.items())
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        system=GENERATION_SYSTEM,
        messages=[{"role": "user", "content": f"Generate a product description.\n{spec_str}"}],
    )
    return resp.content[0].text

products = [
    {"name": "ArcPad Pro Keyboard", "category": "mechanical keyboard",
     "key_features": "hot-swap switches, per-key RGB, aluminium frame",
     "price_point": "premium ($180)", "target_user": "developers and gamers"},
    {"name": "CalmBreeze Air Purifier", "category": "air purifier",
     "key_features": "HEPA H13, 400 sq ft coverage, sleep mode 22dB",
     "price_point": "mid-range ($129)", "target_user": "allergy sufferers, home office"},
]

for product in products:
    print(f"## {product['name']}\n")
    print(generate_description(product))
    print()

Pattern 5: Decomposition #

Decomposition breaks a task too complex for a single prompt into a sequential pipeline of simpler calls. Each call does one thing well. The output of each step becomes the input of the next. This is the foundation of all agentic systems — every multi-step agent is just a decomposition pipeline where the steps are determined dynamically.

Sequential decomposition pipeline
Step 1
Extract
raw text → facts
→
Step 2
Classify
facts → category
→
Step 3
Generate
category → draft
→
Step 4
Refine
draft → final
Each step is a focused prompt. Failures are isolated — if step 2 returns low confidence, you can re-run or route to human without discarding steps 1, 3, and 4.
pythondecomposition.py — automated support reply pipeline
import anthropic, json
client = anthropic.Anthropic()

ticket = """Subject: Can't log in, urgent
Hi, I reset my password twice but still get an 'invalid credentials' error.
I need access for a client demo in 2 hours. My email is sarah@company.com"""

# ── Step 1: Extract structured facts ─────────────────────────────────
extract_resp = client.messages.create(
    model="claude-haiku-4-5-20251001", max_tokens=128,
    system="Extract key facts from the support ticket as JSON. Fields: issue, urgency_hours, user_email, steps_already_tried.",
    messages=[{"role": "user", "content": f"<ticket>{ticket}</ticket>"},
              {"role": "assistant", "content": "{"}],
)
facts = json.loads("{" + extract_resp.content[0].text)

# ── Step 2: Classify priority ─────────────────────────────────────────
classify_resp = client.messages.create(
    model="claude-haiku-4-5-20251001", max_tokens=32,
    system="Classify support priority: low / medium / high / critical. One word.",
    messages=[{"role": "user", "content": json.dumps(facts)}],
)
priority = classify_resp.content[0].text.strip().lower()

# ── Step 3: Generate tailored reply ──────────────────────────────────
reply_resp = client.messages.create(
    model="claude-haiku-4-5-20251001", max_tokens=256,
    system="""You write support replies. Be empathetic, specific, and action-oriented.
Address the user by first name. Reference the specific issue and what they've already tried.
For high/critical priority: acknowledge urgency in the first sentence.""",
    messages=[{"role": "user", "content":
        f"Priority: {priority}\nFacts: {json.dumps(facts)}\nOriginal ticket: {ticket}"}],
)

print(f"Priority: {priority}")
print(f"Facts: {json.dumps(facts, indent=2)}")
print(f"\nReply:\n{reply_resp.content[0].text}")
💡
Use Haiku for every step in a pipeline — it's fast enough When decomposing, each individual step is simple enough for Haiku. The quality comes from the pipeline design, not from using a larger model for each call. Only upgrade to Sonnet for a specific step if that step consistently underperforms — not as a default.

Self-check #

You're building a pipeline that reads customer emails and creates CRM records. Which primitive is the first step and why?
Extraction — the first step reads the unstructured email text and pulls structured fields into a schema: sender name, company, product mentioned, intent, any dates or numbers. This gives you a Python dict that every downstream step (classify intent, generate reply, populate CRM) can operate on reliably. Trying to do extraction + classification + CRM population in one prompt produces inconsistent results and is impossible to debug when one step fails.
A classification pipeline is sending tickets to the wrong team about 8% of the time. You add a confidence score. What should happen to low-confidence results?
Low-confidence results should route to a human review queue — not to the model's best-guess team. Set a threshold (e.g. 0.80 confidence). Results above the threshold are auto-routed; results below trigger a human task. This converts a system that is wrong 8% of the time silently into one that is right 100% of the time on auto-routed tickets, with a known subset flagged for review. Track what percentage hits human review — if it grows, the classification prompt needs improvement or the label set needs refinement.
You run a transformation prompt to simplify a legal clause for consumers, and the model keeps changing specific dollar amounts and deadline dates. What is the fix?
Add explicit preservation constraints to the system prompt. A dedicated section — "Preservation constraints: these must not change — all specific values (dollar amounts, dates, percentages, defined terms in quotes)" — directly tells the model what is off-limits for editing. Without this, the model treats all text as fair game for rewording, and numbers get paraphrased ("approximately $500" instead of "$487.50") or omitted. Pair with XML tags wrapping the source clause so the model has a clear content boundary.
Why is a three-step decomposition pipeline more reliable than one mega-prompt that tries to extract, classify, and generate a reply in a single call?
Four reasons: (1) Isolation — if extraction fails (confidence low, field missing), you catch it before it poisons the classification and generation steps; you can re-run or route to human without discarding everything. (2) Focused prompts — each step has one job and a system prompt tuned for that job; a mega-prompt has competing instructions that dilute each other. (3) Debuggability — you can inspect the output of each step; with a mega-prompt you don't know which sub-task failed. (4) Granular model selection — you can upgrade just the step that underperforms to a larger model, rather than paying for a larger model for the whole task.
