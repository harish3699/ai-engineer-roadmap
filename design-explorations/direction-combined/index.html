Why Python — and why there's no shortcut #

Every framework you'll use in this roadmap — LangChain, LangGraph, FastAPI, Pydantic, the Anthropic SDK, the OpenAI SDK — is written in Python. That's not a coincidence. Python's syntax reads like pseudocode, which let AI researchers express ideas as executable code without becoming full-time software engineers first. That culture compounded for two decades, and now the entire AI toolchain lives here.

The good news: you don't need to master all of Python. You need fluency in the six features that AI frameworks use constantly. Skip any one of them and you'll spend your time debugging framework internals instead of building agents.

Type hints — the feature AI frameworks depend on #

Python is dynamically typed — you write x = 5 and Python figures out the type at runtime. Type hints add optional annotations. They don't change how your code runs, but they unlock three things that matter enormously in AI engineering:

  • Your editor catches bugs before you run anything
  • Pydantic uses type hints to validate data at runtime
  • LangChain reads type hints to auto-generate the JSON Schema it sends to the LLM
⚠️
Production impact Skip type hints on a @tool function and LangChain cannot generate the tool schema. The LLM receives no description of what parameters the tool takes. Tool calls silently fail. This is one of the most common bugs in beginner agent code.
pythontype_hints.py 
# ─ Without type hints: works, but your editor can't help you ─
def greet(name):
    return f"Hello, {name}"

# ─ With type hints: same code, now editor + Pydantic + LangChain understand it ─
def greet(name: str) -> str:
    return f"Hello, {name}"

Mutable default arguments — the most common Python bug

✗ Wrong — produces a silent shared-state bug
def add_item(item, lst=[]):
    lst.append(item)
    return lst

add_item("a")  # ['a']
add_item("b")  # ['a', 'b'] ← bug!

# The [] is created ONCE at definition.
# Every call shares the same list.
✓ Correct — fresh list on every call
def add_item(item, lst=None):
    if lst is None:
        lst = []
    lst.append(item)
    return lst

add_item("a")  # ['a']  ✓
add_item("b")  # ['b']  ✓

# None is a sentinel. Fresh list inside.

Decorators — because agent frameworks are full of them #

A decorator is a function that wraps another function to add behaviour without modifying the original. The syntax looks magic until you build one from scratch. Once you do, every @tool, @app.get, and @dataclass you encounter becomes immediately readable.

🎯
Mental model Think of a decorator like gift wrapping. The gift (your function) doesn't change. The wrapping (the decorator) adds context around it — a label, extra behavior when unwrapped, rules about how it gets used. The LangChain @tool decorator is just gift wrapping that also registers your function in the agent's tool registry.
pythondecorators.py — build one from scratch, then use the real thing 
import functools

# Step 1: build a decorator from scratch to see what's happening
def log_calls(func):
    """Wrapper that prints every call and result."""
    @functools.wraps(func)   # IMPORTANT: preserves func.__name__ + __doc__
    def wrapper(*args, **kwargs):
        print(f"→ calling {func.__name__}({args=}, {kwargs=})")
        result = func(*args, **kwargs)
        print(f"← returned: {result}")
        return result
    return wrapper

@log_calls
def add(a: int, b: int) -> int:
    return a + b

add(2, 3)
# → calling add(args=(2, 3), kwargs={})
# ← returned: 5

🚫
Common mistake Forgetting @functools.wraps(func) inside your decorator's wrapper function. Without it, func.__name__ and func.__doc__ become those of wrapper, not your original function. LangChain uses the function name as the tool name it sends to the LLM — you'd end up with a tool called "wrapper" instead of "search_docs".

Self-check #

If you can answer all five without looking back, you're ready for 1.2.

What does @functools.wraps(func) do and why does it matter for LangChain?
It copies __name__, __doc__, and other metadata from the wrapped function onto the wrapper. Without it, your @tool-decorated function will have the name "wrapper" instead of your function's actual name, and an empty docstring — breaking the tool schema LangChain sends to the LLM.
Why does def f(items=[]) produce unexpected behavior on second call?
The [] default is created once when the function is defined, not freshly on each call. All calls that don't pass items share the same list object. Fix: use items=None and create a fresh list inside the function body.
What's the difference between a list comprehension and a generator expression?
A list comprehension [x for x in ...] evaluates everything immediately and stores all results in memory. A generator expression (x for x in ...) is lazy — it yields one item at a time, using O(1) memory regardless of size. For processing 100k documents in a RAG pipeline, always use a generator.
Why do AI frameworks require type hints on functions they consume?
They use Python's inspect module and type annotations to introspect function signatures at runtime. LangChain builds the JSON Schema for tool calls from type hints. Pydantic validates incoming data against type-annotated model fields. No type hints = no schema = silent failures.
In Python 3.10+, how do you write Optional[str] without importing from typing?
Use the union pipe operator: str | None. Python 3.10 made the X | Y union syntax available natively for type hints, so Optional[str] and str | None are equivalent — the latter is cleaner.
