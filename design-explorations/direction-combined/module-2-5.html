Comparing the Major Models

GPT, Claude, Gemini, Llama, Mistral, DeepSeek, Qwen — seven model families, each with different tradeoffs across cost, quality, speed, context length, and licensing. This module gives you the mental map to pick the right model for any task, understand why engineers switch providers, and stop treating "just use GPT-4" as a strategy. It's also the final module of Phase 2.

~25 min read 1 video · 1 recording script 1 code example Last updated 2026-05-07
Key Takeaways
  • Model choice matters most for: complex reasoning, long-context tasks, tool use, cost-sensitive high-volume pipelines, and data-residency / licensing requirements
  • Model choice rarely matters for: simple extraction, classification, summarisation of short documents — a well-prompted mid-tier model matches a frontier model at a fraction of the cost
  • Open-weight models (Llama, Mistral, DeepSeek, Qwen) can be self-hosted — zero per-token cost at scale, full data control, but you own the infra and the safety layer
  • Provider APIs change constantly — build with an abstraction layer (LiteLLM, LangChain's ChatModel) so swapping providers is a one-line change
  • The model landscape shifts every 2–3 months. Build eval infrastructure (Module 2.4) so you can re-validate your model choice when new releases drop

The major model families #

Think in families, not individual models. Each family has a characteristic tradeoff profile that persists across generations — flagship models for complex tasks, mid-tier for everyday work, small/fast for high-volume pipelines.

When model choice actually matters #

💡
The 80/20 rule of model selection For 80% of tasks — extraction, classification, summarisation, Q&A over retrieved docs — a well-prompted Haiku or GPT-4o-mini will match a frontier model. The remaining 20% — complex reasoning, agentic tool use, ambiguous multi-constraint problems, long-document analysis — is where the frontier model earns its premium. Start cheap, upgrade only when your micro-eval shows the quality gap justifies the cost.
Model choice MATTERS
  • Multi-step reasoning & planning
  • Agentic tool use (BFCL score matters)
  • Complex code generation / debugging
  • Long-document analysis (>50k tokens)
  • Regulated domains (medical, legal, finance)
  • Data residency / compliance requirements
  • High-volume pipelines where cost/token adds up
Model choice rarely matters
  • Entity / keyword extraction
  • Binary or multi-class classification
  • Summarisation of short documents
  • Translation between major languages
  • Simple Q&A over retrieved chunks (RAG)
  • Format conversion (JSON → Markdown, etc.)
  • Sentiment analysis, tone detection

Provider-agnostic code with LiteLLM #

The model landscape changes every 2–3 months. Hardcoding anthropic.Anthropic() everywhere means a provider switch requires touching every file. LiteLLM provides a unified OpenAI-compatible interface across all major providers — swapping models is a one-line change.

pythonlitellm_example.py — swap providers without changing call sites
from litellm import completion
import os

# ── Same call signature for every provider ────────────────────────────
messages = [{"role": "user", "content": "Summarise this document in 3 bullet points."}]

# Claude
resp = completion(model="anthropic/claude-haiku-4-5-20251001", messages=messages)

# OpenAI — identical call, just change the model string
resp = completion(model="gpt-4o-mini", messages=messages)

# Gemini
resp = completion(model="gemini/gemini-2.0-flash", messages=messages)

# Self-hosted Llama via Ollama
resp = completion(model="ollama/llama3.2", messages=messages,
                  api_base="http://localhost:11434")

print(resp.choices[0].message.content)

# ── Use env vars for API keys — never hardcode ────────────────────────
# ANTHROPIC_API_KEY, OPENAI_API_KEY, GEMINI_API_KEY
# LiteLLM picks them up automatically

# ── Cost tracking built in ────────────────────────────────────────────
print(f"Cost: ${resp._hidden_params['response_cost']:.6f}")

Self-check #

Your application processes 200,000 customer support tickets per day with simple intent classification. Which model tier and family would you choose, and why?
A fast, cheap tier model — Claude Haiku, GPT-4o-mini, or Gemini Flash. Classification is a pattern-matching task where mid-tier models match frontier models. At 200k requests/day, the cost difference between a $0.15/M input tier and a $5/M input tier is roughly 33× — potentially thousands of dollars per day for zero quality gain. Always validate with a micro-eval (Module 2.4) first, but the starting hypothesis should always be the cheapest tier that passes your quality bar.
A client requires that all data stays within EU jurisdiction. Which options do you have?
Several options: (1) Claude via AWS Bedrock EU regions — Anthropic's data processing agreement covers EU data residency; (2) Azure OpenAI with EU region deployment — Microsoft's GDPR compliance covers this; (3) Mistral AI — French company, EU-hosted, GDPR-compliant; (4) Self-hosted open-weight models (Llama, Mistral, Qwen) on EU cloud infrastructure — full data control, no third-party processing. The self-hosted option gives maximum control but requires infra expertise. Always check your provider's current Data Processing Agreement before committing — these change.
Why should you use LiteLLM or a similar abstraction layer rather than calling provider SDKs directly?
The model landscape changes every 2–3 months — new models drop, prices change, quality leaderboards shift. If you call anthropic.Anthropic() or openai.OpenAI() directly throughout your codebase, switching providers requires changing every call site. With LiteLLM, the provider and model are a configuration string — swapping from Claude to GPT-4o is a one-line change with no code modifications. It also provides unified cost tracking, retry logic, and fallback routing across providers.
When would you choose a self-hosted open-weight model over a hosted API?
Four main scenarios: (1) Data residency / privacy — data cannot leave your infrastructure, or you're processing PII that can't go to third-party APIs; (2) High-volume cost — at millions of requests per day, the marginal cost of self-hosting (GPU infra) can be lower than per-token API pricing; (3) Latency control — you need guaranteed SLAs and can't rely on provider capacity or rate limits; (4) Fine-tuning — you need to modify model weights for a narrow domain task. The downside is that you own the infra, the scaling, the safety layer, and the updates — a non-trivial operational burden.
