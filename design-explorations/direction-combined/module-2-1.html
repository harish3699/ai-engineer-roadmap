What an LLM Actually Is

Before you write a single prompt, you need the right mental model. Most engineers treat LLMs like a smarter Google — they don't. This module fixes that. It covers what the model was trained on, why it generates rather than retrieves, what a knowledge cutoff actually implies, and why the same prompt can produce different outputs every time.

~20 min read 1 video · 1 recording script 1 code example Last updated 2026-05-07
Key Takeaways
  • An LLM is a statistical model of language trained on a fixed text snapshot — it predicts probable next tokens, it doesn't retrieve facts from a live database
  • Every output is sampled from a probability distribution — the same input can produce different outputs, and that's by design
  • Knowledge cutoff means the model has no concept of events after its training data ended — it won't say "I don't know," it will confabulate confidently
  • Hallucinations are not bugs — they are the expected output of a system that generates plausible continuations rather than retrieves verified facts
  • The correct mental model: a brain in a windowless room with a huge library of books, writing answers from memory, with no internet connection and no way to check if things changed

Trained on a fixed snapshot #

An LLM is trained by showing it billions of sentences and teaching it to predict the next word in each one. It reads Wikipedia, books, code repositories, news articles, forums — a snapshot of written human language — and builds internal statistics about what words follow other words, in what contexts, with what probability.

That training process ended on a specific date. After that date, the model is frozen. It has never seen your internal documents. It has never read this week's news. It cannot look anything up. It only knows what was in the training data.

ℹ
The windowless room analogy Imagine a person locked in a room with a complete library of books — every book published before a fixed date. They can read and remember everything in the library. They can write answers to any question by drawing on that memory. But they have no windows, no internet, no way to know what has happened since the day they were locked in, and no way to check if anything they remember has since turned out to be wrong.

This analogy explains almost every frustrating LLM behaviour:

  • Why it doesn't know about recent events — the library doesn't contain those books yet
  • Why it can't access your company's internal docs — those books were never in the library
  • Why it answers confidently about things that changed — it remembers what was true when the library was built
  • Why RAG exists — you slide new documents under the door; now it can answer about those too

Probabilistic generation, not retrieval #

When you type a prompt, the model doesn't search for a matching answer in a database. It generates the response one token at a time, each token sampled from a probability distribution over the entire vocabulary. At each step, some tokens are more likely than others — but there's always a distribution, not a single certain answer.

Traditional Search / Database
Query
↓ index lookup
Matching records
↓ rank + return
Exact facts retrieved
LLM Generation
Prompt
↓ probability over vocabulary
Sample token 1, 2, 3…
↓ until stop token
Plausible continuation

This is why LLMs hallucinate. The model isn't lying — it's generating the most statistically plausible continuation of your prompt, based on patterns in training data. If no training data clearly addresses your question, the model will still produce something plausible-sounding. It has no mechanism to say "I don't have enough information to answer this confidently."

⚠
Hallucinations are architectural, not a bug to patch You cannot instruct a model to "never make things up." You can reduce hallucinations with grounding techniques (RAG, tool use, structured output, citations) — but the underlying generation mechanism always samples from probabilities. Build systems that verify and cite; don't rely on model self-reporting of uncertainty.

Temperature — controlling randomness #

The temperature parameter scales the probability distribution before sampling. At temperature=0, the model always picks the highest-probability token — deterministic, but sometimes flat or repetitive. At temperature=1, sampling is unscaled. Higher values flatten the distribution, making lower-probability tokens more likely — more creative, but also more likely to go off-track.

pythontemperature_demo.py — same prompt, different temperatures
import anthropic

client = anthropic.Anthropic()

prompt = "Complete this sentence in one sentence: The best way to learn programming is"

for temp in [0.0, 0.7, 1.5]:
    msg = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=60,
        temperature=temp,
        messages=[{"role": "user", "content": prompt}],
    )
    print(f"temp={temp}: {msg.content[0].text}")

# temp=0.0: …to write code every day and build real projects. (same every run)
# temp=0.7: …through hands-on projects solving problems you actually care about.
# temp=1.5: …to embrace frustration as a compass pointing toward growth. (creative, risky)

Rule of thumb: use temperature=0 or 0.1 for extraction, classification, and structured output — tasks where consistency matters. Use 0.5–0.9 for chat and general Q&A. Only go above 1.0 for intentionally creative tasks where quality variance is acceptable.

Knowledge cutoffs in practice #

Every major model has a training cutoff — the date after which no new data was added. GPT-4o's cutoff is April 2024. Claude's is early 2025. But the cutoff isn't a clean line — data from the months immediately before the cutoff is underrepresented (the internet hadn't fully indexed and discussed those events yet). Treat the last 2–3 months before the stated cutoff as unreliable.

💡
What to do about knowledge cutoffs For current events/data: Use web search tools or RAG with live data sources.
For your company's internal data: RAG is mandatory — the model has never seen it regardless of cutoff.
For reasoning tasks with timeless knowledge: (math, code logic, writing) — cutoffs don't matter.
For questions about recent software/APIs: Pass the relevant docs as context; the model may know v1 but not v3.
🚫
Don't ask the model if it knows about X Models often confidently claim to know about things they don't, and claim not to know about things they do. The model has no reliable self-knowledge of what is or isn't in its training data. If recency matters, assume the model doesn't know and provide the information explicitly.

Self-check #

A user asks your agent "What is the current stock price of Apple?" and it gives a confident but wrong answer. What is happening architecturally?
The model is generating a plausible-sounding stock price based on patterns in its training data — it may have seen many examples of stock prices in its training set and is producing something that looks like one. It has no mechanism to distinguish "I know this" from "this seems plausible based on patterns." The fix is to give the agent a stock price tool so it can retrieve real data, not generate it.
Why does asking the model "Do you know about [recent event]?" give unreliable answers?
The model has no reliable introspective access to what is or isn't in its training data. It will often claim to know things it doesn't (generating a plausible-sounding answer) and sometimes claim not to know things it does. The model's self-reported knowledge boundaries are themselves generated text, not a verified database lookup. If recency matters, assume the model doesn't know and provide the information in the prompt.
What is the difference between temperature=0 and temperature=1.5, and when would you use each?
temperature=0 makes the model deterministic — it always picks the highest-probability token. Use this for structured output, classification, extraction, and any task where consistency across runs matters more than variety. temperature=1.5 flattens the probability distribution so lower-probability (more unusual) tokens get sampled more often — produces more creative and varied output but also more likely to go off-track. Use only for intentionally creative tasks where variance is acceptable.
Your agent needs to answer questions about documents your company created after the model's training cutoff. What is the correct architectural solution?
Retrieval-Augmented Generation (RAG) — store the documents in a vector database, retrieve the relevant chunks at query time, and inject them into the model's context window. The model was never trained on these documents and cannot generate accurate answers about them from memory alone, regardless of how it's prompted. RAG is Phase 4 of this roadmap.
