Reading Evals & Benchmarks

Every model release comes with a benchmark table claiming state-of-the-art performance. Most AI engineers take these at face value. The ones who don't build better systems. This module covers the benchmarks that actually matter, the three ways they lie, how to read a leaderboard without being misled, and — most importantly — how to build the only eval that truly matters: one on your own task.

~25 min read 1 video · 1 recording script 1 code example Last updated 2026-05-07
Key Takeaways
  • Public benchmarks measure model performance on a fixed test set — not performance on your task. A model that tops MMLU may be worse than a cheaper model for your specific use case
  • Benchmark contamination is real — test questions appear in training data, inflating scores. Treat any benchmark result for data before 2024 with suspicion
  • The leaderboards worth bookmarking: LM Arena (human preference), Artificial Analysis (speed + cost + quality), Aider (coding), BFCL (function calling)
  • A micro-eval with 50–100 real examples from your actual task is worth more than any public benchmark score
  • LLM-as-judge is the standard evaluation technique for open-ended tasks — use a stronger model to score a weaker one's output against a rubric

The benchmarks worth knowing #

Most public benchmarks are not useful for day-to-day model selection. These seven are worth knowing — not to cite the numbers, but to understand what capability each one actually measures.

Benchmark What it measures Useful for
MMLU 57-subject knowledge MCQ (law, medicine, CS, history…) Broad knowledge — mostly saturated now
GSM8K Grade-school word maths problems requiring multi-step reasoning Basic reasoning — also largely saturated
HumanEval 164 Python coding problems — write a function that passes unit tests Code generation ability (pass@1 rate)
SWE-bench Real GitHub issues — resolve the bug in a real Python repo Agentic coding on real codebases — most realistic
GPQA PhD-level science questions that stump domain experts Expert reasoning — ceiling is still high
MMMU Multimodal understanding — college-level questions with images Vision + reasoning together
BFCL Berkeley Function Calling Leaderboard — tool use accuracy Most relevant for agent engineers — measures exactly what you'll build
💡
Which benchmarks to actually watch If you're building agents: BFCL (function calling) and SWE-bench (agentic coding) are the most predictive. If you're building RAG: none of these — build your own eval (see below). MMLU and GSM8K are mostly saturated — top models score 85–90%+ and the differences are noise.

Why benchmarks lie #

There are three mechanisms that make benchmark scores misleading. Understanding them means you'll never take a model release announcement at face value again.

🚫
1. Contamination — test data in training data If MMLU questions appeared in the training corpus, the model has "seen the answers" before. This is provably happening for many benchmarks — papers have shown that models score 10–30 points higher on contaminated subsets. Any benchmark released before 2022 should be treated with suspicion for models trained after that date.
⚠
2. Prompt sensitivity — scores change with phrasing LLM benchmark scores are sensitive to the exact prompt format used. The same model can score 78% or 86% on the same benchmark depending on whether questions use lettered options (A/B/C/D) or numbered ones (1/2/3/4), whether a chain-of-thought preamble is added, and how the system prompt is worded. Labs optimise their evaluation prompts. The number is not comparable across labs.
⚠
3. Eval gaming — optimising for the test When a benchmark becomes widely used, it becomes a training target. Models are fine-tuned to perform well on specific benchmarks without general capability improvements. This is Goodhart's Law: "When a measure becomes a target, it ceases to be a good measure." The more famous the benchmark, the less it measures true capability over time.

Reading leaderboards skeptically #

These four leaderboards provide genuine signal — each for different reasons.

LM Arena
Human preference ranking — real users rate which response they prefer in A/B blind tests. High signal for overall quality and writing. Low signal for factual accuracy (humans prefer confident wrong answers). lmarena.ai
Artificial Analysis
Speed (tokens/sec), cost per million tokens, latency, quality — plotted together. The best place to compare quality vs cost vs speed for production model selection. artificialanalysis.ai
Aider
Coding leaderboard — percentage of real GitHub issues solved by each model with the Aider coding agent. Measures agentic coding ability on real problems, not toy tasks. aider.chat/docs/leaderboards
BFCL
Function calling accuracy across simple, complex, parallel, and nested tool calls. The closest public proxy for how well a model will work as your agent's brain. gorilla.cs.berkeley.edu/leaderboard

Build your own micro-eval #

The most actionable evaluation is one you build yourself on real examples from your actual task. 50 examples is enough to get meaningful signal. 100 is enough to make confident model selection decisions. The pattern: collect real inputs, write expected outputs or a grading rubric, run the model, score with an LLM judge.

pythonmicro_eval.py — 50-example eval with LLM-as-judge scoring
import anthropic, json
from dataclasses import dataclass

client = anthropic.Anthropic()

# ── Your eval dataset — 50+ real examples from your actual task ───────
# Each example: input the model sees + expected output or rubric criteria
EVAL_DATASET = [
    {"input": "Extract the company name: 'Meeting with Sarah from Acme Corp tomorrow'",
     "expected": "Acme Corp"},
    {"input": "Extract the company name: 'Following up on our Stripe integration'",
     "expected": "Stripe"},
    # ... 48 more real examples from your data ...
]

JUDGE_PROMPT = """You are evaluating an AI extraction task.
Input: {input}
Expected: {expected}
Model output: {output}

Score 1 if the model output is correct (exact or semantically equivalent).
Score 0 if incorrect.
Respond with only the number: 0 or 1."""

def run_model(input_text: str, model: str) -> str:
    resp = client.messages.create(
        model=model, max_tokens=100, temperature=0,
        messages=[{"role": "user", "content": input_text}],
    )
    return resp.content[0].text.strip()

def judge_output(example: dict, output: str) -> int:
    prompt = JUDGE_PROMPT.format(**example, output=output)
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",   # cheap model to judge
        max_tokens=5, temperature=0,
        messages=[{"role": "user", "content": prompt}],
    )
    return int(resp.content[0].text.strip())

def evaluate(model: str) -> float:
    scores = []
    for ex in EVAL_DATASET:
        output = run_model(ex["input"], model)
        score  = judge_output(ex, output)
        scores.append(score)
    accuracy = sum(scores) / len(scores)
    print(f"{model}: {accuracy:.1%} ({sum(scores)}/{len(scores)})")
    return accuracy

# Compare models on YOUR task — not on MMLU
for model in ["claude-haiku-4-5-20251001", "claude-sonnet-4-6"]:
    evaluate(model)

# claude-haiku: 94.0% (47/50)  — probably good enough, 10× cheaper
# claude-sonnet: 96.0% (48/50)  — marginally better, not worth the cost for this task
ℹ
LLM-as-judge guidelines Use a stronger model to judge a weaker one — never the same model judging itself. Use temperature=0 for the judge. Ask for a score on a defined scale (0/1 for binary, 1–5 for rubric), not free text. Always include the expected output in the judge prompt as a reference. For critical evals, add 10–20% human spot-check samples to verify the judge's calibration.

Self-check #

What is benchmark contamination and why does it make scores unreliable?
Benchmark contamination occurs when the test questions from a benchmark appear in the model's training data, either directly or in rephrased form. The model has effectively "seen the answers" before evaluation, inflating its score beyond its true generalisation ability. It's hard to detect because training datasets are enormous and not fully audited. As a rule of thumb, benchmarks released before 2022 should be treated with suspicion for models trained after that date, and score improvements on old benchmarks by newer models often reflect contamination rather than genuine progress.
You're selecting a model for a high-volume document classification task. Which leaderboard gives you the most useful signal?
Artificial Analysis — it plots quality vs cost vs speed together, allowing you to find the best model within your latency and cost constraints. LM Arena shows human preference but not factual/task accuracy. BFCL is for function calling, not classification. MMLU may be relevant for knowledge-heavy classification but is largely saturated. For a production task, the most useful signal is always a micro-eval on your own data — 50 real examples from your actual classification task.
Why should you never use the same model to judge its own outputs in an LLM-as-judge evaluation?
A model judging its own outputs has a systematic bias toward outputs that match its own generation style and preferred answers — it will rate its own mistakes more favourably than a different model would. The judge needs to be independent of the model under evaluation. Use a stronger model (e.g., Claude Sonnet judging Haiku outputs) or a model from a different provider. This is analogous to why peer review in science uses independent reviewers, not the authors themselves.
What is Goodhart's Law and how does it apply to AI benchmarks?
Goodhart's Law states: "When a measure becomes a target, it ceases to be a good measure." Applied to AI benchmarks: once a benchmark becomes widely used for model comparison, labs begin optimising their models specifically to score well on it — through targeted fine-tuning, prompt engineering during evaluation, or including benchmark-adjacent examples in training. The result is that benchmark scores rise without corresponding real-world capability improvements. This is why the most trusted benchmarks are the ones that are hardest to game (SWE-bench with real codebases) or continuously updated with new, unseen examples.
