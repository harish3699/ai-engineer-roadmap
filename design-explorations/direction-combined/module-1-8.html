Async Programming

LLM calls are slow — 1 to 10 seconds each. Without async, your agent processes one thing at a time. With asyncio, you can fan out 10 parallel LLM calls and wait for all of them in the same wall-clock time as one. This module covers the event loop mental model, async/await syntax, gather for parallelism, wait_for for timeouts, and create_task for fire-and-forget logging.

~30 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-06
Key Takeaways
  • The event loop is single-threaded — await yields control back to it, allowing other coroutines to run while you wait on I/O
  • asyncio.gather() runs coroutines concurrently — use it to fan out parallel LLM calls and collect all results at once
  • asyncio.wait_for(coro, timeout=10) raises TimeoutError if the coroutine doesn't finish in time — protects against hanging LLM calls
  • asyncio.create_task() schedules a coroutine to run without blocking the caller — the right pattern for fire-and-forget logging
  • Never call a blocking function (requests, psycopg2, time.sleep) directly in an async context — use run_in_executor or switch to async libraries

Event loop & coroutines #

Python's event loop is a scheduler that runs one coroutine at a time. When a coroutine hits an await on something slow (an HTTP call, a DB query, a file read), it pauses and yields control back to the loop. The loop then runs another coroutine until it awaits something. No threads — one CPU thread, many interleaved tasks.

⚠
Blocking the event loop kills performance Calling requests.get(), time.sleep(), or psycopg2 inside an async def blocks the entire loop — all other coroutines freeze until the call returns. Use httpx.AsyncClient, asyncio.sleep(), and asyncpg instead. Use loop.run_in_executor(None, blocking_fn) as a last resort.
pythonasyncio_basics.py — coroutines, tasks, and the loop
import asyncio
import time

# ── A coroutine is declared with async def ───────────────────────────
async def fetch_reply(query: str) -> str:
    print(f"starting: {query}")
    await asyncio.sleep(1)   # simulates a 1-second LLM call
    print(f"done: {query}")
    return f"reply to {query}"

# ── Sequential — 3 seconds total ─────────────────────────────────────
async def sequential():
    t = time.perf_counter()
    r1 = await fetch_reply("q1")
    r2 = await fetch_reply("q2")
    r3 = await fetch_reply("q3")
    print(f"sequential: {time.perf_counter() - t:.1f}s")   # ~3.0s

# ── Concurrent — 1 second total with asyncio.gather ──────────────────
async def concurrent():
    t = time.perf_counter()
    r1, r2, r3 = await asyncio.gather(
        fetch_reply("q1"),
        fetch_reply("q2"),
        fetch_reply("q3"),
    )
    print(f"concurrent: {time.perf_counter() - t:.1f}s")   # ~1.0s

asyncio.run(concurrent())

asyncio.gather — parallel LLM calls #

asyncio.gather(*coroutines) schedules all coroutines concurrently and returns their results in the same order once all complete. This is the primary tool for fan-out patterns: evaluate multiple prompts in parallel, score chunks in batch, or run multiple agents simultaneously.

pythonparallel_llm.py — evaluate 5 prompts concurrently with httpx
import asyncio
import httpx
import os

async def call_llm(client: httpx.AsyncClient, prompt: str) -> str:
    resp = await client.post(
        "https://api.anthropic.com/v1/messages",
        headers={
            "x-api-key":         os.environ["ANTHROPIC_API_KEY"],
            "anthropic-version": "2023-06-01",
        },
        json={
            "model":      "claude-haiku-4-5-20251001",
            "max_tokens": 256,
            "messages":   [{"role": "user", "content": prompt}],
        },
        timeout=30.0,
    )
    resp.raise_for_status()
    return resp.json()["content"][0]["text"]

async def evaluate_all(prompts: list[str]) -> list[str]:
    # One shared client — connection pool reused across all concurrent calls
    async with httpx.AsyncClient() as client:
        results = await asyncio.gather(
            *[call_llm(client, p) for p in prompts],
            return_exceptions=True,   # don't cancel all if one fails
        )
    return [
        r if not isinstance(r, Exception) else f"ERROR: {r}"
        for r in results
    ]

# 5 LLM calls, ~same wall-clock time as 1
replies = asyncio.run(evaluate_all(["Summarise X", "Score Y", "Classify Z", "Translate A", "Extract B"]))

wait_for timeouts & create_task #

asyncio.wait_for(coro, timeout=N) cancels the coroutine and raises asyncio.TimeoutError if it doesn't complete within N seconds. Always wrap LLM calls with a timeout — a hanging provider will silently block your agent forever without it. asyncio.create_task() schedules a coroutine to run concurrently without awaiting the result — the correct pattern for non-critical side effects like logging.

pythontimeout_task.py — wait_for + create_task patterns
import asyncio

# ── wait_for: raise TimeoutError if LLM takes > 10s ──────────────────
async def safe_llm_call(prompt: str) -> str:
    try:
        reply = await asyncio.wait_for(
            call_llm(prompt),   # the slow coroutine
            timeout=10.0,       # seconds before cancellation
        )
        return reply
    except asyncio.TimeoutError:
        log_warning("LLM call timed out after 10s")
        return "I'm sorry, the response took too long. Please try again."

# ── create_task: fire-and-forget logging ──────────────────────────────
async def write_log(session_id: str, prompt: str, reply: str) -> None:
    await asyncio.sleep(0)    # yield to allow other tasks to proceed
    await db.insert_log(session_id, prompt, reply)

async def handle_chat(req) -> str:
    reply = await safe_llm_call(req.message)

    # Schedule log write — doesn't block the response to the user
    asyncio.create_task(write_log(req.session_id, req.message, reply))

    return reply   # response sent immediately; log writes in background

# ── When to use each ─────────────────────────────────────────────────
# await           → you need the result before continuing
# asyncio.gather  → you need ALL results, can run them in parallel
# asyncio.wait_for → you need the result but must cap max wait time
# asyncio.create_task → fire-and-forget, result not needed by caller

Self-check #

Why doesn't calling time.sleep(5) inside an async def yield to the event loop?
time.sleep() is a synchronous blocking call — it holds the CPU and prevents the Python interpreter from doing anything else. The event loop only gets control back when a coroutine hits an await expression. Use await asyncio.sleep(5) instead, which suspends the current coroutine and lets the loop run other tasks during the wait.
What does return_exceptions=True do in asyncio.gather?
By default, if any coroutine in gather raises an exception, the exception propagates immediately and the other coroutines are cancelled. With return_exceptions=True, exceptions are returned as result values (alongside successful results) rather than being raised. This lets you handle per-call failures without cancelling the entire batch — critical for parallel LLM calls where one provider error shouldn't abort the rest.
What happens to the coroutine passed to asyncio.wait_for when the timeout expires?
The coroutine is cancelled — its next await point raises asyncio.CancelledError internally, which unwinds the coroutine's stack. wait_for then raises asyncio.TimeoutError to the caller. Any resources the coroutine was holding (connections, locks) should be cleaned up in a finally block or context manager inside the coroutine to avoid leaks.
What is the difference between await coro() and asyncio.create_task(coro())?
await coro() suspends the current coroutine until coro() completes — you get the result before moving on. asyncio.create_task(coro()) schedules coro() to run concurrently and returns a Task object immediately — the caller continues without waiting for the result. Use create_task for fire-and-forget side effects (logging, analytics, cache warming) where the caller doesn't need the result and shouldn't be slowed by it.
