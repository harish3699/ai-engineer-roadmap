Calling LLMs via API

The Anthropic and OpenAI SDKs use the same mental model — a messages array with roles, a model name, and a handful of sampling parameters. Once you understand the message contract, streaming and structured output are just two extra flags. This module covers every parameter that matters in production, how to stream tokens to the user, and the three ways to get structured JSON out of a model.

Key Takeaways
  • Every LLM API call is the same contract: a messages array with roles (system, user, assistant), a model, and a token limit — everything else is optional
  • Streaming is a one-parameter change — set stream=True — and changes user experience from "wait 10 seconds" to "words appear instantly"
  • Structured JSON from an LLM has three approaches: JSON mode, forced tool-call schemas, and XML tags — each with different tradeoffs
  • The usage object on every response tells you exactly how many tokens were consumed — track this from day one or you will be surprised by your billing
  • The Anthropic and OpenAI SDKs are 90% conceptually identical — learn one and you know the other

The message contract #

Every API call to any major LLM provider is built on the same abstraction: an ordered list of messages, each with a role and content. The model reads the full list and predicts what comes next. That's it.

The three message roles
system
Instructions for the model — persona, constraints, output format, domain context. Processed before any user message. You write this. The UI would inject its own here; you now control it entirely.
user
Input from the human side of the conversation. In production this is often your code constructing a message — not a real user typing. It can contain retrieved documents, structured data, tool results, or anything you need the model to process.
assistant
Prior model responses. When building multi-turn conversations, you append each model reply as an assistant message before sending the next user message. You can also prefill this role with a partial response to steer the model's output format — covered in module 3.3.
⚠
Messages must alternate correctly After a system message (optional), messages must alternate between user and assistant. Sending two consecutive user messages without an assistant turn in between will throw a validation error. If you're building a stateless endpoint that receives a single prompt, you just send one user message and read the response.

Anthropic SDK in depth #

The Anthropic SDK exposes one primary method: client.messages.create(). The parameters that matter in production are fewer than you'd expect.

pythonsdk_basics.py — every parameter that matters
import anthropic
from dotenv import load_dotenv
load_dotenv()

client = anthropic.Anthropic()  # reads ANTHROPIC_API_KEY from env

resp = client.messages.create(
    # ── Required ──────────────────────────────────────────────────────
    model="claude-sonnet-4-6",   # model ID — pin this in production
    max_tokens=1024,              # hard ceiling on output tokens
    messages=[
        {"role": "user", "content": "Summarise RAG in 3 bullet points."}
    ],

    # ── Strongly recommended ──────────────────────────────────────────
    system="You are a concise technical educator. Use plain text, no markdown.",

    # ── Sampling — tune for your use case ─────────────────────────────
    temperature=0.3,   # 0 = deterministic, 1 = creative. Default 1.
                       # Use 0–0.3 for extraction/classification/code.
                       # Use 0.7–1 for creative/conversational tasks.
)

# ── Reading the response ──────────────────────────────────────────────
text = resp.content[0].text          # always index 0 for a text response
print(text)

# ── Usage — track this from day one ──────────────────────────────────
print(f"Input tokens : {resp.usage.input_tokens}")
print(f"Output tokens: {resp.usage.output_tokens}")
print(f"Stop reason  : {resp.stop_reason}")   # 'end_turn' | 'max_tokens' | 'stop_sequence' | 'tool_use'

# ── Cost estimate (Sonnet 4.6 pricing) ───────────────────────────────
cost = (resp.usage.input_tokens / 1_000_000) * 3.0 \
     + (resp.usage.output_tokens / 1_000_000) * 15.0
print(f"Cost estimate: ${cost:.6f}")
ℹ
stop_reason is your canary If stop_reason == "max_tokens", the model was cut off mid-response. Your output is incomplete. Either increase max_tokens or shorten your input. Always check this in production — truncated JSON or code is a silent failure that causes hard-to-debug downstream errors.

OpenAI SDK — the parallel pattern #

If you know the Anthropic SDK, the OpenAI SDK is a cosmetic difference. The mental model — messages array, roles, usage object — is identical. The differences are in field names and where the text lives in the response object.

SDK differences at a glance
Anthropic
client.messages.create()
param: system="…"
param: max_tokens=1024
text: resp.content[0].text
usage: resp.usage.input_tokens
OpenAI
client.chat.completions.create()
msg: {"role":"system"…}
param: max_completion_tokens=1024
text: resp.choices[0].message.content
usage: resp.usage.prompt_tokens
The main structural difference: Anthropic takes system as a top-level param; OpenAI takes it as the first message in the array with role "system".

Streaming responses #

By default, the API waits until the full response is generated before returning anything. For a 500-token response at 50 tokens/second, that's 10 seconds of silence. Streaming returns each token as it's generated — users see words appearing in real time.

pythonstreaming.py — streaming with the Anthropic SDK
import anthropic
client = anthropic.Anthropic()

# ── Pattern 1: stream context manager (recommended) ───────────────────
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=512,
    system="You are a helpful assistant.",
    messages=[{"role": "user", "content": "Write a short story about a robot."}],
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)  # flush=True shows tokens immediately
    print()                                # newline after stream ends

    # Usage is available after the stream completes
    final = stream.get_final_message()
    print(f"\nTokens: {final.usage.input_tokens} in / {final.usage.output_tokens} out")

# ── Pattern 2: FastAPI endpoint with Server-Sent Events ───────────────
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

def generate_tokens(prompt: str):
    with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=512,
        messages=[{"role": "user", "content": prompt}],
    ) as stream:
        for text in stream.text_stream:
            yield f"data: {text}\n\n"   # SSE format — frontend reads with EventSource

@app.get("/stream")
async def stream_endpoint(prompt: str):
    return StreamingResponse(generate_tokens(prompt), media_type="text/event-stream")

Structured output — three approaches #

Most production systems need JSON back from a model — an extracted entity, a classification with confidence, a structured action for an agent to execute. There are three ways to get it. The right choice depends on whether you need strict schema enforcement or just reliable parsing.

Three ways to get structured JSON
1. Prompt + JSON mode
low reliability
Ask the model to "respond with valid JSON only" in the system prompt. Simple but unreliable — the model occasionally adds prose or malforms the JSON. Never use alone in production.
2. Tool-call schema
recommended
Define a tool with a JSON Schema input_schema. Set tool_choice={"type":"tool","name":"…"} to force the model to always call it. The SDK parses and validates automatically — you get a Python dict, not a string.
3. XML tags + parse
lightweight fallback
Ask the model to wrap its answer in XML tags (e.g. <answer>…</answer>), then extract with a regex or BeautifulSoup. Reliable for simple single-field extraction. Doesn't scale to complex nested schemas.
pythonstructured_output.py — tool-call schema (the right way)
import anthropic, json
client = anthropic.Anthropic()

# ── Define the schema for what you want back ─────────────────────────
extract_tool = {
    "name": "extract_entities",
    "description": "Extract named entities from the provided text.",
    "input_schema": {
        "type": "object",
        "properties": {
            "people":        {"type": "array",  "items": {"type": "string"}},
            "organisations": {"type": "array",  "items": {"type": "string"}},
            "sentiment":     {"type": "string", "enum": ["positive", "neutral", "negative"]},
        },
        "required": ["people", "organisations", "sentiment"],
    },
}

text = "Anthropic released Claude 4 last week. CEO Dario Amodei said the model " \
       "significantly outperforms GPT-5 on reasoning benchmarks."

resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=256,
    tools=[extract_tool],
    tool_choice={"type": "tool", "name": "extract_entities"},  # force the call
    messages=[{"role": "user", "content": text}],
)

# ── SDK gives you a validated dict, not a raw string ─────────────────
tool_block = next(b for b in resp.content if b.type == "tool_use")
result = tool_block.input         # already a Python dict
print(result["people"])          # ['Dario Amodei']
print(result["organisations"])  # ['Anthropic']
print(result["sentiment"])     # 'positive'

# ── XML tags pattern — for simple single-field extraction ─────────────
import re

resp2 = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=64,
    system="Answer inside <answer> tags only. No other text.",
    messages=[{"role": "user", "content": "What is 12 * 14?"}],
)
raw = resp2.content[0].text
match = re.search(r"<answer>(.*?)</answer>", raw, re.DOTALL)
answer = match.group(1).strip() if match else raw
print(answer)  # '168'
💡
Use Haiku for extraction, Sonnet for generation Entity extraction, classification, and sentiment are low-complexity tasks — Claude Haiku handles them at ~10× lower cost than Sonnet. Save Sonnet (and Opus) for tasks that require reasoning, writing quality, or nuanced judgement. The structured output pattern works identically across all model tiers.

Error handling in production #

The SDK raises typed exceptions — catching the right ones lets you recover gracefully instead of crashing.

pythonerror_patterns.py — typed exceptions + retry logic
import anthropic, time
client = anthropic.Anthropic()

def call_with_retry(messages: list, max_retries: int = 3) -> str:
    for attempt in range(max_retries):
        try:
            resp = client.messages.create(
                model="claude-haiku-4-5-20251001",
                max_tokens=512,
                messages=messages,
            )
            return resp.content[0].text

        except anthropic.RateLimitError:
            # 429 — back off and retry
            wait = 2 ** attempt
            print(f"Rate limited. Waiting {wait}s…")
            time.sleep(wait)

        except anthropic.APIStatusError as e:
            if e.status_code == 529:    # API overloaded — retry
                time.sleep(5)
            elif e.status_code >= 500:  # server error — retry
                time.sleep(2)
            else:
                raise                       # 4xx client error — don't retry

        except anthropic.AuthenticationError:
            raise  # bad API key — fail immediately, don't retry

    raise RuntimeError(f"Failed after {max_retries} retries")

Self-check #

What are the three message roles and what is the purpose of each?
system — instructions for the model that apply to the entire conversation: persona, constraints, output format, domain context. Processed before user messages. You write this; in the raw API nothing overrides it.

user — input from the human or (more commonly in production) from your code. Can contain retrieved documents, structured data, tool results, or anything the model needs to process. Not limited to literal user input.

assistant — prior model responses. You include these when building multi-turn conversations. You can also prefill this with a partial string to steer the model's output format — covered in module 3.3.
Your API call returns stop_reason == "max_tokens". What does this mean and what should you do?
It means the model was cut off mid-generation — the output is incomplete. The model had more to say but hit the hard ceiling set by max_tokens. Fix options: (1) increase max_tokens if your budget allows; (2) shorten the input to leave more room for the output; (3) if you're extracting structured data, consider splitting into smaller requests. Always check stop_reason before processing the response — truncated JSON or code is a silent bug.
What is the difference between the three structured output approaches and when should you use each?
Prompt + JSON mode — ask the model to output JSON in the system prompt. Lowest effort but unreliable; the model may add prose or malform the output. Only acceptable for internal scratchpad usage, never in production pipelines.

Tool-call schema — define a JSON Schema as a tool definition and force the model to call it. The SDK validates against the schema automatically and returns a Python dict. Recommended for all production structured extraction. Works across Anthropic, OpenAI, and LiteLLM.

XML tags — ask the model to wrap its answer in named XML tags, then extract with regex. Good for simple single-field extraction where a full tool schema is overkill. Not suitable for nested or multi-field structured output.
You have an endpoint that calls the LLM and needs to show output to the user as fast as possible. What change do you make and what pattern do you use on the backend?
Switch from client.messages.create() to the client.messages.stream() context manager. On the backend, return a StreamingResponse (FastAPI) with media_type="text/event-stream" (Server-Sent Events). Yield each token as f"data: {text}\n\n". On the frontend, use the browser's native EventSource API or a fetch with ReadableStream to consume the chunks and append them to the UI in real time. The user experience changes from "wait 10 seconds for the full answer" to "words appear immediately."
