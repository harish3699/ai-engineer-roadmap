Classes and __init__ — the building block #

A class is a blueprint. An instance is an object built from that blueprint. When you call LangChain's ChatOpenAI(model="gpt-4o"), you are instantiating a class. When you write class SearchInput(BaseModel):, you are extending one. You need to understand what happens inside both.

The __init__ method runs automatically when you create an instance. Its job is simple: store the arguments you pass as instance variables on self. Everything in the class can then access those variables via self.

Diagram · Class (blueprint) → Instance (object in memory)
pythonclasses.py 
# A Message class — the kind you will build in every chat application
class Message:
    # Class variable — shared by ALL instances of Message
    role_options: list[str] = ["human", "assistant", "system"]

    def __init__(self, role: str, content: str) -> None:
        if role not in self.role_options:
            raise ValueError(f"role must be one of {self.role_options}")
        # Instance variables — unique to THIS specific message object
        self.role    = role
        self.content = content
        self._tokens: int | None = None  # _ prefix = private by convention

    def word_count(self) -> int:
        """Instance method — reads this specific message's content."""
        return len(self.content.split())

    def __repr__(self) -> str:
        return f"Message(role={self.role!r}, content={self.content[:40]!r})"


# Instantiate two independent Message objects from the same blueprint
msg1 = Message("human", "What is the capital of France?")
msg2 = Message("assistant", "The capital of France is Paris.")

print(msg1.word_count())   # 6
print(msg2.word_count())   # 6
print(msg1.role)           # "human"  — each has its own data
print(msg2.role)           # "assistant"
💡
What is self? self is just a reference to the specific instance being used. When you call msg1.word_count(), Python passes msg1 as self automatically. The name self is a convention, not a keyword — but break this convention and everyone who reads your code will hate you.

Instance, class, and static methods #

Python has three kinds of methods and choosing the wrong one is a subtle bug that makes frameworks harder to extend. AI framework source code uses all three heavily — especially @classmethod as a factory pattern for constructing objects from JSON and config.

🎯
Decision rule Needs self (this object's data)? → instance method. Needs cls (the class itself, to construct a new instance)? → @classmethod. Needs neither (pure utility function that belongs logically in the class)? → @staticmethod.
pythonmethods.py — all three method types with real-world use cases 
from datetime import datetime

class ChatSession:

    def __init__(self, session_id: str) -> None:
        self.session_id = session_id
        self.messages: list[dict] = []
        self.created_at = datetime.now()

    # ── Instance method — needs self (this session's messages) ──────────
    def add_message(self, role: str, content: str) -> None:
        self.messages.append({"role": role, "content": content})

    # ── Class method — factory constructor (very common in frameworks) ───
    @classmethod
    def from_dict(cls, data: dict) -> "ChatSession":
        """Reconstruct a ChatSession from a JSON-deserialized dict.

        cls is the class itself, not an instance.
        This lets subclasses override from_dict and still work correctly.
        """
        session = cls(data["session_id"])   # calls __init__ via cls()
        for msg in data.get("messages", []):
            session.add_message(msg["role"], msg["content"])
        return session

    # ── Static method — pure utility, no self or cls needed ─────────────
    @staticmethod
    def is_valid_role(role: str) -> bool:
        return role in {"human", "assistant", "system"}


# Instance method — called on an instance
s = ChatSession("sess-001")
s.add_message("human", "Hello")

# Class method — factory, no manual __init__ needed
payload = {"session_id": "sess-002", "messages": [{"role": "human", "content": "Hi"}]}
s2 = ChatSession.from_dict(payload)

# Static method — can be called on class OR instance, no difference
print(ChatSession.is_valid_role("human"))    # True
print(ChatSession.is_valid_role("unknown"))  # False

Inheritance, encapsulation, polymorphism #

LangChain's message objects — HumanMessage, AIMessage, SystemMessage — are all subclasses of BaseMessage. You use them constantly without thinking about this, until you need to extend them or understand why isinstance(msg, BaseMessage) returns True for all of them.

Inheritance lets a child class reuse everything from a parent. Encapsulation hides internal state behind a controlled interface. Polymorphism means the same method name does the right thing for each type. Together these three make framework extension clean instead of copy-paste.

Diagram · LangChain-style message inheritance hierarchy
pythoninheritance.py — mirrors LangChain's actual message hierarchy 
# ─ Base class ─────────────────────────────────────────────────────────
class BaseMessage:

    def __init__(self, content: str, role: str) -> None:
        self.content = content
        self._role   = role  # _ signals: set by subclass, not externally

    def to_dict(self) -> dict:
        return {"role": self._role, "content": self.content}

    def __repr__(self) -> str:
        return f"{type(self).__name__}(content={self.content[:40]!r})"


# ─ Subclasses ─────────────────────────────────────────────────────────
class HumanMessage(BaseMessage):

    def __init__(self, content: str) -> None:
        super().__init__(content, role="human")  # ALWAYS call super().__init__


class AIMessage(BaseMessage):

    def __init__(self, content: str) -> None:
        super().__init__(content, role="assistant")

    # Polymorphism — override the parent's to_dict to add extra fields
    def to_dict(self) -> dict:
        base = super().to_dict()   # start with parent's result
        base["source"] = "llm"     # extend it
        return base


class SystemMessage(BaseMessage):

    def __init__(self, content: str) -> None:
        super().__init__(content, role="system")


# ─ Polymorphism in action ──────────────────────────────────────────────
messages: list[BaseMessage] = [
    SystemMessage("You are a helpful assistant."),
    HumanMessage("What is LangChain?"),
    AIMessage("LangChain is a framework for LLM applications."),
]

for msg in messages:
    print(msg.to_dict())
# {'role': 'system',    'content': 'You are...'}
# {'role': 'human',     'content': 'What is LangChain?'}
# {'role': 'assistant', 'content': 'LangChain is...', 'source': 'llm'}

print(isinstance(messages[2], BaseMessage))  # True — all three are
⚠️
Don't skip super().__init__() Forgetting to call super().__init__() in a subclass is one of the most common OOP bugs. The parent's __init__ sets up internal state the parent's methods depend on. Skip it and you'll get AttributeError deep inside framework code that's hard to trace back to your class.

Dataclasses — boilerplate-free data containers #

A large portion of classes in AI engineering code are just data containers: they hold fields, maybe have one or two methods, and don't need complex logic. For these, writing a full __init__ that assigns every argument to self.field is pure boilerplate. @dataclass generates it for you.

It also solves the mutable default argument bug from Module 1.1. You cannot use = [] as a default in a @dataclass — Python forces you to use field(default_factory=list), which creates a fresh list per instance.

Side-by-side: regular class vs dataclass

Regular class — 12 lines of boilerplate
class Chunk:
    def __init__(
        self,
        text:   str,
        source: str,
        page:   int = 0,
        meta:   dict | None = None,
    ) -> None:
        self.text   = text
        self.source = source
        self.page   = page
        # must handle mutable default manually
        self.meta   = meta if meta is not None else {}
Dataclass — same result, 4 lines
from dataclasses import dataclass, field

@dataclass
class Chunk:
    text:   str
    source: str
    page:   int  = 0
    meta:   dict = field(default_factory=dict)
    # field() forces a fresh {} per instance
    # can't use = {} here — Python will error
pythondataclasses.py — full example with frozen and field() 
from dataclasses import dataclass, field

@dataclass
class Chunk:
    """A document chunk in a RAG pipeline."""
    text:     str
    source:   str
    page:     int  = 0
    metadata: dict = field(default_factory=dict)  # fresh {} per instance

    def word_count(self) -> int:
        return len(self.text.split())


# frozen=True makes the instance immutable and hashable
# Use this when you need to store chunks as dict keys or in sets
@dataclass(frozen=True)
class ChunkId:
    source: str
    page:   int
    index:  int


# Usage
c = Chunk(text="Embeddings map text to vectors.", source="rag-guide.pdf", page=3)
print(c)              # Chunk(text='Embeddings...', source='rag-guide.pdf', page=3, metadata={})
print(c.word_count()) # 5

# Two chunks do NOT share the same metadata dict
c1 = Chunk(text="a", source="x")
c2 = Chunk(text="b", source="y")
c1.metadata["entity"] = "Paris"
print(c2.metadata)    # {} — separate dict, not shared

# Frozen chunk as a dict key
cid = ChunkId(source="rag-guide.pdf", page=3, index=0)
cache = {cid: "cached embedding"}

Pydantic models — the AI engineer's superpower #

Pydantic is the most important library in AI engineering after the model SDKs. LangChain's tools, LangGraph's state, FastAPI's request bodies, and every agent's tool schemas are Pydantic models.

The key difference from a regular class or @dataclass: Pydantic validates data at runtime. Pass a string where an int is expected and Pydantic raises a ValidationError immediately, with a precise error message, before any of your code runs. This is runtime safety on top of the static type hints Python's mypy checks at build time.

🔗
Why this connects back to Module 1.1 Pydantic reads your type hints and docstrings using Python's inspect module — the same mechanism LangChain's @tool decorator uses. BaseModel is also a class, inheriting from Pydantic's internals. Everything in this module connects: classes → inheritance → type hints → Pydantic → tool schemas.
pythonpydantic.py — runtime validation + tool schema pattern 
from pydantic import BaseModel, Field, field_validator

# ─ Tool input schema — LangChain reads this to tell the LLM what your tool needs ─
class SearchInput(BaseModel):
    query:   str            = Field(..., description="The search query")
    top_k:   int            = Field(5, ge=1, le=20, description="Number of results (1–20)")
    filters: dict[str, str] = Field(default_factory=dict)

    @field_validator("query")
    @classmethod
    def query_not_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("query cannot be empty or whitespace")
        return v.strip()  # validators can also transform the value


# ─ Validation at runtime ──────────────────────────────────────────────────────
try:
    bad = SearchInput(query="", top_k=5)
except Exception as e:
    print(e)
    # 1 validation error: query · query cannot be empty or whitespace

try:
    bad2 = SearchInput(query="test", top_k=99)  # violates le=20 constraint
except Exception as e:
    print(e)
    # 1 validation error: top_k · Input should be less than or equal to 20

# ─ Correct usage ──────────────────────────────────────────────────────────────
inp = SearchInput(query="  RAG pipeline best practices  ", top_k=3)
print(inp.query)          # "RAG pipeline best practices" ← stripped by validator

# Serialize to dict (for logging, sending to APIs)
print(inp.model_dump())
# {'query': 'RAG pipeline best practices', 'top_k': 3, 'filters': {}}

# Deserialize from dict (from API payloads, JSON responses)
payload = {"query": "embeddings", "top_k": 10}
validated = SearchInput.model_validate(payload)
print(validated.top_k)    # 10 (int, not string "10" — Pydantic coerces it)

Self-check #

Five questions. If you can answer all five without looking back, you're ready for 1.3.

What is the difference between @classmethod and @staticmethod?
A @classmethod receives the class itself as its first argument (cls), so it can call cls() to construct a new instance — making it ideal as a factory constructor. A @staticmethod receives neither self nor cls; it's a pure function that happens to live in the class namespace. Use @classmethod when you need to create instances. Use @staticmethod for utilities that belong logically in the class but don't need class or instance data.
Why must you call super().__init__() in a subclass __init__?
The parent's __init__ sets up instance variables the parent's other methods depend on. If you skip it, those instance variables are never created, and any parent method that tries to access them will raise an AttributeError. This is especially important when extending Pydantic's BaseModel or LangChain's base classes — their internals rely on __init__ to wire up validation, field tracking, and serialization hooks.
How does field(default_factory=list) in a dataclass fix the mutable default problem?
field(default_factory=list) stores a callable, not a value. Every time a new instance is created, Python calls list() to produce a fresh, independent list. With a plain = [] default, Python would create the list once at class definition time and share that single list across all instances. The @dataclass decorator actually enforces this — it raises a ValueError if you try to use a mutable value directly as a default, forcing you to use field(default_factory=...).
What does Pydantic do at runtime that Python's built-in type hints do NOT?
Python's type hints are purely static annotations — they are completely ignored at runtime unless something explicitly reads them. mypy and your editor check them at build/edit time, but Python itself does not enforce them. Pydantic reads those same type hints at runtime and actively validates incoming data against them — raising a detailed ValidationError with field names and error messages if validation fails. It also coerces compatible types (e.g. the string "5" becomes the integer 5 for an int field).
You pass {"query": "test", "top_k": "five"} to SearchInput.model_validate(). What happens?
Pydantic first tries to coerce the string "five" to int. Since "five" cannot be cast to an integer, it raises a ValidationError with a message like: "top_k: Input should be a valid integer, unable to parse string as an integer". If you had passed "5" instead, Pydantic would silently coerce it to 5 — this coercion behaviour is intentional and can be disabled with model_config = ConfigDict(strict=True) if you want strict type matching.
