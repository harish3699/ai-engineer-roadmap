MCP — Model Context Protocol

Before MCP, connecting an LLM to a tool meant writing custom integration code for every LLM-tool pair. With 10 LLMs and 20 tools, that's 200 integrations. MCP is Anthropic's open standard that decouples LLMs from tools entirely — a tool exposes itself as an MCP server once, and any MCP-compatible LLM client can use it. This module covers what MCP is, the server/client architecture, consuming existing community servers, and building your own.

~24 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • MCP is an open standard (like HTTP, not a framework) — it defines a JSON-RPC 2.0 message protocol for tools (actions), resources (data), and prompts (templates)
  • MCP servers expose capabilities; MCP clients (Claude Desktop, Cursor, your custom agent) consume them — the LLM itself is behind the client
  • stdio transport: server is a subprocess you spawn, communicates via stdin/stdout — simple for local dev. HTTP+SSE transport: server is a network service — required for remote/multi-user deployments
  • Hundreds of community MCP servers already exist (filesystem, GitHub, Slack, Postgres, Brave Search) — before building, check if one already exists
  • MCP is moving fast — spec, registry, auth model and resource semantics are still evolving; bookmark modelcontextprotocol.io and re-read the spec when upgrading

Why MCP exists #

Before MCP, every LLM integration was bespoke: OpenAI function calling has its own schema, Anthropic tool use has a slightly different schema, LangChain has its own abstraction, LlamaIndex has another. A company building a GitHub integration had to write it four different ways for four different LLM stacks. MCP solves this with a single protocol — one GitHub MCP server works with every MCP-compatible client.

MCP architecture
LLM (Claude/GPT)
Generates text + tool call requests
←→
MCP Client
Claude Desktop, Cursor, your agent — manages server connections, translates LLM calls to MCP protocol
←→
stdio / HTTP
MCP Server(s)
Filesystem, GitHub, Slack, Postgres, your custom server — exposes tools, resources, prompts

MCP primitives: tools, resources, prompts #

MCP servers expose three types of capabilities:

MCP primitives
Tools
Functions the LLM can call — equivalent to the function calling tools in module 5.1. Side-effectful actions (write file, send message, query DB). Model-controlled.
Resources
Read-only data endpoints the client can fetch and inject into context — files, database rows, API responses. Application-controlled (not LLM-controlled). URI-addressed.
Prompts
Parameterised prompt templates the user/client can invoke — like slash commands. "Summarise this file as a bug report." User-controlled.

Building a minimal MCP server #

The Python MCP SDK (mcp package) makes building a server straightforward. The FastMCP class handles the JSON-RPC protocol, tool registration, and transport automatically.

pythonweather_mcp_server.py — minimal MCP server with stdio transport
# pip install mcp
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather-server")

# ── Register a tool ───────────────────────────────────────────────────
@mcp.tool()
def get_weather(city: str, unit: str = "celsius") -> dict:
    """
    Get current weather for a city.

    Args:
        city: City name in English, e.g. 'Berlin' or 'Tokyo'
        unit: Temperature unit - 'celsius' or 'fahrenheit'

    Returns weather conditions and temperature.
    """
    # In production: call a real weather API
    return {"city": city, "temperature": 18, "condition": "Partly cloudy", "unit": unit}

# ── Register a resource ───────────────────────────────────────────────
@mcp.resource("weather://cities/supported")
def supported_cities() -> str:
    """List of cities with full weather data coverage."""
    return "Berlin, Tokyo, New York, London, Sydney, Toronto"

# ── Entry point: stdio transport (default for local MCP) ──────────────
if __name__ == "__main__":
    mcp.run(transport="stdio")
ℹ
Docstrings become tool descriptions in MCP In the FastMCP SDK, a tool's docstring becomes its MCP description — the text the model reads to decide when to call it. Write detailed docstrings including the Args section, because those parameter descriptions surface in the MCP tool schema. Same principle as module 5.2: description quality = selection accuracy.

stdio vs HTTP+SSE transport #

MCP supports two transports. Choose based on your deployment model:

Transport comparison
stdio (subprocess)
  • Client spawns server as a subprocess
  • Communicates via stdin/stdout JSON-RPC
  • Simple — no networking, no auth
  • Process lifecycle tied to client
  • Best for: local tools, Claude Desktop config, single-user dev
HTTP + SSE
  • Server runs as a persistent HTTP service
  • Client connects via HTTP + Server-Sent Events
  • Supports multiple concurrent clients
  • Can run remotely (cloud, container)
  • Best for: production, shared team tools, remote servers
pythonmcp_client.py — connecting to an MCP server programmatically
import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def main():
    server_params = StdioServerParameters(
        command="python",
        args=["weather_mcp_server.py"],
    )

    async with stdio_client(server_params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # List available tools
            tools = await session.list_tools()
            print("Available tools:", [t.name for t in tools.tools])

            # Call a tool directly
            result = await session.call_tool("get_weather", {"city": "Berlin"})
            print("Weather result:", result.content)

            # Read a resource
            resource = await session.read_resource("weather://cities/supported")
            print("Supported cities:", resource.contents[0].text)

asyncio.run(main())

Configuring Claude Desktop with MCP servers #

Claude Desktop reads MCP server configuration from ~/Library/Application Support/Claude/claude_desktop_config.json (macOS) or %APPDATA%\Claude\claude_desktop_config.json (Windows). Add any stdio server here and it will be available in Claude Desktop sessions.

jsonclaude_desktop_config.json — adding MCP servers to Claude Desktop
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/harish/projects"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "your-token-here"
      }
    },
    "weather": {
      "command": "python",
      "args": ["/path/to/weather_mcp_server.py"]
    }
  }
}
⚠
MCP is moving fast — re-read the spec before major upgrades As of 2026, MCP's auth model (OAuth for remote servers), resource subscriptions, and the official server registry are still actively evolving. Community servers may use slightly different SDK versions. Before upgrading the mcp package in a production project, check the release notes — breaking changes to message schemas have occurred between minor versions. Bookmark modelcontextprotocol.io for the canonical spec.

Self-check #

Explain the problem MCP solves. Without MCP, what would a company need to do to make a custom database tool available to three different LLM clients?
Without MCP, a company with a custom database query tool would need to implement it three separate times: once using OpenAI's function calling JSON Schema format (for GPT-4 clients), once using Anthropic's tool_use format (for Claude clients), and once for any other LLM's SDK format. If any of those clients change their schemas — which they do between API versions — all implementations need updating. Additionally, if they want to use the tool with LangChain or LlamaIndex, they need two more adapters. With MCP, they implement the database tool as a single MCP server once. Any MCP-compatible client (Claude Desktop, Cursor, a custom agent using the Python MCP SDK) can use it without changes. The protocol handles schema translation — the MCP client converts the server's tool list into whatever format the target LLM requires.
What is the difference between an MCP tool, an MCP resource, and an MCP prompt? Give a concrete example of each for a project management use case.
Tools are model-controlled side-effectful actions — the LLM decides when to call them and they can read or write data. Example: create_task(title, assignee, due_date) — the model calls this when the user says "create a task for Alice due Friday". Resources are application-controlled read-only data sources, addressed by URI, that the client fetches and injects into context. Example: projects://current-sprint/tasks — the client pre-loads this list of tasks into the model's context window at session start, so the model knows what's in the sprint without needing to call a tool. Prompts are user-controlled parameterised templates, like slash commands. Example: a prompt template "Summarise all blocked tasks as a standup update for {team_name}" — the user invokes this, supplies the team name, and gets back a formatted standup. The key distinction: tools are for the LLM to invoke on demand; resources are for the client to pre-load; prompts are for the user to trigger.
When would you choose HTTP+SSE transport over stdio for your MCP server, and what additional concerns does HTTP transport introduce?
Choose HTTP+SSE when: your server needs to support multiple concurrent users (stdio ties the server lifecycle to one client process), your server runs on a remote machine or in a container, you need the server to persist independently of any client session, or you're deploying to a team where multiple developers run the same tools. HTTP+SSE introduces concerns that stdio avoids: authentication — anyone who can reach the HTTP port can list and call your tools, so you need API keys or OAuth (MCP's auth model for remote servers is still evolving); TLS — traffic must be encrypted in transit for production deployments; CORS — if a web client is consuming the server; connection reliability — you need to handle reconnection when SSE streams drop; and rate limiting — without request throttling, a misbehaving client can overwhelm the server. For local development, stdio is the right default — simpler, no networking stack to debug, and the process is automatically cleaned up when Claude Desktop or your agent exits.
