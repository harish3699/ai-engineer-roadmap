Advanced Reasoning Techniques

Basic chain-of-thought solves most multi-step problems. For the ones it doesn't — ambiguous logic, complex planning, tasks where a single wrong step cascades into a wrong answer — there are four advanced techniques. Self-consistency averages out randomness. Least-to-most solves easy subproblems first and stacks the answers. Self-refine uses the model as its own critic. Extended thinking delegates reasoning to a hidden scratchpad. Each has a specific use case and a cost profile.

~25 min read 1 video · 1 recording script 4 code examples Last updated 2026-05-07
Key Takeaways
  • Self-consistency runs the same prompt N times at high temperature and takes a majority vote — it trades cost (N× API calls) for reliability on tasks with a single correct answer
  • Least-to-most decomposition solves the easiest subproblems first and passes each answer into the next call — error propagation is minimised because later steps have correct context
  • Self-refine is structured iterative refinement: the model critiques against an explicit rubric, then rewrites — the rubric is what makes it different from 3.4's simpler refinement loop
  • Claude extended thinking moves all reasoning to a hidden scratchpad — you get a clean final answer at higher accuracy without reasoning tokens polluting your response
  • All four techniques multiply cost — only reach for them when basic CoT fails, the task has a high-stakes correct answer, or quality ceiling matters more than latency

When basic CoT fails #

Basic chain-of-thought ("think step by step") works because it gives the model more context tokens before generating the final answer. It fails when: the problem has multiple valid reasoning paths that lead to different answers; individual reasoning steps are themselves complex enough to fail; or the task requires the model to evaluate its own work against a multi-dimensional rubric. The four techniques below address each failure mode.

Technique vs failure mode
Self-consistency
Multiple reasoning paths exist; single-sample answers are noisy.
N× cost
Least-to-most
Problem has prerequisite subproblems; later steps need earlier answers.
K× cost (K steps)
Self-refine
Output quality needs rubric-driven evaluation and targeted rewriting.
2–3× cost
Extended thinking
Complex reasoning that would pollute the visible response with scratchpad work.
budget_tokens cost

Self-consistency sampling #

Self-consistency runs the same prompt multiple times at a higher temperature and aggregates results. For tasks with a single correct answer — maths, logic, classification — the majority answer across N samples is significantly more accurate than a single greedy sample. The mechanism: different sampling paths find the same answer more often when it's correct than when it's wrong.

pythonself_consistency.py — majority vote across parallel samples
import anthropic, asyncio
from collections import Counter
client = anthropic.AsyncAnthropic()

async def single_sample(question: str, temperature: float) -> str:
    resp = await client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        temperature=temperature,
        system="Solve the problem step by step. End your response with 'Answer: X' on its own line.",
        messages=[{"role": "user", "content": question}],
    )
    text = resp.content[0].text
    # extract the final answer from the last line
    for line in reversed(text.splitlines()):
        if line.startswith("Answer:"):
            return line.replace("Answer:", "").strip()
    return text.strip()

async def self_consistent_answer(question: str, n: int = 5, temperature: float = 0.8) -> str:
    # run N samples in parallel — same latency as 1 call
    samples = await asyncio.gather(*[single_sample(question, temperature) for _ in range(n)])
    counts = Counter(samples)
    majority, votes = counts.most_common(1)[0]
    print(f"Samples: {dict(counts)} → majority: {majority!r} ({votes}/{n})")
    return majority

problem = """A train leaves City A at 9:00 AM travelling at 80 km/h toward City B,
which is 360 km away. Another train leaves City B at 10:00 AM travelling at
100 km/h toward City A. At what time do they meet?"""

answer = asyncio.run(self_consistent_answer(problem, n=7))
print(f"Final answer: {answer}")
# → Samples show 7 samples, majority answer with vote count
ℹ
Run samples in parallel, not serially Using asyncio.gather() fires all N API calls simultaneously — wall-clock time equals one call, not N calls. Serial sampling multiplies your latency by N, which defeats the purpose in production. Always parallelize self-consistency sampling using the async SDK.

Least-to-most prompting #

Least-to-most decomposes a hard problem into an ordered list of subproblems, from easiest to hardest. Each subproblem is solved in a separate call, and the answer is appended to the next call's context. Later steps never need to rediscover what earlier steps already established — the context accumulates correct partial solutions.

Least-to-most: accumulating context
Call 1 → Q1 (easiest) → A1
Call 2 → Q1 + A1 + Q2 → A2
Call 3 → Q1 + A1 + Q2 + A2 + Q3 → A3
Call N → Q1..N-1 + A1..N-1 + Qfinal (hardest) → Final Answer
Each call has a fully correct prior chain. The hardest question is asked last, when all prerequisite answers are already in context.
pythonleast_to_most.py — multi-step word problem decomposition
import anthropic
client = anthropic.Anthropic()

hard_problem = """A bakery sells muffins for $2.50 and cookies for $1.75.
On Monday it sold 48 muffins and 120 cookies.
On Tuesday sales were 30% higher for muffins and 15% lower for cookies.
What was the total revenue across both days, and what percentage came from muffins?"""

# ── Step 1: Ask the model to decompose into ordered subproblems ───────
decompose_resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=256,
    system="Break this problem into 3–5 subproblems ordered from easiest to hardest. Number them. No solutions yet.",
    messages=[{"role": "user", "content": hard_problem}],
)
subproblems = decompose_resp.content[0].text
print("Subproblems:\n", subproblems)

# ── Step 2: Solve each subproblem, accumulating correct context ───────
solve_resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=512,
    system="Solve each subproblem in order. Show working for each. Use each answer in subsequent steps.",
    messages=[
        {"role": "user",      "content": hard_problem},
        {"role": "assistant", "content": f"I'll break this into subproblems:\n{subproblems}"},
        {"role": "user",      "content": "Now solve each subproblem in order and give the final answer."},
    ],
)
print("\nSolution:\n", solve_resp.content[0].text)

Self-refine with a rubric #

Self-refine is a structured critique loop. The difference from the iterative refinement in module 3.4 is the rubric: instead of asking the model to generally improve its output, you give it specific dimensions to score, a minimum passing score per dimension, and instructions to keep refining until it passes. This converts vague "make it better" loops into targeted, measurable improvement cycles.

pythonself_refine.py — rubric-driven critique loop for email copy
import anthropic, json
client = anthropic.Anthropic()

RUBRIC = """Score the email subject line on each dimension from 1–5:
- specificity: does it mention a concrete benefit or number? (5 = very specific)
- urgency: does it create a reason to open now? (5 = strong urgency)
- clarity: is the value proposition immediately clear? (5 = crystal clear)
- spam_risk: does it use trigger words like FREE, URGENT, !!!? (5 = no spam signals)

Respond with JSON: {"specificity": N, "urgency": N, "clarity": N, "spam_risk": N, "pass": bool, "issues": [...]}
pass = true only if all dimensions score 4 or higher."""

brief = "Cold email subject line for a B2B tool that cuts AWS costs by 30% for startups."

# ── Draft ─────────────────────────────────────────────────────────────
draft_resp = client.messages.create(
    model="claude-haiku-4-5-20251001", max_tokens=64,
    system="You write cold email subject lines. Return only the subject line, no quotes.",
    messages=[{"role": "user", "content": brief}],
)
subject = draft_resp.content[0].text.strip()

# ── Refine loop — max 3 iterations ───────────────────────────────────
for iteration in range(1, 4):
    print(f"\nIteration {iteration}: {subject!r}")

    score_resp = client.messages.create(
        model="claude-haiku-4-5-20251001", max_tokens=128,
        system=RUBRIC,
        messages=[{"role": "user", "content": subject},
                  {"role": "assistant", "content": "{"}],
    )
    scores = json.loads("{" + score_resp.content[0].text)
    print(f"  Scores: {scores}")

    if scores.get("pass"):
        print(f"  ✓ Passed on iteration {iteration}")
        break

    # failed — rewrite targeting the specific issues
    rewrite_resp = client.messages.create(
        model="claude-haiku-4-5-20251001", max_tokens=64,
        system="You improve cold email subject lines. Return only the revised subject line.",
        messages=[{"role": "user", "content":
            f"Subject: {subject}\nIssues: {scores['issues']}\nRewrite to fix the issues."}],
    )
    subject = rewrite_resp.content[0].text.strip()
else:
    print("  Best effort after 3 iterations.")

Claude extended thinking #

Extended thinking is Claude's native reasoning mode — activated via the thinking parameter. Instead of chain-of-thought reasoning appearing in the visible response, it happens in a hidden scratchpad. The visible response contains only the final, polished answer. The thinking blocks are available in the response if you want to inspect them, but they don't appear to the end user. This is the cleanest way to apply deep reasoning without contaminating your output.

⚠
Extended thinking requires specific models and disables some params As of 2026, extended thinking is available on claude-sonnet-4-6 and claude-opus-4-7. It is not available on Haiku. When thinking is enabled: temperature must be set to 1 (or omitted); streaming is supported; top_p and top_k cannot be set. budget_tokens controls the thinking token ceiling — minimum 1,024, recommended 8,000–16,000 for complex tasks.
pythonextended_thinking.py — hidden scratchpad reasoning
import anthropic
client = anthropic.Anthropic()

complex_problem = """A startup has 3 pricing tiers: Starter ($49/mo), Pro ($149/mo), Enterprise ($499/mo).
Current distribution: 60% Starter, 30% Pro, 10% Enterprise. Total customers: 2,000.
They want to increase MRR by 40% without acquiring new customers.
What combination of tier upgrades and price increases would achieve this with the lowest churn risk?"""

resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=8000,
    thinking={
        "type": "enabled",
        "budget_tokens": 8000,  # how many tokens the hidden scratchpad can use
    },
    messages=[{"role": "user", "content": complex_problem}],
)

# ── Inspect what's in the response ───────────────────────────────────
for block in resp.content:
    if block.type == "thinking":
        print(f"[THINKING — {len(block.thinking)} chars, not shown to user]\n")
    elif block.type == "text":
        print("[FINAL ANSWER]\n", block.text)

print(f"\nInput tokens: {resp.usage.input_tokens}")
print(f"Output tokens (thinking + text): {resp.usage.output_tokens}")

# ── Streaming with extended thinking ─────────────────────────────────
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=8000,
    thinking={"type": "enabled", "budget_tokens": 5000},
    messages=[{"role": "user", "content": complex_problem}],
) as stream:
    for event in stream:
        if hasattr(event, "type") and event.type == "content_block_start":
            if event.content_block.type == "text":
                print("\n--- Final answer streaming ---")
        if hasattr(event, "type") and event.type == "content_block_delta":
            if hasattr(event.delta, "text"):
                print(event.delta.text, end="", flush=True)
💡
When to use extended thinking vs self-consistency Extended thinking is for a single complex problem that needs deep reasoning — planning, multi-constraint optimisation, step-by-step proofs. Self-consistency is for problems where the correct answer exists but the model is noisy — maths, logic puzzles, classification edge cases. Extended thinking is one expensive call; self-consistency is N cheaper calls. For most production use cases, extended thinking on Sonnet is both cheaper and faster than self-consistency on Haiku with N=7.

Self-check #

You run self-consistency with N=5 on a maths problem and get answers: [42, 42, 41, 42, 40]. What do you return and with what confidence level?
Return 42 — it has 3 of 5 votes (60% majority). Confidence can be expressed as the vote fraction: 3/5 = 0.60. If you need a higher confidence threshold (e.g. 0.80) before auto-routing and not reaching it, you could increase N to 9 or 11 samples — odd numbers avoid ties — or route to a fallback. In practice, 5–7 samples is sufficient for most use cases; beyond 11 samples, the marginal accuracy gain rarely justifies the cost.
What makes least-to-most different from simply asking the model to "break the problem into steps and solve each"?
When you ask the model to break and solve in one call, it generates the decomposition and all solutions in the same context window simultaneously — it can't use the answer to step 1 as verified context when generating step 2, because they're generated in parallel within one completion pass. Least-to-most uses separate API calls — the answer to step 1 is a real completed API response appended to the context before step 2 is requested. This means later steps have verified, correct prior answers as grounding. The model can't generate an internally consistent but wrong chain; it has to use real committed answers.
What is the minimum budget_tokens value for extended thinking and what happens if you set it too low?
The minimum is 1,024 tokens. If budget_tokens is too low for a complex problem, the model's thinking is truncated mid-reasoning and the final answer quality degrades — it may produce a plausible-sounding but wrong response because it didn't finish working through the problem. For simple tasks, 1,024–2,000 is fine. For complex multi-constraint problems (planning, optimisation, proofs), use 8,000–16,000. You can inspect the thinking block length after a call to calibrate the budget for your specific task type. The budget is a ceiling — the model uses less if the task is simpler.
You have a complex single problem that requires deep reasoning. Compare the cost trade-off between self-consistency with N=7 on Haiku vs extended thinking on Sonnet.
Self-consistency N=7 on Haiku: 7 API calls, each charged at Haiku rates (~$0.25/M input, $1.25/M output). For a 500-token problem with 300-token answers: 7 × (500 × $0.25 + 300 × $1.25) / 1M ≈ $0.0034. Extended thinking on Sonnet: 1 API call with 8,000 thinking tokens + ~300 output tokens, charged at Sonnet rates ($3/M input, $15/M output): (500 + 8,000 + 300) × blended rate ≈ roughly $0.02–$0.04 depending on thinking-to-output ratio. Extended thinking on Sonnet is typically 5–10× more expensive for this case, but provides deeper reasoning in one call. For most production use cases on complex single problems, extended thinking wins on quality; self-consistency wins on cost when a deterministic correct answer exists across simple reasoning paths.
