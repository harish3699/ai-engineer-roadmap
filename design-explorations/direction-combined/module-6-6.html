Context Compression

A 20-step agentic workflow produces a transcript that can easily exceed 30,000 tokens — tool calls, retrieved documents, intermediate reasoning, partial results. You cannot fit that history into the next turn without hitting the context limit. Context compression is the art of reducing token volume while preserving decision-relevant information. This module covers three techniques: hierarchical summarisation (progressively compress older turns), selective fact extraction (keep only the actionable conclusions), and trace compression (distill an entire agent run into a structured "what was decided" block).

~22 min read 4 code examples Last updated 2026-05-07
Key Takeaways
  • Compression is lossy by nature — you cannot perfectly summarise 10,000 tokens into 500. The goal is to preserve what the model actually needs to continue correctly, not everything that was said
  • Hierarchical summarisation compresses in layers: old turns → paragraph summary → one-line fact. Applying all three levels lets you represent hundreds of turns in ~300 tokens
  • Tool call results are the biggest compression wins — a 2,000-token API response often has one useful sentence. Extract the conclusion, discard the rest
  • Agent traces (Thought → Action → Observation loops) compress well into structured decision records: what was tried, what was found, what was decided
  • Never compress the most recent turns — the model needs verbatim access to what just happened. Only compress the middle of the context, not the edges

Why compression beats truncation #

Truncation (dropping old turns) is the default for most frameworks — it's simple and has zero latency overhead. The problem: it creates a hard cliff. Turn 1 might contain the user's original goal statement. Dropping it means the agent loses the task framing and starts optimising for local sub-tasks rather than the original objective. Compression replaces the cliff with a gradient: older content is still represented, just at lower resolution.

Hierarchical summarisation #

Hierarchical summarisation applies compression in layers. The oldest turns get the most compression; recent turns stay verbatim. This matches the attention pattern from Module 6.1 — old content receives little attention anyway, so high compression there has minimal impact on quality.

pythonhierarchical_compress.py — tiered compression of long histories
import tiktoken
from openai import OpenAI

enc = tiktoken.get_encoding("cl100k_base")
client = OpenAI()

def compress_chunk(messages: list[dict], level: int) -> str:
    """Level 1: paragraph summary. Level 2: single-sentence summary."""
    instruction = {
        1: "Summarise the following conversation in 1-3 sentences. Preserve key facts, decisions, and user goals.",
        2: "Distill the following summary into ONE sentence capturing the core topic and key decision.",
    }[level]
    raw = "\n".join(f"{m['role'].upper()}: {m['content']}" for m in messages)
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": instruction},
            {"role": "user",   "content": raw},
        ],
        max_tokens=200,
    )
    return resp.choices[0].message.content


def hierarchical_compress(
    history: list[dict],
    budget: int = 2000,
    recent_pairs: int = 6,
) -> list[dict]:
    """Compress a long history into budget tokens, keeping recent_pairs verbatim."""
    total_tokens = sum(len(enc.encode(m["content"])) for m in history)
    if total_tokens <= budget:
        return history  # no compression needed

    recent = history[-(recent_pairs * 2):]
    old     = history[:-(recent_pairs * 2)]
    if not old:
        return history

    # Split old messages into two halves for two-level compression
    mid = len(old) // 2
    very_old = old[:mid]
    less_old  = old[mid:]

    compressed_parts = []
    if very_old:
        # Level 2 compression for oldest chunk — one sentence
        summary = compress_chunk(very_old, level=2)
        compressed_parts.append({"role": "user",      "content": "[earlier context]"})
        compressed_parts.append({"role": "assistant", "content": summary})
    if less_old:
        # Level 1 compression for middle chunk — paragraph
        summary = compress_chunk(less_old, level=1)
        compressed_parts.append({"role": "user",      "content": "[earlier context continued]"})
        compressed_parts.append({"role": "assistant", "content": summary})

    return compressed_parts + recent  # compressed summaries + verbatim recent turns

Tool result compression #

Tool call results are often verbose — a database query returns 50 rows when 1 was needed, an API response includes metadata fields the agent never looks at. The compression pattern: immediately after receiving a tool result, extract the decision-relevant conclusion before appending it to history.

pythontool_compressor.py — extracting conclusions from verbose tool results
COMPRESS_TOOL_PROMPT = """Extract the decision-relevant information from this tool result.
Return 1-2 sentences only. Include: key numbers, status codes, names, dates, counts.
Omit: boilerplate, repeated fields, empty values, debug metadata."""

def compress_tool_result(tool_name: str, raw_result: str, max_tokens: int = 150) -> str:
    import tiktoken
    enc = tiktoken.get_encoding("cl100k_base")
    if len(enc.encode(raw_result)) <= max_tokens:
        return raw_result  # already short enough, don't waste an LLM call
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": COMPRESS_TOOL_PROMPT},
            {"role": "user",   "content": f"Tool: {tool_name}\n\nResult:\n{raw_result}"},
        ],
        max_tokens=100,
    )
    compressed = resp.choices[0].message.content
    return f"[compressed from {len(enc.encode(raw_result))} tokens] {compressed}"

Agent trace compression #

A ReAct loop that runs for 15 steps produces 15 Thought/Action/Observation triples — thousands of tokens of reasoning that was correct at the time but is largely irrelevant to future steps. Compress a completed sub-task into a structured decision record before continuing:

pythontrace_compress.py — distilling a ReAct trace into a decision record
TRACE_COMPRESS_PROMPT = """You are summarising a completed sub-task in an agent's reasoning trace.
Output a structured JSON object with these fields:
- task: the original sub-task
- outcome: "success" | "failure" | "partial"
- key_findings: list of up to 3 key facts discovered (short strings)
- decision: the action decided or conclusion reached
- blockers: any obstacles encountered (empty list if none)

Be extremely concise — this summary will be carried forward in context."""

def compress_trace(trace_messages: list[dict]) -> str:
    raw = "\n".join(f"{m['role'].upper()}: {m['content']}" for m in trace_messages)
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": TRACE_COMPRESS_PROMPT},
            {"role": "user",   "content": raw},
        ],
        response_format={"type": "json_object"},
        max_tokens=200,
    )
    record = json.loads(resp.choices[0].message.content)
    # Format as a readable summary block to inject into context
    findings = "\n".join(f"  - {f}" for f in record.get("key_findings", []))
    return (
        f"[Sub-task: {record.get('task', 'unknown')}]\n"
        f"Outcome: {record.get('outcome', 'unknown')}\n"
        f"Key findings:\n{findings}\n"
        f"Decision: {record.get('decision', '')}"
    )

When to trigger compression #

Trigger Compress Method
History tokens exceed 60% of budget Oldest 30% of history Hierarchical (L1)
Tool result > 150 tokens Tool result immediately Conclusion extraction
Sub-task completes (ReAct finishes a loop) Entire sub-task trace Structured decision record
Session ends Full session Episodic extraction (Module 6.5)
⚠
Compression adds latency — be selective Each compression step is an extra LLM call. In a fast agentic loop, compressing every tool result adds 200–500 ms per tool call. Apply compression selectively: only compress tool results above a token threshold, only compress history when you're approaching the budget limit. Measure the token savings versus the latency cost in your specific workload before making compression the default.
