Networking & Access

The goal of this module is not to turn you into a network engineer — it's to give you the minimum viable VPC setup that keeps your agent services secure, lets them talk to each other, and exposes the right endpoints to the internet. One VPC, two public subnets, two private subnets, sensible security group rules, task-scoped IAM roles, and API Gateway to route traffic. That covers 90% of agent deployment scenarios.

~16 min read 2 code examples Last updated 2026-05-07
Key Takeaways
  • VPC layout: one VPC, two public subnets (ALB lives here), two private subnets (ECS tasks + RDS live here) across two availability zones — this is the standard pattern
  • Security groups as firewalls: ALB SG allows 443 from 0.0.0.0/0; ECS task SG allows traffic only from the ALB SG; RDS SG allows 5432 only from the ECS task SG — each layer only speaks to the next
  • IAM task roles: each ECS task gets its own IAM role with only the permissions it needs — never use an IAM user with static credentials in a container
  • API Gateway: for serverless Lambda endpoints, use HTTP API (cheaper, lower latency than REST API); for WebSocket streaming, use WebSocket API
  • Outbound from private subnets: ECS tasks need to reach external LLM APIs (OpenAI, Anthropic) — add a NAT Gateway in each public subnet, or use VPC endpoints for AWS services to avoid the internet

Minimum viable VPC layout #

textvpc-layout.txt — standard agent deployment network topology
VPC: 10.0.0.0/16

  Public subnets (ALB, NAT Gateway)
  ├── 10.0.0.0/24  (us-east-1a)
  └── 10.0.1.0/24  (us-east-1b)
       ↑ Internet Gateway attached

  Private subnets (ECS tasks, RDS, ElastiCache)
  ├── 10.0.2.0/24  (us-east-1a)
  └── 10.0.3.0/24  (us-east-1b)
       ↑ NAT Gateway routes outbound internet traffic

Security Groups:
  alb-sg:       ingress 443/tcp from 0.0.0.0/0
  ecs-task-sg:  ingress 8000/tcp from alb-sg only
                egress  443/tcp  to 0.0.0.0/0  (LLM API calls)
                egress  5432/tcp to rds-sg
  rds-sg:       ingress 5432/tcp from ecs-task-sg only

Flow:
  Internet → ALB (public subnet) → ECS task (private subnet)
  ECS task → NAT Gateway (public subnet) → OpenAI/Anthropic
  ECS task → RDS (private subnet, same VPC)

IAM roles for ECS tasks #

ECS tasks use two IAM roles: the task execution role (used by the ECS agent to pull the container image and write logs to CloudWatch) and the task role (used by the application code inside the container). Never use a shared admin role — scope each task role to only the AWS resources it actually needs.

jsontask-role-policy.json — principle of least privilege for an agent task
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "S3DocumentAccess",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::my-agent-prod/documents/*"
    },
    {
      "Sid": "SecretsManagerRead",
      "Effect": "Allow",
      "Action": ["secretsmanager:GetSecretValue"],
      "Resource": [
        "arn:aws:secretsmanager:us-east-1:123456789012:secret:agent/db-password-*",
        "arn:aws:secretsmanager:us-east-1:123456789012:secret:agent/openai-key-*"
      ]
    },
    {
      "Sid": "BedrockInvoke",
      "Effect": "Allow",
      "Action": ["bedrock:InvokeModel", "bedrock:InvokeModelWithResponseStream"],
      "Resource": "arn:aws:bedrock:us-east-1::foundation-model/anthropic.*"
    },
    {
      "Sid": "CloudWatchLogs",
      "Effect": "Allow",
      "Action": ["logs:CreateLogStream", "logs:PutLogEvents"],
      "Resource": "arn:aws:logs:us-east-1:123456789012:log-group:/ecs/agent-api:*"
    }
  ]
}
⚠
Never put AWS credentials in environment variables ECS task roles inject temporary credentials automatically via the instance metadata service — your boto3 code picks them up without any configuration. If you put static IAM credentials (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY) in an env var, a container escape vulnerability exposes them permanently. Use task roles instead.

API Gateway for Lambda endpoints #

API Gateway sits in front of Lambda functions and handles routing, authentication, rate limiting, and CORS. Use HTTP API (not REST API) — it's cheaper ($1 vs $3.50 per million requests), lower latency (~10ms vs ~50ms overhead), and supports Lambda proxy integration natively. Use REST API only if you need request/response transformation or a custom domain on an existing REST API.

For ECS Fargate services, an Application Load Balancer (ALB) plays the same role as API Gateway — it handles routing and health checks, and is required for Fargate anyway since tasks get private IPs. API Gateway can also route to ALB via VPC Link for unified traffic management.
