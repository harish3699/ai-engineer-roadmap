Agent-as-Tool

Not every multi-agent system needs a graph. When the hierarchy is clear (parent calls child, never the reverse), the flow is deterministic (parent decides when to call the child), and there is no shared mutable state between agents, wrapping a sub-agent as a @tool gives you all the specialisation benefits of multi-agent with none of the LangGraph overhead. The parent agent calls it like any other function. The sub-agent runs its own internal loop, returns a result, and the parent continues. This is the simplest, most debuggable form of multi-agent composition.

~18 min read 3 code examples Last updated 2026-05-07
Key Takeaways
  • Agent-as-tool: wrap a complete sub-agent (with its own LLM loop, tools, and system prompt) behind a @tool decorated function — the parent agent calls it like any other tool
  • Use this pattern when: the hierarchy is clear and unidirectional, the sub-agent's task has well-defined input/output types, and you don't need to observe or interrupt the sub-agent's internal steps
  • Composing researcher + summariser + critic agents as tools creates a specialist team without graph state machinery
  • The sub-agent's internal LLM calls are invisible to the parent's context window — each agent has its own isolated context, which is the key benefit
  • Choose LangGraph over agent-as-tool when: you need to observe intermediate states, the flow is non-hierarchical, agents need to communicate directly, or you need checkpointing and resumability

The pattern #

pythonagent_as_tool.py — wrapping a sub-agent as a @tool
from langchain_core.tools import tool
from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")

# ── Sub-agent 1: Researcher ───────────────────────────────────────────
researcher_agent = create_react_agent(
    llm,
    tools=[web_search, fetch_url],           # researcher's own tools
    state_modifier="You are a research specialist. Search for facts only.",
)

@tool
def research_topic(topic: str) -> str:
    """Research a topic thoroughly and return a factual summary.
    Use this when you need detailed, sourced information on a specific subject."""
    result = researcher_agent.invoke({"messages": [{"role": "user", "content": topic}]})
    return result["messages"][-1].content  # only final answer returned to parent


# ── Sub-agent 2: Code Generator ───────────────────────────────────────
code_agent = create_react_agent(
    llm,
    tools=[run_tests, check_syntax],
    state_modifier="You are a Python expert. Write clean, tested code.",
)

@tool
def generate_code(specification: str) -> str:
    """Write and test Python code for a given specification.
    Returns the final code only — no explanation unless asked."""
    result = code_agent.invoke({"messages": [{"role": "user", "content": specification}]})
    return result["messages"][-1].content


# ── Parent orchestrator ───────────────────────────────────────────────
orchestrator = create_react_agent(
    llm,
    tools=[research_topic, generate_code],   # sub-agents are just tools
    state_modifier="""You are an orchestrator. For complex tasks:
1. Use research_topic to gather facts
2. Use generate_code to produce implementations
Delegate — do not attempt deep research or coding yourself.""",
)

# The parent treats sub-agents exactly like any other tool call.
# The sub-agent's internal reasoning loop (potentially 10 LLM calls) is
# invisible to the parent — it only sees the final returned string.

Agent-as-tool vs LangGraph — decision table #

Requirement Agent-as-tool LangGraph
Clear parent→child hierarchy ✓ Ideal Works but overkill
Observe sub-agent intermediate steps ✗ Not possible ✓ Via stream()
Interrupt sub-agent mid-execution ✗ Not possible ✓ interrupt_before
Agents communicate bidirectionally ✗ Not supported ✓ Via shared state
Checkpoint and resume workflows ✗ Manual only ✓ Built-in checkpointers
Context isolation between agents ✓ Automatic Requires design
Implementation complexity ✓ Low — just a function Higher — graph DSL

Composing a specialist team #

The agent-as-tool pattern composes naturally into a specialist team — each agent is a callable with a clear contract. The parent orchestrator decides which specialists to engage and in what order, without any graph state infrastructure:

pythonspecialist_team.py — researcher + summariser + critic team
@tool
def deep_research(query: str) -> str:
    """Search and synthesise information on a topic. Returns factual summary with sources."""
    return researcher_agent.invoke({"messages": [{"role":"user","content":query}]})["messages"][-1].content

@tool
def summarise_document(document: str, style: str = "executive") -> str:
    """Summarise a long document. style: 'executive' | 'technical' | 'bullet'"""
    return summariser_agent.invoke({"messages": [{"role":"user","content":f"{style}: {document}"}]})["messages"][-1].content

@tool
def critique_argument(argument: str) -> str:
    """Evaluate an argument for logical flaws, missing evidence, and counterarguments."""
    return critic_agent.invoke({"messages": [{"role":"user","content":argument}]})["messages"][-1].content

# Parent orchestrator with three specialist tools
# It decides the order: research → summarise → critique
analyst = create_react_agent(llm, tools=[deep_research, summarise_document, critique_argument],
    state_modifier="You are an analyst. Use your tools to research, synthesise, and validate claims.")

result = analyst.invoke({"messages": [{"role":"user","content":"Analyse the impact of remote work on productivity"}]})
ℹ
Start with agent-as-tool, graduate to LangGraph When building a new multi-agent system, start with agent-as-tool. It's faster to implement, easier to test, and often sufficient. Only graduate to LangGraph when you hit one of the concrete limitations in the table above — when you need to observe intermediate state, when agents need to communicate sideways, or when you need durable checkpointing. Premature LangGraph adoption is common and adds weeks of debugging without proportional benefit.
