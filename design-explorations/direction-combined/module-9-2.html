Compute

LLM applications have unusual compute requirements: they make external API calls that take 1–30 seconds, they run in loops with unpredictable iteration counts, and they need to hold state across multiple steps. Lambda's 15-minute timeout and cold-start penalties make it the wrong choice for interactive agents. ECS Fargate runs containers indefinitely and is the standard compute target for production agent deployments. This module explains when to use each, and how to manage container images with ECR.

~18 min read 3 code examples Last updated 2026-05-07
Key Takeaways
  • Lambda fits: short, event-driven tasks — document ingestion triggers (S3 event → chunk → embed), webhook handlers, scheduled evaluation jobs. Max 15 min, 10 GB RAM, no persistent connections
  • ECS Fargate fits: long-running interactive agents, streaming API servers, any workflow that holds a database connection open, multi-agent pipelines that run >5 minutes
  • ECR: private container registry in your AWS account — always use ECR for production images, not Docker Hub (rate limits, no IAM integration)
  • The agent compute shape: FastAPI app in a Docker container, deployed as a Fargate task, behind an Application Load Balancer — this covers 90% of production agent deployments
  • Cold start: Lambda cold starts add 1–3 s for Python containers; Fargate tasks take 20–60 s to start but stay warm — provision minimum 1 always-on task for interactive endpoints

Lambda vs Fargate — the decision #

Constraint Lambda ECS Fargate
Max execution time 15 minutes Unlimited
Startup time 1–3 s (cold), <100ms (warm) 20–60 s first start, then instant
Persistent connections Problematic (connection pooling unreliable) Full support — keep RDS/Redis connections alive
Cost model Pay per invocation + ms — zero cost at zero traffic Pay per vCPU/hour + GB/hour while running
Best for agents Ingestion triggers, webhooks, scheduled jobs Interactive chat, streaming API, long pipelines
Scaling Instant to 1000 concurrent — no config Minutes via ECS Service Auto Scaling (target tracking)

ECR and Fargate workflow #

The deployment loop is: build image locally → push to ECR → update ECS service to pull the new image → Fargate drains old tasks and starts new ones. All of this can be triggered from a CI/CD pipeline with a few AWS CLI commands.

bashdeploy.sh — build, push to ECR, update ECS service
AWS_ACCOUNT=123456789012
AWS_REGION=us-east-1
ECR_REPO=${AWS_ACCOUNT}.dkr.ecr.${AWS_REGION}.amazonaws.com
IMAGE_TAG=$(git rev-parse --short HEAD)    # use git SHA as image tag

# Authenticate Docker to ECR
aws ecr get-login-password --region ${AWS_REGION} \
  | docker login --username AWS --password-stdin ${ECR_REPO}

# Build and push
docker build -t agent-api:${IMAGE_TAG} .
docker tag agent-api:${IMAGE_TAG} ${ECR_REPO}/agent-api:${IMAGE_TAG}
docker push ${ECR_REPO}/agent-api:${IMAGE_TAG}

# Update the ECS service to use the new image
# (ECS will register a new task definition revision automatically with --force-new-deployment)
aws ecs update-service \
  --cluster agent-cluster \
  --service agent-api-service \
  --force-new-deployment \
  --region ${AWS_REGION}

echo "Deployed ${IMAGE_TAG} to ECS Fargate"

Lambda for document ingestion #

Lambda is the right choice for S3-triggered ingestion: a user uploads a PDF, S3 fires an event, Lambda picks it up, extracts text, chunks it, embeds it, and writes chunks to Postgres. The whole pipeline runs in under 2 minutes for most documents — well within Lambda's 15-minute limit.

pythonlambda_ingestion.py — S3-triggered document processing
import json, boto3

s3 = boto3.client("s3")

def handler(event: dict, context) -> dict:
    for record in event["Records"]:
        bucket = record["s3"]["bucket"]["name"]
        key    = record["s3"]["object"]["key"]

        try:
            # 1. Fetch document from S3
            obj = s3.get_object(Bucket=bucket, Key=key)
            file_bytes = obj["Body"].read()

            # 2. Extract text (use textract or pypdf depending on format)
            text = extract_text(file_bytes, key)

            # 3. Chunk
            chunks = recursive_chunk(text, chunk_size=512, overlap=64)

            # 4. Embed (batch to stay within rate limits)
            embeddings = embed_batch(chunks)

            # 5. Write to Postgres (use connection pooling via RDS Proxy)
            upsert_chunks(key, chunks, embeddings)

            print(f"Processed {key}: {len(chunks)} chunks")

        except Exception as e:
            # Use ErrorLogger pattern — log and continue processing other records
            log_error("IngestionLambda", "handler", e, key)

    return {"statusCode": 200}

# Lambda function config (SAM template excerpt):
# Timeout: 300  # 5 minutes — sufficient for most documents
# MemorySize: 1024  # enough for PDF parsing without thrashing
# Events:
#   S3Upload:
#     Type: S3
#     Properties:
#       Bucket: !Ref DocumentBucket
#       Events: s3:ObjectCreated:*
#       Filter:
#         S3Key:
#           Rules: [{Name: prefix, Value: "documents/raw/"}]
✦
Use RDS Proxy for Lambda → Postgres connections Lambda functions spin up and down rapidly — each new instance tries to open a Postgres connection. At scale this exhausts the connection pool. RDS Proxy sits between Lambda and RDS, multiplexes connections, and eliminates this problem. Enable it for any Lambda that touches RDS.
