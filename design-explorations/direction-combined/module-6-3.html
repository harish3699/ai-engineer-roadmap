Short-Term Memory — Session History

Every multi-turn agent needs to manage a growing conversation buffer. As the session gets longer, naïve append-forever approaches hit the context limit and either crash or silently drop old turns. This module covers the three strategies for short-term memory management — sliding window, token-based trimming, and summary compression — and shows you when to apply each. You'll build a ConversationBuffer class that handles all three modes with a clean API.

~20 min read 3 code examples Last updated 2026-05-07
Key Takeaways
  • Sliding window: keep only the last N message pairs — fast, simple, loses old context entirely; good for task-focused agents where conversational continuity matters less
  • Token-based trimming: drop oldest messages until total tokens fit the budget — precise, respects the context limit hard, can split pairs awkwardly if you don't trim in pairs
  • Summary compression: periodically compress older turns into a single summary message — preserves semantic content while freeing tokens; requires an extra LLM call and introduces summarisation latency
  • Always trim in message pairs (user + assistant together) — dropping a user message without its response creates a dangling assistant turn that confuses the model
  • The system prompt must never be trimmed — only the history buffer between system and the current user message is eligible for reduction

Why naive history management fails #

The simplest history implementation is a list that grows forever: every user turn gets appended, every assistant response gets appended, and the whole list is sent with every API call. This works for a few turns. By turn 30 in a detailed conversation, you're sending 15,000+ tokens of history that the model barely attends to (the lost-in-the-middle problem from Module 6.1), paying for tokens you're not getting value from, and eventually hitting the context limit where the call either fails or your framework silently drops messages.

Short-term memory strategies at a glance
Sliding Window

Strategy 1 — Sliding window #

Keep only the last k message pairs (user + assistant = 1 pair). The implementation is trivial but the pair constraint is critical — never keep an assistant message without its preceding user message, or you'll send a history that starts with the model talking to itself.

pythonsliding_window.py — pair-safe sliding window buffer
class SlidingWindowBuffer:
    def __init__(self, max_pairs: int = 10):
        self.max_pairs = max_pairs
        self._history: list[dict] = []  # alternating user / assistant messages

    def add_turn(self, user_msg: str, assistant_msg: str) -> None:
        self._history.append({"role": "user",      "content": user_msg})
        self._history.append({"role": "assistant", "content": assistant_msg})
        # Trim to max_pairs — each pair = 2 messages
        max_messages = self.max_pairs * 2
        if len(self._history) > max_messages:
            self._history = self._history[-max_messages:]

    def get_messages(self, system_prompt: str, user_query: str) -> list[dict]:
        return [
            {"role": "system", "content": system_prompt},
            *self._history,
            {"role": "user",   "content": user_query},
        ]

    def __len__(self) -> int:
        return len(self._history) // 2  # number of pairs


# ── Usage ──────────────────────────────────────────────────────────────
buf = SlidingWindowBuffer(max_pairs=8)
# After each turn:
buf.add_turn(user_msg=query, assistant_msg=response)
msgs = buf.get_messages(SYSTEM_PROMPT, current_query)

Strategy 2 — Token-based trimming #

More precise than sliding window — drop the oldest pair until the total token count fits the budget. This adapts to message length, so a session with short messages retains more turns than one with long tool results:

pythontoken_trim.py — token-budget history trimming
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

def count_msg_tokens(messages: list[dict]) -> int:
    total = 3  # reply primer
    for m in messages:
        total += 4 + len(enc.encode(m.get("content", "")))
    return total


class TokenBudgetBuffer:
    def __init__(self, history_budget: int = 2500):
        self.budget = history_budget
        self._history: list[dict] = []

    def add_turn(self, user_msg: str, assistant_msg: str) -> None:
        self._history.append({"role": "user",      "content": user_msg})
        self._history.append({"role": "assistant", "content": assistant_msg})
        self._trim()

    def _trim(self) -> None:
        # Remove pairs from the front until we fit the budget
        while len(self._history) >= 2:
            if count_msg_tokens(self._history) <= self.budget:
                break
            self._history = self._history[2:]  # drop oldest pair atomically

    def get_messages(self, system_prompt: str, user_query: str) -> list[dict]:
        return [
            {"role": "system", "content": system_prompt},
            *self._history,
            {"role": "user",   "content": user_query},
        ]

    def token_usage(self) -> int:
        return count_msg_tokens(self._history)

Strategy 3 — Summary compression #

When the buffer exceeds a threshold, call the LLM to summarise the oldest N turns into a compact summary message. The raw turns are replaced with the summary. Future turns can reference facts from early in the session without consuming the full token budget:

pythonsummary_buffer.py — LLM-driven conversation compression
from openai import OpenAI

client = OpenAI()

SUMMARISE_PROMPT = """Summarise the following conversation segment concisely.
Preserve: key decisions, facts stated by the user, topics discussed.
Omit: pleasantries, repeated information, verbose phrasing.
Output a single paragraph starting with "Earlier in this conversation:"."""

class SummaryBuffer:
    def __init__(
        self,
        history_budget: int = 2500,
        compress_threshold: int = 2000,  # compress when history exceeds this
        keep_recent_pairs: int = 4,       # always keep last N pairs verbatim
    ):
        self.history_budget = history_budget
        self.threshold = compress_threshold
        self.keep_recent = keep_recent_pairs
        self._history: list[dict] = []
        self._summary: str | None = None

    def add_turn(self, user_msg: str, assistant_msg: str) -> None:
        self._history.append({"role": "user",      "content": user_msg})
        self._history.append({"role": "assistant", "content": assistant_msg})
        if count_msg_tokens(self._history) > self.threshold:
            self._compress()

    def _compress(self) -> None:
        # Split: everything except the most recent pairs gets summarised
        recent_count = self.keep_recent * 2
        to_compress = self._history[:-recent_count] if len(self._history) > recent_count else []
        if not to_compress:
            return
        raw = "\n".join(f"{m['role'].upper()}: {m['content']}" for m in to_compress)
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[
                {"role": "system",  "content": SUMMARISE_PROMPT},
                {"role": "user",    "content": raw},
            ],
            max_tokens=250,
        )
        summary_text = resp.choices[0].message.content
        # Prepend to existing summary if there was one
        self._summary = (
            (self._summary + "\n" + summary_text) if self._summary else summary_text
        )
        # Keep only the recent pairs
        self._history = self._history[-recent_count:]

    def get_messages(self, system_prompt: str, user_query: str) -> list[dict]:
        messages = [{"role": "system", "content": system_prompt}]
        if self._summary:
            # Inject the running summary as the first user/assistant exchange
            messages.append({"role": "user",      "content": "[conversation context]"})
            messages.append({"role": "assistant", "content": self._summary})
        messages.extend(self._history)
        messages.append({"role": "user", "content": user_query})
        return messages

Choosing the right strategy #

Use case Recommended strategy Reason
Task-focused agent (coding, data analysis) Sliding window (k=6) Older turns rarely relevant; low latency priority
Customer support bot Token trimming Sessions are bounded; precision matters over semantics
Long-running personal assistant Summary compression Context from hours ago still relevant; user names, preferences
Research / document Q&A agent Token trimming + RAG retrieval History less important than retrieved documents
Agentic workflow with many tool calls Summary + episodic (Module 6.5) Tool call results are verbose; full history unaffordable
⚠
The orphan message bug Always trim history in complete pairs (user + assistant). If your trim logic drops the last user message but keeps the assistant response, you produce a messages array where the model appears to answer a question that isn't there. This causes the next response to seem contextually confused. The fix: trim indices 0:2, 0:4 etc. — always even numbers from the front.

Hybrid approach — trim then compress on demand #

Production agents typically combine all three: use token trimming as the baseline, trigger summary compression only when the token count crosses a higher threshold (to amortise the latency cost of the extra LLM call), and apply a hard sliding window cap as a safety backstop.
