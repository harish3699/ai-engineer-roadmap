Evaluating RAG

Building a RAG pipeline without evaluation is building blind. You cannot know whether a retrieval upgrade actually improved answers, whether a chunking change helped or hurt, or whether your system is hallucinating, without a measurement framework. RAGAS provides four complementary metrics that decompose RAG quality into retrieval precision, retrieval recall, generation faithfulness, and answer relevancy — each one diagnosing a different failure mode. This module covers the metrics, building an eval set, running RAGAS, and using the results to guide concrete pipeline improvements.

~24 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • RAGAS provides four metrics that measure different failure modes: context precision (noise in retrieved chunks), context recall (relevant information missing), faithfulness (hallucination), and answer relevancy (answer addresses the question)
  • Build your eval set from real user queries, not synthetic ones — the distribution of real questions reveals failure modes that contrived examples hide
  • Low faithfulness means the LLM is hallucinating — the fix is the system prompt and generation configuration, not retrieval
  • Low context recall means relevant chunks are not being retrieved — the fix is chunking strategy, embedding model, retrieve_k, or hybrid search
  • Low context precision means retrieved context is noisy — the fix is reranking, final_k reduction, or metadata filtering

The four RAGAS metrics #

RAGAS decomposes RAG quality into two retrieval metrics and two generation metrics. Each metric scores 0–1. You need all four — a system can score perfectly on one while failing badly on another.

RAGAS metrics — what each measures
Context Precision
Retrieval metric
Signal-to-noise
in context
Of all chunks retrieved, what fraction were actually relevant to the question? Low precision = the LLM context is full of noise, increasing hallucination risk. Fix: reranking, lower final_k, metadata filtering.
Context Recall
Retrieval metric
Completeness
of retrieval
Of all the information needed to answer the question, what fraction was present in the retrieved context? Low recall = the answer is in the corpus but retrieval missed it. Fix: hybrid search, larger retrieve_k, better chunking, better embedding model.
Faithfulness
Generation metric
Hallucination
detection
Are all claims in the generated answer supported by the retrieved context? Low faithfulness = hallucination — the LLM added facts not in the context. Fix: system prompt (forbid prior knowledge), lower temperature, smaller max_tokens.
Answer Relevancy
Generation metric
Question
addressed?
Does the generated answer actually address what the user asked? Low relevancy = the answer is on-topic but evasive, or addresses a related but different question. Fix: system prompt clarity, better context (high recall means the right info is available for the answer).

Building an evaluation set #

An eval set is a list of (question, ground_truth_answer, ground_truth_contexts) triples. The ground truth contexts are the specific chunks that contain the information needed to answer the question. You can build this manually (golden standard, most reliable) or synthetically using an LLM (faster, good enough for iteration).

⚠
Use real user queries when possible Synthetic eval sets generated by prompting an LLM to "write questions about this document" miss the distribution of actual user queries — which are often ambiguous, poorly phrased, multi-part, or ask about information that was assumed to be obvious. If you have access to user logs (chat history, support tickets), sample 50–100 real queries as the basis for your eval set. If you have nothing, synthetic is fine to start — but validate it against real queries once you have them.
pythonbuild_eval_set.py — LLM-generated question/answer pairs from corpus chunks
import os, json
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

corpus_chunks = [
    {"text": "Employees may claim $150/mo for home internet.",             "source": "HR Policy v3.2"},
    {"text": "One-time $600 allowance for ergonomic furniture.",            "source": "HR Policy v3.2"},
    {"text": "All claims submitted via Workday by the 5th of each month.",   "source": "HR Policy v3.2"},
    {"text": "Manager approval required for claims over $200.",              "source": "HR Policy v3.2"},
]

eval_set = []

for chunk in corpus_chunks:
    resp = client.chat.completions.create(
        model="claude-haiku-4-5-20251001",
        messages=[{
            "role": "user",
            "content": (
                f"Given this passage:\n\n{chunk['text']}\n\n"
                f"Generate ONE specific question that this passage answers, "
                f"and a concise ground-truth answer. "
                f"Return JSON: {{\"question\": \"...\", \"answer\": \"...\"}}"
            ),
        }],
        max_tokens=200,
    )
    qa = json.loads(resp.choices[0].message.content)
    eval_set.append({
        "question":    qa["question"],
        "ground_truth": qa["answer"],
        "reference_contexts": [chunk["text"]],  # the chunk that answers it
    })

print(json.dumps(eval_set, indent=2))

Running RAGAS #

RAGAS takes a dataset of (question, answer, contexts, ground_truth) rows and computes the four metrics using an LLM-as-judge for the generation metrics and embedding similarity for the retrieval metrics. Connect it to your RAG pipeline, run it over your eval set, and get scores per row and aggregated.

pythonragas_eval.py — run RAGAS metrics on your RAG pipeline output
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    context_precision,
    context_recall,
    faithfulness,
    answer_relevancy,
)
from langchain_openai import ChatOpenAI, OpenAIEmbeddings

# ── 1. Collect pipeline outputs ──────────────────────────────────────
# eval_set was built in build_eval_set.py
eval_set = [
    {
        "question": "What is the monthly internet allowance?",
        "ground_truth": "$150 per month.",
        "reference_contexts": ["Employees may claim $150/mo for home internet."],
    },
    {
        "question": "How much can I claim for a standing desk?",
        "ground_truth": "Up to $600 one-time for ergonomic furniture.",
        "reference_contexts": ["One-time $600 allowance for ergonomic furniture."],
    },
]

# Run your RAG pipeline for each eval question
# (assume `rag` is a RAGPipeline from module 4.7, already indexed)
rows = []
for item in eval_set:
    question = item["question"]
    context_chunks = rag.retrieve(question)        # returns list[Chunk]
    contexts = [c.text for c in context_chunks]
    answer = rag.query(question, stream=False)     # non-streaming for eval
    rows.append({
        "question":    question,
        "answer":      answer,
        "contexts":    contexts,           # what retrieval returned
        "ground_truth": item["ground_truth"],
    })

# ── 2. Build HuggingFace Dataset ─────────────────────────────────────
dataset = Dataset.from_list(rows)

# ── 3. Configure LLM judge + embeddings ─────────────────────────────
llm        = ChatOpenAI(model="gpt-4o-mini")  # judge model
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# ── 4. Run evaluation ─────────────────────────────────────────────────
result = evaluate(
    dataset=dataset,
    metrics=[context_precision, context_recall, faithfulness, answer_relevancy],
    llm=llm,
    embeddings=embeddings,
)

# ── 5. Print scores ───────────────────────────────────────────────────
print(result)
# context_precision    0.92
# context_recall       0.75  ← retrieval is missing some relevant chunks
# faithfulness         0.98  ← generation stays in context
# answer_relevancy     0.89

# Per-question breakdown
df = result.to_pandas()
print(df[["question", "context_recall", "faithfulness"]])

Interpreting scores and fixing failures #

Each metric failure maps to a specific pipeline component. The diagnostic workflow: run RAGAS, find the lowest-scoring metric, identify which questions score worst on that metric, examine those specific (question, context, answer) triples manually, then apply the fix from the table below.

RAGAS score → failure mode → fix
Context Precision < 0.75
Too many irrelevant chunks in context
Add or upgrade reranker · reduce final_k · add metadata pre-filter · raise relevance_threshold
Context Recall < 0.75
Answer in corpus but not retrieved
Increase retrieve_k · add hybrid BM25 search · try better embedding model · fix chunking (chunk too large/small) · try HyDE or multi-query
Faithfulness < 0.8
LLM hallucinating claims
Strengthen system prompt ("do not use training data") · lower temperature · reduce max_tokens · upgrade to a larger LLM with better instruction following
Answer Relevancy < 0.75
Answer doesn't address the question
Improve context recall (answer must be available) · clarify system prompt · check if question is out-of-corpus (should trigger "I don't know")
💡
Target scores and typical production ranges There are no universal thresholds — acceptable scores depend on the risk profile of your application. For a low-stakes FAQ bot: faithfulness >0.85, context recall >0.80. For an enterprise HR or legal policy bot where wrong answers have real consequences: faithfulness >0.95, context recall >0.90. A good baseline target for a well-built initial pipeline: all four metrics >0.80. If any metric is <0.65, that is a systematic failure requiring a specific fix before shipping to users.

Evaluating without ground truth #

Context Precision and Faithfulness do not require labeled ground truth — they can be computed purely from (question, answer, contexts). This means you can run them over real production traffic without a hand-labeled eval set, providing a continuous quality signal.

pythonproduction_monitor.py — faithfulness + precision on live traffic
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import context_precision, faithfulness
from langchain_openai import ChatOpenAI, OpenAIEmbeddings

# Log these fields on every production query
production_log = [
    {
        "question":  "How much is the internet allowance?",
        "answer":    "Employees can claim $150 per month for home internet. (Source: HR Policy v3.2)",
        "contexts":  ["Employees may claim $150/mo for home internet.",
                      "Claims submitted via Workday by the 5th."],
        "ground_truth": "",  # not available for real traffic
    },
    # … hundreds of rows from your production logs
]

dataset = Dataset.from_list(production_log)

result = evaluate(
    dataset=dataset,
    metrics=[context_precision, faithfulness],  # these don't need ground_truth
    llm=ChatOpenAI(model="gpt-4o-mini"),
    embeddings=OpenAIEmbeddings(),
)

print(f"Production faithfulness: {result['faithfulness']:.3f}")
print(f"Production context_precision: {result['context_precision']:.3f}")

# Flag rows where faithfulness drops below threshold
df = result.to_pandas()
low_faith = df[df["faithfulness"] < 0.7]
print(f"Potential hallucinations: {len(low_faith)} / {len(df)} queries")
print(low_faith[["question", "faithfulness"]])
ℹ
RAGAS cost: each metric invokes LLM calls Faithfulness and Answer Relevancy use an LLM judge. For a dataset of 100 questions with 5 context chunks each, expect 300–500 LLM API calls to gpt-4o-mini. At $0.15/1M input tokens this is ~$0.10–0.30 for a full eval run — cheap enough to run on every significant pipeline change. For large corpora (10k+ queries), batch your eval or run RAGAS on a sampled subset of 100–200 representative queries.

Self-check #

Your RAG system has context recall of 0.62 but faithfulness of 0.97. What does this tell you about your pipeline, and where should you focus your improvement effort?
Context recall of 0.62 means that in 38% of questions, the relevant information exists in your corpus but retrieval failed to include it in the context. The LLM cannot answer from context that was never retrieved — so these queries likely receive "I don't know" responses or answers synthesised from partial context. The high faithfulness (0.97) tells you that when the LLM does have context, it stays within it and doesn't hallucinate. So the generation component is working correctly; the problem is retrieval. Focus your effort on: (1) increase retrieve_k — the correct chunk may be ranked just outside the current cutoff; (2) add hybrid BM25 search — keyword mismatches are a common cause of recall failure; (3) examine the low-recall questions to identify the specific failure pattern (vocabulary mismatch, chunking boundary issue, or correct answer split across chunks).
What is the difference between context precision and context recall, and why do you need both?
Context precision measures what fraction of retrieved chunks were actually relevant — it penalises retrieving too many irrelevant chunks (noisy context). Context recall measures what fraction of the information needed to answer was present in the retrieved chunks — it penalises missing relevant information (incomplete context). You need both because they represent opposite failure modes. A system that retrieves every chunk in the corpus has perfect recall (nothing is missed) but precision near zero (99%+ of context is noise). A system that retrieves only 1 highly-relevant chunk has perfect precision but poor recall for complex questions requiring multiple facts. A well-tuned pipeline balances both: retrieve enough to have high recall (large retrieve_k), then rerank aggressively to keep precision high (small final_k). A single metric cannot diagnose which failure mode you have.
A faithfulness score of 0.78 indicates the LLM is sometimes hallucinating. Walk through the three most likely causes and fixes in priority order.
1. Weak system prompt (fix first): The most common cause. If the prompt does not explicitly say "do not use information from your training data," the LLM blends training knowledge with retrieved context. Add this prohibition and re-test — faithfulness often jumps 10–15 points. 2. Noisy context (fix second): If context_precision is also low, the LLM context window contains many irrelevant chunks. With low-signal, high-noise context, the model has less to work with and is more likely to fill gaps from training. Fix: reranking and reducing final_k to deliver only high-relevance chunks. 3. Insufficient context (fix third): Paradoxically, if context_recall is also low, the LLM may have the question but not the answer in its context. When forced to answer without relevant information (because the system prompt doesn't trigger a refusal clearly enough), it hallucinates. Fix: either improve retrieval so the answer is available, or make the "refuse if insufficient context" instruction clearer.
Which two RAGAS metrics can be computed on production traffic without labeled ground truth, and how would you use them in a monitoring system?
Context Precision and Faithfulness can be computed without ground truth — they only require (question, answer, contexts). Context precision asks: "are the retrieved chunks relevant to the question?" — the LLM judge can evaluate this without knowing the correct answer. Faithfulness asks: "are all claims in the answer supported by the contexts?" — again, no ground truth needed. In a monitoring system: (1) Log every production query's (question, answer, contexts) triplet. (2) Run RAGAS nightly or weekly on a sample of 100–500 queries from the log. (3) Alert if faithfulness drops below 0.80 — this signals the LLM has started hallucinating, which may indicate a prompt regression or a corpus update that made context less grounding. (4) Alert if context_precision drops below 0.70 — this signals retrieval quality regression. Track these as time-series metrics per deployment to detect regressions after pipeline changes.
