Reasoning vs Base Models

2025 introduced a split that every AI engineer must understand: standard models vs reasoning models. Reasoning models spend extra compute "thinking" before answering. They're more accurate on hard problems — and 5–20× more expensive and slower. This module explains what thinking tokens are, which tasks actually benefit, and when a well-prompted base model is the smarter choice.

~25 min read 1 video · 1 recording script 2 code examples Last updated 2026-05-07
Key Takeaways
  • Reasoning models generate an internal "scratchpad" of thinking tokens before the final answer — these tokens are billed but usually not shown to the user
  • The major reasoning models in 2025–2026: OpenAI o3/o4-mini, Claude with extended thinking, Gemini 2.5 Pro/Flash thinking, DeepSeek R1, Qwen QwQ
  • Use reasoning models for: multi-step logic, complex code, math, ambiguous instructions, planning — tasks where showing your work helps
  • Use base models for: extraction, classification, summarisation, translation, simple Q&A, high-volume tasks — where extra compute is wasted
  • Most production systems are hybrid: base model for 90% of requests, reasoning model only when complexity is detected

What thinking tokens actually are #

A standard model reads your prompt and immediately begins generating the reply token by token. A reasoning model inserts a hidden step first: it generates a long internal monologue — "thinking tokens" — where it reasons through the problem, considers multiple approaches, and self-corrects before writing the final answer. You pay for those thinking tokens. You usually don't see them.

Standard model vs reasoning model — request flow
STANDARD MODEL
Prompt (input tokens)
↓ immediate generation
Reply (output tokens)

Latency: fast (1–3s)
Cost: input + output only
REASONING MODEL
Prompt (input tokens)
↓ thinking tokens (hidden)
"Let me consider... actually... wait..."
Reply (output tokens)
Latency: slow (5–60s)
Cost: input + thinking + output
ℹ
Thinking tokens are billed at output token rates Output tokens cost 3–5× more than input tokens on most providers. A reasoning model that spends 10,000 thinking tokens on a hard maths problem can cost 10–20× more than a standard model answering the same question. Always check the thinking token count in the API response before scaling.

Using reasoning models via API #

pythonreasoning_models.py — Claude extended thinking + OpenAI o3
import anthropic, openai

# ── Claude extended thinking ─────────────────────────────────────────
client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=16000,
    thinking={
        "type":         "enabled",
        "budget_tokens": 10000,   # how many thinking tokens to allow
    },
    messages=[{"role": "user", "content": "Design a multi-agent RAG system for legal docs"}],
)

for block in response.content:
    if block.type == "thinking":
        print(f"[thinking — {len(block.thinking)} chars]")   # visible scratchpad
    else:
        print(block.text)   # final answer

print(f"Thinking tokens used: {response.usage.cache_creation_input_tokens}")

# ── OpenAI o3 with effort control ────────────────────────────────────
oa = openai.OpenAI()

completion = oa.chat.completions.create(
    model="o3",
    reasoning_effort="high",   # low | medium | high
    messages=[
        {"role": "user", "content": "Write a Python function to solve the knapsack problem"},
    ],
)
print(completion.choices[0].message.content)

Hybrid routing — the production pattern #

Most production systems don't pick one model and use it for everything. They route: simple requests go to a fast, cheap base model; complex requests get upgraded to a reasoning model. The router itself is typically a cheap model classification call or a rule-based heuristic.

pythonmodel_router.py — route to reasoning model only when needed
import anthropic

client = anthropic.Anthropic()

COMPLEXITY_SIGNALS = [
    "step by step", "plan", "architecture", "compare",
    "trade-off", "algorithm", "optimise", "debug",
]

def route_model(message: str) -> tuple[str, dict]:
    if any(sig in message.lower() for sig in COMPLEXITY_SIGNALS) or len(message) > 500:
        return "claude-sonnet-4-6", {"type": "enabled", "budget_tokens": 8000}
    return "claude-haiku-4-5-20251001", None   # fast + cheap for simple tasks

def ask(message: str) -> str:
    model, thinking = route_model(message)
    kwargs = {"model": model, "max_tokens": 4096,
              "messages": [{"role": "user", "content": message}]}
    if thinking:
        kwargs["thinking"] = thinking

    resp = client.messages.create(**kwargs)
    print(f"[routed to {model}]")
    return next(b.text for b in resp.content if b.type == "text")

# Simple → Haiku: "What is a context window?"
# Complex → Sonnet + thinking: "Design a chunking strategy for legal PDFs with tables"
💡
Reasoning effort knobs — budget thoughtfully Low / small budget: quick sanity checks, light planning, simple disambiguation.
Medium / ~4k–8k tokens: most engineering tasks — code review, architecture questions, multi-step reasoning.
High / max budget: reserve for genuinely hard problems — competitive programming, complex legal/medical analysis. The cost difference between medium and high is often 3–5× for marginal quality gains.

Self-check #

What are thinking tokens and why are they expensive?
Thinking tokens are the internal reasoning scratchpad a reasoning model generates before producing its final answer. The model "thinks out loud" — considering approaches, self-correcting, exploring paths — before writing the visible reply. They're expensive because they're billed at output token rates (typically 3–5× the input token price), and a hard problem can consume 5,000–50,000 thinking tokens before the final answer appears. Always check the thinking token count in API responses when building cost-sensitive systems.
Your agent classifies customer support tickets into 12 categories — 50,000 requests per day. Should you use a reasoning model?
No. Classification is a straightforward pattern-matching task where a well-prompted base model (e.g., Claude Haiku, GPT-4o-mini) performs equivalently to a reasoning model. Reasoning models are 5–20× more expensive and 5–10× slower — at 50,000 requests/day that's a massive cost difference for zero quality gain. Use a base model at temperature=0 for classification tasks.
What is the difference between reasoning_effort: "low" and reasoning_effort: "high" on OpenAI o3?
low allocates a small thinking token budget — the model reasons briefly, suitable for moderately complex tasks. high allocates a large budget — the model reasons extensively, exploring more paths and self-correcting more often, giving better results on genuinely hard problems. The cost and latency difference between low and high is typically 3–5×. Use medium as your default and only escalate to high when accuracy on hard tasks is the top priority and cost/latency are secondary.
Describe a hybrid routing architecture that uses both a base model and a reasoning model in the same agent.
Route at the request level: classify incoming queries by complexity (using a cheap model or heuristics like message length, keyword signals, or explicit task type). Simple tasks — extraction, Q&A, summarisation, formatting — go to a fast base model. Complex tasks — multi-step planning, algorithm design, ambiguous multi-constraint problems — get routed to a reasoning model. The router itself is a base model call or a rule-based function. This keeps ~90% of requests on cheap/fast models while applying reasoning compute only where it changes the outcome.
