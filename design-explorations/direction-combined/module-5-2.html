Tool Design Principles

Knowing how to call tools (module 5.1) is one thing. Knowing how to design good tools is the craft that separates brittle prototypes from reliable agents. Poor tool design is the most common cause of agent failures in production: the model calls the wrong tool, passes wrong arguments, gets back a wall of unstructured text it can't reason about, or enters a confusion loop. This module covers four principles that prevent all of those failure modes.

~20 min read 1 video · 1 recording script 4 code examples Last updated 2026-05-07
Key Takeaways
  • One tool, one job — a tool that does three things forces the model to remember which mode it's in; split it and the model always picks the right one
  • Descriptions are your API contract with the LLM — write them as precise, actionable triggers: "Call this when the user wants X, NOT when they want Y"
  • Return structured data (dicts/Pydantic models), never prose — the model reasons over structure far better than parsing narrative sentences
  • Fallbacks belong inside tools, not in the agent loop — if the primary API fails, the tool should try a fallback and return a graceful result, not crash
  • Use Pydantic as your tool schema source of truth — define once, validate inputs and generate JSON Schema automatically

Principle 1: One tool, one job #

The single most impactful tool design rule. When a tool does multiple things based on a mode parameter, the LLM must infer the correct mode from context — and it regularly gets it wrong. Split the tool and the model always calls the right one. The cognitive load shifts from the model to your code, where it belongs.

Bad — one tool, multiple modes
{
  "name": "manage_calendar",
  "description": "Manage calendar events — create, read, update, or delete",
  "parameters": {
    "action": {"type": "string", "enum": ["create","read","update","delete"]},
    "event_id": {"type": "string", "description": "Required for update/delete"},
    "title": {"type": "string", "description": "Required for create/update"},
    "start_time": {"type": "string", "description": "Required for create"},
    "end_time": {"type": "string", "description": "Required for create"}
  }
}

Model must infer action, track which params are required for each mode, and often gets confused — calling "update" with no event_id, or "create" forgetting end_time.

Good — four focused tools
{
  "name": "create_calendar_event",
  "description": "Create a new calendar event. Call when user wants to schedule or book a meeting.",
  "parameters": {
    "title": {"type": "string"},
    "start_time": {"type": "string", "description": "ISO 8601"},
    "end_time": {"type": "string", "description": "ISO 8601"}
  },
  "required": ["title","start_time","end_time"]
}
// + get_calendar_event(event_id)
// + update_calendar_event(event_id, title?, start_time?, end_time?)
// + delete_calendar_event(event_id)

Each tool has one clear purpose, no conditional required fields, and an unambiguous trigger phrase in its description.

Principle 2: Descriptions are your LLM API spec #

The model selects tools by reading their names and descriptions — it does not look at your code. A vague description produces wrong tool selection. A precise description with explicit trigger conditions and exclusions produces reliable selection. Write descriptions as when-to-call contracts: "Call this when X. Do NOT call this when Y."

pythondescriptions.py — bad vs good tool descriptions
# ❌ Vague — model guesses when to use it
{
    "name": "search",
    "description": "Search for information",
    "parameters": {"query": {"type": "string"}},
}

# ✅ Precise — explicit trigger + scope + exclusion
{
    "name": "search_product_catalog",
    "description": (
        "Search the internal product catalog by keyword or SKU. "
        "Call this when the user asks about product availability, pricing, "
        "features, or specifications. Do NOT call this for order status "
        "(use get_order_status) or customer information (use get_customer)."
    ),
    "parameters": {
        "query": {
            "type": "string",
            "description": "Search term: product name, SKU (e.g. 'PRD-1234'), or category",
        },
        "max_results": {
            "type": "integer",
            "description": "Maximum results to return. Default 5.",
            "default": 5,
        },
    },
    "required": ["query"],
}

# Parameter-level descriptions are equally important
# Bad:  "city": {"type": "string"}
# Good: "city": {"type": "string", "description": "City name in English, e.g. 'Berlin', 'New York'"}

Principle 3: Return structured data #

Tools return strings to the model (the content field of a tool message). But the content of that string matters enormously. Return JSON-serialised dicts with clear field names — the model can pick out exactly the fields it needs. Return prose ("The weather in Berlin is 18°C, partly cloudy, with some chance of rain later") and the model must parse and interpret a sentence, introducing room for misreading or hallucinated extraction.

pythonstructured_returns.py — Pydantic tool with structured output
from pydantic import BaseModel, Field
import json

# ── Define input schema with Pydantic ─────────────────────────────────
class WeatherArgs(BaseModel):
    city: str = Field(description="City name in English, e.g. 'Berlin'")
    unit: str = Field(default="celsius", description="Temperature unit: celsius or fahrenheit")

# ── Define output schema — what the model receives back ───────────────
class WeatherResult(BaseModel):
    city: str
    temperature: float
    unit: str
    condition: str
    humidity_pct: int
    wind_kph: float
    error: str | None = None  # None on success, string on failure

def get_weather(city: str, unit: str = "celsius") -> str:
    """Return JSON string — always, even on error."""
    try:
        # In production: call real weather API
        result = WeatherResult(
            city=city, temperature=18.0, unit=unit,
            condition="Partly cloudy", humidity_pct=65, wind_kph=12.5
        )
    except Exception as e:
        result = WeatherResult(
            city=city, temperature=0, unit=unit,
            condition="", humidity_pct=0, wind_kph=0,
            error=f"Failed to fetch weather: {e}"
        )
    return result.model_dump_json()

# ── Auto-generate JSON Schema from Pydantic model ─────────────────────
def pydantic_to_tool(name: str, description: str, model: type[BaseModel]) -> dict:
    schema = model.model_json_schema()
    return {
        "type": "function",
        "function": {
            "name": name,
            "description": description,
            "parameters": schema,
        },
    }

weather_tool = pydantic_to_tool(
    "get_weather",
    "Get current weather for a city. Call when user asks about temperature or conditions.",
    WeatherArgs,
)
💡
Always include an error field in your output schema Rather than having a tool raise an exception, define an error: str | None field in your output schema. On success it's None; on failure it carries the error message. The model reads the error field and can communicate the failure to the user or attempt a different approach — rather than crashing the loop.

Principle 4: Fallbacks inside tools, not around them #

The agent loop should not contain complex error-handling logic for individual tools. If a tool needs a fallback (primary API down, rate limited, cache miss), that logic lives inside the tool — not in the calling loop. The loop's job is to route tool calls and collect results; the tool's job is to return a useful result even under failure conditions.

pythonfallback_tool.py — internal fallback pattern
import httpx, json
from functools import lru_cache

def get_exchange_rate(base: str, quote: str) -> str:
    """
    Get exchange rate. Tries live API first, falls back to cached data.
    Returns JSON string with rate and data_source field.
    """
    try:
        # Primary: live rate from external API
        r = httpx.get(
            f"https://api.exchangerate.host/convert?from={base}&to={quote}",
            timeout=3.0,
        )
        r.raise_for_status()
        data = r.json()
        return json.dumps({
            "base": base, "quote": quote,
            "rate": data["info"]["rate"],
            "data_source": "live",
        })
    except Exception:
        try:
            # Fallback: cached rates (e.g. Redis or file — simulated here)
            cached = {("USD", "EUR"): 0.92, ("EUR", "GBP"): 0.85}
            rate = cached.get((base, quote))
            if rate:
                return json.dumps({
                    "base": base, "quote": quote,
                    "rate": rate,
                    "data_source": "cache (may be stale)",
                })
        except Exception:
            pass
        # Both failed — return structured error, never raise
        return json.dumps({
            "base": base, "quote": quote,
            "rate": None,
            "error": "Exchange rate service unavailable. Please try again later.",
            "data_source": "none",
        })

Keep the tool count low #

Every tool you add to the model's context uses tokens and increases the cognitive load of tool selection. Models start making selection errors as the tool list grows beyond ~10–15 tools. Design guidelines:

⚠
More than 15 tools? Use dynamic tool loading If your agent needs 20+ tools, don't pass them all at once. Use a router pattern: a fast first-pass retrieval (vector search over tool descriptions) selects the 5–8 most relevant tools for the current query, and only those are passed to the model. This is called "tool RAG" — it keeps selection accuracy high even for large tool inventories.

Self-check #

You have a tool called search_docs(query, filter_type) where filter_type can be "policy", "procedure", or "faq". How should you redesign this following the single-responsibility principle, and why?
Split into three focused tools: search_policy_docs(query), search_procedure_docs(query), and search_faq(query). Each has one purpose and a description that clearly states when to call it. The original design with filter_type has two problems: (1) the model must infer the correct filter value from the user's question — if the user asks something that spans categories, the model guesses one; (2) new developers and the model both have to understand the conditional logic. With three separate tools, the model's decision is binary per tool (do I need this or not?), selection accuracy is higher, and the parameter schema is simpler for each. Additionally, if you add a new document type later (e.g., "technical_spec"), you add a new tool rather than extending an enum — no risk of breaking existing routing.
What information should a well-written tool description always include? Write a good description for a tool that sends an email.
A well-written tool description should include: (1) what the tool does — the action and its scope, (2) when to call it — specific trigger conditions or phrases the user might say, (3) when NOT to call it — explicit exclusions to prevent misrouting (especially if there are similar tools), (4) any important constraints or side effects. Example for an email tool: "Send an email to one or more recipients. Call this when the user explicitly asks to send, email, or forward a message to someone. Do NOT call this for drafting or previewing emails (use draft_email) — this action sends immediately and cannot be undone. Always confirm the recipient and subject with the user before calling if they have not been explicitly provided." The description surfaces the irreversibility (sends immediately), routes away similar-sounding tasks (draft vs send), and tells the agent to confirm before acting.
Your tool calls an external weather API that sometimes returns a 429 rate-limit error. Where should the retry/fallback logic live, and how should the tool signal a partial success to the model?
The retry and fallback logic lives inside the tool function, not in the agent loop. The agent loop's job is to route tool calls and build messages — it should not contain per-tool error policies. Inside the tool: catch the 429, implement exponential backoff for 1–2 retries, and if still failing, try a secondary weather data source (e.g., Open-Meteo as fallback). If both fail, return a structured JSON result with "error": "Rate limit exceeded on primary provider, fallback also unavailable. Try again in 60 seconds." and set "temperature": null. Partial success — primary failed but fallback succeeded — is signalled via a "data_source": "fallback (may differ from primary)" field in the response. This lets the model include an appropriate caveat in its final answer ("Based on a secondary weather source...") without the loop needing to know anything about provider-level failure modes.
