The ReAct Pattern

Most LLM tool-calling happens reactively: user asks → model calls tool → model answers. ReAct (Reasoning + Acting) is the pattern that makes agents capable of genuine multi-step problem solving: the model explicitly reasons about what to do, acts with a tool, observes the result, then reasons again — iterating until it has enough information to produce a final answer. Understanding ReAct is understanding how every production agent framework (LangChain, LlamaIndex, AutoGen) works under the hood.

~20 min read 1 video · 1 recording script 2 code examples Last updated 2026-05-07
Key Takeaways
  • ReAct = Thought + Action + Observation, repeated until the agent has enough information — the loop is the agent
  • Thought steps are not function calls — they are the model generating its own reasoning trace, which improves action selection and is invaluable for debugging
  • Modern frontier models (Claude 3.7 Sonnet, GPT-o1/o3) implement "thinking" natively — the scratchpad is private, but the effect is the same: better reasoning before acting
  • Prompt-based ReAct (explicit Thought/Action/Observation format) still works well for smaller models, and gives you full visibility into the reasoning trace
  • Always set a hard max_steps guard — a confused ReAct agent can loop indefinitely; log every step to catch infinite loops early

The problem with direct tool-calling #

In module 5.1's tool loop, the model sees the user's question, decides which tool to call, gets the result, and answers. For simple single-step tasks this is fine. But for complex tasks — "Research our three main competitors, find their pricing pages, and summarise the key differences" — the model needs to plan a sequence of actions, adapt the plan based on intermediate results, and track what it has already learned. Direct tool-calling has no mechanism for this: the model just sees the conversation history and guesses what to do next.

ReAct solves this by giving the model an explicit reasoning step before each action — a Thought where it can plan, reflect on what it learned, and decide what to do next.

The Thought → Action → Observation loop #

ReAct loop — one iteration
Thought
The model generates a reasoning trace: "I need to find competitor pricing. Let me start by searching for the pricing pages of Company A, B, and C separately." This is not a tool call — it's text the model writes to itself.
Action
The model issues a tool call: search_web(query="Company A pricing"). Your code executes the search and returns the result.
Observation
The tool result injected back as context. The model sees: "Search result: Company A charges $49/mo for 5 seats, $99/mo for 20 seats."
pythonreact_agent.py — prompt-based ReAct implementation
import os, re
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

REACT_SYSTEM = """You are a research agent. Solve the task step by step using this format:

Thought: [Reason about what you know and what to do next]
Action: [tool_name(arg1="val1", arg2="val2")]
Observation: [result will be filled in here]
... (repeat Thought/Action/Observation as needed)
Final Answer: [your complete answer to the original question]

Available tools:
- search_web(query: str) → search results as text
- get_page(url: str) → page content as text
- calculate(expression: str) → numeric result

IMPORTANT: Always start with a Thought. Never skip straight to Action."""

# Simulated tools
def search_web(query: str) -> str:
    return {
        "acme pricing": "ACME charges $49/mo (5 seats) and $99/mo (20 seats).",
        "rival pricing": "Rival charges $39/mo flat, unlimited seats.",
    }.get(query.lower(), "No results found.")

def calculate(expression: str) -> str:
    try: return str(eval(expression, {"__builtins__": {}}))
    except: return "Error evaluating expression"

TOOL_MAP = {"search_web": search_web, "calculate": calculate}

def parse_action(text: str) -> tuple[str, dict] | None:
    """Extract Action: tool_name(kwargs) from model output."""
    m = re.search(r'Action:\s*(\w+)\(([^)]*)\)', text)
    if not m: return None
    fn_name = m.group(1)
    args_str = m.group(2)
    # Parse key="value" pairs
    kwargs = {}
    for kv in re.findall(r'(\w+)=["\']([^"\']*)["\']', args_str):
        kwargs[kv[0]] = kv[1]
    return fn_name, kwargs

def react_agent(question: str, max_steps: int = 6) -> str:
    messages = [
        {"role": "system", "content": REACT_SYSTEM},
        {"role": "user", "content": question},
    ]

    for step in range(max_steps):
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
            stop=["Observation:"],  # stop before filling in the observation
        )
        text = resp.choices[0].message.content
        print(f"\n[Step {step+1}]\n{text}")

        if "Final Answer:" in text:
            return text.split("Final Answer:")[1].strip()

        action = parse_action(text)
        if action:
            fn_name, kwargs = action
            print(f"→ Calling {fn_name}({kwargs})")
            result = TOOL_MAP.get(fn_name, lambda **k: "Unknown tool")(**kwargs)
            observation = f"\nObservation: {result}\n"
            messages.append({"role": "assistant", "content": text + observation})
        else:
            break

    return "Max steps reached without final answer."

answer = react_agent("Compare the pricing of ACME and Rival. Which is cheaper for a 10-seat team?")
print(f"\n✅ Final Answer: {answer}")

Native thinking models vs prompt-based ReAct #

Modern frontier models implement thinking natively. Claude 3.7 Sonnet's extended thinking, GPT-o1 and o3's reasoning tokens, and Google Gemini's thinking mode all implement a private scratchpad — the model reasons before generating its visible output. You don't see the thoughts, but the quality of action selection and final answers improves significantly for complex tasks.

Prompt-based vs native thinking
Prompt-based ReAct
  • Thought/Action/Observation in message text
  • Reasoning trace is visible — great for debugging
  • Works on any model, including smaller ones
  • Uses more tokens (thoughts in context)
  • Stop tokens or parsing required
Native thinking (o1/Claude thinking)
  • Private scratchpad before output
  • Reasoning not visible to you
  • Model-specific, higher cost per call
  • Better performance on hard reasoning tasks
  • Just call the model — no prompt engineering
pythonclaude_thinking.py — extended thinking with Anthropic API
import anthropic

client = anthropic.Anthropic()

# Extended thinking — model reasons privately before answering
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=8000,
    thinking={
        "type": "enabled",
        "budget_tokens": 5000,  # max tokens for private reasoning
    },
    messages=[{
        "role": "user",
        "content": "Compare the pricing of ACME ($49/mo 5 seats, $99/mo 20 seats) and Rival ($39/mo unlimited). Which is cheaper for 10, 15, and 25 seats?"
    }],
    tools=[
        {
            "name": "calculate",
            "description": "Evaluate a mathematical expression",
            "input_schema": {
                "type": "object",
                "properties": {"expression": {"type": "string"}},
                "required": ["expression"],
            },
        }
    ],
)

for block in response.content:
    if block.type == "thinking":
        print(f"[thinking — {len(block.thinking)} chars]")
    elif block.type == "text":
        print(block.text)

When to force ReAct vs let the model decide #

💡
Use ReAct for tasks that need planning; direct tool-calling for tasks that don't Direct tool-calling (module 5.1's loop) is simpler, cheaper, and sufficient for single-step or shallow-planning tasks. Add ReAct (explicit thought steps or a thinking model) when: the task requires more than 2–3 sequential decisions, the right tool to call depends on intermediate results, the model needs to evaluate whether it has gathered enough information, or the task is novel/ambiguous enough that explicit reasoning reduces errors. For most production agents, start with direct tool-calling and add thinking/ReAct only when you observe planning failures in your logs.

Self-check #

What is the fundamental difference between a ReAct agent and a simple tool-calling loop? In what type of task does ReAct provide the most benefit?
A simple tool-calling loop gives the model tools and lets it call them — but provides no mechanism for the model to explicitly plan or reason about its approach before acting. The model sees the message history and generates its best guess at the next action. A ReAct agent adds an explicit Thought step between each Observation and next Action — the model generates a reasoning trace about what it knows and what to do next. This thought step is not a tool call; it's text the model writes to itself, reasoning through its situation. ReAct provides the most benefit for: (1) multi-step tasks where the right next action depends on what was learned in previous steps — "research competitors, then compare, then calculate" requires knowing what you found before deciding what to search next; (2) tasks with ambiguous or multiple valid approaches, where reasoning helps the model pick the better path; (3) tasks that may require abandoning a planned approach based on intermediate results. For simple single-step queries ("What's the weather in Berlin?"), ReAct adds overhead without benefit — direct tool calling is better.
Explain the role of the stop parameter in the prompt-based ReAct implementation. What would happen without it?
In prompt-based ReAct, the model generates text following the Thought/Action/Observation template. After it generates an Action, the model would — if not stopped — continue generating and hallucinate the Observation (the tool result) based on its training data rather than waiting for your code to run the actual tool. The stop=["Observation:"] parameter tells the API to stop generation the moment the model outputs the text "Observation:" — at that point, your code extracts the Action, executes the real tool, appends the actual result as "Observation: {result}", and then calls the API again. Without the stop token, the model fills in its own observations — this is catastrophic for factual tasks because the model makes up tool results rather than using real data. The stop token is the mechanism that enforces the "your code runs the tool" contract that makes ReAct useful.
Your agent is supposed to research a topic in 3 steps but sometimes loops 15+ times without reaching a Final Answer. What are the two most likely causes and how do you diagnose each?
Cause 1 — The agent can't find what it needs and keeps retrying: the tool keeps returning "No results found" but the agent keeps searching with slightly modified queries hoping to find something. Diagnose by reading the thought traces — if you see increasingly desperate search queries for the same information, the problem is the knowledge base or search tool has gaps. Fix: add a "I've searched N times without finding X — I'll answer with what I have" instruction in the system prompt, or improve the data source. Cause 2 — The agent is confused about what constitutes "enough information" to produce a Final Answer: no explicit stopping criterion in the system prompt, so the model continues searching even when it already has sufficient data. Diagnose by checking whether the information gathered in steps 1–3 would already be sufficient to answer — if yes, the system prompt needs a clearer stopping rule like "Once you have pricing data for all requested companies, immediately write your Final Answer." Always set max_steps=5–8 as a hard guard regardless, and log every step with the current thought and action to catch both patterns in production.
