Advanced RAG Patterns

The standard retrieve-then-generate pipeline handles most production RAG workloads well. But some problems — multi-hop questions requiring reasoning across multiple documents, corpora dominated by relationships between entities, questions that may require real-time data beyond the knowledge base — demand more sophisticated architectures. This module surveys five advanced patterns: Agentic RAG, GraphRAG, Corrective RAG (CRAG), Self-RAG, and late chunking with ColBERT. For each, you will understand when to reach for it, what it trades off, and how to implement the core idea.

~28 min read 1 video · 1 recording script 4 code examples Last updated 2026-05-07
Key Takeaways
  • Agentic RAG gives the LLM control over when and how to retrieve — it can issue multiple queries, evaluate retrieved context, and decide to search again if the answer is insufficient
  • GraphRAG builds a knowledge graph from the corpus and traverses entity relationships — it excels at questions like "How are X and Y connected?" that standard chunk-based retrieval cannot answer
  • Corrective RAG (CRAG) grades its own retrieved context and falls back to web search when the grader deems retrieval insufficient — it prevents "I don't know" failures on time-sensitive or niche topics
  • Self-RAG inserts special reflection tokens during generation — the model decides when to retrieve, evaluates whether it should, and rates the faithfulness of its own answer
  • ColBERT's late interaction computes per-token similarity rather than pooled vector similarity — it achieves higher retrieval precision on long documents without sacrificing index speed

When the standard pipeline fails #

The retrieve-once-then-generate pipeline covers ~80% of production RAG use cases well. The remaining 20% share specific characteristics that the standard pipeline cannot handle:

When to use advanced patterns
Multi-hop reasoning needed ("Compare X across Y and Z")
Agentic RAG
LLM controls when and what to retrieve — issues multiple targeted queries iteratively.
Relationship-heavy corpus ("Who reports to whom?")
GraphRAG
Knowledge graph traversal finds connected entities that chunk similarity can't surface.
Corpus may be outdated / incomplete
CRAG
Grades retrieved context — falls back to web search when local retrieval is insufficient.
Need fine-grained control over retrieve/generate decisions
Self-RAG
Model decides per-sentence whether to retrieve, uses reflection tokens to self-evaluate faithfulness.
Long documents, fine-grained passage matching
ColBERT / Late interaction
Per-token similarity with MaxSim operator — better recall than pooled bi-encoders on long passages.

Agentic RAG #

In agentic RAG the LLM controls the retrieval loop. Instead of "retrieve top-K once, then generate," the model receives retrieval as a tool — it can call it multiple times with different queries, evaluate whether the retrieved context is sufficient, and decide when it has enough information to generate a final answer. This is especially powerful for multi-hop questions that require gathering information from multiple parts of the corpus and reasoning across them.

pythonagentic_rag.py — LLM-controlled iterative retrieval via tool calling
import os, json
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# Simulated retrieval function (replace with your RAGPipeline.retrieve())
def search_knowledge_base(query: str) -> str:
    kb = {
        "internet": "Employees may claim $150/mo for home internet.",
        "furniture": "One-time $600 allowance for ergonomic furniture.",
        "deadline":  "Claims must be submitted via Workday by the 5th of each month.",
        "approval":  "Manager approval required for claims over $200.",
    }
    results = [v for k, v in kb.items() if k in query.lower()]
    return "\n".join(results) if results else "No results found for this query."

TOOLS = [{
    "type": "function",
    "function": {
        "name": "search_knowledge_base",
        "description": "Search the company knowledge base for relevant policy information.",
        "parameters": {
            "type": "object",
            "properties": {"query": {"type": "string", "description": "Search query"}},
            "required": ["query"],
        },
    },
}]

def agentic_rag(user_question: str, max_iterations: int = 5) -> str:
    messages = [
        {"role": "system", "content": (
            "You are a policy assistant. Use the search_knowledge_base tool to find "
            "relevant information. You may search multiple times with different queries. "
            "When you have enough information, provide a complete answer citing sources."
        )},
        {"role": "user", "content": user_question},
    ]

    for _ in range(max_iterations):
        resp = client.chat.completions.create(
            model="claude-sonnet-4-6", messages=messages, tools=TOOLS, tool_choice="auto"
        )
        msg = resp.choices[0].message

        if msg.tool_calls:
            messages.append(msg)
            for call in msg.tool_calls:
                args = json.loads(call.function.arguments)
                result = search_knowledge_base(args["query"])
                print(f"🔍 Searched: '{args['query']}' → {result[:60]}…")
                messages.append({
                    "role": "tool",
                    "tool_call_id": call.id,
                    "content": result,
                })
        else:
            return msg.content  # LLM decided it has enough — final answer

    return "Max iterations reached."

question = "What are the expense limits for both internet and equipment, and by when must I submit?"
answer = agentic_rag(question)
print(f"\n✅ {answer}")
⚠
Set a max_iterations guard on agentic retrieval Without a hard iteration limit, a confused or poorly-prompted agent can enter an infinite retrieval loop — issuing nearly identical queries, never finding enough context to generate, and burning API budget. Always set max_iterations=4–6. Log every tool call in production — retrieval loops are the most common cost spike in agentic systems. If you hit the limit often, the root cause is usually a poorly-scoped knowledge base (the answer genuinely isn't there) or a vague system prompt that doesn't tell the agent when it has "enough" context.

GraphRAG #

GraphRAG, introduced by Microsoft Research in 2024, builds a knowledge graph from the corpus — entities (people, organisations, policies, concepts) as nodes, relationships (reports to, part of, governs) as edges. Retrieval traverses the graph rather than searching chunk vectors. It excels at questions that require following relationship chains: "Who are the key people in the onboarding process and how do their roles connect?"

ℹ
GraphRAG has two retrieval modes Local search: start from the entities mentioned in the query, traverse their immediate neighbours in the graph, gather related community summaries. Best for specific entity questions. Global search: use community summaries computed over the entire graph to answer high-level questions ("What are the main themes in this corpus?"). Standard chunk-based RAG cannot answer global questions without reading the entire corpus. GraphRAG's community summaries compress that corpus knowledge into a hierarchical structure.
pythongraphrag_sketch.py — entity extraction + graph traversal (simplified)
import os, json, networkx as nx
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# ── Step 1: Extract entities and relationships from chunks ─────────────
chunks = [
    "Sarah Chen is the VP of Engineering and manages the platform team.",
    "The platform team owns the API gateway and authentication service.",
    "James Park is a Staff Engineer on the platform team, reporting to Sarah Chen.",
    "The authentication service uses OAuth 2.0 and is governed by the security policy.",
]

def extract_entities_and_relations(chunk: str) -> dict:
    resp = client.chat.completions.create(
        model="claude-haiku-4-5-20251001",
        messages=[{"role": "user", "content": (
            f"Extract entities and relationships from this text as JSON:\n{chunk}\n"
            f"Format: {{\"entities\": [\"name\"], \"relations\": [[\"from\",\"relation\",\"to\"]]}}"
        )}],
        max_tokens=300,
    )
    return json.loads(resp.choices[0].message.content)

# Build knowledge graph
G = nx.DiGraph()
for chunk in chunks:
    extracted = extract_entities_and_relations(chunk)
    for entity in extracted.get("entities", []):
        G.add_node(entity, chunk_source=chunk[:50])
    for src, rel, tgt in extracted.get("relations", []):
        G.add_edge(src, tgt, relation=rel)

# ── Step 2: Graph traversal retrieval ────────────────────────────────
def graph_retrieve(query_entity: str, depth: int = 2) -> str:
    """Return all neighbours within `depth` hops of query_entity."""
    if query_entity not in G:
        return f"Entity '{query_entity}' not found in knowledge graph."

    neighbours = nx.ego_graph(G, query_entity, radius=depth)
    context_parts = []
    for u, v, data in neighbours.edges(data=True):
        context_parts.append(f"{u} --[{data.get('relation','related')}]--> {v}")
    return "\n".join(context_parts)

# "Who is on the platform team and what do they manage?"
context = graph_retrieve("platform team", depth=2)
print(context)
# → platform team --[manages]--> API gateway
# → platform team --[managed by]--> Sarah Chen
# → James Park --[reports to]--> Sarah Chen

Corrective RAG (CRAG) #

CRAG adds a retrieval grader between retrieval and generation. The grader — typically a lightweight LLM judge or a trained classifier — evaluates whether the retrieved context is relevant enough to answer the question. If the grade is "insufficient," CRAG triggers a fallback: either web search (for real-time data) or a broader corpus query (for niche topics). This prevents the LLM from either hallucinating or refusing when local retrieval fails.

pythoncrag.py — retrieval grader with web-search fallback
import os, json
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

def grade_retrieval(question: str, contexts: list[str]) -> str:
    """Grade retrieved context relevance: 'sufficient' | 'partial' | 'insufficient'."""
    resp = client.chat.completions.create(
        model="claude-haiku-4-5-20251001",
        messages=[{"role": "user", "content": (
            f"Question: {question}\n\n"
            f"Retrieved context:\n{chr(10).join(contexts)}\n\n"
            f"Does the context contain enough information to answer the question? "
            f"Reply with exactly one word: 'sufficient', 'partial', or 'insufficient'."
        )}],
        max_tokens=10,
    )
    grade = resp.choices[0].message.content.strip().lower()
    return grade if grade in {"sufficient", "partial", "insufficient"} else "partial"

def web_search_fallback(query: str) -> str:
    """Stub — replace with Tavily, Serper, or Bing Search API."""
    return f"[Web search result for '{query}'] — integrate a search API here."

def crag_query(question: str, rag_contexts: list[str]) -> str:
    grade = grade_retrieval(question, rag_contexts)
    print(f"Retrieval grade: {grade}")

    if grade == "insufficient":
        # Complete fallback: use web search, discard local context
        rag_contexts = [web_search_fallback(question)]
    elif grade == "partial":
        # Supplement: keep local context AND add web search
        rag_contexts = rag_contexts + [web_search_fallback(question)]

    context_text = "\n".join(rag_contexts)
    resp = client.chat.completions.create(
        model="claude-sonnet-4-6",
        messages=[
            {"role": "system", "content": "Answer only from the provided context. Cite sources."},
            {"role": "user",   "content": f"CONTEXT:\n{context_text}\n\nQUESTION: {question}"},
        ],
        max_tokens=400,
    )
    return resp.choices[0].message.content

answer = crag_query(
    question="What is the current USD/EUR exchange rate?",  # not in local corpus
    rag_contexts=["Our expense policy covers home office equipment."],
)
print(answer)

Self-RAG #

Self-RAG (Asai et al., 2023) is a fine-tuned model that generates four types of special reflection tokens during output: [Retrieve] (should I retrieve?), [ISREL] (is this retrieved passage relevant?), [ISSUP] (does my generated segment align with the passage?), and [ISUSE] (is my response useful?). These tokens let the model decide on-the-fly whether retrieval is needed for each segment, and self-evaluate faithfulness. This is a research-track approach — the model must be fine-tuned to generate these tokens, which is significant infrastructure overhead. In production, teams approximate Self-RAG behaviour by adding explicit reflection steps in their agentic RAG prompt.

ℹ
Self-RAG in practice: prompt-based approximation The original Self-RAG requires model fine-tuning. A practical approximation for production use: after generating a draft answer with your standard RAG pipeline, send a second LLM call asking "Does this answer contain any claims that are not directly supported by the provided context? List any unsupported claims." If the response lists unsupported claims, either regenerate with a stricter system prompt or flag the answer for human review. This adds one LLM call per query but achieves a similar hallucination-detection effect without fine-tuning.

Late interaction: ColBERT #

Standard bi-encoders pool all token representations into one vector per passage — losing per-token information. ColBERT keeps all token representations, then at query time computes the similarity between each query token and the most similar passage token (MaxSim operator). The final score is the sum of MaxSim scores across all query tokens. This late interaction achieves higher retrieval precision on long passages and handles exact-match terms better than pooled encoders.

pythoncolbert_example.py — late interaction retrieval with RAGatouille
from ragatouille import RAGPretrainedModel

# RAGatouille wraps ColBERT with a simple API
RAG = RAGPretrainedModel.from_pretrained("colbert-ir/colbertv2.0")

documents = [
    "Employees may claim $150/mo for home internet connectivity.",
    "A one-time $600 allowance is available for ergonomic furniture.",
    "All claims submitted via Workday by the 5th of each month.",
    "The EBITDA adjustment for Q3 is detailed in the finance report.",
    "AcmeCorp was founded in 2018 in Austin, Texas.",
]

# Index — computes per-token embeddings for every document
index_path = RAG.index(
    collection=documents,
    index_name="hr_policy_colbert",
    max_document_length=256,
    split_documents=False,
)

# Query — MaxSim across all query tokens vs all document tokens
results = RAG.search(query="EBITDA Q3 adjustment", k=3)
for r in results:
    print(f"score={r['score']:.2f}  {r['content']}")
# → EBITDA chunk scores highest — exact token match with MaxSim

# ── When to use ColBERT ───────────────────────────────────────────────
# + Long documents where pooling loses important signal
# + Corpora with technical jargon and exact-match requirements
# + When you can afford 2-3x the index storage (token vectors vs one vector)
# - Not ideal for multilingual or very large corpora (index size)
# - Requires separate serving infrastructure (PLAID search)

Choosing the right pattern #

Advanced patterns add complexity, cost, and latency. Use your RAGAS evaluation results (module 4.8) to decide whether a standard pipeline failure justifies the upgrade. The rule of thumb: exhaust simple fixes first — chunking, embedding model, reranker, retrieve_k — before reaching for GraphRAG or Self-RAG.

Self-check #

What is the key architectural difference between standard RAG and agentic RAG, and what type of question benefits most from agentic retrieval?
Standard RAG retrieves once — a single query produces a fixed context, and the LLM generates from that context. The LLM has no influence over what was retrieved. Agentic RAG gives the LLM control over the retrieval loop: it decides when to retrieve, what query to issue, whether the retrieved context is sufficient, and whether to retrieve again. The LLM is a planner that uses retrieval as a tool. Questions that benefit most are multi-hop questions requiring information from multiple sources ("Compare the expense policies for internet, equipment, and software — and which requires manager approval?"), questions where the right sub-query is not obvious from the original phrasing, and analytical questions where the LLM needs to gather evidence from several corpus locations before reasoning to a conclusion.
Explain the MaxSim operator in ColBERT and how it differs from the pooled cosine similarity used by standard bi-encoders.
Standard bi-encoders produce one fixed-length vector per passage by pooling (averaging or using the CLS token) all token representations. At query time, cosine similarity compares the query pool vector against the passage pool vector — a single score based on averaged representations. ColBERT produces one vector per token — for a 50-token passage, that is 50 vectors. At query time, the MaxSim operator computes, for each query token, the maximum cosine similarity against any passage token. The final score is the sum of these per-query-token max scores. This allows ColBERT to give credit for exact token matches ("EBITDA" in the query matches "EBITDA" in the passage) even when the overall semantic neighbourhood of the passage is different. Pooled bi-encoders lose this per-token information — a passage about general finance may pool to a similar vector as a passage mentioning EBITDA specifically, because both are "financial documents" at the pooled level.
Your RAG system's corpus covers company HR policy but users sometimes ask about real-time data (current exchange rates, live stock prices). Which advanced pattern solves this and how does it work?
CRAG (Corrective RAG) solves this. CRAG adds a retrieval grader between retrieval and generation. After the standard pipeline retrieves HR policy chunks, a lightweight LLM judge evaluates whether those chunks are sufficient to answer the question. For "What is the current USD/EUR exchange rate?" — the HR corpus has nothing relevant — the grader returns "insufficient." CRAG then triggers a web search fallback (Tavily, Serper, Bing API), retrieves live exchange rate data from the web, and substitutes that as the context. For "partial" relevance (some context exists locally but may be incomplete), CRAG merges local chunks and web search results. The LLM then generates from the combined context. This prevents both hallucination (generating a rate from training data) and unhelpful refusal (saying "I don't know" when the answer is readily available via web search).
When should you NOT use GraphRAG, and what simpler alternative should you try first?
Do not use GraphRAG when: (1) Your corpus is primarily factual prose (policy documents, FAQs, product descriptions) with few explicit entity relationships — GraphRAG adds graph extraction complexity but the relationships it finds won't improve retrieval meaningfully. (2) Your eval shows chunking and embedding quality as the bottleneck (low context recall in RAGAS) — GraphRAG won't fix these. (3) You are resource-constrained — GraphRAG requires LLM calls to extract entities from every chunk, significant compute for graph construction, and a separate graph database. (4) Your questions are not relationship-traversal questions ("How are X and Y connected?"). Try first: hybrid search for exact-match improvements, parent-child chunks for context richness, multi-query retrieval for complex questions. GraphRAG has a significant implementation cost and is justified only when your eval shows that relationship traversal is the specific retrieval gap — not as a general "advanced" upgrade.
