Three-Layer Guardrail Architecture

A single guardrail layer is never enough — not because any one layer is weak, but because each failure mode requires a different response time and trust level. Input guardrails must be deterministic and sub-millisecond. Output guardrails can afford an LLM call. Action guardrails must be pure functions that can't be bypassed by a compromised agent. This module builds all three layers from scratch and explains why each constraint exists.

~20 min read 4 code examples Last updated 2026-05-07
Key Takeaways
  • Layer 1 — Input Guardrails: gateway position, sub-1ms, always deterministic code — regex/blocklist/ML classifier. Never an LLM call. Blocks prompt injection, PII, out-of-domain, toxic input
  • Layer 2 — Output Guardrails: post-generation position, can use an LLM judge. Checks faithfulness to source, contradiction with policy, required disclaimers, hallucinated citations
  • Layer 3 — Action Guardrails: inside tool functions, pure and side-effect-limited. Caps retries, caps tool call count, validates query params, enforces read-only DB access, limits top_k
  • Each layer is independent: a bypassed Layer 1 is caught by Layer 2. A jailbroken LLM output is stopped by Layer 3 before it touches a real resource
  • Hard-fail to a safe fallback: when a guardrail fires, the user gets a canned safe response — not an error stack trace, not silence

Layer 1: Input Guardrails #

Input guardrails run before the LLM sees anything. Their job is to reject or sanitise inputs that would cause harm regardless of how the LLM processes them. Because they run on every request they must be fast — budget under 1 ms for regex/blocklist checks, under 20 ms for a lightweight ML classifier. Never call an LLM here.

pythoninput_guardrail.py — deterministic gateway layer
import re
from dataclasses import dataclass
from enum import Enum

class GuardResult(Enum):
    PASS = "pass"
    BLOCK = "block"
    REDACT = "redact"

@dataclass
class InputCheckResult:
    result: GuardResult
    reason: str
    sanitised_input: str | None = None

# ── Prompt injection detection ─────────────────────────────────────────
INJECTION_PATTERNS = [
    r"ignore previous instructions",
    r"disregard (your|all|the above)",
    r"you are now (a|an|my)",
    r"act as (a|an|if)",
    r"forget (everything|your instructions|what)",
    r"new instruction:",
    r"system prompt:",
    r"</?(system|instructions?)>",   # XML injection attempt
]
_INJECTION_RE = re.compile(
    "|".join(INJECTION_PATTERNS), re.IGNORECASE
)

# ── PII patterns — redact, don't block ────────────────────────────────
PII_PATTERNS = {
    "email":   (re.compile(r"\b[\w.+-]+@[\w-]+\.[a-z]{2,}\b", re.I), "[EMAIL]"),
    "ssn":     (re.compile(r"\b\d{3}-\d{2}-\d{4}\b"),              "[SSN]"),
    "credit":  (re.compile(r"\b(?:\d[ -]?){13,16}\b"),            "[CARD]"),
    "phone":   (re.compile(r"\b\+?1?\s?\(?\d{3}\)?[\s.-]\d{3}[\s.-]\d{4}\b"), "[PHONE]"),
}

# ── Out-of-domain blocklist ────────────────────────────────────────────
OUT_OF_DOMAIN_KEYWORDS = frozenset([
    "bomb", "weapon", "synthesise drugs", "hack into", "malware",
])

def check_input(user_input: str) -> InputCheckResult:
    lower = user_input.lower()

    # 1. Prompt injection — hard block
    if _INJECTION_RE.search(user_input):
        return InputCheckResult(GuardResult.BLOCK, "prompt_injection")

    # 2. Out-of-domain — hard block
    if any(kw in lower for kw in OUT_OF_DOMAIN_KEYWORDS):
        return InputCheckResult(GuardResult.BLOCK, "out_of_domain")

    # 3. PII — redact and continue
    sanitised = user_input
    pii_found = []
    for pii_type, (pattern, replacement) in PII_PATTERNS.items():
        new, n = pattern.subn(replacement, sanitised)
        if n > 0:
            sanitised = new
            pii_found.append(pii_type)

    if pii_found:
        return InputCheckResult(GuardResult.REDACT, f"pii:{','.join(pii_found)}", sanitised)

    return InputCheckResult(GuardResult.PASS, "ok", user_input)
⚠
Never use an LLM for Layer 1 An LLM-based input classifier adds 300–800 ms of latency, costs tokens, and — crucially — can itself be prompt-injected. The attacker controls the input you're sending to the classifier. Layer 1 must be deterministic code only.

Layer 2: Output Guardrails #

Output guardrails run after the LLM generates a response but before it's returned to the user. Because the LLM has already paid the latency cost, an extra LLM call here (used as a judge) is often acceptable — but keep it small, and use structured output so you get a boolean back, not prose.

pythonoutput_guardrail.py — LLM judge + hard-fail fallback
from pydantic import BaseModel

SAFE_FALLBACK = (
    "I'm sorry, I wasn't able to generate a safe response for that request. "
    "Please rephrase your question or contact support."
)

class OutputCheck(BaseModel):
    safe: bool
    reason: str     # short label for logging: "faithfulness", "disclaimer", etc.
    confidence: float

JUDGE_SYSTEM = """You are a safety classifier for an AI assistant.
Given a [SOURCE] (retrieved context or policy) and an [OUTPUT] (assistant reply):
- safe=false if the output contradicts the source
- safe=false if the output makes medical/legal/financial claims without appropriate disclaimers
- safe=false if the output includes harmful, offensive, or dangerous content
- safe=true otherwise
Be strict. Return JSON matching the OutputCheck schema."""

judge_llm = llm.with_structured_output(OutputCheck)

def check_output(source: str, assistant_output: str) -> str:
    """Return the output if safe, or SAFE_FALLBACK if not."""
    result: OutputCheck = judge_llm.invoke([
        {"role": "system", "content": JUDGE_SYSTEM},
        {"role": "user",   "content": f"[SOURCE]\n{source}\n\n[OUTPUT]\n{assistant_output}"},
    ])
    if result.safe and result.confidence > 0.7:
        return assistant_output

    # Log the failure before replacing
    log_warning("OutputGuardrail", "check_output",
                f"blocked: reason={result.reason} confidence={result.confidence:.2f}")
    return SAFE_FALLBACK

# ── Disclaimer injection (deterministic, no LLM needed) ───────────────
DISCLAIMER_TRIGGERS = {
    lambda t: any(w in t for w in ["diagnose", "treat", "medication", "symptom"]):
        "\n\n*This is not medical advice. Consult a qualified healthcare professional.*",
    lambda t: any(w in t for w in ["invest", "stock", "portfolio", "financial"]):
        "\n\n*This is not financial advice. Consult a licensed financial advisor.*",
}

def append_disclaimers(text: str) -> str:
    lower = text.lower()
    for trigger, disclaimer in DISCLAIMER_TRIGGERS.items():
        if trigger(lower) and disclaimer.strip() not in text:
            text += disclaimer
    return text

Layer 3: Action Guardrails #

Action guardrails live inside tool functions — they cap what the agent can actually do in the world, regardless of what the LLM asked it to do. They're pure functions with no LLM involvement. They're the last line of defence against a jailbroken model or a logic bug that generates a destructive tool call.

pythonaction_guardrail.py — caps, validators, read-only enforcement
from functools import wraps

# ── Decorator: cap total tool calls per request ────────────────────────
_call_counts: dict[str, int] = {}   # keyed by request_id
MAX_TOOL_CALLS = 20

def tool_call_limit(fn):
    @wraps(fn)
    def wrapper(*args, request_id: str = "default", **kwargs):
        _call_counts[request_id] = _call_counts.get(request_id, 0) + 1
        if _call_counts[request_id] > MAX_TOOL_CALLS:
            raise RuntimeError(f"Tool call budget exceeded ({MAX_TOOL_CALLS}) for request {request_id}")
        return fn(*args, **kwargs)
    return wrapper

# ── SQL query validator — block destructive statements ────────────────
DANGEROUS_SQL = re.compile(
    r"\b(DROP|DELETE|TRUNCATE|UPDATE|INSERT|ALTER|CREATE|GRANT|REVOKE)\b",
    re.IGNORECASE
)

def validate_read_only_sql(query: str) -> str:
    if DANGEROUS_SQL.search(query):
        raise ValueError(f"Non-SELECT SQL blocked by action guardrail: {query[:80]}")
    return query

# ── top_k cap for vector search ───────────────────────────────────────
MAX_RETRIEVAL_K = 20

@tool
def search_knowledge_base(query: str, top_k: int = 5) -> list[str]:
    """Search the knowledge base."""
    top_k = min(top_k, MAX_RETRIEVAL_K)      # cap silently — don't error
    return vectorstore.similarity_search(query, k=top_k)

# ── Retry cap — prevent runaway tool retries ──────────────────────────
def with_retry_cap(fn, max_retries: int = 3):
    def wrapper(*args, **kwargs):
        for attempt in range(max_retries):
            try:
                return fn(*args, **kwargs)
            except Exception as e:
                if attempt == max_retries - 1:
                    raise   # re-raise on final attempt
    return wrapper

Composing all three layers #

In a LangGraph agent the three layers plug in at different points in the graph. Input guardrails run at the entry node. Output guardrails run at the output node before returning to the user. Action guardrails are baked into each tool function.

pythonguarded_agent.py — three layers in a LangGraph
def input_guard_node(state: AgentState) -> AgentState:
    last_user = [m for m in state["messages"] if m["role"] == "user"][-1]
    result = check_input(last_user["content"])

    if result.result == GuardResult.BLOCK:
        return {"blocked": True, "block_reason": result.reason}
    if result.result == GuardResult.REDACT:
        # Replace last user message with sanitised version
        new_messages = state["messages"][:-1] + [{"role": "user", "content": result.sanitised_input}]
        return {"messages": new_messages, "blocked": False}
    return {"blocked": False}

def output_guard_node(state: AgentState) -> AgentState:
    last_ai = state["messages"][-1].content
    source  = state.get("retrieved_context", "")
    safe_output = check_output(source, last_ai)
    safe_output = append_disclaimers(safe_output)
    if safe_output != last_ai:
        return {"messages": [{"role": "assistant", "content": safe_output}]}
    return {}

def route_after_input_guard(state: AgentState) -> str:
    return "blocked" if state.get("blocked") else "agent"

builder = StateGraph(AgentState)
builder.add_node("input_guard", input_guard_node)
builder.add_node("agent", agent_node)          # tools are already Layer 3 guarded
builder.add_node("output_guard", output_guard_node)
builder.add_node("blocked_response", lambda s: {"final": SAFE_FALLBACK})
builder.add_edge(START, "input_guard")
builder.add_conditional_edges("input_guard", route_after_input_guard)
builder.add_edge("agent", "output_guard")
builder.add_edge("output_guard", END)
builder.add_edge("blocked_response", END)
✦
Defence in depth — not defence by a single layer The goal is that no single layer failure causes a user-visible harm. Layer 1 stops 95% of threats cheaply. Layer 2 catches what Layer 1 misses. Layer 3 limits blast radius even if Layers 1 and 2 both fail. Each layer is simple and testable in isolation.
