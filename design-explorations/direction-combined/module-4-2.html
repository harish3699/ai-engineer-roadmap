Chunking Strategies

Before a document can be retrieved, it must be split into chunks and embedded. How you split it — where you cut, how large each piece is, and how much you overlap — determines whether the right information will score high enough in retrieval to make it into the LLM's context. A perfect embedding model and a perfect vector database cannot rescue a bad chunking strategy. This module covers the four main approaches and the heuristics for choosing between them.

~25 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • Chunk size controls a fundamental tradeoff: small chunks retrieve precisely but may miss surrounding context; large chunks carry more context but dilute the relevant signal with noise
  • Fixed-size chunking is the fastest to implement and works well enough for homogeneous, well-structured text — start here before reaching for more complex strategies
  • Overlap (typically 10–20% of chunk size) prevents answers from being split across chunk boundaries; without it you will get retrieval misses on information that spans the cut point
  • Semantic chunking splits on topic shifts detected by embedding similarity — it produces the most coherent chunks but is the most expensive to compute at index time
  • Different document types need different strategies: Markdown → split on headers; code → split on function/class boundaries; conversations → split on speaker turns; PDFs → handle page breaks explicitly

The chunking tradeoff #

Every chunking decision is a tradeoff between two retrieval properties. Precision: when a chunk is retrieved, how much of it is actually relevant to the query? Recall: does the relevant information exist somewhere in the retrieved set at all?

Chunk size vs precision/recall
Small chunks (100–200 tokens)
  • ✓ High precision — dense signal
  • ✓ Many candidate chunks to retrieve
  • ✗ Low recall — answer may span chunks
  • ✗ Loses surrounding context
  • ✗ More chunks = more index storage
Large chunks (800–1500 tokens)
  • ✓ High recall — answer likely inside
  • ✓ Preserves local context well
  • ✗ Low precision — noisy signal
  • ✗ Fewer chunks retrieved (top-K)
  • ✗ More context tokens per call → cost
Starting point: 256–512 tokens with 10–15% overlap. Measure with RAGAS (module 4.8), then tune.

Strategy 1: Fixed-size chunking #

Split the document every N characters (or tokens), with an overlap of M characters between consecutive chunks. It is the simplest strategy, the fastest to implement, and surprisingly effective on homogeneous, well-structured text like knowledge base articles or FAQ pages.

pythonfixed_chunking.py — character splitter with overlap
def fixed_chunk(text: str, chunk_size: int = 500, overlap: int = 50) -> list[str]:
    """Split text into chunks of ~chunk_size chars with overlap."""
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunk = text[start:end]
        if chunk.strip():           # skip empty chunks
            chunks.append(chunk)
        start += chunk_size - overlap  # slide forward by (size - overlap)
    return chunks

# ── Problem: cuts mid-sentence ────────────────────────────────────────
sample = """The remote work policy allows employees to claim up to $150 per month
for home internet. Additionally, a one-time $600 allowance is available for
ergonomic equipment. Claims must be submitted via Workday by the 5th of the
following month. All claims over $200 require manager approval."""

chunks = fixed_chunk(sample, chunk_size=120, overlap=20)
for i, c in enumerate(chunks):
    print(f"[{i}] {c!r}\n")
# → You'll see cuts like "…$150 per month\nfor home" split from "internet."
#   Overlap prevents total loss but mid-sentence cuts hurt embedding quality.

# ── Token-aware variant (avoids sub-word cuts) ────────────────────────
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")

def token_chunk(text: str, chunk_tokens: int = 256, overlap_tokens: int = 32) -> list[str]:
    ids = enc.encode(text)
    chunks, start = [], 0
    while start < len(ids):
        end = start + chunk_tokens
        chunks.append(enc.decode(ids[start:end]))
        start += chunk_tokens - overlap_tokens
    return chunks
⚠
Always chunk by tokens, not characters, when embedding Embedding models have token limits (typically 512 tokens). A 2,000-character chunk might be 400 tokens for English prose but 700+ tokens for dense technical text. Chunking by characters risks silently truncating chunks at the embedding model's limit. Use tiktoken or your embedding model's tokenizer to size chunks accurately.

Strategy 2: Sentence-aware chunking #

Split at sentence boundaries instead of arbitrary character positions. Each chunk contains complete sentences, so embedding quality improves — the model never sees a half-sentence. Group sentences into target-size chunks and overlap by one or two sentences.

pythonsentence_chunking.py — NLTK sentence splitter with sentence overlap
import nltk
nltk.download("punkt", quiet=True)
from nltk.tokenize import sent_tokenize
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")

def sentence_chunk(text: str, max_tokens: int = 256, overlap_sentences: int = 1) -> list[str]:
    sentences = sent_tokenize(text)
    chunks, current, current_tokens = [], [], 0

    for sent in sentences:
        sent_tokens = len(enc.encode(sent))
        if current_tokens + sent_tokens > max_tokens and current:
            chunks.append(" ".join(current))
            # carry over overlap_sentences sentences into the next chunk
            current = current[-overlap_sentences:] if overlap_sentences else []
            current_tokens = sum(len(enc.encode(s)) for s in current)
        current.append(sent)
        current_tokens += sent_tokens

    if current:
        chunks.append(" ".join(current))
    return chunks

sample_long = """AcmeCorp's remote work policy covers three categories of expense.
First, employees may claim up to $150 per month for home internet connectivity.
This allowance applies to primary residence only. Second, a one-time $600 equipment
allowance is available for ergonomic furniture. Approved items include standing desks
and ergonomic chairs. Third, software subscriptions directly related to work may be
expensed up to $50 per month with manager pre-approval. All claims are submitted
through Workday. The deadline is the 5th of the following month."""

for i, chunk in enumerate(sentence_chunk(sample_long, max_tokens=80, overlap_sentences=1)):
    print(f"[{i}] {chunk}\n")

Strategy 3: Recursive character splitting #

The recursive splitter tries a hierarchy of separators in order: double newlines (\n\n), single newlines (\n), sentences, words, characters. It only falls back to the next separator when the current chunk still exceeds the token limit. This produces chunks that respect natural document structure — paragraphs stay together when they fit, sections stay together when paragraphs don't.

pythonrecursive_chunking.py — hierarchical separator splitting
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")

def token_len(text: str) -> int:
    return len(enc.encode(text))

def recursive_chunk(
    text: str,
    max_tokens: int = 256,
    overlap: int = 30,
    separators: list[str] = ["\n\n", "\n", ". ", " ", ""],
) -> list[str]:
    """Split using the first separator that produces chunks under max_tokens."""
    if token_len(text) <= max_tokens:
        return [text]  # already small enough

    for sep in separators:
        parts = text.split(sep) if sep else list(text)
        if len(parts) == 1:
            continue   # separator not found — try next

        chunks, current = [], ""
        for part in parts:
            candidate = (current + sep + part).strip() if current else part
            if token_len(candidate) > max_tokens and current:
                chunks.extend(recursive_chunk(current.strip(), max_tokens, overlap, separators))
                # overlap: carry tail of current into next chunk
                tail = current[-overlap:] if overlap else ""
                current = tail + sep + part
            else:
                current = candidate
        if current.strip():
            chunks.extend(recursive_chunk(current.strip(), max_tokens, overlap, separators))
        return chunks

    return [text]   # fallback — can't split further

# ── Markdown-aware variant — split on headers first ───────────────────
MARKDOWN_SEPARATORS = ["\n## ", "\n### ", "\n\n", "\n", ". ", " "]

Strategy 4: Semantic chunking #

Semantic chunking embeds every sentence, then finds points where consecutive sentences drop sharply in cosine similarity — topic boundaries. The algorithm splits at those drops, producing chunks that each cover one coherent topic. It produces the highest-quality chunks but costs one embedding call per sentence at index time.

pythonsemantic_chunking.py — split on embedding similarity drops
import numpy as np
import nltk
from sentence_transformers import SentenceTransformer

nltk.download("punkt", quiet=True)
from nltk.tokenize import sent_tokenize

embedder = SentenceTransformer("all-MiniLM-L6-v2")

def semantic_chunk(text: str, threshold: float = 0.3) -> list[str]:
    """Split text where consecutive sentence similarity drops below threshold."""
    sentences = sent_tokenize(text)
    if len(sentences) <= 1:
        return sentences

    # embed all sentences in one batched call
    embeddings = embedder.encode(sentences, normalize_embeddings=True)

    # cosine similarity between consecutive sentences
    similarities = [
        float(embeddings[i] @ embeddings[i + 1])
        for i in range(len(embeddings) - 1)
    ]

    # split where similarity drops below threshold — topic boundary
    chunks, current_chunk = [], [sentences[0]]
    for i, sim in enumerate(similarities):
        if sim < threshold:
            chunks.append(" ".join(current_chunk))
            current_chunk = []
        current_chunk.append(sentences[i + 1])
    chunks.append(" ".join(current_chunk))
    return chunks

multi_topic = """AcmeCorp was founded in 2018 and is headquartered in Austin, Texas.
The company employs 400 people across 12 countries. Revenue in 2024 was $42M.

Employees may claim up to $150 per month for home internet expenses.
Equipment allowances cover one-time purchases up to $600 for ergonomic furniture.
All claims must be submitted via Workday by the 5th of each month.

The engineering team uses two-week sprint cycles with Friday retrospectives.
Code reviews are mandatory for all pull requests before merging to main.
The staging environment is refreshed from production data every Sunday night."""

for i, chunk in enumerate(semantic_chunk(multi_topic, threshold=0.4)):
    print(f"[{i}] {chunk[:80]}…\n")
# → [0] Company background (3 sentences)
# → [1] Expense policy (3 sentences)
# → [2] Engineering practices (3 sentences)

Choosing a strategy #

The strategies are not mutually exclusive — production systems often chain them: recursive splitting for structure, then sentence-level overlap for boundary handling. The decision should be driven by document type and how you'll measure quality.

Strategy selection by document type
Markdown / docs site
Recursive
Use header separators (\n## , \n### ) first — each section becomes one or more chunks.
FAQ / knowledge base
Fixed-size
Homogeneous short articles — fixed chunks at 256–512 tokens work well. Fast to compute.
Legal / contracts
Sentence
Clause-level precision needed. Sentence chunking preserves full clauses. Larger overlap (2–3 sentences).
Research papers / books
Semantic
Long documents with topic shifts. Semantic chunking aligns chunk boundaries with concept changes.
Source code
AST-based
Split on function and class boundaries using the AST module. Never cut mid-function.
Chat / transcripts
Speaker turns
Split on speaker changes. Group N consecutive turns per chunk. Keep timestamps as metadata.
💡
Store metadata alongside every chunk The text of a chunk is not enough. Store at minimum: the source document filename, page number or section header, chunk index, and the original character offset. This enables source attribution in your answers ("From: HR Policy v3.2, Section 2") and lets you filter retrieval by document type, date, or department — critical for multi-tenant RAG systems.

Self-check #

You have a 50-page PDF legal contract with many multi-sentence clauses. A user asks about the liability cap clause. Why might fixed-size chunking at 200 tokens fail to retrieve the relevant clause?
Legal clauses are semantically cohesive units — they may span 3–8 sentences and 100–400 tokens. A fixed-size splitter at 200 tokens with no regard for clause boundaries will cut through the middle of the liability cap clause, splitting the condition from the cap amount into separate chunks. The embedding of the first half-clause won't score as highly against the query "liability cap" because it lacks the defining term. The second half-chunk will have the cap amount but not the clause label. Solution: use sentence-aware chunking with a larger target size (400–500 tokens) and 2–3 sentence overlap to ensure clause boundaries are respected and clause text stays together.
Explain why overlap is necessary and what would happen in a RAG system without any overlap.
Without overlap, the answer to a question may straddle two chunk boundaries — part of the relevant information is at the end of chunk N and part is at the start of chunk N+1. The embedding of chunk N doesn't capture the concept well because it's cut mid-thought; the embedding of chunk N+1 is similarly decontextualised. When the user asks about that concept, neither chunk scores high enough to be retrieved, and the model has to answer without the information. Overlap ensures that content near a boundary appears in both the preceding and following chunk, so it has multiple chances to be retrieved. A 15–20% overlap (e.g. 40–50 tokens of overlap on a 256-token chunk) typically captures this without excessive duplication.
In semantic chunking, what does the threshold parameter control and what happens if you set it too high vs too low?
The threshold is the cosine similarity value below which consecutive sentences are considered to have changed topic. Too high (e.g. 0.8): splits too aggressively — even minor style variation between sentences in the same paragraph triggers a split. You get very small, decontextualised chunks that lack the surrounding information needed to answer questions. Too low (e.g. 0.1): almost never splits — genuinely different topics are merged into one large chunk. The embedding averages over multiple topics and retrieval precision drops. Practical starting point: 0.3–0.5. Run your eval set (module 4.8) with a few threshold values and pick the one that maximises context precision and recall together.
What metadata should you store alongside each chunk, and why does it matter for production RAG?
At minimum: (1) source filename / document ID — enables source attribution in answers and lets users verify claims; (2) section or page number — lets users navigate to the exact location; (3) chunk index within document — enables parent-chunk retrieval (fetch surrounding chunks when this one is retrieved); (4) document metadata (date, author, department, access level) — enables pre-retrieval filtering so a finance team query doesn't retrieve HR-confidential documents. In multi-tenant or enterprise deployments, metadata filtering is how you enforce data access controls — the vector similarity search alone cannot distinguish "documents this user is allowed to see" from "all documents". Metadata fields are the mechanism.
