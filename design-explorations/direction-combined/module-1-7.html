FastAPI

Every agent you build will eventually expose an HTTP endpoint — for a chat UI, a webhook, or a scheduled trigger. FastAPI is the de-facto framework for this in the Python AI world: Pydantic built in, automatic OpenAPI docs, async-native, and fast enough for production. This module builds the /chat endpoint you'll reuse throughout the course.

Key Takeaways

  • Pydantic models as request/response types give you free validation, serialisation, and schema documentation in one declaration

Your first /chat endpoint #

A FastAPI app is a Python object. Route handlers are functions decorated with HTTP methods. Request bodies are Pydantic models — that's essentially the entire mental model. Declare a BaseModel subclass, annotate the handler parameter, and FastAPI handles parsing, validation, and serialisation automatically.

💡
Visit /docs after starting Run uvicorn main:app --reload, then open http://localhost:8000/docs. FastAPI generates a fully interactive Swagger UI from your Pydantic models and docstrings — no configuration required. Use Try It Out to POST real requests without Postman.
pythonmain.py — /chat endpoint skeleton
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field
from typing import Optional
import uuid

app = FastAPI(title="Agent Chat API", version="1.0.0")

# ── Request / Response models ─────────────────────────────────────────
class ChatRequest(BaseModel):
    message:    str            = Field(..., min_length=1, max_length=4000)
    session_id: Optional[str] = Field(None, description="Omit to start a new session")
    model:      str            = "claude-sonnet-4-6"

class ChatResponse(BaseModel):
    session_id:    str
    reply:         str
    input_tokens:  int
    output_tokens: int

# ── Route ─────────────────────────────────────────────────────────────
@app.post("/chat", response_model=ChatResponse)
async def chat(req: ChatRequest) -> ChatResponse:
    """Send a message and get an agent reply."""
    session_id = req.session_id or str(uuid.uuid4())

    # LLM call goes here — Phase 2+ wires the real provider
    reply = f"[stub] Echo: {req.message}"

    return ChatResponse(
        session_id=session_id,
        reply=reply,
        input_tokens=len(req.message.split()),
        output_tokens=0,
    )

@app.get("/health")
async def health():

Pydantic request & response models #

The Pydantic models from Module 1.2 plug directly into FastAPI. Annotate a route parameter with a BaseModel subclass and FastAPI automatically parses the JSON body, validates all fields, and returns a 422 Unprocessable Entity with detailed errors if validation fails — no boilerplate needed in the handler.

pythonmodels.py — validation, defaults, and nested models
from pydantic import BaseModel, Field, field_validator
from typing import Literal
from datetime import datetime

class Message(BaseModel):
    role:    Literal["user", "assistant", "system"]
    content: str

class ChatRequest(BaseModel):
    messages:   list[Message]  # nested model — FastAPI handles JSON nesting
    model:      str = "claude-sonnet-4-6"
    max_tokens: int = Field(1024, ge=1, le=8192)   # ge / le = min / max
    stream:     bool = False

    @field_validator("messages")
    @classmethod
    def at_least_one(cls, v):
        if not v:
            raise ValueError("messages list cannot be empty")
        return v

class Usage(BaseModel):
    input_tokens:  int
    output_tokens: int
    total_tokens:  int

class ChatResponse(BaseModel):
    id:         str
    reply:      str
    model:      str
    usage:      Usage
    created_at: datetime = Field(default_factory=datetime.utcnow)

Dependency injection #

Dependencies are callables that FastAPI resolves and injects into route handlers. They're the right place for authentication, database sessions, feature flags, and configuration — anything that every handler needs but shouldn't import globally. FastAPI calls dependencies automatically before the handler and cleans up after.

ℹ
yield vs return in dependencies Use yield when you need cleanup (DB sessions, file handles). FastAPI runs the code after the yield once the response is sent — even if the handler raised an exception. With return, an exception skips your cleanup entirely.
pythondependencies.py — auth token + DB session injection
from fastapi import Depends, HTTPException, status, Header
from sqlalchemy.orm import Session
from .database import engine
import os

# ── Auth dependency ───────────────────────────────────────────────────
def verify_api_key(x_api_key: str = Header(...)) -> str:
    if x_api_key != os.environ["INTERNAL_API_KEY"]:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid API key",
        )
    return x_api_key

# ── DB session dependency — yield so cleanup always runs ──────────────
def get_db():
    db = Session(engine)
    try:
        yield db
    finally:
        db.close()   # returns connection to pool even if handler raises

# ── Route using both dependencies ─────────────────────────────────────
@app.post("/chat", response_model=ChatResponse)
async def chat(
    req: ChatRequest,
    db:  Session = Depends(get_db),
    _:   str     = Depends(verify_api_key),   # _ = only care about side-effect (401 on fail)
) -> ChatResponse:

Running with uvicorn #

uvicorn is the ASGI server FastAPI runs on. The --reload flag watches for file changes and restarts automatically — essential for development. In production, use multiple workers to utilise all CPU cores.

bashterminal — dev and production startup commands
# Development — hot reload, single worker
uvicorn main:app --reload --host 0.0.0.0 --port 8000

# Production — 4 workers (match CPU count), no reload
uvicorn main:app --workers 4 --host 0.0.0.0 --port 8000

# Run from Python (clean for Docker ENTRYPOINT)
if __name__ == "__main__":
    import uvicorn
    uvicorn.run("main:app", host="0.0.0.0", port=8000, workers=4)

# After startup — check these URLs:
#   http://localhost:8000/docs         → Swagger UI (interactive Try It Out)
#   http://localhost:8000/redoc        → ReDoc (cleaner read-only docs)
#   http://localhost:8000/openapi.json → raw OpenAPI schema (for client code-gen)

Self-check #

What happens when a request body fails Pydantic validation in FastAPI?
FastAPI returns a 422 Unprocessable Entity response automatically, before the route handler is called. The response body includes a JSON array of validation errors with the field name, the failing value, and a human-readable message. No try/except needed in your handler — Pydantic and FastAPI handle it completely.
Why use yield in a dependency instead of return?
A yield dependency splits into setup (before yield) and teardown (after yield). FastAPI guarantees the teardown runs even if the route handler raises an exception. This is the correct pattern for DB sessions, file handles, and any resource that must be closed — with return, an exception in the handler would skip the cleanup entirely.
What does response_model=ChatResponse do on a route decorator?
It tells FastAPI to validate the handler's return value against the ChatResponse Pydantic model before serialising it to JSON. Extra fields are stripped, missing required fields raise an error, and the model's schema appears in /docs. It's the output equivalent of the input validation that happens automatically on the request body.
What is the difference between --reload and --workers 4 in uvicorn?
--reload enables hot-reloading — uvicorn watches the filesystem and restarts on file changes. It runs a single worker and is only for development. --workers 4 runs four independent worker processes, each with its own event loop — for production use to utilise multi-core CPUs. They are mutually exclusive: reload mode cannot run with multiple workers.
