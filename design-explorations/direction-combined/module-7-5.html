State Management

Graph state is the spine of a multi-agent system — it's what every node reads from and writes to. Poor state design is the single most common source of multi-agent bugs: too much in state causes bloated context, wrong things in state cause nodes to behave unpredictably, and missing checkpointing means a 30-minute workflow starts over from scratch on the first transient error. This module covers typed state design with Pydantic, the state-vs-context distinction, and the three LangGraph checkpointers you'll use in production: MemorySaver, SqliteSaver, and PostgresSaver.

~22 min read 4 code examples Last updated 2026-05-07
Key Takeaways
  • State should contain coordination metadata (what step we're on, what was decided, what still needs doing) — not the full raw content that nodes process
  • Use Pydantic models for state instead of plain TypedDict when you need validation — Pydantic state gives you type-safe field access with validation at write time
  • MemorySaver is in-process and non-persistent — loses all state on restart. Use for development and tests only
  • SqliteSaver is file-based persistence — durable across restarts, single-process, good for single-node deployments and multi-step tasks that take minutes
  • PostgresSaver is distributed — durable, multi-process, production-ready for multi-user or horizontally scaled deployments

State vs context — the key distinction #

Graph state is persisted across node invocations and stored by the checkpointer. Context (the messages array sent to the LLM) is constructed from state but is not the same thing as state. A common mistake is putting raw LLM responses or large retrieved documents directly into state — this causes checkpoints to balloon and nodes to work with unnecessarily large data structures.

State vs Context
Belongs in graph STATE
  • Current step / phase in the workflow
  • Routing decisions (which worker was called)
  • Structured results: IDs, counts, status flags
  • Error records with retry counts
  • Final outputs (after all processing)
  • Configuration: max iterations, flags
Belongs in node's local CONTEXT
  • Raw retrieved documents (reference them by ID)
  • Full API responses (keep only extracted fields)
  • Intermediate reasoning scratchpad
  • Tool call logs (unless you need to audit them)
  • Large binary data or file contents

Pydantic state model #

pythonpydantic_state.py — typed state with validation
from pydantic import BaseModel, Field, field_validator
from typing import Annotated
from langgraph.graph.message import add_messages

class WorkflowState(BaseModel):
    # Coordination metadata — keeps state small
    task_id: str
    task_description: str
    current_phase: str = "planning"
    completed_phases: list[str] = Field(default_factory=list)
    error_count: int = 0
    max_errors: int = 3
    approved: bool = False

    # Structured results (small, not raw content)
    plan_steps: list[str] = Field(default_factory=list)
    current_step_index: int = 0
    step_results: list[dict] = Field(default_factory=list)
    final_output: str = ""

    # Message history (reducers still work with Pydantic via Annotated)
    messages: Annotated[list, add_messages] = Field(default_factory=list)

    @field_validator("current_phase")
    @classmethod
    def valid_phase(cls, v):
        if v not in {"planning", "executing", "reviewing", "done"}:
            raise ValueError(f"Invalid phase: {v}")
        return v

    def is_over_error_limit(self) -> bool:
        return self.error_count >= self.max_errors

    def has_more_steps(self) -> bool:
        return self.current_step_index < len(self.plan_steps)

# Use WorkflowState as the StateGraph schema
builder = StateGraph(WorkflowState)

Checkpointers — durable state across node executions #

A checkpointer stores the full graph state after every node execution. If the process dies or an exception is thrown, you can resume from the last checkpoint rather than restarting the entire graph. The checkpointer is attached at graph compilation time and uses a config dict with a thread_id to identify which run to save/restore.

pythoncheckpointers.py — MemorySaver, SqliteSaver, PostgresSaver
from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver
from langgraph.checkpoint.postgres import PostgresSaver
import sqlite3, psycopg

# ── Dev / tests — in-memory, resets on restart ────────────────────────
memory_checkpointer = MemorySaver()
graph_dev = builder.compile(checkpointer=memory_checkpointer)

# ── Single-node production — durable SQLite file ───────────────────────
conn = sqlite3.connect("./agent_state.db", check_same_thread=False)
sqlite_checkpointer = SqliteSaver(conn)
graph_prod = builder.compile(checkpointer=sqlite_checkpointer)

# ── Multi-node production — PostgreSQL ────────────────────────────────
pg_conn = psycopg.connect("postgresql://user:pass@host/db")
pg_checkpointer = PostgresSaver(pg_conn)
pg_checkpointer.setup()    # creates checkpoints table on first run
graph_scaled = builder.compile(checkpointer=pg_checkpointer)

# ── Using a checkpointed graph ────────────────────────────────────────
config = {"configurable": {"thread_id": "workflow-job-42"}}

# First run — starts from scratch
result = graph_prod.invoke(initial_state, config=config)

# If the process dies and restarts, resume with same thread_id:
result = graph_prod.invoke(None, config=config)  # None = resume from checkpoint

# Inspect the state of a saved run
state = graph_prod.get_state(config=config)
print(state.values)   # current state dict
print(state.next)     # which nodes would run next

Subgraph state isolation #

When composing graphs (a parent graph that calls a child graph as a node), each graph has its own state schema. The parent must explicitly map the fields it wants to share with the child. This is the LangGraph equivalent of the context isolation benefit from agent-as-tool — each graph operates on its own typed state without accidentally sharing unintended fields.

pythonsubgraph_state.py — parent-child state isolation
class ParentState(TypedDict):
    task: str
    sub_result: str      # receives output from child graph

class ChildState(TypedDict):
    input_text: str     # receives input from parent
    output: str

child_graph = build_child_graph().compile()

def call_child(state: ParentState) -> dict:
    # Explicit field mapping parent → child
    child_result = child_graph.invoke({"input_text": state["task"]})
    # Explicit field mapping child → parent
    return {"sub_result": child_result["output"]}
