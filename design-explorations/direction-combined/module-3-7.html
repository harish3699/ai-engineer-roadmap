Prompts in Production

A prompt that works on Tuesday may silently degrade by Thursday after a model update. An unversioned prompt changed by a teammate breaks your eval suite with no rollback path. A 2,000-token system prompt sent on every request costs ten times what it needs to. This module covers the operational practices that turn prompt engineering from an art into a repeatable engineering discipline: versioning, caching, A/B testing, automated optimisation with DSPy, and the observability layer that tells you when something is wrong before users do.

~30 min read 1 video · 1 recording script 4 code examples Last updated 2026-05-07
Key Takeaways
  • Treat prompts as code — version them in Git, load them from environment-specific config, and never hard-code them inside function bodies where they can't be reviewed or rolled back
  • Anthropic's prompt caching reduces cost by up to 90% for large static content (system prompts, reference documents) — the cache has a 5-minute TTL and charges 10% of normal input price on cache hits
  • A/B testing prompts requires a scoring function, not just a feeling — route a percentage of traffic, score both variants on the same rubric, and promote only when the difference is statistically significant
  • DSPy replaces hand-written prompt text with compiled optimised instructions — when you have an eval set, DSPy can find a better prompt than you can write by hand, automatically
  • The minimum viable observability layer for any LLM system: log every request (prompt version, model, tokens, latency), alert on stop_reason=max_tokens, and track cost per feature per day

Prompt versioning #

A prompt is code. It has the same failure modes as code: silent regressions, undocumented changes, no rollback path. The discipline is the same: version it, review it, test it before deploying it. The minimum viable pattern is a prompt registry — a dict or YAML file where prompts live, keyed by name and version, loaded at runtime from environment-specific config.

pythonprompt_registry.py — versioned prompts loaded from config
import anthropic, os, json
from pathlib import Path
client = anthropic.Anthropic()

# ── prompts.json — lives in Git, reviewed like any code change ────────
# {
#   "support_triage": {
#     "v1": "Classify the support ticket. Return: team, priority.",
#     "v2": "You are a support triage system. Classify tickets accurately.\nReturn JSON: {team, priority, confidence}."
#   },
#   "active": {"support_triage": "v2"}   ← which version is live
# }

class PromptRegistry:
    def __init__(self, path: str):
        self.registry = json.loads(Path(path).read_text())

    def get(self, name: str, version: str = "active") -> str:
        v = self.registry["active"][name] if version == "active" else version
        return self.registry[name][v]

    def active_version(self, name: str) -> str:
        return self.registry["active"][name]

registry = PromptRegistry("prompts.json")

def call_with_version_tag(ticket: str) -> dict:
    prompt_name = "support_triage"
    system = registry.get(prompt_name)
    version = registry.active_version(prompt_name)

    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=128,
        system=system,
        messages=[{"role": "user", "content": ticket}],
    )
    return {
        "result": resp.content[0].text,
        "prompt_version": f"{prompt_name}@{version}",   # log this — critical for debugging
        "model": "claude-haiku-4-5-20251001",
        "tokens_in": resp.usage.input_tokens,
        "tokens_out": resp.usage.output_tokens,
    }
⚠
Always log the prompt version alongside the response When a user reports a bad output two weeks from now, you need to know which prompt version produced it. If you only log the response text, you can't reproduce or debug the issue. Log: prompt name + version, model ID, input/output token counts, latency, and stop_reason. Four fields. Do it from day one.

Prompt caching #

Anthropic's prompt caching lets you mark large static blocks of your prompt — the system prompt, a reference document, few-shot examples — with a cache_control breakpoint. On the first call, the prefix is computed and cached server-side. Subsequent calls within the 5-minute TTL pay 10% of the normal input token price for the cached portion. On a 2,000-token system prompt sent 1,000 times per day, this reduces that portion's cost by 90%.

Break-even: A cache prefix is profitable if you reuse it more than ~2.3 times within the 5-minute TTL. Any system prompt called more than 3× per 5-minute window benefits immediately.
pythonprompt_caching.py — cache_control on system prompt and injected documents
import anthropic
client = anthropic.Anthropic()

large_system_prompt = """You are a financial analysis assistant for AcmeCorp.
[... 1,800 tokens of domain context, rules, and formatting instructions ...]"""

reference_doc = """[Full 3,000-token product specification document injected here]"""

# ── cache_control marks the prefix boundary ───────────────────────────
resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,

    # system as list of blocks to enable cache_control
    system=[{
        "type": "text",
        "text": large_system_prompt,
        "cache_control": {"type": "ephemeral"},  # ← cache this prefix, 5-min TTL
    }],

    messages=[
        # cache the reference document too — it's the same every call
        {"role": "user", "content": [
            {"type": "text", "text": f"<reference>\n{reference_doc}\n</reference>",
             "cache_control": {"type": "ephemeral"}},             # ← also cached
            {"type": "text", "text": "What is the recommended pricing model for SMBs?"}, # not cached — changes every call
        ]},
    ],
)

# ── usage shows cache hit/miss ────────────────────────────────────────
u = resp.usage
print(f"Cache write tokens : {u.cache_creation_input_tokens}")   # first call — writing the cache
print(f"Cache read tokens  : {u.cache_read_input_tokens}")      # subsequent calls — cache hit
print(f"Uncached input     : {u.input_tokens}")                  # user question — not cached

A/B testing prompts #

When you think a new prompt version is better, "better" needs to be measurable. Route a percentage of traffic to each variant, score both with the same rubric or eval function, collect enough samples for statistical confidence, then promote or roll back. Gut feel does not scale. An eval function does.

pythonab_testing.py — traffic split with quality scoring
import anthropic, random, statistics
client = anthropic.Anthropic()

PROMPTS = {
    "v1": "Summarise the text in 2 sentences.",
    "v2": "Summarise the text in exactly 2 sentences. First sentence: main finding. Second sentence: key implication. Plain text only.",
}

def call_variant(text: str, variant: str) -> str:
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001", max_tokens=128,
        system=PROMPTS[variant],
        messages=[{"role": "user", "content": f"<text>{text}</text>"}],
    )
    return resp.content[0].text

def score_response(response: str) -> float:
    # use an LLM as judge — consistent, cheap at Haiku rates
    judge_resp = client.messages.create(
        model="claude-haiku-4-5-20251001", max_tokens=16,
        system="Score this summary 1–5: 5=exactly 2 sentences, plain text, clear main finding + implication. Respond with the number only.",
        messages=[{"role": "user", "content": response},
                  {"role": "assistant", "content": ""}],
    )
    try:
        return float(judge_resp.content[0].text.strip())
    except ValueError:
        return 3.0

# ── Run the A/B test — 20 samples per variant ─────────────────────────
test_texts = ["[sample article text]"] * 20   # replace with real eval set
scores = {"v1": [], "v2": []}

for text in test_texts:
    # route 50/50 for equal sample sizes
    variant = random.choice(["v1", "v2"])
    result = call_variant(text, variant)
    scores[variant].append(score_response(result))

for v, s in scores.items():
    if s:
        print(f"{v}: n={len(s)}  mean={statistics.mean(s):.2f}  stdev={statistics.stdev(s):.2f}")

# ── Decision rule ─────────────────────────────────────────────────────
# promote v2 if: mean(v2) > mean(v1) + 0.3 AND n >= 30 per variant
# 30 samples per variant gives ~80% power to detect a 0.5-point difference

DSPy — automated prompt optimisation #

DSPy (Declarative Self-improving Python) replaces hand-written prompt text with signatures — typed input/output specifications. Given an eval set and a metric function, DSPy's optimisers (called teleprompters) automatically search for the prompt instructions and few-shot examples that maximise your metric. The result: a compiled prompt that often outperforms what you'd write by hand, without manual iteration.

pythondspy_intro.py — signature + optimiser for classification
import dspy

# ── Configure the language model ──────────────────────────────────────
lm = dspy.LM("anthropic/claude-haiku-4-5-20251001")
dspy.configure(lm=lm)

# ── Define a signature: typed inputs and outputs ──────────────────────
class SupportClassifier(dspy.Signature):
    """Classify a support ticket into the correct team and priority."""
    ticket: str    = dspy.InputField(desc="The raw support ticket text")
    team: str      = dspy.OutputField(desc="billing | technical | onboarding | account")
    priority: str  = dspy.OutputField(desc="low | medium | high | critical")

# ── Build the module ──────────────────────────────────────────────────
classifier = dspy.Predict(SupportClassifier)

# ── Run unoptimised — DSPy writes the prompt from the signature ───────
result = classifier(ticket="I was double-charged last month and need a refund.")
print(f"Team: {result.team}, Priority: {result.priority}")

# ── Optimise with a small labelled eval set ───────────────────────────
trainset = [
    dspy.Example(ticket="Charged twice this month",           team="billing",   priority="high").with_inputs("ticket"),
    dspy.Example(ticket="API returning 500 errors",             team="technical", priority="critical").with_inputs("ticket"),
    dspy.Example(ticket="How do I add a team member?",          team="onboarding",priority="low").with_inputs("ticket"),
    dspy.Example(ticket="Need to update billing email address", team="account",   priority="low").with_inputs("ticket"),
]

def accuracy_metric(example, prediction, trace=None):
    return (example.team == prediction.team) and (example.priority == prediction.priority)

# BootstrapFewShot automatically selects the best few-shot examples
optimiser = dspy.teleprompt.BootstrapFewShot(metric=accuracy_metric, max_bootstrapped_demos=3)
optimised_classifier = optimiser.compile(dspy.Predict(SupportClassifier), trainset=trainset)

result2 = optimised_classifier(ticket="I was double-charged last month.")
print(f"Optimised — Team: {result2.team}, Priority: {result2.priority}")

The minimum viable observability layer #

You cannot manage what you don't measure. For any LLM system in production, four data points on every call give you everything you need to debug failures, track costs, and catch regressions before users notice.

Four fields to log on every LLM call
prompt_version
Which prompt produced this output. Enables reproducing any failure. Format: name@version.
tokens_in / out
Input and output token counts. Track cost per feature daily. Alert if average input tokens spike — often indicates a context accumulation bug.
stop_reason
Alert immediately if stop_reason == "max_tokens" for more than 1% of calls. Truncated outputs are silent failures — users get incomplete responses.
latency_ms
Time from request to first token (TTFT) and total response time. Track p50 and p95. Unexplained latency spikes indicate upstream API issues or context window growth.

Self-check #

You have a 2,000-token system prompt sent on 5,000 API calls per day. The cache TTL is 5 minutes. Your traffic is spread evenly across 24 hours. How many cache writes vs cache reads do you expect per day?
With 5,000 calls evenly over 24 hours: ~208 calls per hour, ~3.5 calls per minute. The 5-minute TTL means each cache write covers the next ~17 calls before expiring. Approximately: 5,000 / 17 ≈ 295 cache writes per day, and 5,000 − 295 ≈ 4,705 cache reads. Cache hit rate ≈ 94%. Cost estimate: (295 × 2,000 × $3.75 + 4,705 × 2,000 × $0.30) / 1,000,000 ≈ $2.22 + $2.82 = $5.04/day for the system prompt portion. Without caching that same volume would cost 5,000 × 2,000 × $3.00 / 1,000,000 = $30/day. Caching saves ~$25/day, ~$9,000/year on the system prompt alone.
Your observability dashboard shows stop_reason == "max_tokens" on 4% of calls. What should you investigate first?
4% is well above the 1% alert threshold and indicates a systematic problem, not an edge case. Investigate in order: (1) Context accumulation — check average tokens_in over time. If it's growing, your code is probably appending conversation history without a truncation limit, causing prompts to grow unbounded. (2) max_tokens setting — it may be set too low for the task complexity. Check p95 output length on successful calls and set max_tokens to at least 1.5× that. (3) Task scope creep — if users are sending longer inputs than the system was designed for, you may need input validation. All 4% of truncated calls produced incomplete outputs that users received silently — audit whether any of those caused downstream failures.
What does DSPy's BootstrapFewShot optimiser actually do, and why might it outperform hand-written few-shot examples?
BootstrapFewShot runs the unoptimised module on the training set, collects examples where the module produces correct outputs, and selects the best subset of those as compiled few-shot demonstrations. It outperforms hand-written examples for two reasons: (1) Selection — it evaluates candidate examples against your metric and picks the ones that actually improve accuracy, rather than examples that feel representative to a human; (2) Coverage — it can find edge-case examples that a human wouldn't think to write but that cover the failure modes the model struggles with. The key requirement: you need a labelled eval set and a clear metric function. Without those, DSPy has nothing to optimise against.
A teammate wants to improve the support triage prompt and edits the string directly in the Python file. What is the risk and how does prompt versioning address it?
Two risks: (1) No rollback path — if the edited prompt degrades quality, you can't quickly revert to what was working without digging through git blame, if it was even committed; (2) No audit trail — your logs show the response but not which prompt version produced it, making debugging impossible. Prompt versioning addresses both: the prompt lives in a reviewed, versioned file (e.g. prompts.json) with named versions. The registry always loads the active version. Changing the active version is a one-line config change that can be rolled back instantly. Every API call logs the version tag, so any bad output is traceable to the exact prompt text that produced it.
