Embedding Models

An embedding model converts a chunk of text into a dense vector — a list of floating-point numbers where geometrically close vectors represent semantically similar meaning. Everything downstream in your RAG pipeline depends on the quality of those vectors: retrieval precision, recall, and ultimately the accuracy of your LLM's answers. This module covers what embeddings are, how they are trained, what dimensions mean, how the major models compare, and how to benchmark embedding quality for your specific corpus before you commit to one.

~22 min read 1 video · 1 recording script 4 code examples Last updated 2026-05-07
Key Takeaways
  • An embedding model encodes semantic meaning into a fixed-length vector — two chunks that mean the same thing should have high cosine similarity even if they share no words
  • Embedding models are trained on pairs of similar and dissimilar texts using contrastive loss — the training data distribution determines which domain the model is good at
  • Higher dimensions (1536, 3072) give finer-grained representation but cost more to store and query; start with a lower-dimension model and only upgrade if eval scores justify it
  • MTEB (Massive Text Embedding Benchmark) is the standard leaderboard — filter it to the task type "Retrieval" and the language of your corpus before comparing models
  • General-purpose models (OpenAI, Cohere, BAAI/bge) work well across domains; domain-specific fine-tuning helps for highly specialised corpora like medical or legal text

Text Embeddings Explained — Semantic Search & Vector Similarity

▶ How embedding models convert text to dense vectors and why cosine similarity finds semantic neighbours

What is an embedding? #

An embedding is a dense, fixed-length list of floating-point numbers — a vector — that represents the semantic content of a piece of text. "Dense" means almost every number carries signal (unlike sparse bag-of-words vectors which are mostly zeros). "Fixed-length" means every piece of text produces a vector of the same dimension, regardless of input length.

The key property: geometric proximity encodes semantic similarity. Two texts that mean the same thing produce vectors that are close together in the high-dimensional space — high cosine similarity. Two texts about unrelated topics produce vectors that are far apart — low cosine similarity. The embedding model has learned this mapping from millions of (similar, dissimilar) training pairs.

From text to vector: the embedding pipeline
Input text
"$150/mo for home internet"
→
Tokeniser
[7543, 12, 4820, 601, …]
→
Transformer encoder
Self-attention across all tokens
→
Pooling
Mean or [CLS] token
→
Embedding vector
[0.021, -0.83, 0.41 …] (1536-d)
Pooling collapses the per-token representations into one vector for the whole input sequence. Mean pooling averages all token vectors; [CLS] pooling uses the special classification token inserted at position 0.

How embedding models are trained #

Embedding models are encoder-only transformers (like BERT) trained with contrastive loss. The training signal comes from pairs: a query and a positive passage (an answer or semantically similar text), and negatives (unrelated passages). The loss function rewards high similarity for positive pairs and low similarity for negatives.

ℹ
Why training data determines domain fit A model trained on Wikipedia + web crawl pairs learns general English semantics well. The same model applied to a corpus of pharmaceutical patents or Salesforce SOQL documentation may embed domain jargon poorly — the model has never seen contrastive pairs where two biomedical sentences that use different synonyms for the same compound are marked as similar. This is why domain-specific fine-tuning helps for specialised corpora.

Modern models like OpenAI's text-embedding-3 family and BAAI/bge-m3 also use matryoshka representation learning (MRL): the model is trained so that the first N dimensions of the vector are independently useful. This lets you truncate a 3072-dimension vector to 256 dimensions at query time with only a modest quality loss — useful when storage or query latency is constrained.

Calling embedding APIs #

The OpenAI Embeddings API is the easiest entry point. Pass one or more strings, receive one embedding vector per string. Batch your documents — the API accepts up to 2048 inputs per request, and batching is significantly cheaper than one-request-per-chunk.

pythonopenai_embeddings.py — batch document and query embedding
import os
from openai import OpenAI
import numpy as np

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

def embed_texts(
    texts: list[str],
    model: str = "text-embedding-3-small",
    dimensions: int | None = None,
) -> list[list[float]]:
    """Embed a batch of texts. Returns one vector per text."""
    kwargs = {"model": model, "input": texts}
    if dimensions:
        kwargs["dimensions"] = dimensions  # MRL truncation — only text-embedding-3-*
    resp = client.embeddings.create(**kwargs)
    return [item.embedding for item in resp.data]

def cosine_similarity(a: list[float], b: list[float]) -> float:
    a, b = np.array(a), np.array(b)
    return float(a @ b / (np.linalg.norm(a) * np.linalg.norm(b)))

# ── Embed a corpus (at index time) ───────────────────────────────────
chunks = [
    "Employees may claim up to $150 per month for home internet.",
    "A one-time $600 allowance is available for ergonomic furniture.",
    "All claims must be submitted via Workday by the 5th of the month.",
    "AcmeCorp was founded in Austin, Texas in 2018 and employs 400 people.",
]

chunk_vectors = embed_texts(chunks)

# ── Embed a query (at retrieval time) ────────────────────────────────
query = "How much can I claim for my standing desk?"
query_vector = embed_texts([query])[0]

# ── Rank by similarity ───────────────────────────────────────────────
scores = [(i, cosine_similarity(query_vector, cv)) for i, cv in enumerate(chunk_vectors)]
scores.sort(key=lambda x: x[1], reverse=True)
for rank, (idx, score) in enumerate(scores[:3]):
    print(f"[{rank+1}] score={score:.3f}  {chunks[idx]}")
# → [1] score=0.721  "A one-time $600 allowance for ergonomic furniture."
# → [2] score=0.614  "Employees may claim up to $150 per month for home internet."
# → [3] score=0.482  "All claims must be submitted via Workday..."
💡
Use asymmetric embedding at query time Some models (BAAI/bge, Cohere) support an instruction prefix that tells the model whether it is encoding a query or a document. At index time you embed documents with a "passage:" prefix; at query time you embed with "query:" prefix. This asymmetric encoding improves retrieval precision because the model was trained to match differently-phrased queries against full passages. Always check the model card for recommended prefixes.

Open-source embeddings with sentence-transformers #

When you need zero API cost, offline operation, or data-privacy guarantees, sentence-transformers is the standard library. It wraps Hugging Face models and handles tokenisation, batching, and pooling behind a single encode() call.

pythonlocal_embeddings.py — sentence-transformers with batch encode
from sentence_transformers import SentenceTransformer
import numpy as np

# Downloaded from Hugging Face on first run, cached locally
model = SentenceTransformer("BAAI/bge-small-en-v1.5")

chunks = [
    "Employees may claim up to $150 per month for home internet.",
    "A one-time $600 allowance is available for ergonomic furniture.",
    "All claims must be submitted via Workday by the 5th of the month.",
    "AcmeCorp was founded in Austin, Texas in 2018 and employs 400 people.",
]

# normalize_embeddings=True → vectors are unit-length, dot product == cosine similarity
chunk_vectors = model.encode(chunks, normalize_embeddings=True, batch_size=32)

# BGE requires this instruction prefix for queries (not for documents)
query = "Represent this sentence for searching relevant passages: How much can I claim for my standing desk?"
query_vector = model.encode([query], normalize_embeddings=True)[0]

scores = chunk_vectors @ query_vector  # fast dot product == cosine sim when normalised
ranked = np.argsort(scores)[::-1]
for idx in ranked[:3]:
    print(f"score={scores[idx]:.3f}  {chunks[idx]}")

# ── GPU acceleration (if available) ─────────────────────────────────
import torch
device = "cuda" if torch.cuda.is_available() else "cpu"
model_gpu = SentenceTransformer("BAAI/bge-large-en-v1.5", device=device)
# batch_size=64+ on GPU, vs 16–32 on CPU for large models

Model comparison #

There is no universally best embedding model — the right choice depends on dimensions, latency, cost, and whether your corpus is in-distribution for that model's training data. The table below covers the models most commonly used in production RAG systems as of 2025.

Embedding model comparison
Model Dim Max tokens Cost Best for
text-embedding-3-small 1536 8192 $0.02/1M Most production RAG — great quality/cost balance
text-embedding-3-large 3072 8192 $0.13/1M When retrieval quality is critical and cost is secondary
Cohere embed-english-v3.0 1024 512 $0.10/1M Strong on BEIR benchmarks; Cohere ecosystem
BAAI/bge-m3 1024 8192 Free Multi-lingual, long context, self-hosted
BAAI/bge-small-en-v1.5 384 512 Free Fast local inference, limited memory, prototyping
nomic-embed-text-v1.5 768 8192 Free Long-context open-source, MRL support, competitive quality
all-MiniLM-L6-v2 384 256 Free Very fast, small footprint — semantic chunking (4.2) at index time

What dimensions mean #

Dimensions set the size of the vector space. A 384-dimensional model has 384 numbers per embedding; a 3072-dimensional model has 3072. Higher dimensions allow finer distinctions between semantically close texts — the model has more "room" to place similar but distinct concepts far enough apart to be distinguishable at retrieval time.

Higher dimensions are not always better
  • 3072-d vectors use 8× more storage than 384-d
  • Vector similarity search is O(n·d) — slower queries
  • For simple, homogeneous corpora the extra dimensions are wasted capacity
  • Going from 384-d to 3072-d on a general knowledge base FAQ rarely improves RAGAS context recall by more than 2–3%
When to upgrade dimensions
  • Your corpus has many semantically similar documents that need fine-grained separation (e.g. 10k product descriptions with subtle variations)
  • Your eval shows retrieval recall stalling and context is in the corpus — not a chunking or query problem
  • You are using MRL to truncate at runtime — start with 3072-d model, test with 512-d, promote when quality holds

Benchmarking with MTEB #

MTEB (Massive Text Embedding Benchmark) evaluates embedding models across 56 datasets and 8 task types. For RAG you care about one task type: Retrieval. The metric is NDCG@10 — whether the correct passage appears in the top 10 results.

⚠
MTEB scores are an upper bound, not a guarantee MTEB benchmarks public datasets. If your corpus is domain-specific (legal, medical, proprietary enterprise data), MTEB scores correlate weakly with actual retrieval performance on your data. Always run a domain-specific eval: take 50–100 representative queries from your use case, embed them, retrieve top-K chunks, and measure how often the correct answer appears. This is more predictive than any published benchmark.
pythonbenchmark_embeddings.py — compare two models on your own corpus
from sentence_transformers import SentenceTransformer
import numpy as np

# Ground-truth pairs: (query, correct_chunk_index)
eval_set = [
    ("How much for standing desk?", 1),      # correct answer is chunk[1]
    ("When is the Workday deadline?", 2),
    ("What does AcmeCorp do?", 3),
    ("Internet expense allowance", 0),
]

chunks = [
    "Employees may claim up to $150 per month for home internet.",
    "A one-time $600 allowance is available for ergonomic furniture.",
    "All claims must be submitted via Workday by the 5th of the month.",
    "AcmeCorp was founded in Austin, Texas in 2018 and employs 400 people.",
]

def recall_at_k(model_name: str, k: int = 3) -> float:
    model = SentenceTransformer(model_name)
    doc_vecs = model.encode(chunks, normalize_embeddings=True)
    hits = 0
    for query, correct_idx in eval_set:
        q_vec = model.encode([query], normalize_embeddings=True)[0]
        scores = doc_vecs @ q_vec
        top_k = set(np.argsort(scores)[-k:])
        if correct_idx in top_k:
            hits += 1
    return hits / len(eval_set)

for model_name in ["all-MiniLM-L6-v2", "BAAI/bge-small-en-v1.5"]:
    r = recall_at_k(model_name, k=3)
    print(f"{model_name}: recall@3 = {r:.2%}")
ℹ
Cosine vs dot product vs Euclidean distance When vectors are L2-normalised (unit length), cosine similarity and dot product are identical — use dot product since it is faster. Most vector databases default to cosine; if yours defaults to Euclidean (L2 distance), normalise your vectors first or switch metrics. Euclidean distance is sensitive to vector magnitude, which is irrelevant for semantic similarity.

When to fine-tune an embedding model #

Fine-tuning is warranted when your domain vocabulary is significantly different from the model's training data — medical abbreviations, internal product names, proprietary acronyms. The signal you need is a set of (query, positive_passage, negative_passage) triplets from your domain.

pythonfinetune_embeddings.py — domain fine-tuning with sentence-transformers
from sentence_transformers import SentenceTransformer, InputExample, losses
from torch.utils.data import DataLoader

# Load base model — fine-tune from a good starting point, not from scratch
model = SentenceTransformer("BAAI/bge-small-en-v1.5")

# Domain training pairs: (query, positive_passage) — negatives sampled in-batch
train_examples = [
    InputExample(texts=["internet expense claim", "Employees may claim $150/mo for home internet."]),
    InputExample(texts=["ergonomic equipment allowance", "$600 one-time allowance for ergonomic furniture."]),
    # ... hundreds to thousands of domain-specific pairs
]

# MultipleNegativesRankingLoss: other batch items become in-batch negatives
train_dataloader = DataLoader(train_examples, shuffle=True, batch_size=32)
train_loss = losses.MultipleNegativesRankingLoss(model)

model.fit(
    train_objectives=[(train_dataloader, train_loss)],
    epochs=3,
    warmup_steps=100,
    show_progress_bar=True,
)
model.save("./my-domain-embedding-model")

# Reload later:
# model = SentenceTransformer("./my-domain-embedding-model")
💡
Generate training pairs from your LLM If you have a corpus but no labeled query/answer pairs, use an LLM to generate synthetic training data. For each chunk, prompt: "Write 3 different questions that this passage answers." Collect (generated question, original chunk) as positive pairs. This bootstraps fine-tuning without manual labeling and typically yields a 5–15% recall improvement over the base model on domain-specific retrieval tasks.

Self-check #

What is contrastive training and why does the training data distribution matter for your RAG system?
Contrastive training optimises the model to produce high cosine similarity for positive pairs (semantically similar texts) and low similarity for negative pairs. The model learns what "similar" and "different" mean from the distribution of those pairs. If the training data is general web text, the model learns general semantic similarity well. If your corpus is, say, Salesforce SOQL documentation or pharmaceutical trial reports, the domain jargon — object names, technical abbreviations, domain-specific phrasings — may not appear in the training pairs, so the model's similarity estimates for those concepts will be less accurate. This is why domain-specific fine-tuning (or selecting a model fine-tuned on similar data) can materially improve retrieval recall on specialised corpora.
Your embedding model has a 512-token limit but you are chunking documents at 400 tokens. You switch to a new model with an 8192-token limit. Does this mean you should use much larger chunks?
Not necessarily. The token limit of an embedding model defines the maximum input it can process without truncation — but the optimal chunk size for retrieval is still controlled by the precision/recall tradeoff from module 4.2, not by the model's max tokens. Larger chunks still dilute the embedding signal with noise from multiple topics. The 8192-token limit is valuable when your document units are naturally large (e.g. a long legal clause or a full API documentation page that must not be split), not as a reason to use larger chunks universally. Keep chunking decisions based on your RAGAS eval results, not on the model's context window ceiling.
Explain matryoshka representation learning (MRL) and how you would use it in a production RAG system with a large corpus.
MRL trains the model so that the first N dimensions of the vector are independently useful for retrieval. A 3072-d vector trained with MRL can be truncated to 512-d or 256-d without retraining, with only a modest quality loss. In a production RAG system with millions of chunks, you might: (1) Embed all documents at full 3072-d during initial indexing. (2) Run your RAGAS eval with truncated vectors at 512-d, 256-d, 128-d. (3) Choose the smallest dimension that keeps context recall above your threshold. (4) Store and query at that reduced dimension — lower storage cost, faster similarity search, same model. This avoids reindexing from scratch if you need to scale down later.
What is the difference between cosine similarity and dot product similarity, and when are they equivalent?
Cosine similarity measures the angle between two vectors, ignoring their magnitude: cos(θ) = a·b / (|a| × |b|). Dot product computes a·b directly, which reflects both angle and magnitude. They are equivalent when both vectors are L2-normalised (unit length: |a| = |b| = 1), because the division by magnitudes becomes 1/1. Most embedding models trained with contrastive loss produce near-unit-length vectors, and calling model.encode(..., normalize_embeddings=True) explicitly enforces this. When vectors are normalised you can use dot product (faster in most vector databases and NumPy) instead of computing the full cosine formula. Use Euclidean distance only if you are working with a vector database that doesn't support cosine — it gives different rankings than cosine on non-normalised vectors.
