Vector Databases

You have chunks, you have embeddings — now where do you put them? A vector database is a storage system optimised for one job: given a query vector, find the K most similar stored vectors in milliseconds, even across millions of entries. This module covers why standard databases cannot do this efficiently, how approximate nearest neighbour algorithms (HNSW, IVF) make it fast, how metadata filtering combines vector similarity with structured conditions, and how to choose between Pinecone, Qdrant, Weaviate, pgvector, Chroma, and FAISS for your specific use case.

~24 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • A standard SQL database must compare the query vector against every stored vector (O(n·d)) — at one million vectors this takes seconds; vector databases use ANN indexes to bring that to milliseconds
  • HNSW (Hierarchical Navigable Small World) is the dominant index type — it builds a layered graph of vectors and traverses it to find approximate neighbours; it trades a small recall drop for orders-of-magnitude speed gains
  • Metadata filtering lets you combine vector similarity with structured conditions: "find the 10 most relevant chunks AND department = 'Finance' AND date > 2024-01-01"
  • For prototyping and small corpora (<100k chunks), FAISS or Chroma running in-process is the right choice; for production at scale, a managed service (Pinecone) or self-hosted dedicated store (Qdrant, Weaviate) is needed
  • pgvector adds vector search to PostgreSQL — if you already run Postgres in production, it is the lowest-friction path to adding retrieval without a new infrastructure component

▶ How vector databases index and search embeddings at millisecond speed

The similarity search problem #

Given a query vector q and a set of stored vectors {v₁, v₂, … vₙ}, find the K vectors with highest cosine similarity to q. The brute-force approach computes q · vᵢ for every stored vector — exact nearest neighbour (ENN) search. At 1M vectors with 1536 dimensions, this is 1.5 billion multiplications per query. At 10M vectors it takes several seconds. That is too slow for real-time retrieval.

Exact vs approximate nearest neighbour
Exact (brute-force / Flat index)
  • ✓ 100% recall — always finds the true top-K
  • ✓ No index build time
  • ✗ O(n·d) query time — seconds at 1M+ vectors
  • ✗ Does not scale beyond ~100k vectors for real-time use
Approximate (HNSW / IVF)
  • ✓ Sub-millisecond queries at millions of vectors
  • ✓ Recall@10 typically 95–99% in practice
  • ✗ Build time: minutes to hours for large indexes
  • ✗ Higher memory footprint (HNSW graph structure)
For RAG, the 1–5% recall loss from ANN is acceptable — the embedding model itself introduces more noise than the ANN approximation. Use Flat (exact) only for corpora under ~100k chunks or for offline evaluation.

How HNSW works #

HNSW (Hierarchical Navigable Small World) is the index algorithm used by Qdrant, Weaviate, pgvector, and most production vector stores. It builds a layered graph where each vector is a node. Top layers have a few long-range connections; bottom layers have many short-range connections. A query starts at the top layer, greedily descends toward the query point, and arrives at approximate neighbours at the bottom layer.

ℹ
HNSW tuning parameters: ef_construction and M M controls the number of bidirectional links per node (higher = better recall, more memory). ef_construction controls how many candidate neighbours are evaluated during index build (higher = better recall, slower build). At query time, ef (or search_ef) controls how many candidates the search explores. The practical defaults — M=16, ef_construction=200, ef=128 — work well for most RAG workloads. Only tune if your recall@10 drops below 95% on your eval set.

FAISS: local, in-process vector search #

FAISS (Facebook AI Similarity Search) is a library, not a database — it runs in the same Python process as your application and stores vectors in memory. It is ideal for prototyping, offline batch pipelines, and small corpora (up to a few hundred thousand vectors). For production use cases that need persistence, horizontal scaling, or multi-tenant access control, use a dedicated store.

pythonfaiss_index.py — build, save, load and query a FAISS index
import faiss
import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-small-en-v1.5")

chunks = [
    "Employees may claim up to $150 per month for home internet.",
    "A one-time $600 allowance is available for ergonomic furniture.",
    "All claims must be submitted via Workday by the 5th of the month.",
    "AcmeCorp was founded in Austin, Texas in 2018 and employs 400 people.",
    "Software subscriptions may be expensed up to $50 per month.",
    "Code reviews are mandatory for all pull requests before merging.",
]

# Embed and normalise — cosine similarity = dot product on unit vectors
vectors = model.encode(chunks, normalize_embeddings=True)  # shape: (6, 384)
dim = vectors.shape[1]

# ── Build index ───────────────────────────────────────────────────────
# IndexFlatIP: exact inner product (== cosine on normalised vecs)
# For large corpora use IndexHNSWFlat or IndexIVFFlat with training
index = faiss.IndexFlatIP(dim)
index.add(vectors.astype("float32"))
print(f"Index has {index.ntotal} vectors")  # → 6

# ── Persist to disk ──────────────────────────────────────────────────
faiss.write_index(index, "hr_policy.faiss")

# ── Load from disk ───────────────────────────────────────────────────
index = faiss.read_index("hr_policy.faiss")

# ── Query ─────────────────────────────────────────────────────────────
query = "Represent this sentence for searching relevant passages: standing desk allowance"
q_vec = model.encode([query], normalize_embeddings=True).astype("float32")

scores, indices = index.search(q_vec, k=3)  # returns (1, 3) arrays
for score, idx in zip(scores[0], indices[0]):
    print(f"[{idx}] score={score:.3f}  {chunks[idx]}")

# ── HNSW index (faster at scale) ─────────────────────────────────────
# IndexHNSWFlat: M=16 connections per node, exact bottom layer
hnsw = faiss.IndexHNSWFlat(dim, 16)   # M=16
hnsw.hnsw.efConstruction = 200
hnsw.add(vectors.astype("float32"))
hnsw.hnsw.efSearch = 128              # higher = better recall, slower

Qdrant: self-hosted production vector store #

Qdrant is an open-source vector database written in Rust — fast, memory-efficient, and designed for production RAG workloads. It supports HNSW indexing, metadata filtering, named collections, payload indexing, and a REST + gRPC API. Run it locally with Docker or deploy to Qdrant Cloud.

pythonqdrant_rag.py — upsert chunks with metadata, query with filters
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, PointStruct,
    Filter, FieldCondition, MatchValue
)
from sentence_transformers import SentenceTransformer
import uuid

model = SentenceTransformer("BAAI/bge-small-en-v1.5")
client = QdrantClient(host="localhost", port=6333)

COLLECTION = "hr_policy"
DIM = 384

# ── Create collection (idempotent) ────────────────────────────────────
client.recreate_collection(
    collection_name=COLLECTION,
    vectors_config=VectorParams(size=DIM, distance=Distance.COSINE),
)

# ── Chunks with metadata (payload) ───────────────────────────────────
documents = [
    {"text": "Employees may claim up to $150/mo for home internet.",   "dept": "HR", "section": "expenses"},
    {"text": "One-time $600 allowance for ergonomic furniture.",          "dept": "HR", "section": "expenses"},
    {"text": "Claims must be submitted via Workday by the 5th.",          "dept": "HR", "section": "process"},
    {"text": "AcmeCorp was founded in Austin, Texas in 2018.",             "dept": "About", "section": "company"},
    {"text": "Code reviews are mandatory for all pull requests.",           "dept": "Engineering", "section": "process"},
]

# Embed and upsert with payload
texts = [d["text"] for d in documents]
vectors = model.encode(texts, normalize_embeddings=True)

points = [
    PointStruct(
        id=str(uuid.uuid4()),
        vector=vectors[i].tolist(),
        payload=documents[i],
    )
    for i in range(len(documents))
]
client.upsert(collection_name=COLLECTION, points=points)

# ── Query with metadata filter ────────────────────────────────────────
query = "Represent this sentence for searching relevant passages: expense reimbursement process"
q_vec = model.encode([query], normalize_embeddings=True)[0].tolist()

# Retrieve top-3 from HR department only
results = client.search(
    collection_name=COLLECTION,
    query_vector=q_vec,
    query_filter=Filter(
        must=[FieldCondition(key="dept", match=MatchValue(value="HR"))]
    ),
    limit=3,
    with_payload=True,
)

for r in results:
    print(f"score={r.score:.3f}  [{r.payload['section']}]  {r.payload['text']}")
💡
Index your metadata fields in Qdrant for fast filtering By default, Qdrant applies metadata filters by scanning all candidate vectors after ANN retrieval. For high-cardinality filter fields (department, document_id, date ranges), create a payload index: client.create_payload_index(collection, "dept", PayloadSchemaType.KEYWORD). This enables pre-filtering — only vectors matching the filter are searched — which is faster and more accurate for selective filters.

pgvector: vector search inside PostgreSQL #

pgvector is a PostgreSQL extension that adds a vector column type and operators for cosine similarity (<=>), inner product (<#>), and Euclidean distance (<->). If your application already runs on Postgres, pgvector is often the lowest-friction path to adding retrieval — no new infrastructure, same connection pool, same backup strategy, same access control model.

pythonpgvector_rag.py — store and query embeddings in PostgreSQL
import psycopg2
from pgvector.psycopg2 import register_vector
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("BAAI/bge-small-en-v1.5")

conn = psycopg2.connect("postgresql://user:pass@localhost/mydb")
register_vector(conn)
cur = conn.cursor()

# ── Schema setup (run once) ───────────────────────────────────────────
cur.execute("CREATE EXTENSION IF NOT EXISTS vector")
cur.execute("""
    CREATE TABLE IF NOT EXISTS chunks (
        id          SERIAL PRIMARY KEY,
        content     TEXT NOT NULL,
        dept        TEXT,
        section     TEXT,
        embedding   vector(384)
    )
""")
# HNSW index for fast cosine search — build after initial load
cur.execute("""
    CREATE INDEX IF NOT EXISTS chunks_embedding_hnsw
    ON chunks USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 200)
""")
conn.commit()

# ── Insert chunks ─────────────────────────────────────────────────────
documents = [
    ("Employees may claim $150/mo for home internet.", "HR", "expenses"),
    ("$600 allowance for ergonomic furniture.", "HR", "expenses"),
    ("Claims due via Workday by the 5th of each month.", "HR", "process"),
    ("AcmeCorp founded 2018, 400 employees, Austin TX.", "About", "company"),
]
for text, dept, section in documents:
    vec = model.encode([text], normalize_embeddings=True)[0]
    cur.execute(
        "INSERT INTO chunks (content, dept, section, embedding) VALUES (%s,%s,%s,%s)",
        (text, dept, section, vec.tolist()),
    )
conn.commit()

# ── Query: cosine similarity + metadata filter ────────────────────────
query = "Represent this sentence for searching relevant passages: standing desk allowance"
q_vec = model.encode([query], normalize_embeddings=True)[0]

cur.execute("""
    SELECT content, dept, 1 - (embedding <=> %s::vector) AS similarity
    FROM chunks
    WHERE dept = 'HR'
    ORDER BY embedding <=> %s::vector   -- cosine distance ASC = similarity DESC
    LIMIT 3
""", (q_vec.tolist(), q_vec.tolist()))

for row in cur.fetchall():
    print(f"sim={row[2]:.3f}  [{row[1]}]  {row[0]}")

Choosing a vector store #

The right choice depends on corpus size, deployment constraints, team familiarity, and how much infrastructure complexity you can accept. The table below covers the most commonly used options.

Vector store comparison
Store Index Hosting Best for Limitation
FAISS Flat / HNSW / IVF In-process Prototyping, offline batch, <500k vectors No persistence, no metadata filter, no multi-process
Chroma HNSW In-process / Docker Local RAG apps, LangChain / LlamaIndex integration Not designed for large-scale production
Qdrant HNSW Self-hosted / Cloud Production RAG, rich metadata filtering, Rust performance Managed offering newer; ops overhead if self-hosted
Weaviate HNSW Self-hosted / Cloud Hybrid search (BM25 + vector), GraphQL API, modules More complex setup; higher resource usage
Pinecone Proprietary Managed only Teams that want zero infrastructure ownership Vendor lock-in; cost scales with storage + queries
pgvector HNSW / IVFFlat Self-hosted / RDS Teams already on Postgres; SQL joins across vector + relational data Lower query throughput than dedicated stores at very high scale
⚠
Don't over-engineer the vector store choice early The vector store is almost never the bottleneck in a RAG system — chunking quality, embedding model choice, and retrieval strategy have far larger impact on answer quality. Start with Chroma or FAISS locally. Migrate to Qdrant or pgvector when you need persistence, multi-user access, or metadata filtering in production. Migrate to Pinecone if your team has no infrastructure capacity. The migration is not expensive — it is re-embedding your corpus and re-indexing, which you will likely do anyway when you upgrade your embedding model.

Metadata filtering in practice #

The most important production feature of a vector store is metadata filtering — the ability to combine "semantically similar to this query" with "satisfying these structured conditions". This is how you implement data access control, multi-tenant isolation, and date-range scoping without running separate databases.

▸
Pre-filter vs post-filter Pre-filter: apply the metadata condition first, then run ANN search over the remaining vectors. Accurate for selective filters (e.g. user_id = X where each user has few documents). Requires indexed payload fields. Post-filter: run ANN over all vectors, then discard results that fail the filter. Fast to implement but can return fewer than K results if many matches are filtered out. Most production stores (Qdrant, pgvector HNSW) support both modes — choose pre-filter when the filter is selective, post-filter when it retains most of the corpus.

Self-check #

Why can't a standard SQL database efficiently serve as a vector store for a corpus of 5 million chunks?
SQL databases support indexed lookups for exact matches and range queries on scalar values, but they have no native operator for "find the K vectors most similar to this query vector." A brute-force approach requires computing the dot product between the query and all 5M vectors — 5M × 1536 multiplications per query — which takes several seconds even with optimised compute. There is no standard B-tree or hash index that reduces this to sub-linear time. Vector databases solve this with ANN indexes (HNSW, IVF) that trade a small recall loss for orders-of-magnitude speed improvement, returning approximate nearest neighbours in milliseconds. pgvector is the exception that adds this capability to PostgreSQL via a custom extension and HNSW/IVFFlat indexes.
Explain the HNSW graph structure and why querying it is faster than brute-force search.
HNSW builds a layered graph. The top layer has a few nodes with long-range connections spanning the entire vector space. Each successive lower layer has more nodes with shorter-range connections. A query starts at the top layer, greedily moves toward the nearest neighbour it can find, then descends to the next layer and repeats. By the time it reaches the bottom layer, it has navigated to a neighbourhood of the query point. The search visits only O(log n) layers and a small candidate set at each layer — far fewer vectors than the full corpus. The cost is memory (the graph edges) and index build time. The approximate part: HNSW may miss the true nearest neighbour if it's in a part of the graph the traversal doesn't visit. The M and ef_construction parameters control this quality/cost tradeoff.
You are building a multi-tenant RAG system where each user should only retrieve chunks from their own company's documents. How would you implement this in Qdrant?
Store a tenant_id field in each point's payload and create a payload index on it (PayloadSchemaType.KEYWORD). At query time, always include a filter: Filter(must=[FieldCondition(key="tenant_id", match=MatchValue(value=user.tenant_id))]). Enable pre-filtering so only the requesting tenant's vectors participate in the ANN search — not post-filtering, which could return fewer than K results if most top candidates are from other tenants. The payload index makes pre-filtering efficient. Do not rely on application-level filtering after retrieval as the sole access control mechanism — if a bug causes the filter to be omitted, tenant data leaks into another tenant's context.
When would you choose pgvector over Qdrant, and what are the tradeoffs?
Choose pgvector when: (1) Your application already runs on PostgreSQL and you want to avoid a new infrastructure component, new backup strategy, and new connection management concern. (2) You need SQL JOINs between your vector data and relational data (e.g. join chunk embeddings against an accounts table to filter by subscription tier). (3) Your team has Postgres expertise but no Rust/Go infrastructure experience. Tradeoffs: pgvector's query throughput at very high scale (10M+ vectors, high QPS) is lower than Qdrant or Pinecone because Postgres's multi-process architecture isn't optimised for ANN workloads. Its HNSW implementation is solid but has fewer tuning knobs. The <=> cosine operator is straightforward but mixing vector and SQL queries in one statement requires care about query planning. For most RAG systems under 5M chunks and moderate query volume, pgvector is entirely sufficient.
