LLMOps — Evaluation in Production

Observability tells you something is wrong. Evaluation tells you whether what you shipped is better or worse than what it replaced. This module closes the production feedback loop with four interlocking practices: golden dataset regression tests that run in CI, A/B testing frameworks for prompt and model changes, user feedback collection and surfacing, and automated drift detection on retrieval quality — so you ship with confidence and know the moment a model update silently breaks something.

~22 min read 4 code examples Last updated 2026-05-07
Key Takeaways
  • Golden datasets: 50–200 representative examples with expected outputs; run as pytest cases in CI; fail the build if LLM-judge score drops below a threshold (not string matching — LLM judge handles paraphrase)
  • A/B testing: route a percentage of real traffic to the new prompt/model, compare LLM-judge scores and user feedback rates over 48–72 hours before full rollout
  • Feedback loops: thumbs-up/down is free signal — collect it, tie it to trace IDs, surface it in your observability dashboard
  • Retrieval drift: embedding models change, knowledge bases grow stale — run a nightly eval on 50 fixed queries and alert when NDCG@5 drops more than 5% from the rolling baseline
  • The CI gate: golden dataset pass rate ≥ 90% AND no regression on any priority-1 case is the minimum bar before any prompt or model change ships to production

Golden dataset regression tests in CI #

A golden dataset is a curated set of (input, expected_output) pairs that represent the range of things your application should do correctly. It's your regression safety net — every prompt or model change must pass these tests before merging.

Don't use string matching. Use an LLM judge with structured output that evaluates semantic equivalence — otherwise a paraphrase of the correct answer will fail the test.

pythoneval_ci.py — golden dataset as pytest cases
# eval/golden_dataset.json — checked into the repo
# [{"id": "q1", "priority": 1, "input": "...", "expected": "...", "criteria": "..."}]

import json, pytest
from pydantic import BaseModel

class EvalResult(BaseModel):
    score: float       # 0.0 to 1.0
    passed: bool
    reasoning: str

JUDGE_SYSTEM = """You are an LLM output evaluator.
Given an [EXPECTED] answer and an [ACTUAL] answer, score the actual answer 0.0–1.0.
- 1.0: Correct, complete, and well-phrased
- 0.7–0.9: Correct but incomplete or awkwardly phrased
- 0.4–0.6: Partially correct
- 0.0–0.3: Wrong or misleading
passed=true if score >= 0.7. Be strict about factual accuracy."""

judge_llm = llm.with_structured_output(EvalResult)

def evaluate_response(expected: str, actual: str, criteria: str) -> EvalResult:
    return judge_llm.invoke([
        {"role": "system", "content": JUDGE_SYSTEM},
        {"role": "user", "content":
            f"[CRITERIA]\n{criteria}\n\n[EXPECTED]\n{expected}\n\n[ACTUAL]\n{actual}"},
    ])

with open("eval/golden_dataset.json") as f:
    GOLDEN = json.load(f)

@pytest.mark.parametrize("case", GOLDEN)
def test_golden(case):
    actual = agent.invoke(case["input"])
    result = evaluate_response(case["expected"], actual, case["criteria"])

    # Priority-1 cases must always pass — no exceptions
    if case.get("priority") == 1:
        assert result.passed, f"P1 case {case['id']} failed: {result.reasoning}"
    else:
        assert result.score >= 0.7, f"Case {case['id']} score {result.score:.2f}: {result.reasoning}"

# CI gate: overall pass rate must be ≥ 90%
def test_overall_pass_rate():
    results = [evaluate_response(c["expected"], agent.invoke(c["input"]), c["criteria"]) for c in GOLDEN]
    pass_rate = sum(1 for r in results if r.passed) / len(results)
    assert pass_rate >= 0.90, f"Overall pass rate {pass_rate:.1%} is below 90% threshold"

A/B testing prompt and model changes #

Never ship a prompt change "blind" to 100% of users. Route a percentage of production traffic to the new version, collect LLM-judge scores and user feedback for 48–72 hours, then compare. A change that looks better in offline evals sometimes regresses on the long tail of real user inputs.

pythonab_router.py — percentage-based traffic split with evaluation
import hashlib
from dataclasses import dataclass

@dataclass
class Variant:
    name: str
    system_prompt: str
    model: str
    weight: float   # 0.0–1.0, must sum to 1.0

VARIANTS = [
    Variant("control", CURRENT_SYSTEM_PROMPT, "claude-sonnet-4-6", 0.9),
    Variant("treatment", NEW_SYSTEM_PROMPT, "claude-sonnet-4-6", 0.1),   # 10% traffic
]

def assign_variant(user_id: str) -> Variant:
    """Deterministic assignment — same user always gets the same variant."""
    bucket = int(hashlib.md5(user_id.encode()).hexdigest(), 16) % 1000 / 1000
    cumulative = 0.0
    for variant in VARIANTS:
        cumulative += variant.weight
        if bucket < cumulative:
            return variant
    return VARIANTS[-1]

async def handle_with_ab(user_id: str, user_message: str) -> str:
    variant = assign_variant(user_id)

    # Log variant assignment — needed to compute per-variant metrics
    langfuse.trace(metadata={"ab_variant": variant.name, "ab_experiment": "prompt-v2"})

    response = await llm.ainvoke([
        {"role": "system", "content": variant.system_prompt},
        {"role": "user",   "content": user_message},
    ])

    # Auto-eval in background — don't block the response
    asyncio.create_task(log_auto_eval(user_message, response.content, variant.name))
    return response.content

User feedback loops #

Thumbs-up/down is the simplest feedback signal you can collect and it compounds over time into a training signal and an early-warning system. The key is tying the feedback rating back to the trace ID so you can inspect the exact conversation that drove the rating.

pythonfeedback_api.py — collecting and surfacing user feedback
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class FeedbackPayload(BaseModel):
    trace_id: str       # LangFuse/LangSmith trace ID — sent with every response
    rating: int         # 1 = thumbs-up, -1 = thumbs-down
    comment: str | None = None

@app.post("/feedback")
async def submit_feedback(payload: FeedbackPayload):
    # Post score back to LangFuse — it appears on the trace
    langfuse.score(
        trace_id=payload.trace_id,
        name="user_rating",
        value=payload.rating,
        comment=payload.comment,
        data_type="NUMERIC",
    )
    # Route negative ratings to a review queue
    if payload.rating == -1:
        await enqueue_for_review(payload.trace_id, payload.comment)
    return {"status": "ok"}

Drift detection on retrieval quality #

RAG pipelines degrade silently. The embedding model is the same but the knowledge base grows stale. Or the embedding API updates its model and cosine distances shift. Or your chunking strategy becomes misaligned as document lengths change. None of these fire an error — they just make answers progressively worse.

Fix: run a nightly eval on 50 fixed queries with known correct documents and track NDCG@5. Alert when it drops more than 5% from the 30-day rolling baseline.

⚠
Model version changes are silent regressions When OpenAI or Anthropic updates a model (e.g., gpt-4o to gpt-4o-2024-11-20) your golden dataset and drift monitors may catch regressions that manual testing would miss. Run the full eval suite any time you pin a new model version — before updating the version in production.
