What is RAG

An LLM trained in October 2024 knows nothing about your company's Q1 2025 report, your internal policy documents, or the support ticket your user filed yesterday. You could fine-tune the model — but that's expensive, slow, and the knowledge goes stale the moment anything changes. Retrieval-Augmented Generation solves this differently: at inference time, fetch the relevant documents and inject them into the prompt. The model reads your data fresh on every call, and hallucination on private knowledge drops dramatically.

~25 min read 1 video · 1 recording script 2 code examples Last updated 2026-05-07
Key Takeaways
  • LLMs can only know what was in their training data — they hallucinate confidently on anything outside it, including all of your private and recent knowledge
  • RAG solves the private knowledge problem at inference time: retrieve relevant passages → inject into the prompt → the model reasons over real data it can cite
  • RAG is not a replacement for fine-tuning — they solve different problems; RAG gives dynamic, citable knowledge access; fine-tuning changes how the model reasons and responds
  • The core RAG loop is five steps: chunk, embed, store, retrieve, generate — each step has engineering decisions that determine the system's quality ceiling
  • A minimal working RAG pipeline requires only three libraries and ~50 lines of Python — the complexity comes from scaling and evaluation, not from the concept itself

The knowledge problem #

A language model is a frozen snapshot of everything on the public internet up to a training cutoff date. After that date, its world stops. It knows nothing about events after the cutoff, nothing about your company's internal documents, nothing about your users' data. When asked about any of this, it doesn't say "I don't know" — it generates a plausible-sounding but fabricated answer. This is not a bug to be fixed; it is a fundamental consequence of how these models work.

What the model knows vs what you need it to know
Model knows (training data)
  • Public web content up to cutoff
  • Wikipedia, arXiv, GitHub, books
  • General coding, maths, language
  • Public company filings
Model does NOT know
  • Your internal documentation
  • Support tickets, emails, CRM data
  • Events after training cutoff
  • Your product's current state
pythonwithout_rag.py — the hallucination problem
import anthropic
client = anthropic.Anthropic()

# ── Ask about private internal data — model has never seen this ───────
resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=256,
    system="You are a helpful assistant for AcmeCorp employees.",
    messages=[{"role": "user",
               "content": "What is AcmeCorp's policy on remote work expense reimbursement?"}],
)
print(resp.content[0].text)
# → Model either says "I don't have that information" (honest)
#   OR invents a plausible-sounding policy that doesn't exist (hallucination)
#   You cannot tell which from the response alone — that's the problem.

# ── Solution preview: give the model the document at call time ────────
policy_doc = """AcmeCorp Remote Work Expense Policy (v3.2, effective 2025-01-01):
Employees may claim up to $150/month for home internet.
Standing desk and ergonomic chair: one-time $600 allowance per employee.
Submit claims via Workday by the 5th of the following month.
Manager approval required for claims over $200."""

resp2 = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=256,
    system="You are a helpful assistant for AcmeCorp employees. Answer based only on the provided policy documents.",
    messages=[{"role": "user", "content":
        f"<policy>\n{policy_doc}\n</policy>\n\nWhat is the remote work expense reimbursement policy?"}],
)
print(resp2.content[0].text)
# → Accurate, citable, grounded in the real document.

The RAG pipeline #

RAG is a five-step pipeline split across two phases: an offline indexing phase that runs once when your documents change, and an online retrieval and generation phase that runs on every user query.

The five-step RAG pipeline
Offline — runs when documents change
Step 1
Chunk
split docs into passages
→
Step 2
Embed
text → vector
→
Step 3
Store
vectors in DB
Online — runs on every user query
Step 4
Retrieve
query → top-K chunks
→
Step 5
Generate
LLM with context
Steps 1–3 are a one-time cost. Steps 4–5 run on every query. The quality of your entire system is determined by how well step 4 retrieves the right passages — garbage in, garbage out.

RAG vs fine-tuning #

These are the two most common approaches to giving a model domain-specific capability. They solve fundamentally different problems and are frequently used together.

RAG vs fine-tuning — choose the right tool
Use RAG when…
  • Knowledge changes frequently
  • You need source citations
  • Data is private or proprietary
  • You need to scope to a corpus
  • Fast iteration matters
  • Cost: retrieval infra, no GPU
Use fine-tuning when…
  • You need a new reasoning style
  • Output format must be strict
  • Knowledge is stable and large
  • Latency is critical (no retrieval step)
  • Branded voice or tone
  • Cost: GPU compute, dataset curation
Most production systems use both: fine-tune for tone/format/reasoning style, RAG for dynamic knowledge access. Example: a support bot fine-tuned to respond concisely and always include ticket numbers, with RAG over the latest product documentation.

A minimal working RAG pipeline #

Before adding vector databases, rerankers, and evaluation frameworks, it's worth seeing the core idea in ~50 lines. The key primitive is cosine similarity — a number between -1 and 1 measuring how semantically similar two text passages are when both are represented as dense vectors by an embedding model.

pythonminimal_rag.py — end-to-end RAG with in-memory vector search
import anthropic, numpy as np
from sentence_transformers import SentenceTransformer

client = anthropic.Anthropic()
embedder = SentenceTransformer("all-MiniLM-L6-v2")   # 384-dim, fast, free

# ── Your knowledge base — could be loaded from files/DB ───────────────
documents = [
    "AcmeCorp allows up to $150/month for home internet reimbursement.",
    "Employees receive a one-time $600 home office allowance for a standing desk or chair.",
    "Expense claims must be submitted via Workday by the 5th of the following month.",
    "Claims over $200 require manager approval before submission.",
    "The annual performance review cycle runs from November to January each year.",
    "AcmeCorp's engineering team uses a two-week sprint cycle with Friday retrospectives.",
]

# ── OFFLINE: embed all documents once ────────────────────────────────
doc_embeddings = embedder.encode(documents, normalize_embeddings=True)   # shape: (6, 384)

def retrieve(query: str, top_k: int = 3) -> list[str]:
    # embed the query then find top-K most similar document chunks
    query_vec = embedder.encode([query], normalize_embeddings=True)[0]    # shape: (384,)
    scores = doc_embeddings @ query_vec                                        # cosine similarity (dot product of normalised vectors)
    top_indices = np.argsort(scores)[::-1][:top_k]
    return [documents[i] for i in top_indices]

def rag_answer(question: str) -> str:
    # ONLINE: retrieve → augment → generate
    chunks = retrieve(question, top_k=3)
    context = "\n".join(f"- {c}" for c in chunks)

    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        system="Answer the question using only the provided context. If the answer isn't in the context, say so.",
        messages=[{"role": "user", "content":
            f"<context>\n{context}\n</context>\n\n<question>\n{question}\n</question>"}],
    )
    return resp.content[0].text

# ── Test it ───────────────────────────────────────────────────────────
questions = [
    "How much can I claim for home internet?",
    "When is the expense submission deadline?",
    "What is the company's maternity leave policy?",   # not in corpus → should say so
]

for q in questions:
    print(f"Q: {q}")
    print(f"A: {rag_answer(q)}\n")
ℹ
Why "not in the corpus" is a feature, not a failure When the third question asks about maternity leave and no chunk scores above a similarity threshold, the model correctly says it doesn't have that information. This is RAG working as intended. A model without RAG would confidently hallucinate a policy. A RAG system that fails gracefully on out-of-corpus queries is dramatically safer than a hallucinating model. Module 4.8 covers how to evaluate and measure this behaviour systematically.

Self-check #

Why does an LLM hallucinate on private company data even when the model is highly capable?
Hallucination on private data is not a capability failure — it's an information access failure. The model was trained on public web data up to a cutoff date. Your company's internal documents were never in that training data, so the model has no knowledge of them. When asked about them, it can't say nothing (that's not how autoregressive generation works), so it generates the most statistically plausible continuation — which looks like a real answer but is fabricated. RAG addresses this by injecting the actual documents into the context at inference time, so the model reasons over real data rather than generating from memory.
In the minimal RAG pipeline, what does cosine similarity measure and why do we normalize the embeddings before computing it?
Cosine similarity measures the angle between two vectors — how semantically similar two pieces of text are, independent of their length. Two passages about "expense reimbursement" will point in roughly the same direction in 384-dimensional space; two passages on unrelated topics will point in different directions. Normalizing the vectors to unit length (magnitude = 1) converts cosine similarity into a simple dot product: cosine_sim(a, b) = a · b when both are unit vectors. This is why normalize_embeddings=True + the matrix multiplication doc_embeddings @ query_vec correctly produces cosine similarity scores for every document in one vectorized operation.
Your team is building a customer support bot for a SaaS product. Documentation updates weekly, and users expect accurate answers with links to the source doc. Should you use RAG or fine-tuning?
RAG — for three reasons: (1) documentation updates weekly, so any fine-tuned knowledge would be stale within days of a new release; (2) users expect source citations, which RAG provides naturally (you return the retrieved chunk alongside the answer), while a fine-tuned model has no mechanism to cite where it learned something; (3) "scope to a corpus" — you want the bot to only answer from your docs, not from general training knowledge. Fine-tuning would be appropriate as a complement if you also want the bot to respond in a specific brand voice or always follow a particular answer structure — but for knowledge access on changing content, RAG is the right primary mechanism.
The minimal pipeline passes the top 3 retrieved chunks to the model. What are the tradeoffs of using top 1 vs top 10?
Top 1 — lowest token cost, minimal noise in the context, but if the single most similar chunk doesn't contain the answer (retrieval is imperfect), the model will have no context and either hallucinate or say it doesn't know. High precision requirement on retrieval. Top 10 — much higher chance of including the relevant passage, but: (1) context window cost grows 10×; (2) the "lost in the middle" problem applies — relevant information buried in a long context is attended to less; (3) irrelevant chunks add noise that can confuse the generation. Practical recommendation: start with top 3–5, measure answer quality with an eval set (module 4.8), and adjust. Top 3 is the most common starting point because it balances recall against context noise.
