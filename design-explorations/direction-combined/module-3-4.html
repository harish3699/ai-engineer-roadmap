Core Techniques

Prompt engineering has a short list of canonical techniques that cover 90% of real-world use cases. Zero-shot gets you started. Few-shot teaches the model your format. Role prompting sets the context. COSTAR gives your prompt complete specification. Chain-of-thought unlocks multi-step reasoning. Iterative refinement closes the feedback loop. Learn these six and you can tackle almost any task you'll encounter in production.

~30 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • Zero-shot works for well-defined tasks with clear verbs; reach for few-shot the moment output format or style needs to be precise
  • Few-shot examples are not just helpful — they are the fastest, most token-efficient way to teach a model an entirely new output schema
  • Role prompting goes beyond "you are X" — a rich persona with goals, knowledge, and communication style measurably improves output quality
  • COSTAR (Context · Objective · Style · Tone · Audience · Response) is a complete specification framework — if a prompt keeps underperforming, run COSTAR on it and find the missing dimension
  • Chain-of-thought is a one-sentence addition ("Think step by step") that can double accuracy on multi-step tasks — zero cost, high return
  • Iterative refinement turns the LLM into a reviewer of its own output — a two-call pattern that improves quality without changing the model

Technique selection map #

The six techniques are not competing alternatives — they layer on top of each other. The decision of which to use is determined by the task type and what the model is getting wrong.

When to reach for each technique
Zero-shot
Well-defined task, clear verb, simple output. Start here. Works ~70% of the time on capable models.
Few-shot
Format or style must be precise. Custom schemas. When zero-shot output shape is wrong but content is right.
Role prompting
Output needs a specific perspective, expertise level, or communication style. Add to system prompt.
COSTAR
Prompt keeps underperforming and you don't know why. Run the checklist — find the missing dimension.
Chain-of-thought
Multi-step reasoning, maths, logic, complex decisions. Add "Think step by step" — free accuracy gain.
Iterative refinement
Output quality ceiling too low. Use a second LLM call to review and rewrite. Adds latency and cost but reliably lifts quality.

Zero-shot and few-shot #

Zero-shot means giving the model a task with no examples — just an instruction. It works well when the task is unambiguous and the output format is standard. Few-shot adds 2–5 worked examples before the real input. The examples teach the model your exact format, label set, or style — far more efficiently than a long written description of what you want.

pythonzero_vs_few_shot.py — sentiment classification with both approaches
import anthropic
client = anthropic.Anthropic()

reviews = [
    "The onboarding was smooth but billing support took 3 days to respond.",
    "Absolutely love the product. Switched from a competitor and never looking back.",
    "It works, I guess. Nothing special.",
]

# ── Zero-shot: instruction only ───────────────────────────────────────
def classify_zero_shot(text: str) -> str:
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=16,
        system="Classify the customer review sentiment as POSITIVE, NEGATIVE, or MIXED. One word only.",
        messages=[{"role": "user", "content": text}],
    )
    return resp.content[0].text.strip()

# ── Few-shot: examples teach the exact output schema ──────────────────
FEW_SHOT_EXAMPLES = [
    {"role": "user",      "content": "Fast delivery but the packaging was damaged."},
    {"role": "assistant", "content": '{"label":"MIXED","confidence":0.85,"reason":"positive delivery, negative packaging"}'},
    {"role": "user",      "content": "Best purchase I've made this year. 10/10."},
    {"role": "assistant", "content": '{"label":"POSITIVE","confidence":0.99,"reason":"strong positive superlative"}'},
    {"role": "user",      "content": "Cancelled after the price increase. Disappointed."},
    {"role": "assistant", "content": '{"label":"NEGATIVE","confidence":0.92,"reason":"cancellation due to pricing"}'},
]

def classify_few_shot(text: str) -> str:
    resp = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=64,
        system="Classify customer review sentiment. Output JSON only — no other text.",
        messages=FEW_SHOT_EXAMPLES + [{"role": "user", "content": text}],
    )
    return resp.content[0].text.strip()

for review in reviews:
    print(f"Zero-shot: {classify_zero_shot(review)}")
    print(f"Few-shot : {classify_few_shot(review)}")
    print()
💡
Few-shot example quality beats quantity Two excellent examples outperform eight mediocre ones. Each example should cover a distinct input type, use the exact output schema you need, and avoid edge cases that would confuse the model. Keep examples in the messages array as alternating user/assistant pairs — not as a wall of text in the system prompt.

Role prompting #

Role prompting assigns a persona to the model. The basic version — "you are a helpful assistant" — barely changes behaviour. An effective role prompt specifies expertise level, communication style, and the goal the persona is trying to achieve. The model uses the persona to calibrate vocabulary, depth, assumptions, and tone.

Weak role prompt
"You are a senior developer.
Help the user with their code."
Vague expertise level. No communication constraints. No stated goal.
Strong role prompt
"You are a principal backend engineer
with 10 years of Python experience.
Your goal: find bugs and explain them
so a junior developer can fix and learn.
Style: direct, specific, no jargon."
Expertise level, goal, target audience, and style all specified.

COSTAR framework #

COSTAR is a six-dimensional checklist for prompt specification. When a prompt keeps underperforming, run each dimension — the missing or underspecified one is usually the cause. Each letter is a question you must answer before the model can do its best work.

COSTAR — six dimensions of a complete prompt
C
Context
Background the model needs to make good decisions. "We're a B2B SaaS company, Q3 revenue report, audience is the board."
O
Objective
The precise task — a verb + output type. "Write a 3-paragraph executive summary." Not "help me with the report."
S
Style
Writing style or format. "Formal business prose. No bullet points. Active voice." Can reference an author or brand voice.
T
Tone
Emotional register. "Confident but not aggressive. Acknowledge challenges without alarming." Distinct from style.
A
Audience
Who is reading this. "Non-technical board members familiar with SaaS metrics but not with infrastructure." Calibrates vocabulary.
R
Response
Exact output format. "Three paragraphs, 80 words each. End with a one-sentence call to action." This is module 3.3's format section.
pythoncostar.py — COSTAR applied to an executive summary task
import anthropic
client = anthropic.Anthropic()

# ── COSTAR prompt for executive summary generation ────────────────────
SYSTEM = """## Context
You are writing for AcmeCorp, a B2B SaaS company. The data is from the Q3 2025 board report.
The company grew revenue 18% YoY but missed its 25% target.

## Objective
Write an executive summary of the Q3 financial results.

## Style
Formal business prose. No bullet points. Active voice. Use specific numbers.

## Tone
Confident but not dismissive of the miss. Acknowledge challenges, pivot to action.

## Audience
Non-technical board members. Familiar with ARR and churn but not with engineering terms.

## Response format
Exactly three paragraphs. Each paragraph 60–80 words. No headers. No markdown."""

user_data = """Q3 2025 Results:
- Revenue: $4.2M (target: $5.25M, growth 18% YoY vs 25% target)
- New ARR: $820K (down from $1.1M in Q2)
- Churn: 3.2% (up from 2.8%)
- Gross margin: 74% (stable)
- 3 enterprise deals closed; 2 slipped to Q4
- Headcount: 47 (added 6 engineers)"""

resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    system=SYSTEM,
    messages=[{"role": "user", "content": f"<data>\n{user_data}\n</data>"}],
)
print(resp.content[0].text)

Chain-of-thought #

Chain-of-thought (CoT) prompting asks the model to reason through a problem step by step before giving its final answer. The mechanism: the model's intermediate reasoning tokens become additional context that improves the final token prediction. For multi-step tasks — maths, logic, decision-making — CoT reliably improves accuracy with zero added cost beyond the reasoning tokens.

ℹ
CoT is one sentence, not a technique overhaul The simplest CoT trigger is adding "Think step by step before giving your final answer." to your system prompt or user message. That single instruction activates the behaviour. Module 3.6 covers advanced variants — self-consistency, least-to-most, and self-refine — for cases where basic CoT is not sufficient.
Without CoT vs with CoT
WITHOUT CoT
User: "If I have 3 contracts at
$12K/yr each and one churns,
what % of ARR do I lose?"

Model: "You lose 25%."
← wrong (should be 33.3%)
WITH CoT
User: "…Think step by step."

Model: "Total ARR = 3 × $12K
= $36K. Lost ARR = $12K.
Loss % = 12/36 = 33.3%."
← correct

Iterative refinement #

When a single-pass response quality ceiling is too low, use the model to review and improve its own output. The two-call refinement pattern: first call generates a draft; second call reviews it against specific criteria and rewrites. This separates generation from evaluation — two cognitive modes where using different prompts measurably outperforms a single complex prompt.

pythoniterative_refinement.py — generate then critique and rewrite
import anthropic
client = anthropic.Anthropic()

task = "Write a cold email subject line for a B2B SaaS product that reduces AWS costs by 30%."

# ── Call 1: generate draft ────────────────────────────────────────────
draft_resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=128,
    system="You are a B2B copywriter. Generate 3 subject line options.",
    messages=[{"role": "user", "content": task}],
)
draft = draft_resp.content[0].text
print("Draft:\n", draft)

# ── Call 2: critique then rewrite ─────────────────────────────────────
refinement_resp = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=256,
    system="""You are a conversion copywriting critic.
Evaluate subject lines on: specificity, urgency, and avoiding spam triggers.
Then produce one improved version per option.""",
    messages=[
        {"role": "user", "content": f"<draft_options>\n{draft}\n</draft_options>\n\nCritique each, then rewrite."},
    ],
)
print("\nRefined:\n", refinement_resp.content[0].text)

# ── Pattern variants ──────────────────────────────────────────────────
# - Use a stronger model (Sonnet) for the refinement call only
# - Pass explicit rubric criteria in the critic system prompt
# - Run 2–3 rounds for high-stakes content (diminishing returns after 3)
# - Structured critique: ask critic to return JSON {score, issues, rewrite}
⚠
Refinement loops have diminishing returns Two calls reliably improve quality. Three calls marginally so. Beyond three, you are usually just adding latency and cost with no measurable gain. If three rounds still fall short, the problem is usually the task specification or missing context — not a lack of refinement iterations. Fix the prompt first, then add refinement.

Self-check #

You need to extract structured JSON from 500 support tickets per day. Zero-shot gives inconsistent field names. What technique should you reach for first and why?
Few-shot. The problem is output format inconsistency, not content quality — the model understands the task but doesn't know your exact schema. Add 3–5 worked examples as alternating user/assistant message pairs, each showing the exact JSON structure you want. Two excellent examples with your target field names and types will enforce the schema far more reliably than any amount of textual description. If few-shot still produces occasional schema violations, pair it with assistant prefilling (start the response with '{' ) as a backup.
A prompt keeps producing responses that are technically correct but written at the wrong expertise level for the audience. Which COSTAR dimension is missing?
Audience (A). The model needs to know who is reading the output to calibrate vocabulary, assumed knowledge, and explanation depth. Adding a specific audience description — "mid-level product managers with no engineering background" versus "senior backend engineers" — will shift the vocabulary and assumption level significantly. You may also need to check Style (S) for any conflicting instructions that override the audience calibration.
You add "Think step by step" to a prompt and the model now reasons correctly but the response is too long — the user only needs the final answer. How do you keep CoT accuracy without verbose output?
Keep the CoT instruction but add an output constraint: "Think step by step internally, then output only your final answer on the last line with the prefix 'Answer:'". This gives the model its reasoning tokens (which improve accuracy) but constrains what appears in the visible response. Alternatively, use XML tags: "Think step by step inside <thinking> tags, then give your final answer inside <answer> tags" and extract only the <answer> content. Module 3.6 covers the extended thinking API parameter which moves reasoning entirely off the visible response.
What is the cost-efficiency argument for the iterative refinement pattern? When should you NOT use it?
Cost case for using it: Draft with Haiku, critique with Sonnet. Haiku generates the raw material cheaply; Sonnet's stronger reasoning handles evaluation and rewriting. The blended cost (Haiku + Sonnet × shorter critic call) is usually lower than calling Sonnet twice, while quality matches or exceeds two Sonnet calls.

When NOT to use it: (1) Latency-sensitive endpoints — two serial LLM calls roughly doubles response time; (2) When the root problem is missing context or a poorly specified task — no amount of refinement fixes a bad prompt; (3) Simple tasks — zero-shot extraction or classification does not need refinement and adding it wastes tokens; (4) More than three rounds — diminishing returns make additional rounds economically unjustifiable.
