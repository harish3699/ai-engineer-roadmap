Database Connectivity

Every production agent needs persistent state. Chat history, ingestion progress, evaluation results — all of it lives in a database. This module covers raw PostgreSQL with psycopg2, the SQLAlchemy ORM, and the connection pooling that keeps your agent from exhausting DB connections under load.

~30 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-06
Key Takeaways
  • Always use parameterised queries — string interpolation into SQL is a textbook SQL injection vulnerability
  • Connection pooling prevents "too many connections" errors — never create a new connection per request in production
  • The SQLAlchemy ORM is Pydantic-compatible: your data models from Module 1.2 map cleanly to ORM models
  • Use raw SQL (via session.execute(text(...))) when the ORM generates inefficient queries — don't fight the abstraction, just bypass it
  • pgvector extension turns PostgreSQL into a vector database — your Phase 4 capstone uses it alongside Pinecone

Raw PostgreSQL with psycopg2 #

psycopg2 is the standard PostgreSQL adapter for Python. Use it when you need full control over SQL, when you're writing performance-critical queries, or when you're running database migrations.

🚫
SQL injection — never do this f"SELECT * FROM chunks WHERE source = '{user_input}'" — if user_input is '; DROP TABLE chunks; --, you just deleted your data. Always use parameterised queries with %s placeholders.
pythonpsycopg2_example.py
import psycopg2
from psycopg2.extras import RealDictCursor
import os

# ── Connect ───────────────────────────────────────────────────────────
conn = psycopg2.connect(os.environ["DATABASE_URL"])

with conn.cursor(cursor_factory=RealDictCursor) as cur:
    # ── Parameterised query — %s placeholders, NEVER f-strings ───────
    cur.execute(
        "SELECT id, text, score FROM chunks WHERE source = %s AND score > %s",
        ("policy.pdf", 0.8),   # values as a tuple, not string-interpolated
    )
    rows = cur.fetchall()   # list of RealDictRow — access by column name
    print(rows[0]["text"])

    # ── Insert with RETURNING ─────────────────────────────────────────
    cur.execute(
        """
        INSERT INTO ingestion_jobs (source, status, created_at)
        VALUES (%s, %s, NOW())
        RETURNING id
        """,
        ("new_doc.pdf", "queued"),
    )
    new_id = cur.fetchone()["id"]
    conn.commit()   # DML requires explicit commit

conn.close()

SQLAlchemy ORM #

The ORM lets you work with database rows as Python objects. It integrates with Pydantic (Module 1.2) and FastAPI (Module 1.7), and is what LangGraph uses internally for its PostgreSQL checkpointer.

pythonsqlalchemy_orm.py — model definition + session queries
from sqlalchemy import create_engine, String, Float, Integer, text
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, Session
import os

engine = create_engine(os.environ["DATABASE_URL"], pool_size=10, max_overflow=20)

# ── Model definition ─────────────────────────────────────────────────
class Base(DeclarativeBase): pass

class Chunk(Base):
    __tablename__ = "chunks"

    id:     Mapped[int]   = mapped_column(Integer(), primary_key=True)
    source: Mapped[str]   = mapped_column(String(512), nullable=False)
    text:   Mapped[str]   = mapped_column(String(), nullable=False)
    score:  Mapped[float] = mapped_column(Float(), default=0.0)

Base.metadata.create_all(engine)   # creates table if it doesn't exist

# ── Session: query, insert, update ───────────────────────────────────
with Session(engine) as session:
    # Query — ORM style
    high_score = (
        session.query(Chunk)
        .filter(Chunk.score > 0.8)
        .order_by(Chunk.score.desc())
        .limit(5)
        .all()
    )

    # Insert
    new_chunk = Chunk(source="guide.pdf", text="RAG pipelines...", score=0.92)
    session.add(new_chunk)
    session.commit()

    # Raw SQL when ORM generates inefficient query
    result = session.execute(
        text("SELECT id, score FROM chunks ORDER BY score DESC LIMIT 100")
    )
    rows = result.fetchall()

Connection pooling #

Creating a new DB connection takes 30–100ms. Under concurrent load, creating a connection per request exhausts PostgreSQL's connection limit (default 100) in seconds. A pool maintains a set of open connections and lends them to requests — the request returns the connection to the pool when done.

pythonconnection_pool.py — pool config and why the numbers matter
from sqlalchemy import create_engine

# ── SQLAlchemy pool config ────────────────────────────────────────────
engine = create_engine(
    os.environ["DATABASE_URL"],
    pool_size=10,        # keep 10 connections always open
    max_overflow=20,     # allow 20 more under burst (total 30 max)
    pool_timeout=30,     # wait up to 30s for a connection before raising
    pool_recycle=1800,   # recycle connections after 30min (avoids stale TCP)
    pool_pre_ping=True,  # test connection before lending — auto-reconnects
)

# ── Rule of thumb ─────────────────────────────────────────────────────
# pool_size ≈ number of concurrent workers your app can sustain
# max_overflow for burst headroom — keep pool_size + max_overflow < DB max_connections
# pool_recycle MUST be set for serverless/cloud DBs that drop idle connections

# ── For async (FastAPI + async SQLAlchemy) use create_async_engine ────
from sqlalchemy.ext.asyncio import create_async_engine

async_engine = create_async_engine(
    os.environ["DATABASE_URL"].replace("postgresql://", "postgresql+asyncpg://"),
    pool_size=10,
    max_overflow=20,
)

Self-check #

Why are parameterised queries safe but f-string SQL is not?
Parameterised queries send the SQL template and the values separately to the database driver. The driver handles escaping and quoting — user input cannot break out of the value context. With f-strings, you construct a single string before sending it to the database. User-supplied values become part of the SQL syntax, allowing injection of arbitrary SQL statements.
What does pool_pre_ping=True do and when does it matter?
pool_pre_ping=True tests each connection with a cheap SELECT before lending it to a request. If the connection is stale (dropped by the database or firewall timeout), it reconnects automatically. Without it, your app will get a cryptic "connection closed" error on the first query after a period of inactivity — common on cloud databases (RDS, Neon, Supabase) that drop idle connections after minutes.
When would you use session.execute(text(...)) instead of ORM-style queries?
When the ORM generates inefficient SQL for complex joins, window functions, CTEs, or database-specific features like PostgreSQL's EXPLAIN ANALYZE. Also useful for bulk operations where ORM's per-row model overhead matters, and for pgvector queries (<-> cosine distance operator) which have no ORM equivalent. The rule: start with ORM, drop to raw SQL when you see the generated query and wince.
What happens when all connections in the pool are in use and a new request arrives?
The request waits up to pool_timeout seconds (default 30s) for a connection to be returned to the pool. If no connection is returned within that time, a TimeoutError is raised. This is why pool_size + max_overflow must match your app's actual concurrency — set too low and you get timeouts; set too high and you exhaust PostgreSQL's max_connections limit.
