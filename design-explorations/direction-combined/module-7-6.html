A2A — Agent-to-Agent Protocol

A2A (Agent-to-Agent) is an open protocol proposed by Google in April 2025 for standardising how agents discover and communicate with each other across different frameworks and organisations. Where MCP standardises how agents interact with tools, A2A standardises how agents interact with other agents. An agent publishes an Agent Card (a JSON descriptor of its capabilities), and other agents can discover it, delegate tasks to it, and receive structured responses — regardless of whether the remote agent is built in LangGraph, CrewAI, AutoGen, or a custom framework.

~20 min read 3 code examples Last updated 2026-05-07
Key Takeaways
  • A2A solves the cross-framework agent communication problem — two agents built in different frameworks (LangGraph + CrewAI) can delegate tasks to each other without custom integration code
  • Every A2A agent publishes an Agent Card at /.well-known/agent.json — a JSON descriptor of its name, description, capabilities, and supported modalities
  • The A2A task lifecycle: submitted → working → input-required → completed/failed — agents can stream intermediate updates and request additional input mid-task
  • A2A is complementary to MCP: MCP = agent ↔ tools; A2A = agent ↔ agent. They operate at different levels and are typically used together
  • A2A is still early (released April 2025) — the spec is stable but the ecosystem is maturing. Use it when you have genuine cross-framework or cross-organisation agent delegation needs

Agent Card #

Every A2A agent advertises its capabilities via an Agent Card — a JSON document served at /.well-known/agent.json relative to the agent's base URL. Client agents fetch this to discover what the agent can do before sending it a task.

jsonagent.json — Agent Card for a research specialist agent
{
  "name": "Research Specialist",
  "description": "Web research agent. Searches multiple sources, synthesises findings, returns structured reports.",
  "url": "https://research-agent.internal/a2a",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,
    "pushNotifications": false
  },
  "defaultInputModes": ["text/plain"],
  "defaultOutputModes": ["application/json", "text/plain"],
  "skills": [
    {
      "id": "web-research",
      "name": "Web Research",
      "description": "Search the web and synthesise findings on a topic",
      "tags": ["research", "web-search", "synthesis"],

  ]
}

Task lifecycle and client implementation #

pythona2a_client.py — delegating a task to a remote A2A agent
import httpx, json, uuid

async def delegate_to_agent(agent_url: str, task_message: str) -> str:
    async with httpx.AsyncClient() as client:
        # 1. Discover agent capabilities
        card = (await client.get(f"{agent_url}/.well-known/agent.json")).json()
        print(f"Delegating to: {card['name']}")

        # 2. Submit task
        task_id = str(uuid.uuid4())
        payload = {
            "jsonrpc": "2.0", "method": "tasks/send", "id": task_id,
            "params": {
                "id": task_id,
                "message": {"role": "user", "parts": [{"type": "text", "text": task_message}]},
            },
        }
        response = await client.post(f"{agent_url}/a2a", json=payload)
        task = response.json()["result"]

        # 3. Poll until complete (or stream if agent supports SSE)
        while task["status"]["state"] == "working":
            poll = await client.post(f"{agent_url}/a2a", json={
                "jsonrpc": "2.0", "method": "tasks/get", "id": task_id,
                "params": {"id": task_id},
            })
            task = poll.json()["result"]

        # 4. Extract result
        if task["status"]["state"] == "completed":
            return task["artifacts"][0]["parts"][0]["text"]
        raise RuntimeError(f"Task failed: {task['status']}")

When A2A is the right choice #

ℹ
Use A2A when — and only when
  • Cross-framework delegation: you need to call a CrewAI agent from LangGraph, or an AutoGen agent from a custom Python workflow — without knowing the other framework
  • Cross-organisation agents: the sub-agent is owned and deployed by another team or vendor — you only have their A2A endpoint URL, not access to their code
  • Long-running async tasks: the sub-agent takes minutes or hours and needs to stream progress updates while you wait
Don't use A2A for agents within the same codebase — direct function calls (agent-as-tool or LangGraph nodes) are simpler, faster, and fully observable.
