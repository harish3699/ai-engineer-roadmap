Human in the Loop

Fully autonomous agents are appropriate for low-stakes, reversible tasks. For irreversible actions — sending emails, writing to databases, making payments, deleting records — you need the ability to pause, show the human what the agent is about to do, and resume only after approval. This module covers the interrupt pattern in LangGraph, persistent checkpointing for resumable pauses, and the decision framework for identifying which operations require human oversight.

~20 min read 1 video · 1 recording script 3 code examples Last updated 2026-05-07
Key Takeaways
  • The interrupt pattern pauses the agent graph before a specific node executes — the state is checkpointed, the human reviews and approves or rejects, then the graph resumes from the saved state
  • LangGraph's interrupt_before parameter specifies which nodes to pause before — the graph halts, your UI shows the pending action, and agent.invoke(None, config) resumes it
  • Persistent checkpointers (SqliteSaver, PostgresSaver) let pauses survive server restarts — essential for long-running workflows where approval may take hours
  • The human-in-the-loop decision should be driven by reversibility and blast radius — reads are always safe, DML writes above a size threshold need approval, irreversible actions (emails, payments, deletes) always need approval
  • Design your interrupts to show the human exactly what is about to happen in plain language — agent tool call JSON is not a suitable approval UI

When to pause — the reversibility framework #

Not every tool call needs human approval — that would defeat the purpose of automation. Use reversibility and blast radius to classify tools:

Tool classification by reversibility
✅
Safe — no approval needed
Read operations (search, fetch, query SELECT), calculations, status checks. Reversible by default because nothing is changed.
⚠
Review — context-dependent
Creating records, updating single records, posting to internal systems. Reversible with effort — can be undone but requires manual work. Require approval if above a size threshold (e.g. bulk updates >10 records) or if confidence in model intent is low.
🔴
Always approve
Sending emails/messages, processing payments, deleting records, deploying code, modifying permissions. Irreversible or difficult to undo — human eyes before execution, always.

The interrupt pattern in LangGraph #

LangGraph represents the agent as a state graph. The interrupt_before parameter tells the graph to pause before executing specific nodes — most commonly the "tools" node, which is where tool calls execute. The graph state is checkpointed at the pause point, your code shows the pending action to the user, and agent.invoke(None, config) resumes from the checkpoint.

pythonhitl_agent.py — human-in-the-loop with LangGraph interrupts
from langchain_openai import ChatOpenAI
from langchain.tools import tool
from langgraph.prebuilt import create_react_agent
from langgraph.checkpoint.memory import MemorySaver

@tool
def send_email(to: str, subject: str, body: str) -> dict:
    """Send an email. ALWAYS requires human approval before sending."""
    # In production: call email API (SendGrid, SES, etc.)
    print(f"[EMAIL SENT] To: {to} | Subject: {subject}")
    return {"status": "sent", "to": to, "subject": subject}

@tool
def search_contacts(query: str) -> list:
    """Search contacts by name or email. Safe — read only."""
    return [{"name": "Alice Smith", "email": "alice@example.com"}]

model = ChatOpenAI(model="gpt-4o-mini")
memory = MemorySaver()
config = {"configurable": {"thread_id": "email-task-1"}}

# ── interrupt_before="tools" pauses before ANY tool executes ──────────
agent = create_react_agent(
    model=model,
    tools=[search_contacts, send_email],
    checkpointer=memory,
    interrupt_before=["tools"],
)

# ── First invocation: agent plans, then pauses before tool execution ──
result = agent.invoke(
    {"messages": [{"role": "user", "content": "Email Alice Smith saying the meeting is moved to Thursday"}]},
    config=config,
)

# Check what the agent wants to do
state = agent.get_state(config)
pending = state.next  # ["tools"] — waiting for approval

last_msg = state.values["messages"][-1]
if hasattr(last_msg, "tool_calls"):
    for tc in last_msg.tool_calls:
        print(f"\n⏸ PENDING APPROVAL: {tc['name']}")
        print(f"   Arguments: {tc['args']}")

# ── Human reviews and approves ────────────────────────────────────────
approval = input("\nApprove? [y/n]: ").strip().lower()

if approval == "y":
    # Resume: invoke with None resumes from checkpoint
    final = agent.invoke(None, config=config)
    print("\n✅", final["messages"][-1].content)
else:
    print("❌ Cancelled by user.")
ℹ
interrupt_before="tools" pauses before ALL tools When you use interrupt_before=["tools"], the agent pauses before executing any tool — including safe read-only tools like search_contacts. In production, you typically want selective interrupts: pause only before the dangerous tools, not the safe ones. The cleanest approach: separate your tools into "safe" and "sensitive" lists, or build a custom tool node that checks a per-tool requires_approval flag before executing.

Persistent checkpoints for long pauses #

If approval might take minutes or hours (a manager reviewing a large payment), you can't keep the state in RAM. Use a persistent checkpointer that saves to SQLite or Postgres, so the pause survives server restarts.

pythonpersistent_hitl.py — SqliteSaver for durable pauses
from langgraph.checkpoint.sqlite import SqliteSaver
import sqlite3

# SQLite-backed checkpointer — survives server restarts
conn = sqlite3.connect("agent_checkpoints.db", check_same_thread=False)
memory = SqliteSaver(conn)

agent = create_react_agent(
    model=model,
    tools=[search_contacts, send_email],
    checkpointer=memory,
    interrupt_before=["tools"],
)

# Pause, shut down server, restart, resume from checkpoint:
thread_id = "approval-task-42"
config = {"configurable": {"thread_id": thread_id}}

# On restart — check for pending approvals:
state = agent.get_state(config)
if state.next == ["tools"]:
    print("Resuming approval for task", thread_id)
    # Show the pending action, get approval, then:
    final = agent.invoke(None, config=config)  # resume

# For production: use PostgresSaver for distributed deployments

💡
Show intent in plain language, not tool call JSON The approval UI should present the pending action in human-readable form: "The agent wants to send an email to alice@example.com with subject 'Meeting rescheduled to Thursday'. Do you approve?" — not the raw {"name": "send_email", "args": {...}} JSON. Add a translation layer between the tool call and the approval screen. If the tool arguments include recipient, subject, and body, render them in a readable preview. Humans approve intent, not JSON objects.

Self-check #

Explain the LangGraph interrupt mechanism. What happens to the agent's state when it hits an interrupt_before node, and what is the sequence of calls to resume it?
When a LangGraph agent hits a node listed in interrupt_before, the graph execution pauses at that node without executing it. The graph's current state — including all messages, tool call requests, and any intermediate data — is serialised and saved to the checkpointer (MemorySaver, SqliteSaver, etc.) under the current thread_id. The agent.invoke() call returns with the current state, but the agent has not yet executed the pending tool calls. To inspect the pending action, call agent.get_state(config) — this returns the state object with state.next telling you which node is waiting and state.values["messages"][-1].tool_calls showing the pending tool call details. To resume, call agent.invoke(None, config) — passing None as the input tells LangGraph to resume from the checkpoint rather than starting a new conversation. The graph continues from the paused node, executes the tool calls, and continues the ReAct loop. To cancel, call agent.update_state(config, {"messages": [...]}, as_node="tools") to inject a cancellation message before resuming.
Your agent has 8 tools: 5 read-only lookups and 3 write operations. How would you implement selective interrupts that only pause for the write operations?
LangGraph's interrupt_before=["tools"] pauses before the entire tools node, affecting all tools. To selectively pause only for writes, replace the default tools node with a custom function that inspects each tool call before executing: define a custom call_tools node that iterates over state["messages"][-1].tool_calls, checks whether each tool is in a predefined "requires_approval" set, and raises an NodeInterrupt exception (LangGraph's mechanism for conditional interrupts) only for those tools. Another approach: add a requires_approval: bool attribute to each tool definition and check it in the custom tools node. A third approach: use two separate tool nodes — a "safe_tools" node and a "sensitive_tools" node — with routing logic that sends the model's tool calls to the correct node, and only apply interrupt_before to the "sensitive_tools" node. The custom tools node approach is the most flexible for production.
Why is MemorySaver insufficient for a production approval workflow where managers might take hours to respond, and what should you use instead?
MemorySaver stores the checkpointed agent state in the Python process's heap memory. This has two critical production problems: (1) Process restarts destroy all state — if the server crashes or restarts during deployment while an approval is pending, the approval workflow is lost and the task must be restarted from scratch; (2) Horizontal scaling is impossible — if you run multiple instances of the agent service behind a load balancer, each instance has its own MemorySaver and instances can't see each other's checkpoints. A manager on Instance A approves a task, but the resume call lands on Instance B which has no knowledge of the pending state. For production approval workflows, use SqliteSaver (single instance, survives restarts) or PostgresSaver/RedisSaver (multi-instance, distributed). The checkpointer interface is the same — only the constructor changes — so switching from MemorySaver to a persistent backend requires changing one line.
