Debugging Multi-Agent Systems

Multi-agent bugs don't look like regular bugs. The system runs — it just produces wrong output, runs forever, or costs ten times what it should. This module covers the four failure modes that account for the majority of production incidents: agents that talk past each other (state design flaws), cycles that won't terminate (missing or broken exit conditions), cost explosions (unbounded token usage), and the observability tooling — LangSmith — that makes all of these diagnosable in under five minutes instead of five hours.

~22 min read 5 code examples Last updated 2026-05-07
Key Takeaways
  • LangSmith is non-negotiable: three env vars and you get a full trace of every node, every LLM call, and every tool invocation — debugging blind is not a viable alternative
  • Agents talking past each other: root cause is almost always a state field that overwrites instead of accumulates, or a system prompt that doesn't tell the agent what inputs to expect
  • Infinite loops: every cycle in a LangGraph must have a hard integer counter cap — never trust the LLM's self-assessment of "am I done?"
  • Cost explosions: message lists grow with every node hop; token-count and truncate before the LLM call, not after
  • The debugging workflow: trace first → isolate the failing node → reproduce with a unit test → fix state or routing → re-run full trace to confirm

LangSmith tracing #

LangSmith is Langchain's observability platform. For a LangGraph workflow it captures every graph run as a nested trace: each node is a span, each LLM call is a child span, each tool invocation is a grandchild span. You can see exact inputs and outputs at every level, latency per node, and total cost per run.

Three environment variables are all it takes:

bash.env — enable LangSmith tracing
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=lsv2_pt_...          # from smith.langchain.com → Settings → API Keys
LANGCHAIN_PROJECT=my-agent-prod         # group traces by project

After that every graph.invoke() and graph.stream() call auto-instruments. No code changes needed. In LangSmith's UI, open a trace and look for:

  • High-latency spans: a node taking 15 s while others take 1 s is your bottleneck
  • Unexpected node visits: if a node appears 20 times in one run, your cycle isn't terminating
  • Token counts on LLM spans: if input tokens are unexpectedly large, your message list is growing unchecked
  • Tool errors: tool spans show the raw exception — no need to re-run locally to reproduce

Why agents talk past each other #

The symptom: Agent B produces output that doesn't use Agent A's work. The report ignores the research. The writer rewrites the brief from scratch. The validator approves without reading the content.

There are three root causes, in order of frequency:

pythontalking_past_debug.py — three causes and their fixes
# ── CAUSE 1: state field overwrites instead of accumulates ───────────
# Bug: each node sets state["output"] — later nodes overwrite earlier ones
class BrokenState(TypedDict):
    messages: Annotated[list, add_messages]
    output: str                        # ← single string, last writer wins

def research_node(state):
    return {"output": research_result}  # written here...

def write_node(state):
    # state["output"] is still the research_result here — good
    return {"output": report}            # ...overwritten here

# Fix: use named fields per concern, or a dict reducer
class FixedState(TypedDict):
    messages:  Annotated[list, add_messages]
    research:  str       # set by research_node, read by write_node
    report:    str       # set by write_node, read by validate_node
    approved:  bool
    feedback:  str

# ── CAUSE 2: agent system prompt doesn't mention the state fields ─────
# Bug: write_node builds its own content from the message history,
# ignoring state["research"] entirely

WRITER_PROMPT_BAD = """You are a technical writer. Write a report."""
# The writer has no idea that state["research"] even exists

WRITER_PROMPT_GOOD = """You are a technical writer.
You will receive a research summary in the human message tagged <research>.
Your job is to turn that summary into a polished report.
Do not perform additional research — only use what is provided."""

def write_node_fixed(state: FixedState):
    messages = [
        {"role": "system", "content": WRITER_PROMPT_GOOD},
        {"role": "user", "content": f"<research>\n{state['research']}\n</research>"},
    ]
    response = llm.invoke(messages)
    return {"report": response.content}

# ── CAUSE 3: wrong message role — "user" when "assistant" is expected ──
# Bug: supervisor routes a worker's output back as a "user" message
# The next worker sees "the user said X" instead of "assistant said X"
# Fix: always preserve the original role when forwarding messages
def format_worker_result(worker_response: str) -> dict:
    return {"role": "assistant", "content": worker_response}  # not "user"
ℹ
How to diagnose in LangSmith Open the failing trace. Click the write_node span. Check "Inputs" — if state["research"] is present but the LLM's user message doesn't include it, that's Cause 2. If state["research"] is empty, that's Cause 1 (overwrite). If the role on an incoming message is wrong, that's Cause 3.

Cycles that won't terminate #

Reflection loops and ReAct agents both contain explicit cycles. They're supposed to terminate when the LLM says it's done or the critic approves. In practice the LLM will occasionally stay in a loop indefinitely — the critic always finds something to improve, or the ReAct agent keeps calling the same tool with the same arguments.

The fix is always the same: a hard integer counter that the routing function checks before consulting the LLM's judgment.

pythonloop_guard.py — every cycle needs a hard cap
from typing import Annotated
import operator

class ReflectionState(TypedDict):
    messages:      Annotated[list, add_messages]
    draft:         str
    approved:      bool
    feedback:      str
    revisions:     Annotated[int, operator.add]   # accumulates, never resets
    max_revisions: int                              # set once at graph entry

def should_revise(state: ReflectionState) -> str:
    # Hard cap checked FIRST — before asking the LLM anything
    if state["revisions"] >= state["max_revisions"]:
        return "end"                    # forced exit regardless of approval status
    if state["approved"]:
        return "end"
    return "generate"

# For ReAct agents: use recursion_limit on the graph itself
graph = builder.compile(checkpointer=checkpointer)

# Set per-invocation — different workflows may need different limits
config = {
    "configurable": {"thread_id": thread_id},
    "recursion_limit": 25,    # LangGraph raises GraphRecursionError if exceeded
}
try:
    result = graph.invoke(state, config=config)
except GraphRecursionError:
    # Surface this to monitoring — it means the termination logic is broken
    log_error("graph_runner", "invoke", e, thread_id)
    result = {"error": "max iterations exceeded", "partial_state": graph.get_state(config)}

# ── Detecting same-tool-same-args loops ───────────────────────────────
def detect_tool_loop(messages: list, window: int = 4) -> bool:
    """Return True if the last `window` tool calls are identical (name + args)."""
    tool_calls = [
        (m.tool_calls[0].function.name, m.tool_calls[0].function.arguments)
        for m in messages[-window:]
        if hasattr(m, "tool_calls") and m.tool_calls
    ]
    return len(tool_calls) == window and len(set(tool_calls)) == 1
⚠
Never use the LLM's judgment as the only exit condition LLMs will occasionally claim they are done while continuing to produce tool calls, or claim they need more iterations when they've already found the answer. The integer cap is the safety net — the LLM's judgment is the happy path. Both must exist.

Cost explosions #

In a single-agent loop, each iteration adds one user message and one assistant message. In a multi-agent system, each node hop can add messages from multiple agents. After ten nodes the input to each subsequent LLM call has grown by the full output of every prior node. Token costs compound.

Three places to intervene:

pythoncost_guard.py — three-layer cost defence
import tiktoken

ENCODER = tiktoken.encoding_for_model("gpt-4o")

def count_tokens(messages: list[dict]) -> int:
    return sum(len(ENCODER.encode(m.get("content", ""))) for m in messages)

# ── Layer 1: trim the message list before each LLM call ───────────────
def trim_to_budget(messages: list[dict], budget: int = 8000) -> list[dict]:
    """Keep the system message + most recent exchanges that fit in budget."""
    system = [m for m in messages if m.get("role") == "system"]
    rest   = [m for m in messages if m.get("role") != "system"]
    result = system[:]
    for msg in reversed(rest):         # most recent first
        if count_tokens(result + [msg]) > budget:
            break
        result.insert(len(system), msg)  # insert after system messages
    return result

# ── Layer 2: compress tool results before appending ───────────────────
def compress_tool_result(tool_name: str, raw: str, max_tokens: int = 200) -> str:
    if len(ENCODER.encode(raw)) <= max_tokens:
        return raw
    # Ask the LLM to summarise the result — cheaper than passing the full raw output
    summary = llm.invoke([{
        "role": "user",
        "content": f"Summarise this {tool_name} result in ≤{max_tokens} tokens:\n\n{raw}"
    }])
    return summary.content

# ── Layer 3: per-run budget with early exit ───────────────────────────
class BudgetState(TypedDict):
    messages:        Annotated[list, add_messages]
    tokens_used:     Annotated[int, operator.add]
    token_budget:    int             # set at graph entry, e.g. 50_000
    budget_exceeded: bool

def check_budget(state: BudgetState) -> str:
    if state["tokens_used"] > state["token_budget"]:
        return "budget_exceeded"
    return "continue"

def any_node(state: BudgetState):
    messages = trim_to_budget(state["messages"], budget=8000)
    response = llm.invoke(messages)
    tokens_this_call = response.usage_metadata["total_tokens"]
    return {"messages": [response], "tokens_used": tokens_this_call}

The five-step debugging workflow #

When a multi-agent run produces wrong output, runs too long, or costs too much, follow this sequence:

Step Action Tool What you're looking for
1. Trace Open the LangSmith run LangSmith UI Unexpected node count, high-latency spans, large token counts
2. Isolate Find the first node whose output is wrong Trace inputs/outputs Empty state field, wrong message role, missing context
3. Reproduce Unit-test the failing node in isolation pytest Does the node produce wrong output given the exact state from the trace?
4. Fix Fix state schema, prompt, or routing Editor Named state fields, explicit context injection, hard loop caps
5. Verify Run the full graph and compare traces LangSmith compare Fewer node hops, smaller token counts, correct final output
✦
LangSmith "compare" mode After a fix, run both the broken and the fixed version on the same input, then open them side-by-side in LangSmith's compare view. You can see token reduction, latency improvement, and node hop count changes in a single screen — useful for convincing yourself the fix didn't just move the bug.
