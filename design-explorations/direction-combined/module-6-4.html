Semantic Caching

Exact-match caching doesn't work for LLM applications — "what is the return policy?" and "how do I return an item?" are the same question. Semantic caching stores responses keyed by embedding vectors and retrieves them by cosine similarity, so near-duplicate queries hit the cache rather than the model. In FAQ-heavy workloads this cuts LLM costs by 30–70% and reduces p50 latency to under 50 ms. This module covers the full implementation: embedding queries, similarity thresholds, TTL management, and cache invalidation strategies.

~22 min read 3 code examples Last updated 2026-05-07
Key Takeaways
  • Exact-match caching misses semantically identical questions phrased differently — semantic caching matches by cosine similarity on query embeddings instead
  • The similarity threshold is the critical hyperparameter — too high (0.99) → almost no cache hits; too low (0.80) → stale/wrong answers returned
  • Cache entries need TTL (time-to-live) — facts change, and a cached "what is the price?" from yesterday may be wrong today
  • Only cache on the current query, not the full conversation history — two users asking the same question in different conversation contexts should get the same cached answer
  • Never cache personalised, sensitive, or user-specific responses — cache is shared across users; a user-specific answer is a data leak waiting to happen

How semantic caching works #

The standard cache lookup is a dictionary key lookup: hash the query string, return the stored value if it exists. Semantic caching replaces the key lookup with an embedding similarity search. On every incoming query, you embed the query text, search the cache store for the most similar stored embedding, and if the cosine similarity exceeds a threshold, return the cached response. Otherwise, call the LLM, store the new (query embedding, response) pair, and return the fresh response.

Semantic Cache Flow
1
Incoming query → embed with text-embedding-3-small → get 1536-dim vector
2
Search cache store: find entry with highest cosine similarity to query vector
3
similarity ≥ threshold → return cached response (cache HIT)
4
similarity < threshold → call LLM → store (embedding, response, TTL) → return

Core implementation #

pythonsemantic_cache.py — embedding-based cache with TTL
import time, numpy as np
from dataclasses import dataclass, field
from openai import OpenAI

client = OpenAI()

@dataclass
class CacheEntry:
    query: str
    embedding: np.ndarray
    response: str
    created_at: float = field(default_factory=time.time)
    hits: int = 0


class SemanticCache:
    def __init__(
        self,
        threshold: float = 0.92,    # cosine similarity required for a cache hit
        ttl_seconds: int = 3600,    # 1-hour TTL by default
        embed_model: str = "text-embedding-3-small",
    ):
        self.threshold = threshold
        self.ttl = ttl_seconds
        self.embed_model = embed_model
        self._entries: list[CacheEntry] = []

    def _embed(self, text: str) -> np.ndarray:
        resp = client.embeddings.create(model=self.embed_model, input=text)
        vec = np.array(resp.data[0].embedding, dtype=np.float32)
        return vec / np.linalg.norm(vec)  # normalise for cosine similarity

    def _cosine(self, a: np.ndarray, b: np.ndarray) -> float:
        return float(np.dot(a, b))  # both vectors normalised → dot = cosine

    def _evict_expired(self) -> None:
        now = time.time()
        self._entries = [e for e in self._entries if now - e.created_at < self.ttl]

    def lookup(self, query: str) -> str | None:
        self._evict_expired()
        if not self._entries:
            return None
        q_vec = self._embed(query)
        best_entry, best_score = None, -1.0
        for entry in self._entries:
            score = self._cosine(q_vec, entry.embedding)
            if score > best_score:
                best_score, best_entry = score, entry
        if best_entry and best_score >= self.threshold:
            best_entry.hits += 1
            return best_entry.response
        return None

    def store(self, query: str, response: str) -> None:
        vec = self._embed(query)
        self._entries.append(CacheEntry(query=query, embedding=vec, response=response))

    def invalidate(self, keyword: str) -> int:
        """Remove all entries whose query contains keyword (for manual invalidation)."""
        before = len(self._entries)
        self._entries = [e for e in self._entries if keyword.lower() not in e.query.lower()]
        return before - len(self._entries)

    @property
    def stats(self) -> dict:
        return {"entries": len(self._entries), "total_hits": sum(e.hits for e in self._entries)}

Wrapping the LLM call with semantic cache #

pythoncached_agent.py — LLM call wrapped with semantic cache lookup
cache = SemanticCache(threshold=0.92, ttl_seconds=3600)

async def ask(user_query: str, system_prompt: str) -> str:
    # Step 1: check cache (embedding lookup — ~10-30 ms vs 800-2000 ms for LLM call)
    cached = cache.lookup(user_query)
    if cached:
        return cached  # cache HIT — return immediately, no LLM call

    # Step 2: cache MISS — call the LLM
    resp = await client.chat.completions.acreate(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user",   "content": user_query},
        ],
    )
    response_text = resp.choices[0].message.content

    # Step 3: store in cache for future similar queries
    cache.store(user_query, response_text)
    return response_text


# Behaviour in practice:
# "What is the return policy?"       → cache MISS (first time) → LLM call
# "How do I return a product?"       → similarity 0.94 → cache HIT (no LLM)
# "Can I return something I bought?" → similarity 0.91 → cache HIT (no LLM)
# "What is the weather today?"       → similarity 0.31 → cache MISS → LLM call

Tuning the similarity threshold #

The threshold is a precision-recall tradeoff. A high threshold (0.97+) means only near-identical queries hit the cache — safe but low hit rate. A low threshold (0.80) caches aggressively — high hit rate but risks returning answers to slightly different questions.

Threshold Behaviour Risk Best for
≥ 0.97 Near-exact rewording only Very low hit rate General-purpose Q&A
0.92–0.96 Same intent, different phrasing Low — recommended range Customer support FAQ
0.85–0.91 Topically related queries Medium — test carefully Static domain knowledge
< 0.85 Broad topic match High — wrong answers served Avoid in production
⚠
What must never be cached
  • User-specific questions: "What is my account balance?" — the answer differs per user; caching it leaks one user's data to another
  • Real-time queries: "What is the weather right now?" — the answer changes by the minute
  • Queries with user-provided context: If the user says "given that my order was placed yesterday, what's the status?" — the context makes the question unique
Add a cacheable(query: str) → bool function that checks for personalisation signals before performing the cache lookup.

Production backends #

The in-memory implementation above is good for development. In production, use a vector store as the cache backend — this gives you persistence across restarts, horizontal scaling, and faster ANN (approximate nearest-neighbour) lookups at large cache sizes:

pythonredis_cache.py — using Redis with vector search as production cache backend
# GPTCache (pip install gptcache) wraps any LLM call with semantic caching
# and supports Redis, Milvus, Weaviate, and Qdrant as backends.

from gptcache import cache
from gptcache.adapter import openai as cached_openai
from gptcache.embedding import OpenAI as CacheEmbedding
from gptcache.manager import get_data_manager, CacheBase, VectorBase
from gptcache.similarity_evaluation.distance import SearchDistanceEvaluation

# Initialise once at startup
cache.init(
    embedding_func=CacheEmbedding().to_embeddings,
    data_manager=get_data_manager(
        CacheBase("sqlite"),              # metadata store
        VectorBase("faiss", dimension=1536), # vector store
    ),
    similarity_evaluation=SearchDistanceEvaluation(),
)

# Drop-in replacement for the openai client — same API, cache applied transparently
response = cached_openai.ChatCompletion.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": user_query}],
)

# First call: cache miss → LLM call → result stored in SQLite + FAISS
# Second call (similar query): cache hit → result returned from FAISS lookup
ℹ
Cache vs prompt caching — two different things Don't confuse semantic caching (this module) with prompt prefix caching (Module 6.2). Semantic caching caches the full response and avoids the LLM call entirely. Prompt caching (Anthropic/OpenAI feature) still calls the LLM but caches the KV state for the stable prefix of your prompt, reducing input token costs. Use both simultaneously — they are complementary.
