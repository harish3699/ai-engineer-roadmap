Data Structures

The Python containers you reach for every day — and the collections module that replaces half your custom logic. Choosing the right structure determines whether your RAG pipeline runs in milliseconds or minutes.

~25 min read 1 video · 1 recording script 4 code examples Last updated 2026-05-06
Key Takeaways
  • List vs tuple vs set vs dict — each has O(1) operations the others don't; picking wrong costs you production throughput
  • defaultdict eliminates the if key not in d: d[key] = [] pattern that clutters every data-grouping function
  • Counter turns word/token frequency analysis from 10 lines into 1
  • deque is the correct structure for sliding-window chat history — O(1) append and pop from both ends
  • Sets give you O(1) membership checks — use them for deduplication, not lists with in

Lists and the operations that matter #

A list is an ordered, mutable sequence. The operations you'll use in AI engineering code daily are slicing, sorting, filtering (comprehensions), and flattening. Everything else is rarely worth memorising — look it up.

⚠️
O(n) membership check item in my_list scans every element — O(n). For large sets of seen IDs, chunk hashes, or document URLs, use a set instead. item in my_set is O(1) hash lookup.
pythonlists.py
from typing import Any

# ── Slicing ──────────────────────────────────────────────────────────
chunks = ["a", "b", "c", "d", "e"]
print(chunks[1:3])   # ['b', 'c']    — start inclusive, end exclusive
print(chunks[-2:])   # ['d', 'e']   — last two
print(chunks[::2])   # ['a', 'c', 'e'] — every other

# ── Sorting ──────────────────────────────────────────────────────────
docs = [{"score": 0.9, "text": "RAG"}, {"score": 0.7, "text": "LLM"}, {"score": 0.95, "text": "agent"}]
ranked = sorted(docs, key=lambda d: d["score"], reverse=True)
# [{'score': 0.95, 'text': 'agent'}, ...]

# ── List comprehensions — filter + transform in one line ─────────────
scores: list[float] = [0.9, 0.4, 0.85, 0.3, 0.95]
high_quality = [s for s in scores if s > 0.8]   # [0.9, 0.85, 0.95]
normalised   = [round(s, 2) for s in scores]       # [0.9, 0.4, 0.85, 0.3, 0.95]

# ── Flattening nested lists — common when chunking PDFs ──────────────
pages: list[list[str]] = [["chunk1", "chunk2"], ["chunk3"], ["chunk4", "chunk5"]]
flat  = [chunk for page in pages for chunk in page]
# ['chunk1', 'chunk2', 'chunk3', 'chunk4', 'chunk5']

# ── Generator — same pattern, O(1) memory (use for large doc sets) ───
flat_gen = (chunk for page in pages for chunk in page)

Tuples and sets #

Tuples are immutable sequences — use them for fixed-size structured data (coordinates, return pairs, function signatures). Sets are unordered collections of unique hashable items — use them whenever you need O(1) membership or deduplication.

pythontuples_sets.py
# ── Tuples ───────────────────────────────────────────────────────────
# Good: fixed-size return values — coordinates, scores with ids
def top_result(results: list[dict]) -> tuple[str, float]:
    best = max(results, key=lambda r: r["score"])
    return best["text"], best["score"]

text, score = top_result([{"text": "RAG", "score": 0.9}])  # unpacking

# ── Named tuples — self-documenting, better than plain tuples ────────
from typing import NamedTuple

class RetrievedChunk(NamedTuple):
    text:   str
    source: str
    score:  float

chunk = RetrievedChunk(text="Embeddings...", source="rag.pdf", score=0.92)
print(chunk.score)   # 0.92 — named access, not chunk[2]

# ── Sets — O(1) dedup and membership ─────────────────────────────────
seen_urls: set[str] = set()

def process_url(url: str) -> bool:
    if url in seen_urls:   # O(1) — not a list scan
        return False       # already processed
    seen_urls.add(url)
    return True

# Set operations — useful for comparing retrieval results
retrieved_a = {"doc1", "doc2", "doc3"}
retrieved_b = {"doc2", "doc3", "doc4"}
print(retrieved_a & retrieved_b)  # {'doc2', 'doc3'} — intersection
print(retrieved_a | retrieved_b)  # {'doc1','doc2','doc3','doc4'} — union
print(retrieved_a - retrieved_b)  # {'doc1'} — in A but not B

Dicts — the backbone of LLM payloads #

Every LLM API request is a dict. Every tool call result is a dict. Every chunk of metadata is a dict. You need dict operations to be muscle memory.

pythondicts.py
# ── Safe access patterns ─────────────────────────────────────────────
metadata = {"source": "policy.pdf", "page": 3}

source  = metadata.get("source", "unknown")       # safe — default if missing
chapter = metadata.get("chapter")                   # None — no KeyError

# ── setdefault — initialise if absent, return existing if present ────
# Without setdefault:
cache = {}
if "embeddings" not in cache:
    cache["embeddings"] = []
cache["embeddings"].append([0.1, 0.2])

# With setdefault — one line:
cache.setdefault("embeddings", []).append([0.1, 0.2])

# ── Merging dicts (Python 3.9+) ───────────────────────────────────────
base_payload = {"model": "claude-3-5-sonnet", "max_tokens": 4096}
overrides    = {"max_tokens": 1024, "temperature": 0.3}
final        = base_payload | overrides   # overrides wins on conflicts
# {'model': 'claude-3-5-sonnet', 'max_tokens': 1024, 'temperature': 0.3}

# ── Dict comprehension — transform API responses ──────────────────────
raw_scores = [("doc_a", 0.91), ("doc_b", 0.74), ("doc_c", 0.88)]
score_map: dict[str, float] = {doc: score for doc, score in raw_scores if score > 0.8}
# {'doc_a': 0.91, 'doc_c': 0.88}

The collections module #

Three classes from collections that every AI engineer should use weekly: defaultdict for grouping, Counter for frequency, deque for sliding windows.

pythoncollections_module.py
from collections import defaultdict, Counter, deque

# ── defaultdict — group chunks by source without if-checks ───────────
chunks = [
    {"source": "policy.pdf", "text": "chunk1"},
    {"source": "guide.pdf",  "text": "chunk2"},
    {"source": "policy.pdf", "text": "chunk3"},
]

by_source: dict[str, list] = defaultdict(list)
for c in chunks:
    by_source[c["source"]].append(c["text"])
# defaultdict(list, {'policy.pdf': ['chunk1','chunk3'], 'guide.pdf': ['chunk2']})

# ── Counter — token/word frequency in one call ───────────────────────
tokens = ["agent", "RAG", "agent", "LLM", "agent", "RAG"]
freq = Counter(tokens)
print(freq.most_common(2))   # [('agent', 3), ('RAG', 2)]

# Counter arithmetic — combine multiple batches
batch1 = Counter(["a", "b", "a"])
batch2 = Counter(["b", "c"])
combined = batch1 + batch2   # Counter({'a': 2, 'b': 2, 'c': 1})

# ── deque — sliding-window chat history, O(1) both ends ─────────────
history: deque[dict] = deque(maxlen=10)  # auto-drops oldest when full

history.append({"role": "human",     "content": "What is RAG?"})
history.append({"role": "assistant", "content": "RAG is..."})

# Convert to list when sending to LLM API
messages_payload = list(history)

# appendleft / popleft — add system prompt at front without shifting all
history.appendleft({"role": "system", "content": "You are a helpful assistant."})
🎯
When to use which List: ordered, needs indexing or sorting. Tuple: fixed-size, immutable, hashable (dict keys, set members). Set: membership check, dedup, set operations. Dict: key→value lookup. defaultdict: grouping. Counter: frequencies. deque: queues, sliding windows with bounded size.

Self-check #

Four questions — if you can answer without looking back, you're ready for 1.4.

Why is item in my_set O(1) but item in my_list O(n)?
A set is implemented as a hash table. Python computes the hash of item and checks that bucket — a constant-time operation regardless of set size. A list is a contiguous array; Python must compare item against every element in order until a match is found. For a list of 1 million items, that's up to 1 million comparisons.
What is deque(maxlen=10) doing in a chat history context?
It creates a double-ended queue capped at 10 items. When you append() an 11th message, the oldest message (at position 0) is automatically discarded. This gives you a sliding window of the last 10 messages with O(1) append and automatic eviction — no manual slicing or copying needed. It's the correct structure for bounded chat history because both appending to the right and popping from the left are O(1), unlike a list where left-pop is O(n).
What does defaultdict(list) do that a regular dict doesn't?
When you access a key that doesn't exist, a regular dict raises KeyError. A defaultdict(list) automatically creates a new empty list for that key and returns it. This eliminates the if key not in d: d[key] = [] boilerplate before every append — grouping operations become a single line.
You have two sets of retrieved document IDs from two different retrieval strategies. How do you find documents retrieved by BOTH?
Use the intersection operator: set_a & set_b. This returns a new set containing only elements present in both sets — O(min(len(a), len(b))) on average. This pattern is common in hybrid retrieval (vector + keyword) to find documents that scored well on both signals.
