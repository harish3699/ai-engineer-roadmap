Error & File Handling

Production AI pipelines fail. The question is whether they fail loudly (you know immediately and can fix it) or silently (bad data flows downstream for days). This module teaches you to control which one happens.

~25 min read 1 video · 1 recording script 4 code examples Last updated 2026-05-06
Key Takeaways
  • Catch specific exceptions — catching Exception broadly swallows bugs you need to know about
  • The finally block always runs — use it to release resources even if an exception occurred
  • Custom exception classes make error handling self-documenting and allow callers to catch specific failure types
  • Context managers (with) guarantee cleanup — files close, DB connections return to pool, even on exception
  • JSON and CSV are the two formats every RAG ingestion pipeline reads — know json.load and csv.DictReader cold

try / except / else / finally #

Python's exception handling has four clauses. Most developers only use two. All four serve distinct purposes and skipping them means either swallowing errors or duplicating cleanup code.

pythonexceptions.py — all four clauses with real meanings
import json, logging

def parse_llm_response(raw: str) -> dict:
    try:
        # Only code that might raise goes here — nothing else
        result = json.loads(raw)
    except json.JSONDecodeError as e:
        # Catch specific exceptions — not bare `except:` or `except Exception:`
        logging.error(f"LLM returned non-JSON: {raw[:100]!r} — {e}")
        raise   # re-raise so the caller knows it failed
    except (TypeError, ValueError) as e:
        # Multiple exception types in one clause
        logging.error(f"Unexpected parse error: {e}")
        raise
    else:
        # Runs ONLY if no exception occurred — put success logic here
        logging.info("Parsed successfully")
        return result
    finally:
        # ALWAYS runs — exception or not. Use for cleanup.
        logging.debug("parse_llm_response exiting")


# ── Exception hierarchy — catch parent to catch all children ─────────
# OSError is the parent of FileNotFoundError, PermissionError, etc.
try:
    with open("config.json") as f:
        config = json.load(f)
except FileNotFoundError:
    config = {}  # safe default
except json.JSONDecodeError as e:
    raise RuntimeError(f"config.json is corrupt: {e}") from e
🚫
Never do this except Exception: pass — silently swallows every error including bugs in your own code. except: — catches even KeyboardInterrupt and SystemExit. Always catch the most specific exception you expect, and always either handle it or re-raise.

Custom exception classes #

When you build a pipeline, create a small exception hierarchy for it. Callers can then catch PipelineError to handle any pipeline failure, or ChunkingError specifically. This makes error handling expressive and avoids catching unrelated exceptions.

pythoncustom_exceptions.py
# ── Exception hierarchy for a RAG ingestion pipeline ─────────────────
class PipelineError(Exception):
    """Base class — catch this to handle any pipeline failure."""

class IngestionError(PipelineError):
    """Raised when a document cannot be read or parsed."""

class ChunkingError(PipelineError):
    """Raised when chunking logic fails on a specific document."""
    def __init__(self, source: str, reason: str) -> None:
        self.source = source
        super().__init__(f"Chunking failed for {source!r}: {reason}")

class EmbeddingError(PipelineError):
    """Raised when the embedding API returns a non-200 response."""


# Usage — raise with context
def chunk_document(source: str, text: str) -> list[str]:
    if not text.strip():
        raise ChunkingError(source, "document is empty after stripping whitespace")
    return text.split("\n\n")

# Caller can be as specific or broad as needed
try:
    chunks = chunk_document("policy.pdf", "")
except ChunkingError as e:
    print(f"Document skipped: {e.source}")   # access structured attributes
except PipelineError:
    print("Any other pipeline failure")

Context managers #

The with statement guarantees that cleanup code runs even when exceptions occur. Files, database connections, locks, and HTTP sessions should always be opened inside with blocks.

pythoncontext_managers.py
from contextlib import contextmanager
import time, logging

# ── Built-in context manager: file ───────────────────────────────────
with open("data.json", "r", encoding="utf-8") as f:
    data = f.read()
# f.close() is called automatically — even if an exception occurred

# ── Build your own with @contextmanager ─────────────────────────────
@contextmanager
def timer(label: str):
    """Logs how long a block of code takes — useful for LLM call latency."""
    start = time.perf_counter()
    try:
        yield           # code inside `with` runs here
    finally:
        elapsed = time.perf_counter() - start
        logging.info(f"{label}: {elapsed:.3f}s")

with timer("embed_batch"):
    # your embedding call here
    time.sleep(0.1)   # simulated
# logs: embed_batch: 0.100s

# ── Multiple context managers on one line ────────────────────────────
with open("input.json") as src, open("output.json", "w") as dst:
    dst.write(src.read())   # both files close cleanly

File I/O — JSON, CSV, binary #

Every RAG ingestion pipeline reads files. JSON for configs and API payloads. CSV for structured data exports. Binary for PDFs (handled by Docling in Phase 4). Know the standard library patterns cold.

pythonfile_io.py
import json, csv
from pathlib import Path

# ── JSON ─────────────────────────────────────────────────────────────
# Read
with open("config.json", encoding="utf-8") as f:
    config: dict = json.load(f)        # file object → Python dict

# Write
with open("output.json", "w", encoding="utf-8") as f:
    json.dump(config, f, indent=2, ensure_ascii=False)

# String ↔ dict (for API payloads, LLM responses)
json_str  = json.dumps(config)          # dict → JSON string
back_dict = json.loads(json_str)        # JSON string → dict

# ── CSV — DictReader gives you each row as a dict ────────────────────
with open("documents.csv", newline="", encoding="utf-8") as f:
    reader = csv.DictReader(f)
    rows: list[dict] = list(reader)     # [{'title': '...', 'text': '...'}, ...]

# Write CSV
fieldnames = ["source", "chunk_id", "score"]
with open("results.csv", "w", newline="", encoding="utf-8") as f:
    writer = csv.DictWriter(f, fieldnames=fieldnames)
    writer.writeheader()
    writer.writerows([{"source": "a.pdf", "chunk_id": 0, "score": 0.91}])

# ── pathlib — cleaner than os.path ───────────────────────────────────
data_dir = Path("data")
pdf_files = list(data_dir.glob("*.pdf"))   # [Path('data/policy.pdf'), ...]
for pdf in pdf_files:
    print(pdf.stem, pdf.suffix)              # 'policy'  '.pdf'

Self-check #

Four questions — ready for 1.5 when you can answer without hesitation.

What is the difference between else and finally in a try/except block?
else runs only when no exception was raised in the try block. finally always runs — whether an exception occurred, whether the code returned early, or whether everything succeeded. Use else for "what to do on success" and finally for "what to clean up no matter what."
Why should you re-raise after logging an exception in most cases?
Logging the exception but swallowing it (not re-raising) hides the failure from the caller. The caller has no way to know the operation failed and may continue with invalid data or state. The rule: log AND raise — or handle AND recover. Never log AND silently continue unless you have an explicit fallback path that is safe.
Why is with open(...) as f: better than f = open(...) followed by f.close()?
If an exception occurs between open() and close(), the file handle is never closed — leaking a file descriptor. The with statement calls __exit__ (which calls close()) even when an exception occurs, so the file is always properly closed. On Windows, an unclosed file handle also prevents other processes from writing to the file.
What does json.load(f) do that json.loads(f.read()) does not?
json.load(f) reads and parses the file in one operation using an iterable API — it can stream large JSON files without loading the entire content into a string first. json.loads(f.read()) reads the entire file into a Python string, then parses it — requiring twice the memory peak (file bytes in string + parsed object). For large config files this rarely matters, but for large JSON datasets it can be significant.
